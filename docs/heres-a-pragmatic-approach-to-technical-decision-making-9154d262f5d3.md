# 这里有一个技术决策的实用方法

## 平衡您的解决方案的现在和未来

![](img/7b0f2c274b39ca05171c8a0c5ec3f9ec.png)

软件工程最重要的一个方面是技术决策。决策与您的角色无关，并且是您已经经历过或将经历的事情，作为您软件开发旅程的一部分。技术建议是丰富的，并且不缺乏 SaaS 和开源产品，以知识和最佳实践淹没你。

技术决策是选择技术、工具、框架、编程语言、服务、架构或模式来科学地解决问题的艺术。一个决定可以让你未来的生活变得更容易、更简单、快或慢、复杂或痛苦。

花时间、资源和精力来做好这件事对你未来的成功至关重要。此外，技术领域呈指数级迭代和发展，增加了选择正确解决方案的困惑和复杂性。我没有灵丹妙药，也没有在所有情况下都有效的决策流程图。

作为这个过程的一部分，我已经获得了很多知识，这些知识帮助我客观地决定解决方案，一只眼睛看着现在，另一只眼睛看着未来。

# 问题启发

一个被很好理解的问题已经解决了一半，并且会不断寻求更好的解决方案。然而，几乎每一次，我都看到了范围蔓延。是的，考虑未来的影响是必要的，但同时，扩大问题的范围只会导致选择过度设计的解决方案或解决您当前没有的挑战。

> “如果我有一个小时来解决一个问题，我会花 55 分钟思考问题，5 分钟思考解决方案。”―阿尔伯特·爱因斯坦

每当人们走进一个房间解决一个问题时，他们会先谈论好的，然后才是必须的。在你的车里有加热座椅是很棒的，但是我们先谈谈制造引擎怎么样？我最喜欢的是那些在看到问题之前就提供解决方案的人，最终将问题改装成解决方案，导致浪费时间、精力和不可逾越的技术债务。

这里需要学习的是确保每个人都同意问题陈述、范围和目标，然后再解决问题。

# 复制粘贴

公司雇佣与他们的问题有相似经历的人，这样他们就可以运用他们过去的经验。这是一把双刃剑，因为你希望拥有相关知识和技能的人分享他们的智慧，以做出明智的技术决策。尽管如此，它最终还是没有尽职地复制和粘贴解决方案。

这些问题看似相同，但在这两个组织中却很难完全相同。模式匹配是最容易做到的事情，新员工更倾向于这样做，以显示他们在相关领域的知识和经验。

我支持让有相关经验的人分享他们从过去学到的东西，尤其是当你试图解决的问题与他们已经解决的问题重叠时。正如他们所说的，细节决定成败，所以最好是从你以前的解决方案中解决挑战，设计一些与问题相关的东西，而不是仅仅复制上一个答案。

# 情感选择

人类是有情感的，作为技术人员，很容易对技术、语言、框架或模式产生依恋。此外，人们很容易被最新的科技产品所吸引。我喜欢了解行业内的最新动态，并评估即将到来的趋势，但同时，我也不喜欢它。

我和一些人有过多次对话，他们提出某件事是因为他们对这件事太依恋或太舒服了。

> 我们用 React 吧，因为它很优秀。我过去用过它，非常喜欢用它开发应用程序。

这不是一个有效的论点，对于做决定来说也很幼稚。偏好没有明显的原因，更多的是情感而不是逻辑。相反，我们应该支持源于分析性讨论和数据的选择。

做出有倾向性的选择会导致技术债务、无绩效的解决方案、采用率下降，以及对提议者的依赖。这既不明智，也不适合解决问题。

总是要求基于数据的评估，以消除其中的情感因素。解决同一问题的多种方法，但从逻辑上评估每个选项只会支持你的事业。作为一名开发人员，对您有利的可能对您的组织不利。

# 长钉

快速、肮脏、粗糙的评估远胜于没有对您的解决方案选择进行早期评估。作为极限编程(XP)的一项创新，spikes 已经成为敏捷团队在投入生产之前评估技术、框架和设计的一种规范。

“快速失败”概述了您的技术偏好的缺点，让您可以继续前进，或者在没有盲点的情况下接受它。事先了解这些限制总是比以后遇到障碍更明智。

由于时间紧迫，我参与了跳过这一步的实现，它们最终成为了阻碍。如果你在没有投入太多资源的情况下就失败了，你可能会错过高峰。

对于较大的项目或计划，为每个部分做一个概念验证可能是一个挑战；这有助于确定最重要的方面，然后指出我们需要更多地关注这些方面。

# 可用、可扩展、可维护

技术讨论中最被忽视的部分是可用性、可伸缩性和可维护性。采用率低的技术决策不是一个好的决策。你的工具、框架或语言的易用性和它提供的其他特性一样重要。开发体验不应该受到影响，因为它直接影响团队士气。

假设你平衡了当前的需求和未来的需求，明智的决定将为你的明天打下基础。您正在设计一个当前负载为 1，000 用户/秒的系统。将比例点考虑在内，以便您可以管理系统上的额外负担。然而，设计一个满足 10 亿用户/秒的解决方案只会使其成为一种昂贵且低效的资源使用。以后有问题是好事，所以你的服务受欢迎。

> “大规模可行的方法可能与大规模可行的方法不同。试点往往会成功，而当环境发生变化时，扩大规模往往会失败。”—罗希尼·尼勒卡尼

没有人喜欢在凌晨 3 点醒来，对事件进行分类和解决。这就是解决方案的可维护性发挥作用的地方。花时间评估你的选择的 nfr。“你是在引入风险吗？这个解决方案优化了吗？是自愈吗？”是需要评估的问题。

# 现有技能、技术、框架

评估您当前的技术体系及其组件有助于推动更好的技术决策。如果您的技术团队已经在使用 Python，那么引入一种新的编程语言是没有意义的，除非您正在进行迁移。类似地，如果您的团队已经熟练掌握了一个框架或技术，那么考虑它对您的解决方案的影响。

> “最好的部分是没有部分。最好的过程就是没有过程。”——埃隆·马斯克

重用现有的模式和行为是更快交付价值的明智而快速的方法。不要把这和复制粘贴混淆，复制粘贴传递了解决问题方法的信息。

而是不要多此一举；利用现有的知识、技术和流程为您的问题找到创新的解决方案。从长远来看，向解决方案中添加不必要的组件意味着增加复杂性和降低可维护性。

# 社会化你的决定

一旦你做了一个决定，传播它，并与利益相关者进行交流。组织中最常见的陷阱是达成一个解决方案而不共享它。最终，它会对受影响的成员造成冲击，导致不信任和不和谐。相反，如果你提前与他们分享，你会赢得他们的信任、忠诚和反馈，让他们觉得自己是这个过程的一部分。

早期反馈有助于消除潜在问题，并获得对决策的认可。不要忘记，实施和决策同样重要。发号施令的各方可能不会执行它。在早期，让所有利益相关者参与进来将会停止指责游戏，并在他们之间创造一种主人翁感。最后，为你的决策创建一个 ADR，以确保未来的成员能够理解其基本原理。

# 结论

技术决策是一个复杂的过程，每个组织都采取这个过程来持续一致地交付价值。对个人和团队来说，这在本质上是具有挑战性的，但也是有益的。平衡过去、现在和未来并非易事，需要耐心、理解和解决问题的积极态度。明智地决定。

> "你不能基于恐惧和可能发生的事情来做决定。"―米歇尔·奥巴马
