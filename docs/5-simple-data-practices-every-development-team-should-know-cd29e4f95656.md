数据可以成为改进产品的有力工具。如果使用正确，它可以帮助我们修复错误，识别功能差距，并突出 UX 需要改进的地方。

然而，收集数据并不自动等同于洞察。从原始数据到真实的、可操作的见解需要大量的工作，这些见解可以帮助我们确定下一步工作的优先顺序，以及如何更好地为用户服务。如果您已经编写了代码来收集事件，那么如何从拥有一个数据池到有意义的获取呢？这些实践会有所帮助。

# 1.保持诊断日志的可读性

当我第一次开始为我的团队进行随叫随到的轮换时，我经常需要读取客户提供给我们的日志，或者从我们的一个微服务中获取大量日志。我经常有一种挥之不去的感觉，那就是我看到的并不是人类的眼睛，而是某个地方的某个人工智能。因为我不知道什么更好，所以我接受了这种情况，即使我必须努力理解这些神秘的日志，并深入了解哪里出了问题。

当我的现任经理加入团队时，他的第一个贡献就是标准化我们所有组件的日志记录。对我们来说，行之有效的方法是在每一行前面加上“ <timestamp>| <component name="">| <severity level="">”，这样每一行信息都在同一个地方开始，这样您就可以快速浏览文件，查找任何严重程度较高的问题。</severity></component></timestamp>

如果您的团队已经有了良好的数据实践，这一点对您来说可能是显而易见的。然而，如果到目前为止我在一个大组织中工作学到了什么的话，那就是团队可以长时间使用糟糕的过程，仅仅是因为没有人指出它们。

像使日志更易读这样的小变化可以节省开发人员的时间，使团队能够发现错误，并更快地减少随叫随到事件。

# 2.根据严重性区分错误的优先级

虽然对每个开发团队来说，知道代码中抛出的异常数量很重要，但并不是所有的异常都是一样的。在尝试的操作成功之前，可能会多次引发和记录一些异常。在其他地方，一个简单的`null ref`可能就足以关闭整个会话。

简而言之，查看任何给定异常的命中次数都不能很好地反映问题的严重性。

在我的团队在我们的日志中做出改变以指示关键和非关键异常之间的差异之前，我们有时会浪费时间来修复非关键错误，因为它们在我们的错误图表中有更高的命中数。我们通过在代码中指出哪个异常导致了崩溃并指出所有异常是否被处理来解决这个问题。

今天，我们的图表主要关注关键异常的频率，而不是简单地显示按命中次数聚集的异常。这样，我们可以将大部分精力用于解决最严重的问题。

# 3.根据对客户的影响确定 bug 的优先级

虽然了解错误的总体数量很有价值，但我们发现了解受影响的用户数量也同样重要。在处理只对一小部分用户重现的边缘情况之前，您可能想花精力修复影响大多数人的错误。

要做到这一点，您需要有一种方法来区分不同的用户(但不是以个人可识别的方式)。对于每种产品来说，您这样做的方式是不同的。对于客户端应用程序，您可能会散列某种硬件 ID，或者在磁盘上保存一个安装 ID。通过区分不同的用户，您可以在图表中表示用户影响，并做出更好的决策。

# 4.分析客户群

一旦您能够查看您的产品中出现了哪些错误以及它们影响了多少人，您还需要开始考虑您的客户会陷入哪些可能影响他们体验的困境。你同时为 Mac 和 Linux 用户服务吗？不同的客户依赖不同的体验或应用不同的设置会影响他们的会话吗？

通过识别您的重要客户群，您可以在报告中添加代表他们的可视化效果。下一次你发现一个重要的问题时，你很快就能判断出它是针对所有人还是只针对某个特定的用户群。

# 5.做一些漏斗

漏斗图是表示一个操作阶段的图表。它用于可视化构成产品中重要体验的不同事件，以及有多少人进入了每个阶段。通常，漏斗在顶部是宽的，在每一个连续的步骤中会变窄一点，因为客户会因为错误或可用性问题而减少。

如果有任何特别引人注目的下降，您已经确定了一个许多客户面临问题的领域——要么是由于错误，要么是由于糟糕的 UX。例如，一个在线商店可能会使用一个漏斗来查看“添加到购物车”但从未到达“结账”阶段的人之间是否有落差。

# 结论

如果你在一个成熟的、数据驱动的组织或团队中，这些见解可能对你来说很明显。然而，许多团队在如何使用数据方面仍在成长，甚至可能正在向数据驱动的文化迈出第一步。你可以成为组织中的一员，通过指出流程和报告中可以改进的地方来增加价值。

如果利用得当，数据可以帮助我们做出为客户提供最佳服务的决策。只需要一点迭代就可以实现。
