三角龙不是恐龙。它既不是首字母缩略词，也不是流行语。基本上，当我每天看到新的 DevOps 趋势出现时，这是我消除每个人的不足感的方法。换句话说，这是一些虚假和幽默的东西。

三角龙这个词来自于我在 Reddit 上发布文章 [DevOps 已死，AppOps](/devops-appops-f096cdbb02ac) 万岁】后收到的一个搞笑评论[它很有幽默感，我很欣赏它。](https://www.reddit.com/r/devops/comments/ov2afq/appops_it_is_possible_to_focus_only_on_app/)

事实上，评论的作者回复了我的帖子，批评了围绕 DevOps 话题的流行语的数量。老实说，当我听到 DevSecOps，NoOps，GitOps，AppOps 等时，我也有同样的感觉。我们应该有硕士学位才能把它们都列出来！这就是为什么，当我们被流行词汇轰炸时，我们害怕在我们的列表中有一只三角龙。

玩笑归玩笑，我们可以开始谈论 DevOps 了。在这篇文章中，我想解释最重要的流行词，并试图找出什么是带回家的根本教训。标题和有趣的故事只是一个借口，用来开始分析传播最广的 DevOps 流行语。

# DevOps 和流行语

算法很简单，取 Ops 这个词，加上一些主题，就会产生 DevSecOps(开发+安全+运营)、NoOps(无运营)、GitOps (Git 运营)和 AppOps(应用+运营)。

让我们继续描述吧！

## DevSecOps

安全性已经成为应用程序开发中最重要的主题之一。我们不仅讨论采取所有预防措施来避免事故，还讨论将安全任务集成到 DevOps 流程中。事实上，我们不能允许这样的情况，我们完成一个项目，然后我们必须将它提交给安全团队，冒着看到特性被拒绝的风险。这种方法导致返工的高成本和大量时间的浪费。这就是为什么我们应该从一开始就将安全性包括在流程中。换句话说，DevSecOps 要求我们*在开发过程中包含安全检查和最佳实践，并将其作为管道的一部分进行管理。*

好吧，让我们看看下一个！

## NoOps

NoOps 表示*没有操作*。这很有煽动性，但是 NoOps 的目标是通过简单地消除开发和操作之间的所有摩擦。

几年前，我在文章“ [DevOps 死了，NoOps 万岁](https://medium.com/better-programming/devop-noops-difference-504dfc4e9faa)”中写过 NoOps 这篇文章详细解释了什么是 NoOps，以及为什么您应该接受它。我们可以用一句话来概括 NoOps 的精髓:

DevOps […]意味着你总是需要一些手工劳动。你仍然有一个人在流程的大部分后面，这意味着你在以旧的方式工作。

既然我们知道必须减少人力，下一个问题是我们如何安全地存储描述基础设施的配置。让我们来发现 GitOps！

## GitOps

顾名思义，GitOps 是 Git 和 operations 的组合，为所有配置创建一个单一的事实来源。“配置”这个词很模糊，但是你可以把它理解为应用程序配置、基础设施配置或者任何你想要的东西。这种方法跳过了许多关于如何在 Git 存储库中安全存储信息的技术细节，使信息变得安全、可访问和可复制。

这种方法与 monorepo 策略相结合，使配置变得透明，因此对系统管理员和开发人员来说一目了然。当然，您可以使用 secrets 和 vaults 将敏感数据与源代码分开。无论如何，如果你不想在基础设施管理上浪费精力，你可以阅读 AppOps，这种趋势旨在只关注开发过程！

## AppOps

正如我在上一篇文章*中解释的那样，“AppOps 的目的是将注意力集中在应用范围上，包括源代码和基础设施。”*

由于技术是使事情成为可能的使能器，DevOps 是使事情变得高效的方法。但是我们不能忘记我们的应用程序应该做的“事情”。每当我们在基础设施上花费精力时，我们就在为生产优秀软件的目标减少时间。

# 用不同的方式说同样的事情

在本文中，我提到了 DevOps 场景中最流行的术语。为了避免离题，我省略了 DataOps 和 MLOps，但是我们有足够的信息来得出一些结论。

在我看来，所有这些流行语最终都是一回事。自 2009 年 Patrick Debois 创造了 DevOps 这一术语以来，十多年过去了。

他创造了这个术语，以创造一种消除摩擦和重复任务自动化的文化。这过去是，现在仍然是我们的使命。

我们应该把 Git 作为真理的来源吗？

是的，我们应该。我们不能在纸上存储配置，因此，如果我们有一个文件，我看不到任何其他解决方案比存储在 Git 回购。

*我们应该像在 NoOps 一样取消操作吗？*

当然不是。NoOps 是一种尽可能减少人力的挑衅性方法，随着云的采用，这变得简单而自动。

*我们是否应该专注于应用程序开发，而不是忍受基础设施维护的痛苦？*

是的，我们应该。因此，如果有软件可以抽象基础设施，或者我们可以依靠云服务，我们必须这样做。这是 AppOps。

最后，你会同意我的看法，这都是 DevOps。

# 带什么回家

DevOps 是一种心态。一旦你加入了这种方法，你会发现所有这些趋势都是同一种独特哲学的一部分。然后，您需要找到一个解决方案来实施正确的策略。在市场上，您可以找到许多解决方案，其中大多数都是开源的，成本有限。

同样，这不是工具的问题，而是正确接受 DevOps 理念的意愿，用工具进行实验，并找到适合您的场景的更好的配置。
