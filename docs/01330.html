如何通过提交、推动和拉动成为团队成员

Git 是主流的源代码控制系统——软件开发人员事实上的工具。它用于管理在一个公司中一起工作的团队，或者运行开发人员分散在世界各地的开源项目。

习惯在团队中工作是新开发人员必须学习的第一件事。共享代码库是其中的一部分。因此，Git是新开发人员工具箱中的一个重要工具。学习如何正确地管理您的本地和远程存储库是团队成员需要掌握的一项关键技能。

这里有六个技巧来帮助确保你是一个好的Git公民，并且尽你所能保持团队的代码库尽可能的干净和无冲突。

顺便提一下，我假设你和你的团队已经为你的Git库选择了某种形式的工作流。如果你没有这样做，那么我下面说的很多都没有意义。如果您还没有选择工作流程，我强烈建议您立即选择。

推之前一定要先拉

这是一条基本规则。在您尝试将代码推送到存储库之前，您应该总是将所有当前的更改从远程存储库拉到您的本地机器上。这样做将确保您的本地副本与远程存储库同步。请记住，其他人一直在推送远程副本，如果您在同步之前推送，您可能会在推送时出现多个头或合并冲突。

默认情况下，Git不允许您将更改推送到具有远程提交的分支上。您可以关闭此功能，但不建议这样做。

经常拉

在大型团队中，中央存储库将不断变化。您应该努力让您的本地机器尽可能靠近远程存储库。方法是将远程服务器上的所有更改都拉到您的本地副本中。你可以经常这样做，并且应该经常这样做——至少每天一次，如果不是更多的话。

不经常推

当你推动时，你是在把你的改变强加给别人。只有当您知道您不会破坏构建，当您已经仔细检查了您正在推动的内容，并且当您觉得您的本地代码处于对他人有用的状态时，您才应该这样做。

记住，当你推的时候，每个人都会看到你做了什么。一定要对。

经常提交

许多开发人员没有经常提交。他们在做了许多更改后最终提交，结果，他们的注释不足以描述所有的更改。试图审查大而笨重的提交是令人恼火的，而且会适得其反。

相反，您应该经常使用精心制作的提交消息来提交小的、单个的更改，这些消息会告诉您小的、离散的更改是什么。许多容易理解的小错误并没有错。每一次提交都应该是对代码的逻辑的、独特的改变。如果你觉得有必要在你的提交信息中使用“和”这个词，你可能没有经常提交。小的提交更容易理解，如果需要的话可以稍后恢复。

经常合并“前进”

假设您正在使用一个标准的Git工作流，您可能会将大部分时间花在一个起源于develop分支的分支上。无论您的分支源自何处，您都应该经常将原始分支合并到您的分支中。

通过这样做，您可以确保您的分支不会偏离原始分支太远——记住，其他人也会改变原始分支。通过这样做，您将限制合并冲突，并且通常使您的生活更容易。

不经常创建拉请求

虽然您可以尽可能地“向前”合并，但是在创建pull请求以将您的代码合并回develop分支时，您应该更加谨慎。

您的组织可能有一个关于拉请求的策略，但是您应该小心创建它们。拉请求是要求人们仔细检查你的代码，所以要小心。只有当您的代码处于可以清晰、离散、完整地添加到代码库中的状态时，您才应该发出pull请求。同样，如果你觉得有必要用“和”这个词来描述你的“拉”请求，它可能太大了。

不经常创建拉请求，并且要小心。

结论

本文仅仅触及了使用Git的皮毛。掌握Git是所有开发者的义务。本文涵盖了您应该遵循的一些高级规则，以确保您的代码库保持整洁和无冲突。
