# 如何在技术文章中恰当地引用你的资料

## 避免媒体抄袭指南

![](img/92ef085660f6dd65acfdf32e32aa0581.png)

尼克·莫里森在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片

在[更好的编程](http://medium.com/better-programming)，我们读了很多文章。

我们与成千上万的作者合作过，我们的团队中有 15 名专业的文字编辑润色我们发表的每一篇文章。我们在适当的地方编辑语法和拼写，确保每篇文章都符合[更好的编程风格指南](https://medium.com/better-programming/better-programming-styleguide-suggestions-eada98e35150)。

最常让我们无法发表文章的问题是**抄袭**。剽窃通常表现为作者从文章中没有引用的来源复制并粘贴文本。

当我们发现抄袭的文本时，我们给作者一个机会用适当的归属来更新他们的文章。如果随后提交的文章带有抄袭的作品，我们继续与该作者合作将变得更加困难，因为我们无法相信他们提交的作品完全是他们自己的作品。

当然，每个人都会犯错，并不是每个复制粘贴作业都是恶意的。没有人会单独做任何事情，但是如果我们基于他人的作品，我们必须尽最大努力确保我们发布的作品属于该作者。

# 定义

韦氏词典的字典对 [*抄袭*](https://www.merriam-webster.com/dictionary/plagiarizing) 的定义是:

> 剽窃盗用或冒充(他人的思想或语言)作为自己的 **:** 使用(他人的作品)而不注明出处
> 
> 盗窃文学作品:把一个来源于现有资源的想法或产品呈现为新的和原创的

例如，一篇标题为“立即开始使用苹果 Swift”的文章可能会这样开头:

> Swift 编程语言于 2014 年发布，迅速成为历史上发展最快的语言之一。Swift 使编写速度惊人且安全的软件变得容易。
> 
> (然后文章继续)

快速的谷歌搜索会显示那两句话是直接从[Swift.org](https://swift.org)的第二段复制粘贴过来的，没有引用也没有注明出处。

可惜这是抄袭。不是作者的原著。它不使用引号或属性的原始来源，Swift.org 的文本。

如果不注明出处，我们将不得不拒绝这篇文章发表在《更好的编程》杂志上。

# 那么作者应该怎么做呢？

好消息是这很容易修复，而且做起来很快！

通过将作者复制和粘贴的文本用引号括起来，然后添加出处，作者已经纠正了他们的错误:

> “自 2014 年发布以来，Swift 编程语言迅速成为历史上发展最快的语言之一。据苹果公司的[Swift.org](http://swift.org)称:“Swift 使得编写速度惊人且安全的软件变得容易。”。

他们用引号来表明这些文字是别人写的，然后用署名(根据)来表明它们实际上是谁的文字。他们还链接到了来源，因此读者可以访问 Swift.org 了解更多关于 Swift 的细节，直接来自苹果。

# 改写

除了引用和引述他们的来源，作者可以*意译*。转述的形式是阅读一些东西，并根据原文写出你自己的版本，而不是逐字复制。

例如，作者可以把 Swift.org 的话解释为:

> 苹果在 2014 年推出了 Swift。此后，它成为发展最快的语言之一。Swift 的类型安全使得在其设计中编写快速且安全的代码变得容易。

注意它没有被引用，也没有引用 Swift.org。也不算抄袭。

那是因为这是作者的原创作品。他们提炼所读的内容，为他们的观众写一些原创的东西。

# 你应该引用什么样的东西？

*   任何不是你自己写的文章，不管多长或多短
*   你在文章中使用的网络照片
*   您从另一个源代码复制并粘贴的代码，带有指向该源代码的链接

# 结论

我希望这有助于弄清楚如何给你在文章中使用的内容正确的归属。我们不想让你偷任何人的作品。他们努力写作。

网络的美妙之处在于我们一起构建它。链接很容易做到，它很快，并帮助我们与我们的观众分享我们的资源。

就像在物理世界一样，在数字世界里偷窃是不好的。如果一个作者不能或者不愿意更新他们的文章以给出正确的归属，剽窃会阻止我们发表一些伟大的作品。
