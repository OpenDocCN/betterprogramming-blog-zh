## 因为没人知道你的缩写是什么意思

我感到失落。

在没有任何指导的情况下，我被扔进了一个团队，花时间破译首字母缩略词，并从一个会议跳到另一个会议。我们实际上在建造什么？我在这里做什么？

加入一个新团队是令人害怕的，入职过程可能会成就或毁掉你的新员工。

在十个不同的团队工作后，我经历了好的和坏的入职，我是组织另一个人到来的人。作为开发人员，我们经常偏离琐碎的任务，比如为某人的入职体验设置结构。

然而，这是团队中最重要的过程之一——它是让新成员感到受欢迎的大门。

你如何成功地吸纳新的开发人员？

这里有五种方法可以确保你的新团队成员顺利开始:

# 1.创建入职文档

一份完整的入职文档包括背景情况、他们需要访问的系统、团队成员职责、技术概念、首字母缩写词等。

让它成为一份活的文件，成为你的团队的真理之源。新手应该在文档上留下评论并进行编辑，这样下一个人的入职过程会更加顺利。

通过接收团队中每个人的意见，使文档具有跨职能性。思考:我们为什么要开发这个产品？什么是技术堆栈？我们的团队仪式是什么？

注意:入职文档不应只交给新成员。应该有人带他们走一遍，并随时可以消除任何困惑。

# 2.提供一个入职伙伴

你的新开发人员会有很多问题。找一个乐于在这个旅程中帮助他们的人，成为洞察项目和流程的关键人物。

虽然让入职伙伴成为开发伙伴很有帮助，但是团队中的任何人都可以担任这个角色。这位朋友应该明确表示他们可以提供帮助，甚至应该打几个电话来提供时间。

# 3.安排与产品和项目经理的会议

我坚信，在开始编写代码之前，您需要理解您正在构建的东西背后的目的。

因此，我强烈建议经理们在开发人员陷入技术泥沼之前，确保他们能够跟上产品的速度。在这些会议中，产品经理可以回顾产品路线图，任何用户研究见解，以及用户将如何与产品交互。

项目经理可以审查截止日期、涉众期望和团队惯例。

# 4.将他们与其他开发人员配对

作为开发人员，没有什么比看到代码更好的了！尽快让你的新成员和另一个开发者一起开始一个特性。

虽然他们不会完全理解代码库，并且构建给定的功能需要更长的时间，但在实际代码上工作而不是阅读代码的好处将会得到回报。

[结对编程](https://martinfowler.com/articles/on-pair-programming.html)将教会你的新开发人员，并帮助你的老开发人员巩固他们的知识。开发人员将一起对代码有更好的理解。

# 5.主持交互式架构会议

老实说，没有什么比交互式架构会议更棒的了。这对团队来说是一个合作和考虑全局的机会。

在白板的配合下，一名团队成员画出软件架构，并根据他们的理解进行解释。其他人可以贡献更多细节，帮助大家学习。

这个会议对每个人都有好处，因为在技术杂草中一段时间后，一些开发人员会忘记*为什么*团队做出某些决定。

作为奖励，定期运行此会话，并交换谁有机会绘制和解释图表。

入职是为您的开发团队创造良好体验的第一步。检查以下五个项目，让您的新开发人员了解:

1.  创建入职文档
2.  给他们配一个入职伙伴
3.  安排产品和项目经理会议
4.  将开发人员分配给功能配对
5.  安排交互式架构会议
