# **无所不知**

作为一名开发新手，我犯的第一个新奇的错误是，要成为一名优秀的程序员，你必须知道所有的事情。这是一个诱人的兔子洞，但如果你能避开它，那么你就能在这个领域更快地成功。

请记住，在您正在使用的特定应用程序中，一组代码或函数可以有唯一的引用。试图了解一切会让你感到孤立无援，因为你的努力没有什么可展示的。应用程序代码是高度相关的，扎实的基础知识加上学习的意愿是学习如何编码的更好方法。

基本上，要用 C#、Java、Python 等语言编写简单的程序，不需要对语言有很深的理解。在我看来，编程是一系列执行良好的简单步骤。是的，程序可能非常复杂，但这不是火箭科学。

**而不是什么都想学。一定要找到一个好的方法来跟踪你解决问题的过程。**

这可以像任何文本编辑器或文字处理器一样简单地使用剪贴板来记录笔记。或者，设计系统流程图的网站，如 visual-paradigm、MIRO 等。作为一名开发人员，你会惊讶于你必须定期搜索的内容。

> **提示:**一边走一边追踪你的脚步，这样你在寻找答案的过程中会更有效率。想象你在外面觅食，你想记住富饶的地区。

> 建立一个适合你的系统，或者努力去发现这意味着什么。

# 算法或破产

算法可能会令人生畏。如果你像我一样，没有计算机科学(CS)、数学或工程背景。*(这些就像编程中的三巨头。接近这些笨蛋的最好方法是把他们当成一个谜题，或者对游戏玩家来说，就像对游戏更深层次的探索。我当然没有写很多，但是我对排序算法有一个基本的了解。*

算法对于更多的理论知识是有好处的，但作为一个 app-support-dev，我通常会重新分解 API 调用，编写 CRUD ops，或者做一些更简单的 HTML/CSS 前端优化。

*注:教材* [*【算法导论】*](https://mitpress.mit.edu/books/introduction-algorithms-third-edition) *也让一众班长大开眼界。:-)*

不要认为你需要很深的 CS 水平才能编程。当你在做教程时，一定要想出一个跟踪方法和调试过程，比如搜索关键词。

例如，如果您正在训练的教程应用程序有一个引用或依赖关系发生了变化，现在导致了一个错误，并且没有为您安排修复。*你会看到这种模式在 dev 领域发生。*错误消息关键字将最终建立足够的上下文来解决问题。

我花了无数个小时在教程中试图找到一个解决办法，有时是 8-10 个小时。它最终只是基于更新的依赖项所需的一些简单的更改。

> 小贴士:从一开始就开始你的面包屑之旅，当你完成后，你会知道如何回家。您会注意到这里与第一个技巧的相似之处。

> 作为一名开发人员，为自己创建一个工作系统是你能为自己提供的最好的工具。

许多编程框架都有内置的方法来帮助完成繁重的工作。如果你能跟踪堆栈回调，或者理解你所在的应用环境的基本构建模块，你会好很多。很快，您将能够像不关任何人的事一样调用框架方法！然后，在你尝试去森林远足之前，也许可以试着在你的后院赤脚散步。:-)

有一个关键词*应用环境。*这就像当你读一个故事或玩一个角色扮演游戏时，人物拱门和情节都与你所在的章节或区域相关。字符拱门就像你的方法被调用来讲述你的应用程序的故事。应用程序中的高阶活动是引导这些人物拱门前进的情节。你的应用环境将是这些主题发挥作用的地方。

> **谈到*如何*穿越那片地形，你是决策者。**

# **分支策略**

*…每个 app 都像一棵树。*

分支结构看起来是独特的，但在它们背后都有相同的基本结构。主干是所有其他分支汇合的地方。你的团队将有自己的方式来理解合并为“开发”或“主”以保持应用程序的繁荣。越有条理越好，但是你必须愿意和你加入的团队一起工作。

看看下面这棵树的图片。现在想象节点是树的神经系统。当和*发生时，树会知道*如何对它茁壮成长所需的活动做出反应。这些内部信使就像分支是如何工作的。当每个信使节点响应另一个时，操作的顺序将是简洁的。总之，*团队内部沟通*将是关键。所以，像树一样思考…这是整个过程的一个非常抽象的版本。或者，如果你想得到技术，谷歌“git 分支策略”…**

# 角色名称

*成为全栈开发者意味着什么？*

当然，这个标题听起来不错，但在你的日常职责方面也会产生误导。我认为这是一个多面手，无所不能的角色。

作为一名新的开发人员，这个角色听起来像是一个目标或抱负，但是不要让谷歌搜索吓到你。这是我作为开发人员的第一个角色，你也可以！！

我的日常工作主要是跟踪数据是如何从前端转移到后端的，以及一路上的每一点中间层逻辑。当然，这听起来可能很多，但当你只是简单地做工作时，就不那么令人生畏了。

# **分析是我工作的 75%,而编码和协作是另外的 25%**

# 工作职责和期望

**快速对比**:系统架构师可能会遵循与应用支持开发人员相同的流程，反之亦然。他们做的任务可能是可互换的，或者架构师做一些开发人员可能支持的更高级的活动。相互作用是动态的，而不是一个静态的角色来界定每种能力。乐于接受团队的需求，并有信心站出来说话。你的经理不会读心术。

# **你是自己最好的拥护者**

> 注意:同样，大多数工作清单上的需求清单也是一个理想的场景。这就像当我们看电视，看到有人拥有理想的身材时，他们会得到很多报酬。让我们不要自欺欺人，把这个清单作为对开发者的现实期望。

# **相信自己，坦诚自己的能力。**

**不要把你的努力集中在适合一个特定的角色上，*围绕你的经验和兴趣建立你的理想角色。***

*我认为这是一种更有机、更有意义的方式来理解作为一名开发者意味着什么。*

> *提示:建立你的角色可能需要一路自我反省，但你可能会对自己的结局更满意。*

*例如，如果你对 UX/前端设计感兴趣，你可能想从平面设计或视觉艺术的角度来构建。如果你喜欢画画，并且有丰富的空间智慧，这个区域会非常适合你。*

*或者，如果你是一个喜欢搞清楚事物如何工作的分析师(我)，你可能会发现后端或全栈开发是你最好的选择。有大量的角色，它们的意义与你参与的项目相关——不管头衔如何。*

# *什么打开了发展职业的大门？*

*毅力、耐心、勤奋和解决问题的能力。*

*如果你有这些核心特质，那么发展可能就是你的囊中之物。如果纯粹是为了钱，那也可以。知道你接近这个领域的角度会帮助你找到一份更好的工作，让你的日常生活变得更好。*

*最终，这不正是我们都在寻找的吗？**如果你为工作所做的事情在某种程度上改善了你的生活，那就继续做下去。作为一名开发人员，这种动力将驱使你在软件开发的荒野中跋涉。轻踩！***
