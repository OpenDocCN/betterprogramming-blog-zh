关于编码的3个不方便的事实

编程世界

接受它们，以便进行实用的编码

当我学习计算机科学时，我的目标是学习如何开发优秀的软件。编码是其中的主要部分。

我喜欢编码，因为它具有逻辑性和分析性。对我来说，这是“科学”,这意味着有一个终极真理，使我们能够编写最好和最持久的代码。

在这个领域工作了二十年后，我现在可以接受一些过去不会接受的编码事实。这是其中的三个。

1.没有一种正确的编码方式

“在一个满是顶级软件设计师的房间里，如果两个人在同一件事情上意见一致，那就是多数。”— 比尔·柯蒂斯

在我的软件开发生涯中，我目睹了许多关于不同编码实践的有见地的讨论和辩论——无论它们是关于风格、模式，甚至仅仅是命名。

有时我赞同一方，有时我看到两者的观点，有时我不赞同其中任何一方。也有第一次不同意某人的时候，事后才意识到自己是多么的盲目。

现在，我开始接受的一件事是，所有的编码方式都有优点和缺点。人们可以永远捍卫他们的观点。

让我们看一个简单的Java例子if-else和三元…

if (a > b) { 
   print(a);
} else { 
   print(b); 
}
// OR
(a > b)? print(a): print(b)

关于为什么一个比另一个好，可以有无数的争论。我敢肯定，甚至有人不同意以上两者。

如果我们想让每个人都完全同意用一种单一的方式来编码一件作品，我们将没有答案。没有一种正确的编码方式。

但是如果我们愿意把自我放在一边，我们可以解决手头的问题——即使认为代码并不完美。"正确的编码方式不止一种."这不是解放吗？

2.所有代码都是临时的

硬件是为了经久耐用。软件是用来改变的。改变是唯一永恒的东西。

当我第一次学习编码时，我的想法是直截了当的:解决眼前的问题。完全没有建筑。

随着我继续学习，我学会了考虑假设场景，探索如何抽象出所有可能场景的公共逻辑，以便代码可以轻松扩展以适应未来的变化。这有助于我更通用、更清晰地构建我的代码。

然而，我后来发现，即使将代码抽象出来，也不足以迎合一些新的意料之外的变化。我想也许我应该把它进一步抽象化。抽象中的抽象…想法是拥有一些永远不需要再改变的抽象部分。

我大错特错了。过度抽象的代码不仅使代码变得更加复杂，而且还浪费了大量时间过度设计一些不太可能被使用的东西。有了这么多的抽象，做出错误抽象的风险就更高了。

我们无法预料所有的变化。这就是我们有软件的原因。随着变化的出现，我们可以改变它。这就是为什么它是一种可娇惯的东西。我们写的所有代码，包括基础架构，都应该是可变的。

永远不要编码一些东西，并期望它保持不变。如果它是永久性的，它应该被转换成硬件。

也许说“一个软件架构有多好，它就有多容易被重构”并不过分。

3.当我们编码的时候，我们也会犯错误

“如果调试是去除bug的过程，那么编程一定是把bug放进去的过程。”
- 埃德格·w·迪杰斯特拉

在电影中，我们有英雄和恶棍。但在编程界，编码的人也是引入bug的人。英雄也是反派。

一次，一位执行经理问我，“你能向我保证这个软件版本没有错误吗？”虽然我很想回答“是的”，但我知道我在撒谎。

编码就像打开和关闭数十亿个开关。缺少其中任何一个都可以认为是bug。没有编码人员会故意制造bug，但这是不可避免的。

我们不愿意承认我们创造了代码和错误，但这是事实。

程序员需要保持警惕

因此，我们需要接受我们的工作不仅仅是编码。每个人都可以编码。但是作为专业人员，我们需要做的不仅仅是编码来最小化过程中的bug创建。

这就是为什么我完全支持开发人员创建他们自己的单元测试，并对他们开发的软件进行大量手动测试。不要把这些责任推给所谓的QA组。

有QA人员来帮助进一步测试是很好的，但是我们对代码的质量全权负责。

当我们估算一件工作时，不要仅仅估算编码它所需的时间。测试的时间因素——包括执行手工测试和创建自动化测试。事实上，测试时间可能会占据整个任务时间的50%甚至更多。请把它考虑到工作中去。

除此之外，不断寻找方法来改进消除我们创建的bug的过程(例如，在CI中使用问题检测自动化、结对编程、代码审查、使用现代编程功能等)。).

程序员需要务实

即使竭尽所能，我们仍然不得不接受生活并不完美的事实。bug时有发生。

我们可以提高测试和bug预防的成本，但这将达到不再可行的程度。软件是要被改变的，减缓软件可能的改变只会阻碍进步和创新。

与硬件不同，软件中变更和错误修复的成本要低得多。所以软件开发的风险承受能力更强。在某些情况下，一个bug可以在几分钟内修复并分发给每个人。

因此，我们必须从不同的角度看待软件质量。在软件中，“预防胜于治疗”并不总是适用的。当预防的成本高于治疗的成本时，冒险是可以的。

我们应该学会优先考虑最重要的事情。区分职业限制错误和职业限制过程的能力至关重要。保持警惕和务实的平衡是关键。

结论

交付一个伟大的软件不仅仅需要编码，因为编码本身是不完整的。一个伟大的程序员需要更多的东西。处理不同视图以及处理模糊性和不完美性的能力对于我们做出最佳编码决策至关重要。

对于我们所做的一切工作，没有一个可以遵循的规则。这从来都不容易，但我们必须坚持下去。

这是一门艺术——一门不断变化的艺术。
