即使是有经验的开发人员也可能在编码面试中失败的6种方式

原文：https://betterprogramming.pub/6-ways-even-an-experienced-developer-can-fail-a-coding-interview-8571120b5480?source=collection_archive---------4-----------------------#2021-10-19

下一次工作面试的常见错误和制胜策略

我每周都在面试软件工程师。他们中有太多的人在编码面试中犯同样的错误，并且未能以预期的水平展示他们的技能。结果，他们得不到工作机会。

这很可悲，因为任何软件开发人员，不管他们的经验如何，都可以避免这些错误。以下是我最常看到的几个:

  1. 直接开始写代码
  2. 不接受面试官的暗示
  3. 过于复杂的事情
  4. 选择错误的编程语言
  5. 放弃解决问题
  6. 毫无准备地来面试

让我们看看这些错误是如何导致面试失败的，以及你可以做些什么来成功通过面试。

1.直接开始写代码

通常候选人一接到任务就开始编码。我很少看到这种方法能产生有效的解决方案。

一些开发者认为这会给面试官留下深刻印象。然而，事实远非如此。其他人想出一个主意；通常是错误的。他们认为没有必要解释它，并继续实施。

这里有三个问题:

  • 由于压力，很容易错过任务的关键细节，并选择错误的方法。
  • 面试官可能会认为候选人没有意识到分享想法和获得早期反馈的重要性。换句话说，候选人表现出糟糕的沟通和协作技能。
  • 候选人可能显得过于自信或倾向于仓促做出决定。

最好的方法是:

  • 首先，暂停几分钟，想一个可能的解决方案。在提出解决方案之前，花时间了解问题的各个方面。
  • 然后和你的面试官谈谈这个想法，以获得反馈。你将展示良好的沟通技巧和解决问题的深思熟虑的方法。此外，你会确认你的想法是可行的。

大多数面试官希望你在编写代码之前提出澄清性的问题并讨论你的方法。

2.不接受面试官的暗示

你的面试官见过其他人做同样的工作。他们可能知道一两个解决方案和典型错误。他们可能会通过放弃不会产生正确结果的想法来暗示候选人节省时间。他们也可能注意到候选人已经在正确的轨道上，并决定加快这个过程。

当候选人没有领会他们的暗示时，面试官可能会认为:

  • 这位候选人总的来说过于执着于自己的想法，不会考虑其他人提出的替代方案。这种性格特征会让你很难与候选人在一个团队中共事。
  • 候选人没有正确的沟通技巧:没有很好地倾听，没有接受指示，没有提出澄清问题。

当面试官给你一个暗示时——接受它:

  • 如果你不明白——要求阐明想法，这样你就能看到好处。
  • 如果你不同意，解释你的观点。他们有可能误解了你。不过，如果面试官坚持的话，最好尝试一下他们的提议。

记住面试官通常是站在你这边的。只要他们确认你符合他们的期望，他们就会雇佣你。因此，他们可能会给你提示，问一些引导性的问题。

3.过于复杂的事情

有些考生似乎觉得简单的东西不够“好”。他们试图煮沸海洋，将他们的解决方案过度复杂化，以用他们的知识打动面试官。

在我的一次面试中，候选人为他们的解决方案建立了一个测试套件。那花了一段时间。然后，候选人不得不在剩余的时间内匆忙编写实现代码。更快的选择是将测试数据硬编码到解决方案代码中，或者使用命令行界面传递它。无论如何，解决方案代码只是一个单一的函数。

当候选人的实现过于复杂时:

  • 他们把面试时间花在对解决任务没有帮助的事情上。
  • 它们增加了出错的风险，并且由于不确定问题是在代码中还是在设置中，使得即使是微不足道的问题也难以调试。
  • 他们让面试官认为候选人会在他们的新角色中摆弄工具和配置，而不是解决业务问题。

以下是应该做的事情:

  • 在你的实现中应用 KISS原则，避免分心，专注于解决编码问题。
  • 从简单安全的一步开始。如果你想出了一个幼稚但次优的解决方案，你可以稍后再完善它。请随意向你的面试官解释。

4.选择错误的编程语言

有时候，考生可以选择编程语言来解决问题。在某些情况下，他们选择一种不熟悉的语言或者一种需要太多样板代码的语言。

在一次面试中，候选人选择了Typescript，并且不得不花太多时间定义类型和修复神秘的编译器错误，而不是编写实际的代码。最终，我们没有时间了。优秀的旧Javascript可能是更安全的选择。

当候选人选择一种语言来打动面试官，而不是有效地解决问题时:

  • 他们冒着花太多时间在语言结构上而不是实现一个解决方案的风险。这无助于展示他们的最佳表现。
  • 他们努力调试他们的代码。
  • 面试官可能无法判断候选人是否因为不熟悉语言或不会编码而陷入困境。

最佳策略是:

  • 如果可以，选择你最熟悉的编程语言——最理想的是你每天都在使用的语言。没有必要增加失败的风险。
  • 如果你精通几种编程语言，选择“最简单”的一种。例如，如果您使用C#和Javascript，请选择Javascript。如果您使用Java和Python，请选择Python。

5.放弃解决问题

永远不要在面试中放弃解决编码难题。当候选人这样做时，面试官可能会认为他们也将放弃工作中的挑战性任务。此外，很难证明向一位在面试中认输的候选人提供职位是合理的。

如果你卡住了:

  • 回到起点，开始和面试官讨论你的方法。你可能会突然灵光一现。
  • 请面试官给你一个提示。

只要你一直在寻找解决办法，承认自己碰壁并寻求帮助是没问题的。

请注意，有时编码问题允许面试官在候选人实现他们的解决方案的过程中添加需求。他们用这种技术来观察候选人能走多远。在这种情况下要求提示可能是意料之中的。如果你承认你需要帮助——你可能会因为你的透明和开放而加分。

6.毫无准备地来面试

一些候选人认为他们无需准备就能解决编码问题。毕竟他们每天都在写代码！然而，典型的编码面试挑战不同于常规的软件开发任务。

例如，许多组织希望候选人熟悉基本的数据结构、排序和搜索算法、递归等。，来解决问题。

毫无准备地参加面试的问题是:

  • 面试官可能会把缺乏准备解释为对这个职位不感兴趣，不会建议雇佣这个候选人。
  • 面试官可能无法判断候选人是否对特定技能生疏(TDD、数据结构的使用、搜索算法、估计O(n)复杂度等)。)或者对他们不熟悉。他们很可能会选择后者。

要准备编码面试，请尝试以下方法:

  • 问问你的招聘人员有什么期望，并为此做好准备。例如，如果招聘人员说面试将包括现有代码库上的配对，考虑和朋友一起练习。如果他们告诉你它将专注于解决问题——找一个网站在线练习解决编码挑战。
  • 温习基本的数据结构和算法并练习估计算法的O(n)复杂度不会有什么坏处。

外卖食品

要增加您通过编码面试的机会:

  • 花点时间准备，解决编码问题，复习记忆中的基本Comp Sci。
  • 面试的时候，不要急着写代码。先想一想，然后向面试官解释你的想法。
  • 不要把事情弄得太复杂。不要把时间花在建立框架、库、写样板代码上。如果可以选择编程语言，挑一个最适合自己的。
  • 永远不要忽视面试官的暗示。带他们上船。如果你不明白他们，要求澄清。
  • 永远不要放弃解决问题。如果拐错了弯，后退几步，或者承认自己被困住了，寻求帮助。如果你觉得在任何时候都需要停下来想一想。

祝你下次编码面试好运！
