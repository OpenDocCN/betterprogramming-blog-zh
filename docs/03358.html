2020年你该选择哪种Java微服务框架？

探索微诺特和夸库斯与Spring Boot——他们有多好？

截至2020年，Java仍然是构建web应用程序最受欢迎的编程语言之一——尽管它必须面对来自Go、Python和TypeScript等新语言的激烈竞争。

在Java世界中，Spring框架已经成为微服务开发事实上的标准。通过像【Spring Boot】和 Spring Data 这样的库，该框架易于使用，并且允许高效且在很大程度上无痛的开发。

然而，近年来新的框架已经被引入，声称可以改善Java应用程序的启动时间和内存占用。由于我目前正在使用Java开发一个更大的基于微服务的应用程序，所以我想检查哪种Java框架最适合这样的架构。

因此，我的主要关注点将是开发的简易性以及生成的微服务的资源管理。

就资源管理而言，Spring(实际上是大多数Java平台)从来没有获得过最好的声誉，尤其是在单个进程所需的开销方面。在应用服务器时代，这不是一个主要问题，因为实例数量很少。然而，随着微服务架构及其大量小实例的兴起，这变得越来越成问题——或者如 Christian Lusardi 最近所说:

“我发现，在Spring Boot上运行的基本Java应用程序至少需要1GB的RAM才能运行，这在开发中间件应用程序时是可以的，但在微服务架构中这就非常糟糕了！”

候选人

春天

作为对早期Java企业复杂性的回应，Spring在2003年应运而生。从本质上讲，Spring最初是一个依赖注入(DI)和面向方面编程(AOP)框架，后来发展成为一个易于使用的web应用程序框架。通过其丰富的文档、广泛的使用和无数的库，Spring允许开发人员高效地创建和维护应用程序，并提供了一个平坦的学习曲线。

Spring使用反射在运行时执行DI。因此，当spring应用程序启动时，会扫描类路径以查找带注释的类。在此基础上，具体对象被实例化和链接。

虽然这非常灵活且对开发人员友好，但它会使启动速度变慢，并且非常消耗内存。此外，将这种机制移植到GraalVM非常困难，因为它不支持反射。

微型机器人

Micronaut 是一个现代的全栈-微服务框架，由Grails框架的创建者于2018年推出。

它提供了构建全功能微服务应用所需的所有工具。同时，它旨在提供快速启动和减少内存占用。这个目标是通过使用Java注释处理器来执行DI、创建面向方面的代理以及在编译时而不是运行时配置应用程序来实现的。

Micronaut中的许多API都是受Spring和Grails的启发。这是设计出来的，有助于快速引入新的开发者。因此，Micronaut提供了像Micronaut HTTP、数据、安全和各种其他技术的连接器这样的模块。然而，这些库的成熟度仍然落后于Spring。

夸库斯

Quarkus 是红帽在2019年推出的Kubernetes-native Java框架。它构建在MicroProfile、Vert.x、Netty和Hibernate等标准之上。

Quarkus的目标是通过允许更快的启动、低内存消耗和容器编排平台中近乎即时的扩展，使Java成为Kubernetes中的领先平台。Quarkus通过使用定制的Maven插件在编译时而不是编译时执行尽可能多的工作来实现这一点(在Quarkus中，这也被称为编译时引导)。

Quarkus使用的主要是现有的标准技术，但它对扩展开放。然而，由于该项目一年前才开始，这些扩展的成熟度和兼容性并不总是很清楚。随着平台的发展，这种情况可能会在未来发生变化。

Helidon微型剖面图

MicroProfile 项目开始于2016年，当时还不清楚甲骨文是否会以及如何继续进行Java Enterprise的工作。

像它的前身JEE一样，MicroProfile是一个可以由不同厂商实现的规范。

此后出现了多个这样的实现，最著名的是Payara Micro和Helidon MP。Payara是源自GlassFish的Jakarte EE服务器，Payara Micro是它的MicroProfile实现。Helidon是Oracle在2018年启动的一个运行时，提供了自己的MicroProfile规范实现。

因为它们源自JEE，所以MicroProfile规范是成熟的，并且有很好的文档记录。然而，缺少现代技术的连接器或像Spring Data和Spring Security这样的库的替代品。

此外，MicroProfile的未来还不明朗，因为与此同时，Jakarta EE(也在Eclipse Foundation内)的开发已经开始。因此，这两个项目很可能会在未来合并，或者至少是紧密协调。

比较框架

为了比较提到的框架，我使用它们中的每一个实现了一个简单的应用程序。示例应用程序由一个创建、读取、更新和删除对象的REST接口和一个将这些对象存储到表中的关系数据库连接器组成。

如果一个框架支持不同的访问数据库的方式，我试图为不同的变体实现示例项目。然后，我比较了这些应用程序的性能。

我使用OpenJDK Docker映像运行这个应用程序。如果一个框架支持生成原生GraalVM映像，我还比较了它们的性能。此外，请查看我的文章“使用R2DBC、Micronaut和GraalVM 进行反应式数据库访问”，了解有关GraalVM的更多信息。所有这些应用的源代码都可以在GitHub 上找到。

我比较了这些应用在三个关键阶段的性能:

  • 实现示例应用程序有多容易？为了实现这些框架，我不得不查阅文档以及搜索有关Stack Overflow等平台的信息。
  • 编译应用程序需要多长时间？我测量了执行一次干净构建所花费的时间，包括Docker映像的生成。对于GraalVM，这包括生成本机映像的时间。
  • 启动一个应用需要多长时间？这里我测量了运行docker up到应用程序正确响应第一个HTTP请求之间的时间。此外，我比较了刚启动后的空闲应用程序的内存占用量。
  • Load:应用程序在peek下可以处理多少请求？我使用JMeter执行负载测试，并测试了应用程序，其中25%的请求执行数据库写操作，75%的请求只执行数据库读操作。然后，我再次测量了应用程序在其峰值性能下的内存占用。

我在运行Ubuntu 19.01的谷歌云平台虚拟机上执行了所有测试，该虚拟机配有四个英特尔Haswell CPUs和15 GB内存。为了避免干扰因素，所有的测量都重复了多次。你可以在 GitHub 上找到使用的脚本以及原始数据。

结果

易于开发

由于我以前只知道使用Spring Boot，这是一个有点不公平的比较。然而，当检查文档和可用的信息和例子时，Spring是最容易开始的框架。

Micronaut的文档做得很好，有类似Spring和Grail的API。因此，对于一个Spring开发者来说很容易上手。

在我看来，Quarkus的学习曲线有点陡峭，因为与Spring和Micronaut相比，它们的库和API还不太成熟。我特别想念简单的数据库访问。

但在我看来，Helidon显然是最后一个，因为我费了很大劲才让应用程序运行起来。

汇编

使用OpenJDK时，所有框架的编译时间都非常相似，都在6.98秒(Spring使用JDBC)和10.7秒(Quarkus)之间。

然而，生成原生GraalVM映像非常耗时，需要231.2秒(Micronaut使用JDBC)到351.7秒(Micronaut使用JPA)。这使得本机映像基本上对开发毫无用处，因为等待四分钟来编译一个简单的应用程序实在是太长了。

启动

使用Spring数据的Spring Boot应用程序平均需要8.16秒才能启动。去除JPA和Spring数据后，这个时间缩短到了5.8秒。

在这里，Micronaut(使用JPA时为5.08秒，使用JDBC时为3.8秒)和Quarkus (5.7秒)兑现了他们缩短启动时间的承诺。

只有Helidon MP比Spring还慢——平均8.27秒。

然而，真正的赢家是GraalVM。本机映像的启动时间在1.39秒(Quarkus)和1.46秒(Micronaut使用JDBC)之间，明显快于OpenJDK实现。

启动后的内存使用情况非常相似。Spring分配了420 MB内存(使用Spring数据)和261 MB内存(使用JDBC)。

Micronaut使用JPA为262 MB，使用JDBC为178 MB。

197 MB的Quarkus性能更好。赫利登MP，414 MB，和Spring Boot差不多。

同样在这里，原生GraalVM映像大大优于OpenJDK实现，仅使用7 MB (Quarkus)到27 MB (Micronaut使用JPA)的内存。

最佳性能

在负载下，Spring Boot表现得相当好，每秒能够处理342个(使用Spring数据)和216个(JDBC)请求，使用581 MB (Spring数据)和484 MB (JDBC)内存。Helidon显然是最后一个，只能提供175 r/s的服务，同时分配超过1 GB的内存。

其他框架能够提供400 r/s (Quarkus作为本地映像运行)到197 r/s (Quarkus on OpenJDK)的服务。各种Micronaut实现介于两者之间，JDBC比JPA略胜一筹，本机映像比OpenJDK略胜一筹。

在内存使用方面，OpenJDK上的Quarkus表现得出奇的好，只消耗了255 MB的内存。这甚至比作为本机映像运行的相同应用程序还要少，后者平均占用368 MB内存。

然而，Micronaut被证明是相当浪费的。OpenJDK中运行的JPA实现平均使用了880 MB，比Spring的内存使用量高出50%以上。然而，使用JDBC和本机映像帮助Micronaut将其内存占用减少到367.8 MB。

结论

与Spring和MicroProfile等现有框架相比，新的Java框架Micronaut和Quarkus承诺了更快的启动时间和更低的内存占用。

他们确实兑现了这个承诺——但只是在空闲或负荷较轻的时候。在这方面，它们优于Spring，尤其是当它们与本机GraalVM映像结合使用时。然而，在负载下，它们并没有提供太多的优势，甚至在作为本机映像运行时也是如此。

由于Spring仍然提供了迄今为止最好的开发者体验，在我看来，它仍然是最适合微服务应用的Java框架——即使考虑到它在启动时的糟糕性能。

令我吃惊的是使用Hibernate/JPA/Spring数据的巨大成本。即使对于这个非常简单的应用程序，内存开销(以及r/s)也是巨大的。在这里，我特别喜欢Micronaut数据的解决方案，它可以自动生成存储库代码，而不需要JPA。这确实是可以添加到Spring数据中的东西。

原生GraalVM映像在启动时表现出惊人的速度和内存效率，但是在负载情况下，它们并没有提供显著的优势。由于原生GraalVM的生成带来了一些额外的困难，并且编译时间急剧增加，因此这种技术目前仅在需要快速启动时有用，例如，在无服务器架构中或者非常快速地向上扩展。在所有其他情况下，与负载下的类似性能相比，成本仍然太高。
