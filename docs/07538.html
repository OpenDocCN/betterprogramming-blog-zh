在我的上一篇关于最慢的现代编程语言的文章中，我讨论了像Python这样的动态类型语言是如何用户友好和不易出错的。

事实证明，并非所有人都同意这一观点。

许多高级开发人员表示，他们发现使用动态类型语言是一件痛苦的事情。下面的回复收集了大部分的掌声。

“请问，什么？动态类型语言比静态类型语言更不容易出错？很抱歉，但是，在我21年的软件开发生涯中没有。”— 拉斯穆斯·舒尔茨

根据这个答案，我决定收集让高级开发人员回避动态类型语言的主要原因，并在这里列出这些原因来澄清误解。

为了更好地说明这个问题，我们将Python作为动态类型语言的一个例子，因为它越来越受欢迎。

动态打字

在这个上下文中键入与击键无关。这个词来源于数据类型。

说到编程，很多类型化的类，比如强类型化，鸭类型化就派上用场了。但我们将仅限于最常见的:

  • 动态打字
  • 静态打字

动态类型是在运行时标记类型错误。比如说，它也不必像Python、Ruby和JavaScript那样显式声明数据类型。

与动态类型相反，静态类型是编译时类型错误的报告和数据类型的显式声明。C、C++和Java都是这种情况。

关于Python和编程语言有趣的是，它们的一些优点也是它们的缺点。

动态类型被设置为通过隐式数据类型声明删除一些代码行来简化编码过程。尽管如此，这个特性有一个很大的缺陷。

全局解释器锁

高级开发人员会对性能表示惊讶。

与只处理几行代码的初级开发人员不同，维护和编写健壮的产品代码(通常有数百或数千行代码)的任务落在了经验丰富的开发人员的肩上。因此，拥有一种高效的编程语言可能变得毫无疑问。

也就是说，因为全局解释器锁(GIL)禁止充分利用计算机资源，所以它是Python和MRI Ruby等编程语言的性能瓶颈。受GIL影响的资源是CPU线程。

然而，不使用GIL的编程语言充分利用了CPU的能力。因此它们支持并行计算。

并行计算只不过是让所有线程同时运行。由于需要处理的数据量巨大，这种类型的计算现在比以往任何时候都更加重要。

并行计算的一个例子如下图所示:

假设相同的CPU时钟速度，你的计算机有越多的线程，你的程序运行得越快，这是合理的。

然而，GIL的出现终结了并行计算。

GIL是一个一次只允许一个线程使用GIL的锁。线程的选择遵循排队方式。这意味着，当优先级最高的线程正在使用GIL时，其他线程将处于等待状态，直到GIL被释放。

最重要的是，用户无法控制线程的选择。相反，操作系统是负责线程优先级排序的。

为了解决这个问题，许多程序员，或者至少是聪明的程序员，尝试使用Python的multithreading模块在线程间手动拆分进程，希望获得更好的性能。他们最终会表现得更差。

虽然结果可能看起来很奇怪，但这是计算机科学，而不是计算机推测。如果你深究此事，一切都会水落石出。

尽管Python的核心开发团队完全意识到了这个问题，但是很难摆脱GIL，因为它是Python许多细节的支柱，比如内存管理和C扩展等等。

Python的官方作者吉多·范·罗苏姆说，他对Python是否会支持并行计算没有信心，因为这最终是该语言的设计方式。

然而，像C++这样的静态类型语言不受GIL的限制。这使得它们在比较中更有效率。

空白区分

使用一种会根据放错位置和缺少空白标记错误的编程语言可能不是每个人都喜欢的。这些空白是空格、制表符、换行符、回车符或换页符。例如，与C不同，Python确实对空格很敏感。

我们将通过下面C和Python代码的比较来证明这一点。

Python版本:

i = 50
if i % 2 == 0:
    print("inside if statement")  print("i is even")

输出:

print("inside if statement")  print("i is even")
                                  ^
SyntaxError: invalid syntax

c版本:

#include<stdio.h>int main(void)
{int i = 50;if (i % 2 == 0)
{printf("inside if statement\n");printf("i is even\n");}}

输出:

~/ $ ./test1
inside if statement
i is even

与C版本不同，C版本抵制了混乱的代码结构，并以正确的输出结束，Python生成了最初由放错位置的语句引起的语法错误。所以有人可能会说Python不如C++或者raw C健壮。

具有讽刺意味的是，虽然许多专业程序员认为空白敏感令人讨厌，但许多Pythonistas认为处理空白问题比最后必须进行列追踪要好。

最后，在处理大块代码时，空格敏感性问题很烦人。然而，如果良好的编码习惯被灌输到您的团队中，空白敏感性问题可以很容易地被克服。

向后兼容性

不支持向后兼容意味着旧版本的Python代码可能无法在新版本下工作。换句话说，我们绝对需要查找新版本中出现的语法变化，并相应地重写代码。

向后兼容性成为严重问题的一个很好的例子是第一次从Python 2到3的过渡。

Python核心开发团队认为，人们将Python 2代码转换成Python 3代码不成问题。但是他们错了。

Python的作者自己也承认了这一点，他说:

“我们低估了有多少人已经编写了大量的Python代码，然后基本上忘记了它是如何工作的。所以，他们没有能力升级它。我们意识到我们有一个问题。”

对这个问题的结论是延长Python 2.7的生命周期。

外卖食品

编程语言一直是一个热门话题，所以没有对错之分。至少一个人有一些理由喜欢一种语言胜过另一种语言。

一般来说，每种通用编程语言都适合特定的人群。Python的官方作者说:

“学习Python编程比学习Java或Swift编程容易得多。对于学习计算机科学的专业软件开发人员来说，Java和Swift是很好的编程语言。然后开始在一家软件开发公司工作。但是Python可以教给中学的孩子。”

除此之外，最好的情况是拥有一种能够结合C++和Python优点的编程语言。

如今，最接近这个目标的人是朱莉娅。
