# 为什么不应该总是使用 IDE

> 原文：<https://betterprogramming.pub/why-you-shouldnt-always-use-an-ide-28ed8c7e6843>

## IDEs 让你的生活变得更加简单——直到它们不再简单

![](img/8d25455df33e12fe753abf1f81614b1a.png)

杰佛森·桑多斯在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片

集成开发环境，简称 IDE，是一个帮助你写代码的工具。它们支持许多特性，比如调试、代码完成、语法高亮、构建自动化、重构、版本控制等等。然而，在开发过程中使用 IDE 可能并不总是最好的选择——尤其是如果您刚刚起步的话。

在我们深入探讨不使用 IDE 的原因之前，让我先搞清楚一件事。我喜欢 ide(尤其是来自 JetBrains 的那些)，为它们付了很多钱，并且每天都在使用它们。它们让我的生活变得更加轻松，并帮助我更快地编写代码。

然而，当我十多年前开始编码时，我开始在非常基本的文本编辑器中这样做(在真正与一些 ide 斗争之后)。这真的帮助我成为今天的开发人员。因此，让我们深入探讨一下为什么不应该使用 IDE 的原因！

## 语言和框架理解

当你写代码的时候，ide 在支持你方面是惊人的。但是，如果您正在使用一种您并不熟悉的编程语言或框架，IDE 可能会在您甚至不知道您正在编写什么之前自动完成您的代码。这样，你不会完全明白你在做什么，也许只是明白你正在实现的结果。

如果您没有 IDE 的自动完成功能，您将被迫阅读文档、检查日志和错误消息，从而彻底了解您正在使用的编程语言或框架。这极大地帮助您理解一种语言或框架是如何工作的，以及如何利用它来获得优势——而不是用 IDE 作为武器来对抗这种语言或框架。

## 太多，太快

如果你开始学习用一种新的语言或框架进行开发，你首先必须从基础开始。然而，IDE 提供了许多复杂的选项，如代码片段、自动完成、智能重构和其他接管代码控制权的选项。通过利用所有这些复杂的功能，你可能在学会如何正确游泳之前就已经潜入了游泳池的深水区，否则你会在很久以后才看到这些复杂的功能。

相反，应该专注于学习你正在使用的语言或框架的基础知识，并在这些知识的基础上进行构建。当我试图用 HTML 创建我的第一个网站时，我开始在 Dreamweaver 中拖放一些元素。然而，过了一段时间，我挣扎着对齐一个图像。因为我只使用可视化编辑器，所以我缺乏关于底层 HTML 的适当知识来解决这个问题，并且我完全被生成的代码难倒了。

失败后，我又开始了同一个项目，但这次是我自己编写 HTML 代码。这一次，我不仅学到了更多，而且还完全避免了以前遇到的问题。所以，当你开始学习一门语言时，不要依赖 IDE 来帮助你了解它的特性。首先，自己尝试一下，只有在你探索了这门语言的深度之后，才依靠 IDE 来完成繁重的工作。

## IDEs 是肥猫

ide 是非常广泛的工具:它们在你写代码的时候分析你的代码，它们能够知道你的代码库中的每一个类、函数和变量，它们通过运行调试器来帮助你，它们跟踪改变的文件等等。正如您所料，这可能会消耗大量资源。

如果您在一个资源有限的系统上工作，您宝贵的资源可能已经被您打开来研究您正在探索的语言或框架的几十个浏览器标签消耗掉了。这可能会为驱动您的 IDE 的后端进程留下很少的资源，导致性能缓慢，并在您进入新世界的第一步中带来许多烦恼。探索一种新的语言或框架可能会令人望而生畏。不得不与缓慢的工作环境作斗争可能是压垮骆驼的最后一根稻草，让你放弃发展。

## 隐藏的构建过程

从源代码构建应用程序可能是一个复杂的过程。IDE 可以为您处理这个复杂的过程，让您有时间专注于应用程序的开发。但是，如果您计划将您的应用程序部署到某种服务器或应用程序商店，您可能需要采取一些您的编辑器无法为您采取的步骤。

因此，知道如何采取构建应用程序所需的步骤是您知识工具箱中一个非常强大的工具。了解依赖项、构建链、Makefiles、编译器、优化、标志，以及用您选择的语言或框架构建应用程序所需的任何东西。通过这种方式，您将了解 IDE 为您执行的一系列复杂步骤，并且当您的 IDE 让您失望时，您会更好地理解该做什么。

## IDEs 会把你引向错误的道路

快速修复并不总是正确的修复。仅仅因为您的 IDE 可能建议或敦促您通过重命名变量、移动代码或将类放入新的命名空间来解决问题，并不意味着这就是您在项目中看到的问题的正确解决方案。

当您应用 IDE 建议的简单修复时，它甚至可能进一步破坏您的应用程序。项目中的问题可能与您的业务逻辑或数据存储有关，而您的 IDE 可能对此一无所知。

如果您依赖 IDE 来修复代码中的错误，您可能会认为代码中的简单修复就能解决问题，即使更大的问题可能会破坏您的代码库。在依靠 IDE 解决问题之前，您应该了解问题的原因，并知道如何自己解决问题。

## 您的 IDE 可能不理解您的项目

ide 在理解你的代码库方面做得非常好——只要你在某种程度上遵守语言或框架的标准。如果您正在编写一个简单的应用程序，并且您只是想尝试一些功能，IDE 可能会因为不理解您正在设置的项目的结构而使您烦恼。

当然，您应该继续坚持编码标准。但是，当你刚开始使用一种新的语言或框架时，如果不去应付一个不断要求你重构代码的 IDE，记住所有的需求已经够难的了。

即使你知道所有的行业标准，你也可能有一些不遵守它们的理由。

在我以前的雇主那里，我看到了一个并不符合行业标准的代码库。因此，试图在这个代码库上使用 IDE 带来了更多的烦恼，而不是好处:它不断告诉我被调用的函数不存在，自动完成功能不工作，并且它试图修复符合公司标准的代码。在这个代码库上使用 IDE 就像逆水行舟，让我的工作变得更加困难。

# 结论

让我们以诚实来结束。如果你正在半专业地编写代码，你可能有更多的理由使用 IDE 而不是不使用。正如我以前说过的，我每天都使用一个，如果只允许我使用常规的文本编辑器，我可能会发疯。

但是，如果你从这个故事中学到了什么，那就是你应该知道 IDE 为你处理的所有复杂任务——不仅这样你就知道当你的 IDE 失败或不可用时该做什么，而且还要感谢你不必日复一日地担心这些复杂的任务，这样你就可以专注于保持理智。

编码快乐！
