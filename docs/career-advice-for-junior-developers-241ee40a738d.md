# 初级开发人员的职业建议

## 软件工程师在他们职业生涯的早期经常问我，“我怎样才能快速提升？”

软件工程师在他们职业生涯的早期经常问我，“我怎样才能快速提升？”

换句话说，我怎样才能在尽可能短的时间内成为一名有效的贡献者？如何才能熟悉我们海量的代码库？我如何学习所有我应该知道的事情？

开始一份新工作可能会有压力——即使是作为一名高级工程师——尤其是当这是你的第一份编程工作时。所以，让我们来看看四个提升初级开发人员的策略，这样你就可以立即投入工作。

# 1.问很多问题

![](img/0465ee60457acd302cbcdf80cc4e0183.png)

照片由 [Camylla Battani](https://unsplash.com/@camylla93?utm_source=medium&utm_medium=referral) 在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 拍摄。

首先，问问题——很多问题。或许更重要的是，不要害怕提问。鼓起勇气问问题可能很难，因为这样做意味着承认你并不知道所有的事情。但是你为什么要这样做呢？软件工程是一个非常广阔的领域，任何人都不可能知道所有的事情。

提问，尤其是在一个公开的论坛上，比如你公司的一个松散的渠道，会让你处于一个易受攻击的位置。你可能会疑惑“别人会不会因为我不知道这个东西而看不起我？我会失去作为软件工程师的可信度吗？”在那些时刻，提醒自己一些事情是很重要的。

首先，每个人都必须从某个地方开始。即使是拥有 30 年经验的软件工程师也曾处于你的位置，试图在这个广阔的行业中找到自己的位置。

第二，如果你有一个问题，很可能其他几个人也有同样的问题。有勇气去问对你和你的同事都有帮助。

三、[冒名顶替综合征](https://en.wikipedia.org/wiki/Impostor_syndrome)影响所有人。每个人都会在某个时候觉得自己不够好，觉得自己不值得呆在现在的位置上，如果他们的同事意识到他们知道的很少，他们就会被发现是个骗子。别听那个声音。

第四，当你在公共论坛上提出你的问题时，它会成为你以后可以参考的文档。这就是为什么我总是鼓励那些在 Slack 上给我发私人信息的工程师，转而在公共 Slack 频道上发布他们的问题。然后，整个团队或组织可以插话帮助回答，可能有相同问题的其他人将受益，并且该对话可供将来有此问题的其他人搜索。

现在，问题接收端的工程师们需要注意:当有人在公共论坛上提问时，就会产生一个漏洞，将神圣的信任交到你的手中。由你来决定你公司的文化。这家公司是一个心理安全的地方吗？在这里，人们可以自由提问，而不会被斥责或看不起。在你如何回应时要明智，以免你创造了一个让你的同事不敢说话的环境。

当谷歌进行研究以确定哪些因素有助于高绩效团队时，心理安全是首要因素。团队成员需要感到安全，他们能够在彼此身边变得脆弱。

也就是说，作为一名初级开发人员，为了提高效率，你会问什么样的问题呢？这里有几个例子:

*   你能给我介绍一下我们应用的架构吗？我们使用什么框架和库？
*   你能给我介绍一下我们的代码基本目录结构吗？代码在哪里？它是如何组织的？
*   开发过程是什么样的？我们使用什么类型的 [Git 工作流](https://www.atlassian.com/git/tutorials/comparing-workflows)？
*   [发布过程](https://devcenter.heroku.com/articles/git)是什么样子的？新代码如何投入生产？新代码发布的频率如何？
*   为什么[Feature X]是这样实现的？
*   为什么我们用[库 A]而不用[库 B]？

事实上，这些都是任何人在开始一份新工作时都应该问的绝妙问题——不仅仅是初级开发人员。

# 2.当你需要帮助的时候寻求帮助

![](img/23ea7bc39379024b1466cc34441cf395.png)

Amy Hirschi 在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片。

同样，当你需要帮助时，寻求帮助也很重要。

与困难的任务搏斗是学习如何解决问题和排除故障的关键部分。如果有人一直在那里握着你的手，你就不可能进步得那么快。但是有一点是更明智的，那就是接受你需要帮助。

这里有一个很好的经验法则:如果你被某事卡住了，再尝试 15 分钟，如果你仍然被卡住，你必须寻求帮助。

这给你的任务设置了一个时间框。它还要求你在寻求帮助之前多花一点时间在这个问题上，这样你就不会马上放弃。更重要的是，你可能会受到时间限制的激励去解决它，因为也许你*真的*不想寻求帮助！

如果 15 分钟的时限已经过了，你仍然觉得卡住了，你*必须*寻求帮助。你可能认为继续独自奋斗是高尚的，但是请记住，你的工作是有报酬的。你在某件事情上花费数小时却毫无进展，这在财务上是不负责任的——尤其是如果一个同事可以给你一个快速的提示让你畅通无阻。你的队友可以成为你的巨大资源，并随时提供帮助。

不管他们是否知道，经验丰富的老师和导师在帮助初级开发人员时，会经常使用维果茨基的最近发展区理论以及脚手架理论。最近发展区(ZPD)是“学习者在没有帮助的情况下可以做的事情和他们在拥有更多知识或专业技能的人的支持下可以做的事情之间的距离。”脚手架是一种向学生提供指导，帮助他们在 ZPD 内操作的技术。

因此，一个有帮助的导师给初级开发人员足够的指导，使他们能够完成他们的任务，或者足够的信息，在他们可能处于的任何开发阶段解除他们的障碍。

# 3.不断学习

![](img/83d65628c34d24a0941e60a73b959744.png)

[Adeolu Eletu](https://unsplash.com/@adeolueletu?utm_source=medium&utm_medium=referral) 在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片。

软件工程领域是不断变化的。新的语言被创造出来，曾经流行的库和框架被新来者取代，设计趋势来来去去。为了跟上这个快节奏的世界，你必须不断学习。软件工程师不仅仅是上大学或参加开发训练营，毕业，找到工作，然后就再也学不到任何新东西。学习是我们每天都在做的事情。

早在 2008 年，马尔科姆·格拉德威尔的书《异常者[](https://www.amazon.com/Outliers-Story-Success-Malcolm-Gladwell/dp/0316017930)*》推广了“一万小时法则”。这条规则表明，做一件事大约需要 10，000 个小时才能成为这方面的专家。虽然你练习得越多，你就会做得越好，这是有道理的，[自从这本书出版以来，一万小时法则已经被其他人揭穿了几次。](https://www.6seconds.org/2018/02/09/the-great-practice-myth-debunking-the-10000-hour-rule-and-what-you-actually-need-to-know-about-practice/)*

*原来真正重要的不仅仅是*练多少*，而是*怎么*练。“练习”和“刻意练习”是有区别的。*

*当学习演奏一种乐器时，你必须仔细考虑如何练习和练习什么。如果你正在学习一首特定的歌曲，你不只是一遍又一遍地浏览这首歌。每次都只是简单的从头到尾播放，效果并不好。这首歌可能会有比其他部分更具挑战性的部分。如果你是在刻意练习，你可能会一遍又一遍地弹奏同样的四个小节，直到你把它们固定下来，然后再转到歌曲的另一部分。*

*同样的概念也适用于软件工程。不要敷衍了事。对你想学的东西要深思熟虑。*

*如果你觉得自己在编写单元测试时遇到了困难，那就去看两个小时的 [Pluralsight](https://www.pluralsight.com/) 课程，用 [Jest](https://jestjs.io/) 进行单元测试(或者任何与你的编程语言相关的测试框架)。*

*如果你试图学习[反应](https://reactjs.org/)，去阅读文档。 [React 文档](https://reactjs.org/docs/getting-started.html)其实真的很好看！*

*请务必了解贵公司使用的一些基本技术。熟悉 [AWS](https://aws.amazon.com/) 、 [Heroku](https://www.heroku.com/) ，或者你使用的任何 [IaaS/PaaS 提供商](https://www.heroku.com/developers)。如果你是前端开发人员，学习你公司使用的 UI 框架或库，比如 [Angular](https://angular.io/) 、 [React](https://reactjs.org/) 或 [Vue](https://vuejs.org/) 。如果您经常使用数据库，请了解一下 [SQL 和 NoSQL 数据库](https://www.geeksforgeeks.org/difference-between-sql-and-nosql/)之间的区别以及它们各自的优缺点。*

*换句话说，花时间“磨利锯子”这个术语是斯蒂芬·R·科维创造的，是他的《高效能人士的 7 个习惯》一书中的最后一个习惯。引用柯维博士的话，“我们绝不能忙着锯东西而没有时间磨快锯子。”*

*很容易陷入上班时间只完成任务工作的陷阱。毕竟，你在上班，你的雇主付钱让你去上班，所以你应该“工作”，对吗？然而，这种想法是短视的。这就像试图用锯子锯倒一棵大树，却从不花时间磨利锯子。你花在磨锯子上的时间不是花在砍树上的时间。但是用更锋利的锯，你的切割会更有效。事实上，如果你没有停下来磨利锯子，你可以用比砍树更少的时间砍倒这棵树。*

*好的雇主认识到这个真理，并积极鼓励员工每周花几个小时进行一些有意识的学习。不要因为在工作时间花时间阅读文章或观看视频教程而感到内疚。当你把这些事情作为你有意识的技能强化练习的一部分时，你会成为一个比你 100%的时间都专注于任务工作时更有效率的工程师。*

# *4.参与代码评审*

*![](img/dd06125ad0ab71a22a1cf3b670ef5e78.png)*

*在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上由[海拉戈斯蒂奇](https://unsplash.com/@heylagostechie?utm_source=medium&utm_medium=referral)拍摄的照片。*

*最后，参与代码评审。所有值得尊敬的软件公司都有一个代码审查过程，以帮助保持他们代码库的高质量。代码审查通常被认为是一种把关的实践——你想要确保使用好的设计模式，确保[代码是干净的](https://www.amazon.com/Clean-Code-Handbook-Software-Craftsmanship/dp/0132350882)，确保它被充分测试，以及避免潜在的负面后果。*

*但是也许同样重要的是，代码评审也是一个知识共享的过程。当您创建一个新的合并请求并要求其他人评审您的代码时，您是在邀请他们给您反馈。他们可能会带着各种各样的东西回来:你可以[重构你的代码](https://martinfowler.com/books/refactoring.html)的方法，可以使用的更好的数据结构或设计模式，违反你还没有学到的最佳实践。代码审查是最好的学习机会之一，而且它们已经融入了开发过程！*

*代码审查可能是一种有点情绪化的体验。你在要求人们批评你的创作，所以很容易让你的自我卷入其中。当这种情况发生时，后退一步并记住:即使你的一些代码是糟糕的，这并不意味着你是一个糟糕的开发人员。把你的自我放在一边，记住生产高质量代码和共享知识的最终目标。*

*当准备合并请求时，一定要尊重你的审阅者的时间。他们会抽出时间来帮助您，所以请确保您有好的提交消息、有用的合并请求描述，并且您自己已经审查了代码。作为一个评审者，没有什么比评审没有上下文的代码或者到处看到注释掉的代码和格式错误的代码更令人沮丧的了。关于这个问题，我读过的最好的建议来自迈克尔·林奇的文章[如何让你的代码审查员爱上你](https://mtlynch.io/code-review-love/)。*

*另一方面，不要害怕审查其他工程师的代码——甚至是来自更高级工程师的代码。人无完人，高级工程师也会犯错。通过审查更有经验的工程师的代码，您将能够看到他们如何编写代码、构造代码、命名变量以及处理困难的问题。通过观察并模仿你的前辈的编码风格，你可以快速提高你自己的代码质量。*

*谷歌为代码评审者和代码被评审的 T2 代码作者提供了很好的指导。我强烈推荐两本都读。*

# *结论*

*如果你从这篇文章中只学到一样东西，我希望你记得在学习中要有目的性。找出你需要知道的和你想知道的，并专注于那些事情。在实践中要深思熟虑。好奇。渴望学习。采取这种心态会让你走上一条通往长期成功职业生涯的正确道路。*

*祝你好运！*
