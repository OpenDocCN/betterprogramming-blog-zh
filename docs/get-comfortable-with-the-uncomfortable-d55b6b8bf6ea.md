# 对不舒服的事感到舒服

> 原文：<https://betterprogramming.pub/get-comfortable-with-the-uncomfortable-d55b6b8bf6ea>

## 作为程序员，我们需要停止抱怨，开始解决问题。即使这些事情“不是我们的工作”

![](img/4038fad9a2dbe9b8333b4ab6d9eb1fd0.png)

马库斯·斯皮斯克在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的照片

抱怨很舒服。

我们大多数人在出生的那一刻就明白了这一点。当我们还是婴儿的时候，我们很快就发现，大声哭闹*，可以得到我们想要的东西。在以后的生活中，当我们想吃冰淇淋，想多看一会儿电视，或者想晚睡一会儿时，我们也运用了同样的经验。*

*即使作为初出茅庐的软件工程师，我们也会因为抱怨而得到回报。指出一个组织的缺陷——其糟糕的编程实践，其非结构化的代码库，等等——最初可能会得到我们积极的关注。*

*但是，面对现实吧。虽然抱怨很容易，但提出问题只会让你走得更远。在某些时候，需要有人来解决这些问题。*

*过不了多久，你会发现你的抱怨正在变成一种负担。你因为爱抱怨而名声大噪。是时候停止抱怨问题，开始做些什么了。*

# *我们可以解决哪些问题？*

*如果你像我们大多数人一样，你会选择舒适的东西。毕竟你是写代码的，所以为什么不找一些代码来修复呢？您将花一个下午的时间重构一个类，使它更易于测试。*

*您将花一个晚上的时间将命令设计模式应用于一些数据访问对象。见鬼，也许你会花几天时间编写一个小框架来支持你新的命令-设计-模式-数据访问代码。*

*你会炫耀你的工作，一些人会告诉你你做得多好。你会感觉很棒，因为你向每个人展示了你愿意尽一切努力帮助公司成功。*

*除了，你还没有真正做到这一切。你所展示的是你可以为自己想出有趣的编码项目。*

*用不了多久，人们就会开始厌倦看你花时间做的最新的宠物项目，而他们已经在埋头苦干那些无趣的任务了。*

*那么，一个真正想解决问题并证明自己价值的有事业心的工程师该怎么做呢？*

# *寻找需要的东西，而不是迷人的东西*

*答案很简单:找点难的。每个人都知道需要完成的事情，但没有人想做。一些超出你舒适区的事情。甚至可能到了你不知道如何完成的地步。*

*几年前，我加入了一家公司，一家 Java 商店，它在一个遗留版本的 Java(确切地说是 7)上运行应用程序。因为那时我已经在 Java 8 上开发了大约一年，所以我有点失望。*

*因此，很自然地，我对我的新同事说了些溜须拍马的话，得到的回报是一些会心的抱怨。接下来的几个月里，这种情况或类似的情况时有发生。*

*有人会在工程会议上问我们什么时候会最终迁移到 Java 8。答案总是一样的:*必须有人站出来让它发生*。*

*终于有一天，我决定我要成为那个人。我知道任务会很艰巨。当然，我的工作量很大，所以这项工作超出了我的工作范围。*

*这有很大的风险。即使进展顺利，这种努力也必然会导致周期性的构建问题，并且每个人都知道该责怪谁。在最坏的情况下，我可能会引入潜在的运行时问题，这些问题会在生产中暴露出来。*

*另外，我完全不知道该怎么做。在我自己的 Mac 上安装 Java 8 是一回事。但是让整个组织——它的各种开发、QA 和生产环境；单片应用和微服务；自主开发的库和框架——全部升级了？*

*我的意思是，我是一个软件架构师，不是系统管理员、政治家，也不是巫师！*

*但另一方面，我会为组织提供巨大的服务，使其现代化，并让许多工程师感到高兴。此外，我想使用 Java 8 的流和 lambdas，该死的！*

*因此，我宣布我将领导这项工作。这的确是一项艰巨的任务。我研究了做同样事情的公司经常遇到的问题。*

*我创建了一个依赖项的详细列表，它决定了应用程序升级的顺序。我招募了 DevOps 和 QA 团队的成员，以及来自单个工程 scrum 团队的成员，作为这项工作的一部分。*

*事实上，随着项目的进展，不止一个工程团队遇到了令人烦恼的构建问题。*

# *日益增长的影响力和信心*

*但最令我惊讶的是，每个人都全力以赴。尽管付出了努力，尽管遇到了道路颠簸，但每一个参与的人都愿意投入并帮助度过难关。*

*我注意到我为记录常见问题和解决方案而建立的 Wiki 页面开始随着其他团队的输入而繁荣起来。几个月后，公司的每个系统都运行在 Java 8 上。*

*几年后，我仍然会偶尔收到公司工程师的感谢信。尽管作为我在公司“正式”工作的一部分，我承担了更复杂的任务，但我总是作为将公司过渡到 Java 8 的人而引人注目。*

*此外，成功升级给了我处理其他复杂任务的信心。*

*后来，在同一家公司，我发现自己——以及一些工程师和工程经理——在批评工程组织采用的各种招聘和面试政策。*

*答案回来了:“那么，需要有人想出更好的解决方案来引进优秀的工程候选人。”我停顿了一下，然后同意:如果我在抱怨，那么我应该愿意找到解决办法。*

*同样，我对手头的主题没有任何特别的爱好或技能。我当然不是招聘人员。但这是一个很多人都认识到的问题。*

*因此，我收集了一系列议程，并召集了小团队来解决每一个问题。慢慢地，我们开始改进我们的招聘和雇佣流程。*

*从那时起，我只是想当然地认为，如果我在组织中发现了一个合理的痛点，作为该组织的一员，我的工作就是帮助解决它——不管它是否与编写软件有关。*

*当然，随着我花在编写代码上的时间越来越少，花在管理和领导角色上的时间越来越多，我发现自己每天都要处理各种非工程问题。*

*但是，即使你打算以写代码为生，直到你死去(或者更乐观地说，退休)，接受非技术性的挑战总是一个好主意。*

*你会提升你自己的自信和你在组织中的地位。你将和不同的人一起工作，学习一些新的东西。*

*你会发现自己抱怨的事情更少了。*
