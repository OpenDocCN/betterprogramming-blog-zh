## 反思、合作和谦逊

今天正好是我开始在一家中型软件公司实习的一周。这不是我第一次在职业上有所发展；然而，我以前的经历只是在学术、开放源码软件和自由职业环境中。接下来是对是什么让我的第一周成功的反思。

# 做笔记

记笔记的想法会让你想起讲座、会议和研究:第一天的工作不在场景列表中。然而，在第一天，人们会被来自迎新会、参观和公司历史的重要信息轰炸。它实际上是上述所有内容的结合，但有一点需要注意的是，它发生在一个外国环境中。因此，人们很容易因信息超载而陷入困境，有用的信息可能无法被吸收。因此，在第一天做笔记是至关重要的，这样新信息的财富就不会与周围铺天盖地的兴奋争夺大脑空间。

除了第一天，软件开发——以及任何与此相关的职业——包括计划、学习和创造力。关于规划，关键任务由项目经理安排，并在数字应用程序中跟踪。但是日常任务通常是自我管理的，习惯性地花两到五分钟写下你下一次工作的目标是值得的。我建议在到达后或前一天离开前立即为当天做计划。当学习新技术或从事设计时，蓝图、草图和草图是很常见的。但是，我们必须更进一步，挖掘这些活动的全部价值。个人成长的长期实现从这些活动的*反映*开始。当我在散步或通勤途中独处时，我的思维会自然而然地转向思考。然而，有些人更喜欢预定的“签到”无论哪种方式，重要的是要抓住这些反思的要点。

对于记笔记，我推荐一个基本的直线 [Moleskine](https://us.moleskine.com/en/) 或任何非常接近直线的东西。然而，如果你更喜欢使用高度便携的笔记本电脑(尤其是如果公司给了你一台的话)，打字会更快。无论使用哪种笔记记录设备，记录笔记的机制都是相同的:

*   设备应该是便携式的。(我经常带着 Moleskine 去散步——即使是在周末。)
*   鼓励在自发的思考、顿悟和有趣的想法期间或之后立即写一篇短文。
*   从你的笔记中提炼出要点。我经常发现使用带有要点的问答对话很有帮助。
*   **总是** *标题*和*日期*你的条目，以便捕捉主题和成长时间表。

# 提问

这一点怎么强调都不为过:提问是正在进行的*对话*的关键部分，也就是所谓的“入职”许多人担心提问会导致负面结果，比如惹恼同事或显得无能。此外，我们中的许多人都曾在充满敌意的同事环境中工作过。如果你因提问而被记下(特别是作为一名新员工，来自一个权力之旅的人)，我会实事求是地说:T **他们错了**(在某些情况下，他们只是混蛋)。健康的工作环境需要协作和团队精神，只有每个成员都自信地参与讨论，才能实现这一点。

与同事建立融洽的关系需要时间，初次接触拖延得越久，这种关系就越难维系。我的办公室由从事不同项目的团队共享，包括不同角色的人。我是一个专门项目的全栈开发人员，但我身边的数据库管理员主要监管一组不同的项目。尽管如此，我还是努力向他们做了自我介绍，并问他们是否能给我简单介绍一下他们的工作。因此，现在每当我们擦肩而过时，我们都会随意交谈。如果我的任何任务依赖于他们的指导或者可以从他们的指导中受益，我也有信心和他们一起完成。

你应该立即解决的一系列关键问题是关于你的工作环境的设置。通过问正确的问题，我不仅升级了我的笔记本电脑，而且我的同事发现设置开发环境的`README`已经过时了。相反，如果我卷起袖子，埋头工作一个星期，仅仅是试图让项目在我的本地机器上运行，当我透露我没有完成任何事情时，我会显得相当无能。相反，我设法在第一天就提交了一份 PR！问了这么多等同于无能的问题。

# 接受评论和建设性的批评

我上班的第二个 PR 是~300 行，花了三天才合并。之所以花了这么长时间，是因为它被审查了五次。那是整整五次我在彻底检查后自信地推送代码，结果却收到了许多评论、建议和亮点。当然，这些评论中的一些是对需求的修改(这是一个敏捷商店)——很公平，逻辑也相应地更新了。但是很多时候，我的评审者留下的评论是我改进代码的方法。这种建设性批评和协作算法设计的经历是我迄今为止职业生涯中最有收获的经历之一。

对于那些对自己的代码过于自豪的人来说，记住软件是协作构建的是很重要的。此外，每个开发人员都带来了独特的见解，每个人都可以从中受益。在软件工程中，我们使用定义良好的工具，并尽最大努力遵守标准和协议。尽管做出了这些努力，但总会有一些临时的解决方案和启发式方法。当面临评审时，代码本身(而不是你)和软件作为一个整体，将根据所有这些以及你可能不知道的内部标准进行评估。因此，你仍然可以感到自豪——但是这种自豪是因为你作为团队的一员，以及你共同开发的高质量软件。

最后，代码是开发人员的通用语言。需求、模拟和设计讨论有明显的用途，但并不总是容易转化为代码。一旦编写了代码，团队就可以缩小并理解彼此的观点和挑战。由于这个原因，大多数开发周期都是非常迭代的；通常，在首次尝试某些逻辑之前，需求是不完整的。然后，它们被提炼。然后，评审就成了这个过程的一个自然部分。事实上，在文章“GitHub Flow”中，[斯科特·沙孔](http://scottchacon.com/about.html) [1]描述了 PRs 的这种迭代性质:“在我们真正想要合并代码之前，我们使用【拉请求】来审查代码。”因此，回顾是这个循环的一个重要方面，也是一件值得公开接受的事情。最终，它是用特定领域的语言进行的对话。

开始一份新工作同样令人兴奋和具有挑战性——有许多新的人和内部事实要处理。在加入软件团队的背景下，我相信自我反省、协作和向同事学习不仅是开始正确方向的基础，也是长期保持正确方向的基础。毕竟，我们不仅应该努力成为他人的终身学习者，也应该成为自己的终身学习者。
