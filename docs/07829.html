无服务器是惊人的，但这里有一个大问题

原文：https://betterprogramming.pub/serverless-is-amazing-but-heres-the-big-problem-9e76b65f23c6?source=collection_archive---------3-----------------------#2021-02-24

尽管有这么多好处，它的采用率却呈现出一幅完全不同的画面

首先是多克。然后是微服务。然后是Kubernetes。现在，没有服务器。服务器让人头疼，对吧？有哪个web开发者想学习Apache Linux来部署SPA？而这还只是部署。维护是另一回事。负载平衡、安全补丁、资源调配不足、资源调配过度，这些问题层出不穷。

无服务器是答案吗？

近年来，无服务器计算越来越受欢迎。但是它实际上解决问题的效果如何呢？首先，我们来定义一下。用我自己的话来说，无服务器计算将服务器的功能抽象化，这样开发者就不必担心他们的操作了。您可以使用服务器，而无需手动使用服务器。从技术上讲，它不是字面上的“无服务器”你不必担心你正在使用的服务器。您提交您的代码，提供商将确保它在需要时运行，无论数量多少。自动扩展和服务器管理不再是您关心的问题。

表面上看，这听起来很酷。你可以简单的写一些代码，捆绑成Docker镜像，然后通过AWS Lambda或者Google Cloud Run部署成无服务器功能。只需点击几下鼠标。没有混乱的安装。没有Linux命令行。你所要担心的就是编写你的应用程序代码。

但是我的问题是:它不能非常有效地解决原来的问题。请记住，问题是开发人员不想管理服务器来部署他们的代码。无服务器应该可以解决这个问题。理论上，这是可能的。但实际上，很少有开发人员使用无服务器部署。太吓人了。

例如，当我在为我的交易机器人微服务寻找部署选项时，我偶然发现了无服务器宣传列车。当时，我已经在建立Kubernetes，因为我知道这是健壮的、可扩展的应用程序的行业标准。对优步足够好，对我也足够好。出于某种原因，无服务器听起来更令人生畏。

我想，“你说的无服务器是什么意思？没有服务器？这到底是怎么回事？”所以我就不管了。我觉得太复杂了。实际上，无服务器部署可能比Kubernetes集群更容易设置。笑话我了。但我不认为我是唯一有这种反应的人。无服务器对新来者来说是一种威胁，这很讽刺，因为这些人应该是受益最大的。

我的意思是，他们应该在初学者编程课程中教授无服务器。以这种方式部署很容易。只需编写一些代码，从互联网上复制一个通用docker文件，通过网站点击几下就可以部署。这比部署到一些DigitalOcean droplet要容易得多，更不用说有用得多，因为小的无服务器应用程序可以用来轻松扩展任何应用程序的功能。而且开发起来很简单。然而，所有这些好处从未被发现，因为这种东西让人们害怕。

这就是无服务器的问题:太可怕了。也许是围绕它的内容。在行业会议上谈论这个问题的大多数人都是高级DevOps类型的人。这给人一种错觉，以为这是大人物的先进概念，但这不一定是真的。也可能是领养的问题。也许无服务器解决方案只是需要更好地向初学者推销。

尽管如此，无服务器还是很酷的。我会更多地使用它，并关注空间。缓慢但肯定的是，一些云功能将会进入我的生产机器人。这才是真正的乐趣开始的时候。
