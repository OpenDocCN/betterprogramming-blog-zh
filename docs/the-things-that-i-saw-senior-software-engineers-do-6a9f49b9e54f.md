我很幸运地以一名初级开发人员的身份开始工作，一名高级开发人员是我的导师。这意味着我有机会向走过我将要走的路的人学习，他在这段旅程中比我更有经验。

在你的团队中有一个愿意教导和指导你的高级工程师是一种福气。如果你的职业目标是成为一名高级开发人员，这可以节省你大量的时间。

在这篇文章中，我想分享我看到我的导师和其他高级开发人员做的事情，以及我从他们身上学到的经验。

# 他们挑战顾客

在我们开始实施产品之前，必须与客户一起决定产品的特性。高级开发人员通常是技术领导，并参与这些会议。

有时候，客户实际上并不知道他们需要什么。他们可能认为他们脑海中的功能是他们所需要的，但事实并非如此。我见过高级开发人员质疑某个功能的可行性，作为回报，他们会提出另一个能更好地满足客户真正需求的想法。

正如史蒂夫·乔布斯曾经说过的:

> **人们不知道他们想要什么，直到你展示给他们看。—史蒂夫·乔布斯**

这并不一定意味着客户什么都不知道，但是作为一名开发人员，很多时候你知道一个特性是/应该如何构建的，并且你可能能够提出一个更适合的想法。

## **我的外卖:**

*   始终考虑你的工作提供的商业价值。
*   思考所请求功能的可行性——是否有更好的解决方案？
*   不要总是认同客户。不要害怕礼貌地挑战，表达自己的想法。即使他们是错的，但是在一个有良好文化的公司里，每个意见都是重要的。

# 他们永远不会没有工作可做

在我与我的经理的一对一谈话中，他向我解释了这个概念。他说:

> 高级开发人员永远不会没有事情可做，因为他们对组织内的代码库有如此大的影响。他们是创造和加强回购的一部分，所以他们记得积累的技术债务。他们总能找到值得改变和优化的地方。

当然，我的公司没有谷歌那么大，但是对于一个已经在公司工作了一段时间的高级开发人员来说，他们对代码库的影响是巨大的。

一些回购可能是由他们发起的，这使得他们在其他开发者遇到问题时成为“技术联络员”。他们可能记得由于赶一个项目而被推迟的事情，但是当他们有空闲时间时，他们仍然可以增强它们。

只有初级开发人员有时会觉得好像无事可做。因为作为大三学生，我们的首要任务是完成手头的任务，完成这个冲刺(我写完这篇文章后会开一个 PR)。

## **我的外卖:**

*   有空的时候，看看其他团队在做什么。如果可以的话，也许可以帮忙。这将让你了解组织的其他部分，并产生更大的影响。也许你甚至可以认识到管道中可以优化的部分，并开始新的项目。
*   如果你真的找不到任何东西，记录你做过的一个复杂的项目或者使用过的一个库总是有帮助的。

# 他们可以看到其他团队

这与前一点有联系。我经常注意到，我的高级导师知道其他团队正在做什么，并在我们做出影响其他团队的改变时考虑他们。

不仅如此，我们还以适应其他团队发布时间表的方式来区分工作的优先级。

不可避免的是，您将在其他团队使用的共享库中工作，或者您可能在另一个团队拥有的 repo 中工作。通过了解每个团队如何运作，可以减少团队之间的摩擦，并使工程部门的每个人受益。

当然，这在亚马逊或脸书这样有数百个不同团队的公司是不可能的。但是你至少应该在你的团队之外有一些可见性，而不只是埋头于自己的工作，对周围发生的事情视而不见。

通过这样做，如果你想转到另一个团队就很容易了。如果有足够的资源，而你已经在帮助其他团队的工作，为什么你的经理不允许你调动？

## **我的外卖:**

*   和上一点一样。

# 他们倾听，他们解释

我有时会问一些愚蠢的问题。答案往往就在我眼皮底下。每当我陷入困境后，我的导师为我指出解决方案时，我经常想，我自己怎么没有想到这一点。

有时，我认为我正在做一些事情，并与他们讨论如何以某种方式实现一个特性。我想我找到了一个为我们节省一些 AWS 成本的方法，或者这是一个展示我所知道的一点算法知识的好方法。

他们总是倾听，然后冷静地反驳我所有的观点，证明我错了。但是，他们从来没有让我感到失败，没有写大段文字来解释为什么。

如果我真的有所发现，他们也会倾听，让我做我想做的事情。

但不管怎样，高级开发人员**会倾听别人的意见，并乐于解释**。无论是在辅导低年级学生时还是在解释图书馆功能的文档中。

## **我的外卖:**

*   即使不要求你这么做，也要愿意记录和解释事情是如何运作的。
*   交流是关键，如果有机会，可以通过写文档或解释概念的形式来练习。

# 他们不怕看源代码

我们使用了如此多的 JavaScript 包，我见过的最老的 git commit 是八年前的。鉴于库更新的速度，并不是所有的库都想支持八年前写的东西。所以我们有很多很多的错误。

StackOverflow 上有很多 bug，但不幸的是，我们在试图支持一个八年前的 web 应用程序时看到的 bug 通常在网上找不到。不过，这个库仍然存在于 GitHub 上，源代码也在那里。使用它们。

很多时候当我们的构建管道出现 bug 时，在我在 Google 上找不到任何东西(或者可能只是我搜索得不够努力)之后，我的导师在 Slack 中提到我们使用的库有更新，出现了 bug。他已经为他们提出了一个问题。与此同时，我们可以做一些废话…

我敢打赌，开源图书馆就是这样列出开放问题的。这是因为人们实际上进去了，看了源代码，发现了库的一个 bug。而不是像我这样害怕查看 React 源代码并认为开源库总是正确的开发人员。

要知道开源库公开他们的代码是有原因的。在调试和全面理解一项技术时，它们是一个很好的工具。

## **我的外卖:**

*   如果你正在使用的某个库出现了 bug，而谷歌搜索没有返回任何有用的东西？看一看你正在使用的函数的源代码。你也许能找到问题，或者至少能洞察解决方案。
*   在阅读源代码时找到乐趣。**源代码是你所使用的库的唯一真实来源。**

# 最后的想法

通往高级开发人员的道路不仅仅是提高编码水平，还有很多其他的东西。如果我们想有一天成为其中一员，为什么不向高级开发人员本身学习呢？

当与更有经验的人交流时，观察他们的所作所为，并加以分析。你可以学到一些真正有价值的东西，它们可以为你指明一个明确的努力方向。

即使是高级开发人员也会向其他人学习——可能是其他经理或员工工程师。如果你周围都是更有经验的开发人员，你没有理由不进步。
