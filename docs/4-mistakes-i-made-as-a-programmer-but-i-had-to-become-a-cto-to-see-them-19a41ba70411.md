我做了五年多的程序员。这并不令人印象深刻，因为你们中的一些人可能有三倍以上的工作经验，但我喜欢把自己想象成一名高级开发人员。这听起来很严肃，很重要，不是吗？

有一天，我得到了一个提议，在一家医疗技术初创公司担任首席技术官(CTO)。在这个新的工作岗位上工作一段时间后，我可以回头说，我不是高级开发人员。不要误解我——我仍然相信我拥有优秀的编程知识，尤其是 web 开发——但是，如果是这样的话，我为什么说我不是大四学生呢？

因为我现在脑子里有这四样东西。

# 1.用户是傻逼

不，他们不是。

是的，用户以一种意想不到的，通常是怪异的方式使用这款应用。

是的，用户可以问一些看起来很愚蠢的问题。

是的，有时候用户要求看起来毫无意义的功能。

是的，用户对那些似乎不言自明的功能有困难。

用户不是专家。我的医生不要求我知道低密度脂蛋白和高密度脂蛋白的区别。那么，为什么我过去认为用户应该知道他们使用什么样的浏览器？这对我和你来说是显而易见的，但我妈妈认为谷歌和互联网是同义词。她会说她不使用任何浏览器，因为她使用谷歌🙈。

有时为了让用户高兴，我不得不覆盖框架部分来改变它的默认行为。有时我不得不添加对我不想要的浏览器的支持(向 Safari 用户问好😅 ).当我今天谈论它的时候，它是愚蠢的，但是那些日子我真的认为这是客户的错误，因为他们的定制需求，我不得不在我的代码中做一些变通。

# 2.我的代码是一门艺术，必须完美

干净的代码、单元测试、优秀的文档——这些无疑是重要的东西。作为一名程序员，我总是希望使用现代模式编写干净的代码，我经常检查项目中的所有依赖项是否都是最新的，等等。我想成为优秀的程序员。

当我的产品经理要求我放弃单元测试以提高开发速度时，我感到很生气——难道他没有意识到单元测试有多重要吗？我们没有任何其他的自动测试，所以 UTs 是我们使产品稳定和无回归的唯一希望。

这个决定对我来说似乎是目光短浅的。此外，他建议我们应该停止编写文档，将代码转换成不太复杂的架构(我们可以这样做，因为我们是在项目的开始)。

好吧，我同意这会在一段时间内加速开发，但我们在未来会有很多问题。我们会浪费很多时间修复回归 bug，项目长大后新架构会太简单！如果没有好的自述文件，我们将如何向项目介绍新的程序员？

我们花了几个小时讨论这个决定有多糟糕，以及它在未来会让我们付出多少代价。

这个项目几个月后失败了，因为它大大超出了预算。

多年后，我不得不承认一个事实:我们的团队犯了一个巨大的错误。我们想到了未来，忘记了现在。我们完全忽略了环境——小预算和在短时间内创建 MVP 的需要。

编写可以向他人展示并引以为豪的代码是一件很棒的事情。但是成功完成项目就更好了。毕竟编程不是艺术。

# 3.我将在这个项目中使用“X ”,因为我知道它

在我之前的公司，我们用相同的技术栈创建每个项目:Symfony 和 Angular。为什么？Symfony 是最好的后端框架吗？没有。也许棱角分明是创造现代前端的唯一方式？没有。我们总是选择那套技术，因为我们不知道其他人以及这些。这是我们的舒适区，但是为新项目选择众所周知的技术有错吗？看情况。

在很多情况下，你的下一个项目或多或少与之前的项目相似。既然你已经有了行之有效的解决方案，那么花大量时间学习新技术是没有意义的。但有时，这可能是个错误的决定。

我记得在一个项目中，一个运行良好的 WebSocket 是最重要的需求。我们选择什么来创建我们的后端？当然是 Symfony。也许今天用 PHP 创建 WS 更容易，但那是一场噩梦。我们花了很多时间让它工作。我是说很多。我们知道基于 PHP 构建 WS 需要花费多少时间(和金钱),但是我们放弃了使用 Node 的想法。为什么？我真的不知道。在 Node 中，我们构建 API 的速度会快 10 倍，但这不是我们团队的技术。

我真的很高兴我现在团队中的程序员比我更开明。上周，我们决定彻底改变用于构建我们系统的部分技术。我相信这个决定会为我们节省很多时间，即使这意味着从头开始学习一些东西。

# 4.我的产品负责人/经理错了，我会做得更好

当我在做程序员的时候，我和产品经理的关系很…不好。每当他告诉我任务范围的新变化时，我就想:

为什么你不能在我开始工作之前做好你的工作，确定范围？！预先决定该特性应该如何工作很难吗？

这太天真了，但我真的认为这很容易。现在我完全理解计划项目的每个细节是多么具有挑战性。你必须考虑技术和预算的限制(这实际上是一回事)，你必须考虑将使用你的产品的用户，你不能忘记业务和营销需求。有时，一些需求在开始时并不知道；有时，业务环境会发生变化，有时你必须先构建一些东西，以发现它可以做得更好。

另一件事——产品经理也会犯错。事情就这么简单。程序员制造 bug，pm 也是。现在想起来是那么的明显。如果我以前意识到这一点，我会成为一名更好的程序员。我应该专注于寻找解决方案，而不是试图证明他们有多错。

这很讽刺，也很悲哀，但在某个时候，我忘记了我和经理们有着相同的目标——做出伟大的产品。他们只是在预算、商业环境、客户需求、期限和优先级方面比我有更多的知识。这就是为什么我不理解他们的一些决定。

# 摘要

对你们中的一些人来说，这四件事可能是显而易见的。如果你在一个伟大的、有组织的、有好领导的敏捷团队中工作，并且牢记 UX 的基本规则——我真的很高兴。我认为你可以成为比我更好的程序员。因为“成为一名优秀的程序员”不仅仅是技术技能。更重要的是要明白你能给公司带来的价值是什么，怎么做。下面是我现在对高级开发者这个术语的理解:

> 高级开发人员并不了解技术的每个方面。这个人将帮助我们公司开发出伟大的产品，即使这需要他们跨越舒适区的边界。解决问题的方法。
