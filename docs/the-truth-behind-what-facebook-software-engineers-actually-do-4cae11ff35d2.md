## 我们并不总是做软件工程！

[莱昂](https://unsplash.com/@myleon?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍照。

我作为软件工程师在脸书工作了四年。

但是我在我的上一个团队中花了将近一年的时间，却没有发布一行代码。我实际上是在做产品经理和技术主管的工作。

我与用户交谈，澄清我们想要解决的问题，定义成功标准，集思广益并区分想法的优先级，形成路线图，与其他团队合作，确定工作范围，管理项目。

我开始真正怀念能够坐下来不受干扰地写一整天代码的日子。毕竟那是我进入这个行业的原因。

我也厌倦了在公司的成熟部门工作，那里的问题更多是组织性的，而不是技术性的。

我的工作不再有趣了。

所以我换了一个团队，现在我的工作完全不同了。

这是我喜欢在脸书工作的原因。软件工程团队在这里实际做的工作有很大的不同。

你可以把时间花在编码、运行实验、分析数据、与用户交谈、计划和管理项目、面试、招聘、指导以及这之间的任何事情上。

确切的平衡取决于你的团队和你在团队中的*角色*。最棒的是你可以两者都选！

# 产品

如果你在脸书的一个产品团队中，你会编写和发布大量的代码。但是你通常不需要解决很难的技术问题。所以实际代码可能没那么有趣。你只是想尽可能快地把一个产品推出去，这样你就能看到人们是如何使用它的，然后不断重复。

真正的问题不是弄清楚*如何*制造产品。它在计算*要制造什么*产品。按照脸书的标准，这意味着要进行大量的实验，看看哪些可行，哪些不可行。

大多数新产品都失败了。对于每一个 Facebook Marketplace，都有许多其他产品没有得到关注，被砍掉了。但是在高层，你会因为成功的*结果* — 而不是努力而获得奖励。所以做产品工程师可能是一个高风险、高回报的游戏。

# 基础设施

脸书的基础设施团队就像产品团队，但你不是为数十亿消费者构建，而是为其他工程团队构建。

由于您的“产品”被其他开发人员使用，构建它通常涉及更多的技术问题，比如找出合适的 API 和抽象来提供给他们。你可以直接与你的同事交流，找出哪些有效，哪些无效，而不是在大众身上做实验。

与产品工程师相比，获得成功的*成果*可能更容易。通常，基础设施问题得到了更好的定义，您可以更理性地推导出它们的解决方案。作为一名开发人员，你可以更好地理解你的客户。因为他们是你的同事，你可以从他们那里得到即时的口头反馈。一旦他们建立在你的 API 之上，他们就很难迁移，所以基础设施往往比消费产品更加 T2 化。这些差异也解释了为什么许多创始人更喜欢创办 B2B 公司，而不是消费类公司。

# 等级

最后还有排名队伍。这些通常是产品团队，但核心产品使用机器学习。所以你在这里更像是一个数据科学家，而不是软件工程师。您构建培训数据管道、监控数据质量、培训和评估模型，以及运行实验。

产品和基础设施工程师倾向于通过编写大量新代码来改进他们的产品。但是排名工程师通过改进他们的模型参数、训练数据和损失函数来改进他们的产品。所以如果你喜欢写很多代码，你可能不喜欢排名。

然而，与常规的产品团队相比，它更容易获得成功的结果。排名团队通常只是拿一个已经*成功*的现有产品或功能，用机器学习来改进它。因此，构建用户根本不想要的东西的风险更低。在脸书的规模上，有大量成功的产品可以通过机器学习来改进，也有大量的*数据*来训练高质量的模型。

在脸书的四年里，我与所有三种类型的团队合作过。我目前的团队正在为脸书商店构建前端框架。这是产品*和*基础设施的结合，我很喜欢它，因为我得到了两个世界的最好的东西。

我可以解决有趣的工程问题，为我的同事构建，并编写大量代码。但与此同时，我离产品很近，所以我可以思考消费者的问题，进行大规模的实验，并参与一个大的 0 比 1 产品赌注。

但是还有另一个关键因素让我现在的团队如此伟大:它是公司相对较新的部门。

# 成熟

当你的团队处于公司的成熟部分时，你往往会遇到更多的组织、技术和兴奋方面的摩擦。所以构建你的产品会成为一种拖累。

## 组织摩擦

不要把大部分时间花在编码上，你可以把大部分时间花在会议上，与无数的利益相关者“结盟”，他们对你的产品应该如何工作以及如何与他们的产品集成有意见。不仅仅是工程和设计团队，还有隐私、政策和法律团队。

## 技术摩擦

当你最终坐下来构建你的产品时，你可以花更多的时间去理解现有的代码库，而不是实际编写代码。当其他产品团队已经蜂拥而至，留下一堆边缘案例和丑陋的黑客时，这就不好玩了。

当你最终开始编写代码时，你必须确保它与每一个相邻产品的每一种可能状态兼容。这种复杂性不仅减缓了产品开发。这使得产品一旦推出就更容易出问题。所以你也必须在维护上花更多的时间。

## 兴奋摩擦

在公司成熟的部分，你会觉得你只是在优化已经存在的东西，而不是从根本上创造新的东西。许多想法已经被尝试过了，所以你没有多少发挥想象力的空间了。如果你像我一样，这会让你的工作不那么令人兴奋。

# 技术领导

还有最后一个因素让我现在的角色与上一个角色非常不同:我不再是团队的“技术领导”。

作为一名技术主管，你通常不会花太多时间去编码。相反，你把大部分时间花在计划别人的工作和审查他们的代码上。

你负责团队的整体技术方向和执行。这意味着你不能只做你觉得最有趣的项目。事实上，不愉快的技术工作往往会落到你身上。

你是团队的最后一道防线，所以当事情出错时，你就要挺身而出解决问题。当人们陷入困境时，你需要解除他们的障碍——无论是直接帮助他们完成任务，还是将他们与合适的人联系起来。你工作的一部分也是发现盲点，填补别人遗漏的空白。您需要主动识别风险并降低风险。

在团队内部和伙伴团队之间协调人员的组织工作也会落到你的头上。这可能会占用你很多时间，尤其是在公司成熟的部分。

然而，尽管这份工作可能没那么技术性，你可以通过扮演*到*其他人来产生更大的影响。因为你负责设定整个*团队*的方向，所以你的决定受到高度*影响*。你的行动可能是团队成败的关键。这可能是非常有益的，尤其是如果你真的关心团队的使命。

当许多工程师想要提升自己的职业生涯时，他们会被更多的组织工作所吸引。我知道我是。但是在保持深度技术的同时有可能进步。

# 包扎

今天，我的工作与几个月前大不相同。

之前，我在公司的成熟部门领导一个排名团队。没有太多机会写代码，因为技术工作更多的是数据科学而不是软件工程。

我的角色是设定方向，帮助计划和审查其他人的工作，并协调团队内部和跨合作伙伴团队的人员。这意味着编写代码的机会更少了。

我在脸书广告公司——公司的成熟部门——遇到了许多组织和技术上的摩擦。这意味着*甚至更少*编写代码的机会。

我的工作不再有趣。

现在，我在公司一个新部门的产品基础设施团队工作。我花了大量不受干扰的时间为困难而有趣的技术问题编写解决方案。我不是技术负责人，所以我在工作上有更多的自由。但是我的工作仍然是高度杠杆化的，因为我是为其他工程师做的。我不必处理组织或技术上的摩擦。我对团队的使命以及我们支持的产品感到非常兴奋。

作为一名软件工程师，你可以选择做的工作有很大的变化——尤其是在大型科技公司。提升你的职业生涯并不只有一条路。因此，找出你觉得有趣、令人兴奋和充满活力的事情，然后投入其中是值得的。对我来说，这意味着精疲力尽和玩得比我想象的更开心之间的区别！
