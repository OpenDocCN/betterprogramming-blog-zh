了解如何做出您的第一份开源贡献

为开源做贡献是提高编程技能、帮助他人以及学习如何在协作环境中工作的好方法。

然而，许多人对项目的复杂性望而却步。他们还认为为开源做贡献就是编码。

但这不是真的！您可以通过其他方式为开源项目做出贡献，例如:

  • 编写文档
  • 创建支持材料
  • 审查代码
  • 翻译成其他语言
  • 构建和重新构建代码和项目结构

因此，在本文中，您将学习如何在Hashnode文档的帮助下做出您的第一份开源贡献。 Hashnode 是一个开源文档的博客平台，这是开始开源之旅的好地方。

关于开源软件和贡献的好处

在进一步深入之前，我们先明确一下开源是什么意思，作为一个开发者为开源做贡献有什么好处。

简而言之，开源软件是任何人都可以获得其源代码的软件。因此，人们可以看到软件的实现，甚至可以为之贡献自己的力量。一个例子是代码编辑器Visual Studio代码，你可以在这里看到它的库。

另一方面，Github是一个闭源软件的例子。这意味着只有特定的人可以访问源代码。通常具体的人是创始组织/人和团队。

现在你知道了什么是开源，让我们来看看为开源软件做贡献的好处。最显著的优势如下:

  • 知道有很多人使用你的代码，为流行软件做贡献会带来巨大的满足感。
  • 你加快了编程技能的进步。为人们使用的真实世界的应用程序编写代码与构建辅助项目非常不同。
  • 通过为开源做贡献，你学会了如何在团队和协作环境中工作。
  • 可以带来工作机会；公司可能会注意到你的贡献并给你一份工作。
  • 通过贡献，你可以学习新的技能。例如，如果你学习了一门新的编程语言，你可以用这门语言为一个项目做贡献。
  • 你发展非技术技能，如沟通、写作、解决问题、创造力。

因此，你可以看到为开源软件做贡献可以带来很多有价值的好处。这是加速您的开发人员职业生涯的好方法。

投稿前

通常，开源项目有一个行为准则和贡献指南。行为准则是一份设定贡献者行为期望的文件。其目的是创造一个健康、积极和有建设性的社区。

此外，投稿指南描述了您应该如何创建拉式请求、未决问题以及投稿所需的工作流程。

行为准则和捐款指南都很重要，所以强烈建议您在捐款前查看它们。

如何投稿

要对开源做出贡献，您应该熟悉基本的Git工作流。对于本教程，您将使用以下工作流:

  • 派生存储库——在本例中，是Hashnode文档
  • 通过克隆将存储库下载到您的机器上
  • 为您的工作创建一个单独的分支
  • 进行必要的更改
  • 提交更改
  • 推动变革
  • 打开拉取请求

这是最基本的Git工作流，在大多数情况下已经足够了。我成功地用它为复杂的代码库做出了贡献。

接下来就是找项目，开始投稿了！因此，我们就这么做吧。

找到一个项目

第一步是找到一个开源项目。找到一个可以贡献的项目可能很困难，但是有很多方法可以选择一个项目。

你可以选择一个你日常使用的软件或者一个激起你兴趣的项目。或者，有一些网站为初学者策划合适的项目/问题。

然而，如前所述，您将使用 Hashnode文档来做出您的第一份贡献。

放弃这个项目

要派生这个项目，您需要去Hashnode文档库，您可以在这里访问。

一旦进入存储库，您应该会看到一个“Fork”按钮。单击按钮，在您的帐户中创建存储库的副本。

请参考图1。

分叉需要一段时间，但是一旦完成，您应该会在您的帐户中看到存储库。现在，您可能会问“为什么派生项目而不直接在机器上克隆它？”。

派生项目会在您的帐户上创建一个独立的项目副本。这意味着您可以在不影响原始存储库的情况下试验分叉项目。此外，当您派生一个项目时，您可以通过创建拉请求来为原始项目做贡献。因此，在这种情况下，您可以派生Hashnode存储库，进行更改，然后创建pull请求来修改原始存储库。

另一方面，当您克隆一个项目时，您创建了一个远程存储库的本地副本。更简单地说，您可以在自己的机器上从GitHub“下载”存储库。此外，如果项目由除您之外的任何人所有，除非您被添加为协作者，否则您不能参与该项目。因此，当您是存储库的所有者，或者当您想要检查存储库，但是您不希望做出贡献时，克隆是可行的。

也就是说，您应该在您的帐户中看到分叉的存储库。请参见下面的图2以供参考。

下一步是在您的机器上克隆存储库。

克隆项目

因为您分叉了存储库，现在您拥有了副本，所以您可以在您的机器上克隆它并进行更改。

要获得您的存储库链接，请转到分叉的存储库，并单击绿色按钮“代码”。单击按钮后，您应该会获得存储库URL。请参考图3。

复制URL，打开您的终端并运行以下命令:

git clone https://github.com/catalinstech/support.git

注意:用你的网址替换上面的网址。如果您复制我的存储库URL，它将无法正常工作。

等待存储库下载到您的机器上，一旦完成，在您最喜欢的代码编辑器中打开它。然后，是时候做一些改变了！

创建分支

当您对代码库进行更改时，通常会在单独的分支中进行。

每个分支都独立于其他分支，这意味着一个分支中的变化对其他分支是不可见的。因此，分支允许人们在项目中工作而不会互相冲突。

此外，每个项目都使用分支命名约定，这指定了您应该如何命名您的分支。正确命名您的分支非常重要，因为它加快了代码审查和协作的速度。

分支命名约定的一些示例如下:

  • username/branchName-issueID -例如，catalinpit/add-log in-1234就是一个例子
  • issueID-issue -一个例子是7777-修复-破坏-认证

issueID代表Github发行的编号/ID。每个存储库都有一个专门的问题页面，每个问题都有一个唯一的ID。因此，您可以将分支和问题联系起来，这有助于避免任何混淆。

接下来，您可以创建一个新分支并切换到它，如下所示:

git checkout -b branch_name

或者，您可以创建一个新分支，如下所示:

git branch branch_name
git checkout branch_name

这两个命令有什么区别？您可以将第一个命令视为第二个示例的快捷方式。

创建新分支并切换到它之后，您就可以开始进行更改了。

说到做出改变，这取决于你所参与的项目。在本文中，您将看到一个修复Hashnode文档中一个拼写错误的小例子。

在下面的图4中，你可以看到有一个错别字——“CodeSanbox”应该是“CodeSan d box”。

因此，在添加正确的名称后，您保存文件并提交它。这将带我们进入下一步！

发布您的更改

发布更改的第一步是将它们添加到临时区域。临时区域是一个空间，您可以在其中添加要在下次提交中包含的更改。

命令git add允许我们这样做——它将您的更改添加到暂存区。更简单地说，它指定在下一次提交中包含哪些文件。

如果您在终端中运行git status,它会显示哪些文件没有准备提交。请参考图5。

有两种方法可以将您的更改添加到临时区域。第一种选择如下:

git add .

如果您运行上面的命令，它会将所有更改添加到临时区域。例如，如果您修改了五个文件，它会将所有这些文件添加到临时区域。

然而，有时你可能不想这样做。例如，您可能只想添加特定的更改。您可以按如下方式操作:

git add your_file_name

使用上面的命令，您将特定的文件添加到临时区域，而不是包含所有的更改。在这个例子中，我将运行命令git add faqs.md，因为我在faqs.md文件中做了更改。

将您的更改添加到临时区域后，您就可以提交它们了！

提交变更

下一步是提交更改。但是什么是“git提交”呢？

简单地保存文件对git来说是不够的。因此，命令“git commit”允许我们保存项目更改。根据 Atlassian网站——“git commit命令捕获项目当前阶段变更的快照。”

以下命令是您提交更改的方式:

git commit -m "Fixed CodeSandbox typo"

当您提交您的更改时，您还需要提供描述您的更改的消息。上面命令中的-m标志代表“消息”,它允许你添加你的修改的摘要。

一旦您运行了“commit”命令，您应该会得到您的更改的摘要，如下面的图6所示。

现在您已经准备好推送更改并创建拉取请求了！

推送您的更改

提交的更改目前只在您的本地存储库中可用。这意味着它们只对你可用。

如果您想将它们发送到远程存储库，您需要“推送”它们。您可以使用以下命令将更改推送到远程存储库:

git push --set-upstream origin catalinpit/fix-type-faqs

注意:用你的分行名称替换catalinpit/fix-type-faqs！

运行该命令后，您应该在终端中得到一条消息，您可以从这里创建一个拉请求。请参见下面的图7以供参考。

点击链接，它会带你到GitHub打开拉请求。

打开拉取请求

让我们先来谈谈为什么你首先需要一个拉请求。那么，为什么需要拉取请求呢？

当您在协作环境中工作时，您不应该对代码库进行直接更改。通过在不打开拉请求的情况下进行直接更改，很有可能会引入低效、糟糕的代码、错误和其他问题。

但是，通过创建拉请求，您可以避免这些问题。当您打开“拉”请求时，团队成员和开发人员可以看到您的更改。他们可以检查您的代码，指出问题或提出改进建议。因此，这对每个人来说都是双赢的——您提高了编程技能，避免了不正确的代码进入数据库。

因此，通过打开拉式请求，团队:

  • 保持代码库的高质量标准
  • 避免糟糕、低效的代码、错误和其他问题
  • 可以获得有价值的反馈，从而提高他们的技能

现在您已经知道了拉请求的重要性，让我们创建一个吧！

创建拉动式请求

当您从终端点击链接时(如图7所示)，您将被带到Github打开pull请求。

在打开拉取请求之前，请阅读贡献指南，并确保您遵守这些指南。大多数项目都有一个模板，您可以将它用于拉取请求。

在图8中，您可以看到我对Hashnode文档所做的更改的pull请求。

添加描述性名称和描述后，单击绿色按钮“创建拉式请求”来创建它。

现在，您已经完成了，您需要等待项目维护人员审查您的代码。如果一切正常，评审人员会将您的更改合并到代码库中。如果需要改进，您必须在合并代码之前实现它们。

代码评审的一个小技巧——不要对你的代码产生依赖，不要把反馈当成个人意见。其他开发者只想帮你！

结论

在本文中，您了解了进行开源贡献所需的基本Git工作流。这不是唯一的方法，还有其他的方法。但是，工作流足以开始贡献。

此外，请记住，为开源做贡献不仅仅是写代码。你也可以通过其他方式为开源做出贡献。

你准备好做出贡献了吗？开始打开您对文档的第一个拉动请求！
