你好。这是一系列文章中的第一篇，在这些文章中，我将用简单易懂的术语介绍计算机科学概念。从我个人的经验来看，尽管有大量的资源，作为初学者学习新的编程概念可能是困难的。这是因为许多资料来源经常用过于专业的术语来解释它们。

因此，我在这里的目标是尽我所能将事情分解成类比/日常例子，以便为那些可能正在努力理解这些核心概念的初学者提供服务。

让我们进入第一个话题:大 O 符号。

考虑我的一个例子:你在一个大型会议上做志愿者，当每个与会者出现时，你负责给他们姓名标签。你得到了一堆无序的名片。你将如何着手这项任务？

完成这项任务的第一个非常低效的方法是，对于每一个出现的人，你要仔细检查一堆卡片中的每一张，直到找到他们的名片。第二种方法，也许是你刚刚想到的，是根据出席者的姓氏将卡片分类。这可能会占用你桌子上更多的空间，但是你只需要知道与会者的姓氏，这样你就可以更快地取回他们的名片。

在这个例子中，你可以看到有许多不同的方法来解决一个问题。我们考虑的两件事是一个解决方案有多快，以及一个解决方案需要多少空间，以确定它是否是“最佳”选项。在测量一个东西有多“快”时，我们可以精确地*计时*秒或分钟的精确数值，但这可能不是最好的方法。在第一种解决方案中，您要遍历姓名堆中的每一张卡片来查找姓名，您可能会很幸运地立即找到它，而在其他情况下，您可能需要花时间遍历几乎整个堆栈才能找到它。因此，测量*实际时间*来确定解决方案的有效性是不可行的。

相反，我们可以用大 O 符号来解释有效性。简而言之，大 O 回答了这个问题:随着输入规模的增加，一个算法解决方案在时间和空间方面的表现如何？在上面的例子中，如果会议现在多接纳 1000 人会怎么样？你的第一个解决方案与第二个相比如何？提示:那会更糟！

以下是一些常见的*时间复杂度*案例和真实世界的例子，帮助你思考它们。在这些例子中， *n* 代表输入的大小。

*注意:你可以在网上找到很多分析实际代码复杂性的例子，所以我不会在这里赘述。我最喜欢的一课是来自 Coderbyte* *的* [*Alvin。*](https://www.youtube.com/watch?v=HfIH3czXc-8)

# O(1) —常数

*   不管输入大小如何，算法解都需要恒定的时间。
*   例如，不管你预计会有多少客人，你打扫客厅准备迎接客人所花的时间是一样的。

# O(log n) —对数

*   执行求解所需的时间被描述为输入大小的对数。对数可以被认为是指数的反义词:指数是重复的乘法，而对数是重复的除法。
*   例如，您正在物理电话簿中查找某人的姓名和电话号码。你要做的是把书分成两半，看看这个人的名字是在前半部分还是后半部分，从头到尾看一遍，然后重复。每一次，你都是分成两半来得出你的答案。现在，如果电话簿变得更厚一点(输入大小*增加*)，就不会花*太多的*时间来找到这个人的名字和号码。把这比作一种解决方案，你可以从一页翻到另一页，在书中找到这个人的信息。随着书的长度增加，你要花*更长的时间*才能找到它们。这就把我们带到了 *O(n)* 。

# O(n) —线性

*   执行一个解决方案所需的时间与输入大小的成比例增加*。以我们之前的例子为例，这就是“翻阅每一页，直到找到那个人”的方法。*
*   很多生活任务都是用大 O(n)运行时完成的。例如，你叠衣服所花的时间取决于你的衣物量，你读完一本书所花的时间取决于这本书有多少页。

# O(n ) —二次

*   随着输入大小的增加，执行该算法所需的时间以二次方的方式增加。时间是输入大小的平方。
*   例如，假设你和其他几个人一起参加了一个精心设计的面试技巧练习小组。在每个人都练习回答一个面试问题之后，房间里的每个人都要给那个人一个反馈。随着越来越多的人加入面试准备小组，给出反馈所需的总时间将呈指数级增长，因为每个人都需要去，每个人都需要在每个回合后给出反馈。
