# 不能在开发运维上偷工减料的 5 个理由

> 原文：<https://betterprogramming.pub/5-reasons-why-you-cant-afford-to-cut-corners-on-devops-5a205fce3034>

## 想在 DevOps 上省钱？那就花掉它

![](img/a2c9035ad0f505f96c8c3367d1eb4d85.png)

持续集成和持续部署现在非常流行，但是除非你知道如何正确地使用它们，否则它可能会让你暴跳如雷。这种观念当然与 CI 和 CD 的意图不一致。

尽管如此，许多公司已经排队提供闪亮的，新的工具来完成这个过程，他们都承诺，它将注入你的团队前所未见的新水平的生产力。

作为工程师，我们已经看到了太多未实现的童话，这个誓言也不例外。这些工具有什么可以区分转瞬即逝的幻想和可靠的现实吗？我的专业经验表明，有一个关键:一个涵盖所有提供者和方法的概念。

它是所有 CI/CD 服务的核心，是区分好服务和优秀服务的关键。这是对你的团队和公司的投资，你不能错过。

# 中央钥匙

关键不在于提供者。我不打算向你展示为什么 GitHub Actions 比 Jenkins 或 pit Azure Pipelines 更适合 GitLab 持续集成。

无论您使用哪个提供者，或者您的构建脚本有多聪明，它们的性能总是受到您的 CI 运行程序的能力的限制。如果你在一台高性能的机器上有一个可靠的运行者，可以可靠地运行相同的步骤，得到相同的结果，你的工作将会变得更加容易管理。

然而，这种说法说起来容易做起来难，而且没有单一的答案可以满足这种需求…除非你改变你的策略。

# 将您的跑步者置于云端

也许这听起来太容易或太昂贵。毕竟，大多数主要的 CI 提供商都提供了一种在本地硬件上安装 runner 应用程序的方法，为您提供充足的使用时间，并且不收取月费。这是您团队最重要的运营之一的成本削减措施。如果你期望降低成本的努力有所回报，那么你不妨试着找一个有能力的、愿意免费工作的工程师。

你可能会质疑这个解决方案是这里对那里的问题。这会有什么不同呢？很多。以下是五个原因。

## 1.没有维护

工程师不应该对与 IT 相关的问题负责。

*   你的线人下线了吗？派一名工程师到办公室调查原因。
*   你忘了能量调节吗？最后一场雷雨刚刚切断了你办公室的电源，你所有的构建都被卡住了，直到你的跑步者重新上线。你没忘记开机并在电源恢复时自动登录吧？对吗？
*   主要的操作系统或工具链更新呢？工程师必须抽出一天的时间来修补这些转轮(或所有这些转轮)以符合最新的要求。

最后一点与 iOS 项目尤其相关，在 iOS 项目中，目标移动设备通常与构建它的 Xcode 版本紧密相关，而 Xcode 版本又与支持的 macOS 版本相关联。

> “主要的操作系统升级总是无缝的。”——真的没有人曾经

如果您在云中，所有主要的供应商都支持构建模板，这些模板为您的运行者提供了一个基础操作系统，并为各种项目和语言提供了许多必要的工具链。云运行者由一群解决上述问题的工程师维护，他们的配置可能会更加圆滑，因为他们有数千个类似的项目要支持。

## 2.它更便宜

云运行者的月成本需要很长时间才能超过新硬件的前期成本。

*   除非你的团队或公司有专门用于此目的的优秀硬件，否则你可能不得不提交一份数千美元设备的费用报告。任何值得买的硬件都不会便宜。
*   为此，您需要几台机器，因为即使是一台不错的机器实际上也不可能运行多个并行构建。
*   不要忘记外围设备，如显示器、KVM 交换机，可能还有备用电池，这样您的团队就不会因为小停电而离线。

您的公司有旧硬件吗？它可以为此目的借给您的团队吗？为延长构建时间做好准备，因为遗留机器会在构建的高要求任务中搅动。这种廉价的解决方案将在一段时间内由工程师的工资支付，这些工程师需要等待更长的时间来完成每一次建造或完成堵塞的管道。工程师最好把宝贵的时间花在构建软件上。

如果你期望一个降低成本的措施能有回报，那么你不妨试着找一个有能力的、愿意免费工作的工程师。

相反，请你的经理在 CI/CD 提供商的云服务器上购买时间段，并观察支付给专业人员的红利会给你的团队和公司带来什么。

## 3.它更稳定

稳定性无疑是减少 CI/CD 过程引起的摩擦的关键。

*   即使是最好的机器也需要不时重启。每当事情开始变慢或变得没有反应时，谁来管理呢？
*   机器之间设置的微小差异可能会以通常的方式出现，经常导致调试会话延长。
*   如果你选择的硬件是苹果设备，你准备好处理英特尔和 M1 苹果电脑之间的细微差别了吗？

没有必要重启云，因为您不负责它们的硬件性能。至于硬件和软件中的意外差异，除非您更改模板类型，否则不会发生这种情况。

许多公司已经将其面向消费者的产品委托给云的稳定性。对工程团队构建这些应用程序所依赖的工具做同样的事情难道没有意义吗？

## 4.即时满足

亚马逊的当天送达无法与云提供商在一瞬间为您的团队提供的服务相比。当你的团队现在需要一个 CI 形象大使*时，instant 是你能得到的最接近的了。*

*运送硬件需要时间。设置操作系统和必要的工具链可能会花费更多的时间，这通常是以软件工程师的日常工作为代价的。*

*如前所述，大多数云提供商提供开箱即用的模板，可以完美地满足您的操作系统和工具链需求。这涵盖了前两个步骤，留给团队最后一步:为他们独特的项目定制构建。*

## *5.它随你而变*

*您的项目的构建时间增长了吗？是否有更多工程师对现有资源提出要求？购买更多硬件就像从头再来。*

*当然，除非你可以登录提供商的控制面板，增加分配的分钟数或你可以访问的虚拟跑步者的数量。完成了。不需要设置这些，因为它们也有与您的第一台机器相同的模板。*

# *从哪里开始？*

*就像上大学或把钱投入股市一样，有时存钱或挣钱的最好方法就是花对钱。这一原则在 DevOps 世界中是正确的，CI 和 CD 在其中扮演着重要的角色。*

*不管您在哪里托管您的源代码控制库，该领域的一些杰出的参与者已经在几乎所有的软件工程学科中证明了他们的价值，尤其是如果您的团队已经使用了 Git。*

*以下是一些我专门使用过的，并且可以推荐给个人的通用适用性:*

*   *[**GitHub 动作**](https://go.oliverspryn.com/github-actions) **:** 对托管在 GitHub 上的库的一流支持*
*   *[**Azure Pipelines**](https://go.oliverspryn.com/azure-pipelines)**:**与市场上所有主要的 Git 提供商集成良好*
*   *[**GitLab 持续集成**](https://go.oliverspryn.com/gitlab-ci) **:** 允许 GitLab 的用户将他们所有的工具放在一个屋檐下*
*   *[**TeamCity Cloud**](https://go.oliverspryn.com/teamcity)**:**CI/CD 界的中流砥柱，最近带到了云上*

*其他荣誉奖包括:*

*   *[**Bitrise**](https://go.oliverspryn.com/bitrise) **:** 非常适合开发移动应用的团队*
*   *[**Travis CI**](https://go.oliverspryn.com/travis-ci)**:**多年来一直忠实地为 GitHub 社区服务，是最重要的 CI 提供商之一*
*   *[**位桶管道**](https://go.oliverspryn.com/bitbucket-pipelines) **:** 为已经在使用位桶的用户提供卓越的一流支持*

*这些提供商以一定的价格提供云运行者，这些价格在可靠性和免维护性方面是值得的。如果你还没有从你旁边的盒子跳到你上面的云，我劝你考虑一下。你会很高兴你放弃了做 CI 和 CD 的老方法。你的公司也一样。*
