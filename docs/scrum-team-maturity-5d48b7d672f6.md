如果团队对过程没有纪律，或者在技术上没有成熟到实际交付 sprint 的工作，Scrum 过程注定会失败。

这就是为什么 Scrum 可能最适合至少由处于相似成熟度水平的人组成的团队。

Scrum 不是一个项目管理过程，而是一个产品开发框架，因此，最好开始测量每个 sprint 结果的有效性。

在每次冲刺中，有多少最终用户的价值被添加到产品中？

然后，让团队找出他们需要衡量自己什么来提高价值。

我这里所说的*成熟*实际上是什么意思呢？

让我快速解释几个 KPI，它们将向你展示 Scrum 团队是如何进展的。

# 速度

这是一个趋势，一旦团队开始运转，并且没有明显的障碍阻止他们加速，这个趋势在一段时间内似乎太高了。

对于软件开发人员来说，速度是通过团队在每个 sprint 中交付的故事数量来表示的。

这样，故事点将是一个很好的工具，可以帮助团队在分享每个团队成员对故事的理解时估计问题。这样，团队就可以对故事进行分类，并用“复杂”、“bug”、“琐事”、“需求研究(spike)”、“简单”来标记

这种优先级排序不仅允许团队用每个用户故事的精确故事点数来精确地评估时间估计，还允许团队在冲刺阶段更快乐、更专注于最终价值。

# 团队中的关系

这里，我指的是 Scrum master、首席工程师、开发团队和产品负责人之间的关系。

如果很好地理解了关注点的分离，那么带有任务的标签就被清楚地描述和书写了。每个团队成员都知道其他人应该做什么，并在 sprint 结束时为用户带来最终价值。

然后你会得到一个相对成熟的团队，并且会相当成功地执行。

无论是小公司还是大公司，团队建设都有助于发展团队关系。

据[加州大学](https://hr.berkeley.edu/hr-network/central-guide-managing-hr/managing-hr/interaction/team-building/introduction)报道:

> “团队建设是一个持续的过程，有助于一个工作组发展成为一个有凝聚力的单位。团队成员不仅对完成团队任务有着共同的期望，而且相互信任和支持，尊重彼此的个体差异。”

由于这个戏剧性的全球性问题，我们都必须远程工作，这使得离线团队建设活动变得不可能。

# 追溯问题

这就是团队如何使用回顾，以及他们如何通过学习和增量改进来有效地利用它。如果团队关心成长和持续的工作改进，回顾是最重要的 scrum 过程之一。

在整个回顾会议中，团队找出哪些进展顺利，哪些在去年春天出了错，以及哪些在未来可以做得更好。

这些每周/双周会议的主要目标是认识到团队做得不好的地方，并停止这样做或调整，认识到团队做得好的地方，并做得更多。

此外，如果过程的某个部分在团队环境中工作得不好，选择不同的实践、方法等。这向团队交付了相同的价值，但是更适合团队的环境。

回顾的主要结果是分配给整个小组、子小组或个人的行动项目列表。

# 可测量的指标

团队是否有有效且有影响力的度量标准，他们是否意识到为什么他们在做出贡献的同时试图改进度量标准？

理想的情况是 scrum 团队关注客户的产品使用和客户行为。本质上，用户流，例如，通过应用内分析、反馈信息、聊天机器人的客户请求或对产品的评论。

只是回答这个问题:“你的客户快乐吗？”因为使用 scrum 成为敏捷的主要标准是这样的。

另一方面，绩效改进计划有一些通用的指标，如之前提到的速度、燃尽图等。你可以在这里阅读[著名的坏的和好的敏捷指标](https://www.extremeuncertainty.com/agile-metrics-good-bad-ugly/)。

# 团队中的危机

在我看来，是从团队内部互相的微观管理开始的。而且只有在团队内部，成员之间没有信任的时候才会开始。

如果你注意到荒谬的频繁的关于门票状况的会议和无数关于消除障碍的谈话，你很容易就能认识到这一点。对于擅长自己工作的人来说很难。不得不花几个小时甚至几天来证明他们的工作是对他们的侮辱。

因此，了解和承认团队危机的存在并准备好尽快处理它是至关重要的。这将使团队高度融合、成熟和有经验。

# 结论

然而，不幸的是，很多时候，测量集中在产品的速度和效率上，而没有测量有效性。

如果我们没有构建正确的东西(没有效率)，我们将永远无法提高产品开发速度。

否则，产品开发只是像仓鼠轮子里的仓鼠。快速移动，不向前移动。
