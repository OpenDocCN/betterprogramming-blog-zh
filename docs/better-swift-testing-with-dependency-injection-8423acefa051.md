在这篇文章中，我们将看到[依赖注入](https://en.wikipedia.org/wiki/Dependency_injection)如何帮助你预先构建你的应用并解决你的一些测试问题。它甚至会帮助你以后进行重构。

# 依赖注入简介

依赖注入(DI)的主要思想是，它是一种技术，其中某个对象(例如视图控制器)可以通过某种配置方法或属性接收它所依赖的对象(例如视图模型)。它的伟大之处在于，它不依赖于任何特定的软件架构(MVC 或 MVVM)，也不需要任何第三方框架来应用它(即使这样的框架确实存在)。它实际上是一个非常基本的设计模式，但是它给了你很大的灵活性。

通常，有三种方法可以注入依赖项:

1.  通过自定义初始化器——如果您完全控制对象的初始化(自定义类、结构等),这是一个很好的选择。)，但如果您不这样做，则不是一个选项(例如视图控制器)。
2.  通过属性—只需创建一个允许您设置依赖关系的公共可访问属性。很好也很简单，但是如果您不希望其他人可以访问这个依赖项，就不要这样做。
3.  通过配置方法—允许您轻松设置依赖关系，还允许您对外界隐藏依赖关系。如果您想确保其他代码不直接访问您的依赖项，这是一个很好的方法。

每个或多或少是相等的，但是提供稍微不同的(不)优势。我们将使用一些将 web 服务注入视图模型的示例代码来探索 DI。

# 概括起来

依赖注入可能是一个很难理解的概念——不是因为它本身很难，而是因为框架声称可以促进这一点。他们试图提供一种非常僵化、集中和结构化的方法来做到这一点，这常常导致令人困惑的样板文件和一种压倒性的“对抗系统”的感觉。

本文展示了从 DI 开始的一种非常简单(也很常见)的方法，并展示了使用它的好处——如果不是为了测试目的，那么是为了可维护性。DI 并不强迫你为你的应用使用某种架构，也不要求你使用 MVC、MVVM、Clean 或者任何其他流行的架构。它仅仅是一种设计模式，可以帮助您将代码分解成定义良好的片段，每个片段都有自己的职责。
