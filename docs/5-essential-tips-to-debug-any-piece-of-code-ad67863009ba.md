是的，我又带着虫子来了。bug 仍然是你在编码时遇到的最令人沮丧的事情之一。我知道我发现他们在某种程度上令人沮丧。所有的程序员都会同意，没有什么比构建一个你认为可以工作的东西却在你面前崩溃更好的了。

但是，不要误解我的意思:bug 并不完全可怕。当你试着调试任何程序时，你总能学到一些东西。我有过的一些最有价值的经历包括修复一个讨厌的 bug。当你看着它在指尖融化时的那种感觉是无价的。有一句名言是这样说的:

> “如果调试是去除软件 bug 的过程，那么编程一定是把 bug 放进去的过程。——埃德格·w·迪杰斯特拉

这是相当刻薄的，但我同意大多数程序并非没有错误的事实。

很难知道每一个场景，并且在你写代码的时候声明你的代码将会运行。正因为如此，bug 是编码的一个自然副作用。不幸的是，这并没有减少处理这些 bug 的挫败感，尤其是当它们完全破坏了你的程序的时候。

也就是说，调试中有一些技巧和方法可以让这个过程变得更加顺利。在这篇文章中，我将分享我关于如何轻松调试的五个技巧。这些技巧大部分来自大卫·j·阿甘斯的教科书“[调试:找到最难以捉摸的软件和硬件问题的 9 个不可或缺的规则](https://www.amazon.com/Debugging-Indispensable-Software-Hardware-Problems/dp/0814474578)其中一些是我处理 bug 的经验。让我们开始吧。

# 1.持续重现该行为

可靠地复制 bug 是成功的一半。如果你每次都能以同样的方式重现错误，你就有更大的机会识别和修复错误。当您发现一个 bug 时，甚至不要去想可能有问题的代码。只要把代码想象成一个黑盒，找出是什么输入导致了错误。一旦你能复制它，缩短复制它的步骤。想一想让这个 bug 出现的最简单的方法。这有助于在你试图修复错误时减少变量的数量简化，简化，简化。一旦你可以复制它，写下你如何做的步骤，尤其是如果有很多这样的步骤。万一你忘了，你会很高兴你做了。

# 2.各个击破

假设您可以一致地重现 bug，那么您现在必须找出代码的哪一部分导致了错误。这是您必须尝试深入代码的特定区域的地方。一到五行代码是你应该努力达到的准确水平。那么，你从哪里开始呢？根据代码库和你对它的熟练程度，你可能马上就知道了。

如果你不知道，我有一些步骤可以帮助你:

首先，找到任何可以搜索的字符串。例如，UI 中可能有一个字符串，或者错误消息是唯一的。使用该唯一术语搜索整个代码库。你会感觉到应该关注什么样的社区或区域。

第二，问问你的团队成员去哪里找，以防你和其他人在同一个代码库上工作。我保证，他们不会看不起你。相信我，这样做会节省很多时间。

最后，设置断点并打印语句。如果您已经将范围缩小到代码库的一个一般区域，这将特别有帮助。设置断点并将 print 语句放在该区域的不同部分，以获得值和状态的快照，这可以让您更好地了解 bug 可能在哪里。

# 3.观察风景

缩小代码库的一般区域后，检查该区域内的代码。彻底了解代码正在做什么，可以让您做出明智的更改，这样您就不会在尝试修复一个 bug 时导致更多的 bug。因此，请务必仔细阅读代码，并找出您脑海中可能出现的问题的答案。为什么会这样写？特定的代码行何时运行？为什么该变量或函数在实现中是必需？如果有错误信息，也要仔细阅读。质疑你的假设——因为你可能对代码做了一些不真实的假设。

另一个窍门是通过查看各种书面记录来确定时间线。自上次运行以来，发生了哪些变化？看看你的版本控制历史。在这一点上，你就像一个侦探一样在寻找错误。如果你愿意，可以做一个角色扮演。任何能让你理解代码的东西都是值得花时间去做的。逻辑思维是解决这部分问题需要磨练的技能之一。

# 4.一步一步地做出改变

速度和效率是编码人员应该具备的良好素质。然而，在处理 bug 时，你需要彻底。

> “虫子很娇气。”

令人沮丧但仍很微妙。一次改变太多东西不会帮助你找到问题所在。你甚至可以决定只改变一件事，然后运行程序，看看这个 bug 是否以同样的方式运行。更改其他内容，然后重新运行程序。冲洗，起泡，重复。

如果可能，创建一个代码沙箱。看看你是否能通过把那部分代码拉进一些不复杂的东西来复制同样的错误。它可以是一个独立的程序，一个应用程序，一个类，或者任何对你的环境有意义的东西。排除了所有其他变量。这样做有助于你专注于代码的这一部分，并且通过排除所有其他干扰，一次只改变一件事会更容易。

# 5.获得新的视角

你是否曾经被困几个小时甚至几天试图解决同一件事，你不再知道你是饿了还是渴了？当你处于这种状态时，知道如何识别是至关重要的，因为经过一段时间后，毫无结果地敲打同样的东西实际上是有害的。

阿尔伯特·爱因斯坦说:“精神错乱的定义就是一遍又一遍地做同样的事情，却期待不同的结果。”也就是说，休息一段时间，用新的眼光去观察这个 bug。如果你在同一个 bug 上工作了一天以上，那么是时候休息一下，从不同的角度来看待它了。你甚至可以尝试从你的朋友或同事那里获得信息。你的朋友可以向你介绍你以前可能没有考虑过的新观点或新的解决问题的方法。我知道这很难，但是不要害怕寻求帮助。当每个人都能一起解决问题时，团队会变得更好。无论你从这个 bug 中学到了什么，都可以和其他人分享。当然，修复这个错误可能需要更多的时间，但是没问题。如果你是一个内向的人，走橡胶鸭子的路。我一直都这样！大声思考很有帮助，可以理清你的思路。

总之，以上就是我关于如何轻松调试的五个步骤。现在，这些只是我经常使用的；它们并不是调试艺术的全部。还有比这些更多的方式，但我不会在本文中涉及。万一你有更多，请分享。我们互相学习会更好。当我们互相帮助提高技能时，编程社区会更强大。
