解决VS代码、IntelliJ、Nano和Sourcetree中的git冲突

冲突发生了

你修改了一个文件，你的同事也修改了。冲突会发生，很可能会在你的桌子上结束。

你落后于快速通道分支(例如develop)，你以冲突结束。解决这些问题可能需要一天的时间，因此找到一种快速解决问题的方法非常重要。

在本文中，我编译了五个快速方法 来解决git冲突。

1.VS代码

VS代码是迄今为止我最喜欢的冲突解决者。你会在冲突上方看到一个可爱的界面。解决冲突有三种选择:

  1. 接受当前的变化。
  2. 接受传入的更改。
  3. 接受这两个更改。

这些选项不言自明。单击选项将应用它们描述的操作。冲突可以是内联的。对于这些，您可以在VS代码中编辑文件。️️️

2.IntelliJ

如果你点击Merge Conflicts后的Resolve，你会看到这个窗口。您会看到一些类似于VS代码中的选项。接受您的意味着您保留了来自您的分支的变更(在我们的例子中，他们的变更来自于master)。

假设你对此不满意。然后双击有问题的文件:

在这一步，你可以看到到底会发生什么。第一个标签是你的分支，feature-2。中间的选项卡将是合并结果。第三个选项卡是您试图合并的分支(master)。

您可以选择>>添加更改，或点击X放弃更改。

在右下角，您有两个选项:Accept Left将使用来自您的分支的变更，而Accept Right将使用来自master的变更。

3.毫微；纤（10的负九次方）

你们有小冲突，想用 Nano 。用命令nano index.js加载有问题的文件。像在任何文本编辑器中一样，冲突是固定的——只留下你需要的。完成后，执行CTRL+X保存更改。

执行git status打印出all conflicts fixed。提交后，我们合并成功。

4.带合并的源代码树

你会喜欢上源码树。它有一个干净的用户界面，在Git战斗中非常有用。

右键单击有问题的文件会给你一些选项。Resolve Using ‘Mine’会用你的变化作为真理的源泉。Resolve Using ‘Theirs’将使用您正在合并的分支的更改。

完成外部冲突解决后，您可以选择Mark Resolved选项。

5.基于Rebase的源代码树

自定义操作可以为您节省一些时间。要添加新的自定义操作，请转到操作>自定义操作>编辑。您可以添加脚本并通过快捷方式使用它们。我正在使用update_branches.sh来重置到develop上。

冲突解决后，点击左上角的Commit按钮。你需要Continue Rebase，它相当于git rebase --continue。
