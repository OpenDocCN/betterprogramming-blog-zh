随着数据行业随着新技术的发展而发展，数据工程也面临着挑战。新工程师能期待什么？

作为数据工程师，通过收集和分析数据，您在您的领域中扮演着重要的角色。但如今必要的数据工程师技能与过去几年不一样了，这个角色正在经历一些严重的成长烦恼。让我们来关注一下数据工程师面临的一些挑战。

# 数据工程师必须脚踏实地地学习

最大的挑战之一，也是许多其他挑战的根源，是数据工程是一个相对较新的动态学科。虽然它起源于数据库维护和商业智能，但近年来它有了自己的生命。你不会找到很多关于这个主题的大学课程，也不会期望很快找到一个“数据工程训练营”。这意味着工程师在工作中学习大量的最佳实践。

更复杂的是，数据工程领域[偏离了它最初的道路](https://rudderstack.com/blog/the-data-engineering-megatrend-a-brief-history)。过去的工程师更关注创建数据管道和将数据收集到仓库中。现在的工作要复杂得多，数据分析和构建算法的责任也增加了。工程师们处理的数据比过去大得多(但以后会更多。)

数据工程是一个真正的混合角色，诞生于数据爆炸和技术进步。这些进步正在改变行业，而且这种改变仍在继续。我们可以预期数据工程的角色会随着它而不断变化，它最终会走向何方还有待观察。

# 要处理的数据太多

标题有点夸张，但术语“大数据”并不夸张。如今，数据工程师必须处理比以往更多的数据，而且没有停滞不前的迹象。尽管海量数据对该行业来说是一件好事，但数据增长的速度超出了大多数人的预期，这导致了一系列问题。

# 性能差

所有这些信息对最先进的机器都是一种压力。报告和模型慢得像爬行一样，因为它们很难处理通过它们的大量数据。如果您不小心，您的数据需求可能会超出您的机器的能力。

作为一名数据工程师，您的时间非常宝贵。你不能在几份报告上花几个小时。不过，还是有办法解决这个问题的。如果您还没有，那么迁移到云可能是一个可行的选择。云数据仓库有几个好处，比如比传统仓库更具可扩展性和弹性。此外，不在本地部署服务器意味着您可以节省数据库管理的时间和资源。

# 无法获取数据

所有这些数据会让工程师超负荷工作，他们很难足够快地获取数据集。没有帮助的是旧的 ETL 技术，它可能是代码繁重的，并且进一步使您的过程陷入困境。一个可能的解决方案是切换到 ELT 系统——提取、加载和*然后*转换——根据需要处理数据。这可能会与您的数据治理策略相冲突(下面将详细介绍)，但是在开发更大的数据图并指导您为核心模型开发更好的数据集时，这可能会很有用。

# 数据管道维护

随着对更多数据管道的需求以及大数据浪潮的兴起看起来更像一场海啸，最大的数据工程挑战之一是保持现有管道的正常工作。

幸运的是，在代码层面也有一个转变。[命令式编程让位于声明式编程](https://codeburst.io/declarative-vs-imperative-programming-a8a7c93d9ad2)，对低代码甚至无代码系统的日益重视减轻了数据工程师肩上的巨大负担，并减少了维护负担。

虽然其他行业害怕自动化，但在这种情况下，它是数据工程师的朋友。

# 数据治理，或者缺乏数据治理

数据治理并不有趣。它给数据工程增加了一定程度的官僚作风，这是您可能不想要的。但是这种选择会导致关键数据值和定义的不一致。这意味着坏数据在各种集成和报告中浮动的可能性。

考虑一下您的企业中有多少集成系统。如果程序之间的某些字段不同步，那么如果在错误的时间从错误的地方获取报告，可能会导致数据不准确。如果字段没有实时更新，情况尤其如此。

对此的一个解决方案是实施某种数据治理计划。这可能是一本手册中的一页，也可能是一个更大的委员会，这取决于你企业的规模。重要的是你要有一个保持数据输入和输出一致的计划。好消息是，您可能至少已经有了一些数据治理策略。

不幸的是，这给数据工程师带来了新的挑战，还有前面提到的要处理的数据太多的问题。您现在需要在快速获得数据和“足够好”之间取得平衡，并保持数据足够准确以做出合理的业务决策。

# 人的因素

有时，您的数据工程挑战不是来自数据，而是来自其他人。客户或雇主可以有意或无意地设置障碍。有时候我们就是无法摆脱自己的方式。

# 不明确的战略

一个清晰的商业战略将是任何公司的基础。但有些人在行业中发现了一个新玩具，并希望实现它，而没有考虑它会如何影响他们的业务或战略。机器学习可能是企业想要采用的新玩具，但他们是否考虑过如何让它为自己服务？

改变不一定要像一个新的人工智能那样激进。这可能是您想要实现的新集成。但是，如果不考虑如何将这种增加融入到你的商业计划中，你可能会花费比你想的更多的时间来完成这个拼图。

绕过这个障碍的一个方法是每次都把你的商业目标放在第一位。考虑您在数据战略中所处的位置，您想要达到的位置，以及最终您将如何实现。关于这方面的更多指导，这里有另一篇关于[开发数据分析策略](https://www.theseattledataguy.com/developing-a-data-analytics-strategy-for-small-businesses-and-start-ups/#page-content)的文章。

# 变革阻力

一些遗留的程序和系统持续存在几乎令人不舒服。他们在湍急的河流中扮演着岩石的角色。但是面对一个不断变化的行业，有时这些系统会带来一些问题，这些问题可以通过一点软件升级来解决。

一个例子是使用 Excel。几十年来，它一直是办公室的中流砥柱，这是有充分理由的——它简单而有效。但它也不是没有缺点，这些缺点即使对最大的公司来说也是代价高昂的。想想[巴克莱银行在 2008 年因为一个 Excel 电子表格的重新格式化错误而购买了比预期多得多的东西。这些错误并不常见，但并非闻所未闻，它们可能会发生在你身上。](https://www.computerworld.com/article/2533631/excel-error-leaves-barclays-with-more-lehman-assets-than-it-bargained-for.html)

如果您仍在使用 Excel 并希望避免类似的错误，请考虑将其视为自己的编码语言。这意味着实现评审和测试用例。与您的数据治理策略非常相似，这可能看起来是对您的工作负载的一种昂贵且乏味的添加。但是你知道离开的风险。

另一种选择是研究和考虑其他软件。只要确保它符合你的商业战略。
