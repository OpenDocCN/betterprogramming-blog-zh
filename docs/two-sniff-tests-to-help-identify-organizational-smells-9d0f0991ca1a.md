## 你的团队结构有味道吗？

在软件开发领域，我们有[代码气味](https://en.wikipedia.org/wiki/Code_smell)的概念。这种想法认为有些事情是隐约错误的，它们可能表明一个更深层次的问题。

根据 Martin Fowler 的说法，代码气味必须是容易察觉的——可以嗅到的——它会指出我们代码中的问题。

> 我们需要一种类似的简单方法来确定我们的组织中是否存在问题。

下面我提出两个简单的问题来帮助识别组织的气味。我提出这些问题是为了让你问自己或询问你的团队成员。一旦你回答了这些问题，你就会知道你是否有什么要解决的。我还会列出每种情况的一些潜在原因。

# 问题 1:

# 你在正确的时间范围内工作吗？

要理解我的意思，考虑一下你把注意力花在哪里了。你关注的是今天/明天、下周、下个月发生的事情吗？

> 有些角色的时间尺度是一天，有些是一周，有些是一个月。

举例来说，开发人员可能是按天计算的。高层领导中的某个人应该在月尺度上。一个经理，建筑师，项目经理大概一周。

这里的问题不是你的过程对突然的变化有多大的弹性，关键是——人们在做正确的工作吗？

# 你有气味，如果:

*   你是一名副总裁，为了团队的正常运转，你必须参与每一个电话，而不是参与团队战略、年度预算和业务关系。
*   你是日常琐事的主管，而不是管理团队成长、培训和技术指导。
*   你是一个产品经理，参与日常运营，而不是创建路线图，会见客户，做市场分析。
*   你是一个真正在月尺度上工作的开发人员，如果是这样，那么听起来你的角色实际上是一个战略家、传道者、架构师或者类似的角色。

# 这里有一些观察结果:

*   如果你没有在正确的时间刻度上工作，那么你可以向上或向下工作。如果你应该在月尺度上工作，而你实际上是在周尺度上工作，那么你是在向下工作。如果你应该按日工作，而你实际上是按月工作，那么你就是在按日工作。
*   这些是气味——不一定是不好的。有时候，你需要合法地上下移动。当有人外出时。关键时刻到了。当你刚进入一个角色，并在旧角色中培训某人时。在这里，你只是想确定它是故意的，并且是暂时的。

# 在错误的时间范围内工作的潜在原因

*   促销。领导和管理的一个主要目标是帮助人们在公司中成长和晋升。然而，这必须得到很好的管理——否则人们会倾向于提升，但仍然保留他们以前角色的元素。
*   缺乏授权。这与晋升事业有关，但可能发生在任何人身上。如果你正在按时间表工作，那么你可能需要委派更多的人。
*   人太少了。可能是因为招聘缓慢，人员流动缓慢，或者只是职位不够。这可能是缺乏授权的一个原因，或者可能只是时间紧迫，但你会经常看到人们跳下时间表投入工作。如果这是习惯性的，那么你需要解决它。
*   一些应该存在的职位或角色并没有存在。我认为这是一个常见的原因，如果一个人正在工作的时间表。有些人会跳出来做急需的工作，他们会直接接受。也许团队成长了，或者工作增加了，团队需要一个产品人、一个项目经理或者一个领导。
*   缺乏晋升。也许有人是事实上的 *<在这里插入角色>* 但是他们还没有正式取得这个角色。这绝对是一场灾难。

# 问题 2:

# 你回答了正确的问题吗？

这对于非常小的团队来说没有什么意义，因为人们不得不扮演许多角色。然而，如果你的团队有一套传统的规程——工程、项目管理、产品管理和人员管理——那么通常这些规程中的人应该问这些类型的问题:

# 项目/计划—何时？

*   神器什么时候送到？
*   我们能约会吗？
*   发射所需的所有组件会同时准备好吗？
*   如期完成有什么风险？
*   在这个时间框架内，这可能吗？

# 工程——怎么做？

*   这个新特性的所有组件是如何组合在一起的？
*   架构图是什么样子的？
*   怎样才能让系统做到*<X>？*
*   这可能吗*(根本)*？

# 产品——什么？为什么？

*   我们试图解决什么问题？为什么要解决？
*   路线图上有什么？
*   下周我们应该做什么？下个月？明年？
*   什么特性更重要？
*   我们能做什么更有价值？
*   为什么我们要做这个而不是别的？
*   为什么这会为组织提供价值？
*   我们应该做什么实验来帮助回答这些问题？

# 人员管理-谁？

*   我们应该指派谁去做这个项目？
*   谁是应该将其知识更广泛传播的单点故障？
*   谁最了解/正在学习这项新技术？
*   谁是我们团队领导的继任者？
*   当我们的质量保证工程师离开两周的时候，谁来代替他们？
*   我们需要雇用谁？我们需要为谁创造一个新的职位或角色？
*   谁的人际团队或家庭问题没有得到解决或承认？
*   如果我们不去干涉他们，哪些贡献者会茁壮成长？我们应该更密切地关注和鼓励哪些人？

# 潜在原因

*   角色定义。这是这里最明显也是最常见的原因。人们不知道谁应该做什么。我见过详细的角色和职责的大型演示。如果工作大到足以保证同一学科的多人参与，这一点尤其有用。如果你只是需要在学科之间划一条线，以使学科清晰——这些问题可能有助于将这些线聚焦。
*   工作没有完成。也许谁应该做这项工作是非常清楚的，但工作没有完成，所以有人在插手。这可能比就同事的产出进行艰难的对话要容易。
*   缺少一个位置。如果有人经常回答与其角色无关的问题，那么团队可能需要增加另一个职位或角色。也许你缺少一个产品人员或 QA 或架构师。

这些测试你做得怎么样？你在正确的范围内工作吗？你回答了正确的问题吗？你团队的其他人呢？
