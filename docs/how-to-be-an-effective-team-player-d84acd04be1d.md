# 如何成为一名有效的团队成员

> 原文：<https://betterprogramming.pub/how-to-be-an-effective-team-player-d84acd04be1d>

## 最好的开发人员能有效地与他们的团队合作。他们就是这样做的

![](img/53fce94650e94dc2386e1090e5f656df.png)

安妮·斯普拉特在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片。

我做过开发人员，我的队友都是很棒的人，但是我们合作得并不好。不是因为我们不擅长，而是因为那个工作场所的文化没有奖励或激励团队工作——或者团队工作出色。

然而，总有一些人会脱颖而出，设法克服这样的工作文化，并证明团队合作不仅有效，而且回报丰厚。如今，我很荣幸能够在一个团队合作更加顺畅、稳定和有回报的环境中工作，我可以看到差异是巨大的。

> “有效合作的感觉很好。对一些人来说，这可能是职场上的一种新体验。”——Kent Beck 在[极限编程中解释](https://books.google.ca/books?id=-DNcBAAAQBAJ&pg=PA43&lpg=PA43&dq=%22Working+effectively+together+feels+good.+It+may+be+a+new+experience+in+the+workplace+for%C2%A0some%22&source=bl&ots=EXGTm9jG0Y&sig=ACfU3U33eOKj2l0aUzcT4PnkBqjwkU_Niw&hl=en&sa=X&ved=2ahUKEwirr_2P7ePqAhXRlHIEHc5iCroQ6AEwAHoECAIQAQ#v=onepage&q=%22Working%20effectively%20together%20feels%20good.%20It%20may%20be%20a%20new%20experience%20in%20the%20workplace%20for%C2%A0some%22&f=false)

不管你在什么样的工作环境中，能够与他人合作总是会让你脱颖而出，从你的努力中获得最好的结果，并成为你团队中的参考。所以，这是伟大的团队成员实现这一目标的方法。

# 1.他们能有效地与非技术人员交流

每当与非技术人员交谈时，避免使用过多的技术术语或不必要的细节。

乍一看，这似乎是显而易见的，但是我很确定你已经告诉了某个 PM 或 PO 关于一个 bug 或一个技术挑战，却得到了一个困惑的面部表情，然后是一个类似“好的，这是一个拦截器吗？

*每当你发现自己处于这种情况下，总是提供小块的信息，代表你要说的一切的全面而简化的版本。如果和你谈话的人仍然好奇或不相信，重复这个过程，增加细节层次。继续这样做，直到你表明你的观点。这也会让你有机会把注意力集中在听者最难理解或认为最重要的部分。*

*这将为你和任何想知道你正在研究的技术的深度发生了什么的人节省时间。*

# *2.他们与其他开发人员沟通良好，并发现学习机会*

*现在，每当你和其他开发人员交谈时，你当然有更多的自由去使用那些不需要任何解释就能运行良好的语句。但是，并不是所有的开发人员都有相同的资历，最重要的是，对你们所从事的技术都很熟悉。*

*为了在这种情况下有效地沟通，调整交付的速度很重要，这样其他开发人员至少有时间告诉你，你是否经历了他们不太理解的事情。尤其是当你和经验比你少的人一起工作时，重要的是你能找出他们知识上的差距，这样你就能创造出短暂而简洁的教学时间。*

*如果你是一个缺乏经验的开发人员，或者只是对正在讨论的事情不太了解，不要忘记说出来，因为没有人能够猜到这一点。*

*正在学习的人很可能会在他们的整个职业生涯中带着这些非常精确和独立的时刻，所以确保你在说话时不要匆忙，并且让这种对话非常互动，这样你就不会失去这样的机会。*

*如果你碰巧在一个很少合作的团队中工作，试着以某种方式创造这样的分享时刻。作为一名开发人员，教和学对于积累知识都是至关重要的。*

# *3.他们是自私的程序员*

*这听起来很奇特，但它简单地意味着将你的个人自我从你的工作中分离出来——无论是做你自己的工作还是评论别人的工作。解决了这个问题的开发人员能够持续地接受对他们工作的批评，而不会感到沮丧或产生怨恨。*

> *“在一个充满自我的环境中竞争工作的最优秀的程序员不能像作为一个自律和自组织的团队合作的普通程序员那样完成很多工作。你需要一个团队授权和协作蓬勃发展的过程来发挥你的全部潜力。”— [唐·威尔斯](http://www.agile-process.org/)谈极限编程*

*另一方面，在审查代码或讨论解决方案时，不要傲慢或刻薄。不要使用贬低或任何形式的攻击性交流。找出代码中的问题，批评作品——而不是作者。不要陈述事实。陈述观点，这就是它们的本质。根据支持他们的证据或想法提出建议。*

> *“不要使用贬义词，例如‘这是我见过的最糟糕的代码’当涉及到不属于你的代码时，要客观。更好:“有几个 bug。”“— [维基百科](https://wiki.c2.com/?EgolessProgramming)*

*这也包括不要试图证明你从语言或设计模式中了解一些奇特的特性。有些程序员写复杂的东西只是为了炫耀，或者可能是因为这样代码会超级简洁、精彩，而且…你猜怎么着:别人很难理解。不要这样。好的代码通常是愚蠢而简单的。让模式自然地出现，并在将它们强加到代码库之前与其他人讨论大的抽象。*

# *4.它们防止并摧毁知识孤岛*

*优秀的团队成员对知识仓库有敏锐的洞察力。当某个开发人员(或者甚至是某个不同角色的人)知道某个主题的所有事情，而团队的其他人不知道时，这种孤岛就会出现——可能是由于孤立工作的缘故。*

*无论何时，当你意识到你团队中的某个人最终积累了关于某个特性或其他你团队负责的事情的所有知识，邀请他们来分享。这非常简单，不需要复杂的文档或演示。一个 30 分钟的会议展示你自己的电脑屏幕就足够了。如果你真的需要异步，Google Docs 中一个简单但写得很好的文档也很好。*

# *包扎*

*有效地一起工作，并意识到一个团队不仅仅是其成员的简单相加，这是非常有益和令人上瘾的。*

*记得尽量用小噪音交流。试着抓住每天的学习和教学机会，记得把你的自我放在门外。成长和实现的环境不仅是由公司文化和激励机制创造的，也是由日常的小互动创造的。意识到这一点并自己实施小的改变已经是一大步了！*
