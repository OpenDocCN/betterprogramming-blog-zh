# 眼球震颤如何让我成为更好的程序员

> 原文：<https://betterprogramming.pub/how-nystagmus-made-me-a-better-programmer-1504cf983a45>

## 我输了一些，我赢了一些——这都是观点的问题

![](img/22b5ef8fdc53e1ea2bc03569d38f8229.png)

丹尼尔·库切列夫在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片

所以我有眼震。

但是你可能不知道那是什么，所以让我简单介绍一下。

眼球震颤是一种不自主眼球运动的情况。基本上，我的眼睛时不时会开始不受控制地移动，我会本能地倾斜头部来补偿这种移动。

而且，我必须看着另一个方向大约 10 到 20 秒，而不聚焦在任何物体上才能恢复视力。

好的一面是这不是常数。我每小时只有大约六次这样的发作。但这取决于我昨晚睡了多长时间，我的压力有多大，我有多努力去读一些东西，不同的光线条件等等。

在最糟糕的情况下，我每隔 2-3 分钟就会发作一次。你可以想象到目前为止这对我的生活产生了怎样的影响。

我很难保持目光接触。

如果文本在我面前不清晰，我很难阅读。

我很难在街上或在某个地方认识人。

我真的不会开车。

这样的例子不胜枚举。你可以想象我在编程时经历的所有其他场景——紧迫的期限、糟糕的代码格式、过于明亮的显示器。

但讽刺的是，眼球震颤让我成为了更好的程序员。所有这些不利因素迫使我以不同的眼光看待事物。

让我告诉你怎么做。

# 1.快速阅读

虽然违反直觉，但无法持续关注文本让我成为了一个更快的读者。

我学会了通过快速浏览一行文本来阅读，而不是通过眼睛跟踪一行中的每个单词。如果我没有跟踪每个单词，我也没有默读每个单词——这也提高了我的阅读速度。

所以，当我阅读一本书时，我只是在页面的每一行中间垂直移动。对于每一行，我都在脑海中创建具有整体意义的图像，而不是关注每个单词的独立语义。

当我阅读一段代码时也是如此。每行 80 个字符的标准限制实际上帮助了像我这样的人。我不用从左到右去读一行代码。

你可能认为这意味着我错过了重要的信息。事实是每行只有大约 2-4 个重要的单词。剩下的话只是把重要的连接起来。

一行代码也是如此:80%是语法，20%是有意义的信息。

通过练习，我开始感觉到哪些单词在一行中是重要的，这样我就不会错过它们。

# 2.用模式思考

当我加入一个项目时，我会花一些时间去理解其中使用的架构和模式。如果我不马上这样做，它会迫使我一遍又一遍地重读代码库。

例如，我没有关注传递给函数的实际参数。相反，我关注的是函数是如何定义的，以及以何种方式调用它。

这使我能够用模式思考。

现在，当我看到一段代码时——我已经知道它是如何定义的，以及它的使用方式。我不必重读整个代码来理解它。

这种心态在很多方面帮助了我:

*   当我发现一个 bug 时，我很可能已经知道了根本原因。图案上有一个瑕疵。
*   当我遇到技术挑战时，我很可能已经知道一个可能的解决方案。我以前也见过类似的问题。
*   当有人问我问题时，我很可能会马上回答。我不需要看代码，它已经在我脑子里了。

眼球震颤迫使我更好地理解事物，所以我不会被迫再次学习它们。

# 3.使用快捷方式

我认为自己是一个速度编码员。

当我编码的时候，我几乎不碰鼠标。

当我共享屏幕时，你可能不明白我在做什么，因为事情就发生在屏幕上——变量改变名称，文本被选中，多个标签被打开。

我在用捷径。

这是因为我不能总是依赖 IDE 提供的菜单项。如果我在寻找菜单操作时出现了眼震，我会很难找到它。

而是有了捷径，我真的可以闭着眼睛做操作。我不必在复杂的菜单层次中搜索。这有助于我将注意力完全集中在最终目标上，而不是集中在为实现目标必须采取的所有步骤上。

这很有趣，因为当眼球震颤发作时，我有时真的闭着眼睛做事情。

如果你知道如何触摸字体，你就能理解我。你不必看着键盘打字。如果你有很好的准确性，你甚至不需要看屏幕来检查你的打字。

好的一面是捷径对每个人都适用。你不必受眼球震颤的折磨，今天就开始从中受益。

# 结论

眼球震颤让我的生活更加艰难。

但这也让我对生活有了不同的看法。如果我有一个完美的愿景，我会不断寻找替代方法来做我本可以做的事情。

替代方法通常比标准方法更好。但是如果我不是被迫去寻找的话，我永远也不会发现其他的选择。

在编程中，眼球震颤帮助我对系统有了更好的理解，并提高了编码速度。我可以说眼球震颤确实让我成为了一名更好的程序员。

每一个劣势都只是一种取舍。有输有赢。

专注于胜利，你永远不会对失败感到难过。
