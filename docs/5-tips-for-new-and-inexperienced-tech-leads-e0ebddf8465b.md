## 新的技术领导者没有理由必须自己解决所有问题

在我成为技术负责人的前六个月，我耗尽了自己的精力。成为一名技术领导者一直是我的目标，但在我实现这个目标后，我花了一些时间真正后悔。

我白天花了很多时间做技术方面的事情，晚上赶着做票务工作。我不是很清楚这个角色需要什么，我只是在做我看到的上一个技术领导做的事情。我对此毫无准备。

那是五年前的事了，那时我已经是三个团队的技术主管和工程经理了。随着时间的推移，这种体验变得越来越容易，对技术领导者的期望也变得越来越清晰。新的技术领导者没有理由必须自己解决所有问题——以下是我对成为更好的技术领导者的建议。

# 首先，什么是技术领先？

在不同的公司，技术领先的含义可能略有不同。

技术领导的一种形式是团队领导——你负责开发团队的日常运营和高质量产品的交付，但没有雇佣/解雇的权力。

技术领导的第二种形式是工程经理——你是一名拥有雇佣/解雇权力和人员管理职责(如定期绩效评估)的经理。有时，这些重叠成一个角色。我也认为这是两个不同的角色。

技术领导的第三种形式是首席工程师——你是团队的高级成员，有时领导较小的项目，但主要是在技术方面，如进行代码审查，为新项目提出数据模型，或设计项目。

本文将重点关注技术领导的第一种形式，也称为团队领导。弄清楚这一点后，这里有一些指导你的技术领导力的原则。

# 做你团队的挡箭牌

处理任何可能会打断团队的事情，例如回复对您的团队或即将到来的项目有疑问的闲散人员，与内部支持团队合作解决上报的问题，以及在必要的会议上代表团队的利益。

这通过防止其他人(如高层管理人员)直接向您团队的开发人员提出请求来保护团队的时间。这会分散他们的注意力，把开发人员从重要的计划工作中拉出来。所以你的目标是发现这种情况是否发生，并向那些人清楚地传达这些请求必须首先通过正确的渠道。

很多这种工作实际上只是试图对手头的问题进行分类，并根据需要重定向到正确的人。

# 了解其他团队在做什么

会议是必要的罪恶之一，我们喜欢谈论我们如何淹没在无用的会议中，但偶尔，一些会议中有有用的信息。

在一个理想的世界里，我会得到一个我没有积极参与的每一次会议的五分钟回顾，但这永远不会发生。

他们有机会听到关于其他团队正在发生的事情的信息，无论是你直接工作的团队，还是与你的工作更不相关的其他团队。你可能无意中听到某个团队正在构建一个新的服务，这给了你一个机会说:“嘿！我的团队正在研究类似的东西。大家说说吧！”每隔一段时间，这种互动会为您节省四分之一的工作量。

# 促进与团队的讨论

你是否曾经在团队会议中，大部分时间都在沉默中度过，等待有人插话(在远程环境中更是如此)？“我们应该探索这个图书馆吗？”有人可能会问，听到蟋蟀的回应。有时候，人们不知道如何回应，害怕看起来很傻。

作为一名技术负责人，你的工作是促进与团队的讨论，以便团队可以拥有决策权，并做出决策。帮助人们获得尽可能多的信息来帮助他们做决定。你可以问得更清楚，或者问一些愚蠢的问题。一半的时候，其他人也有同样的问题，但他们太害羞而不敢问，因为他们不想成为问愚蠢问题的人。

在您的角色的早期阶段，您可能需要依赖您的开发人员一点。你不会知道代码库的所有不同部分，但你仍然需要能够回答诸如“特性 X 可能吗？我们在 system Z 中有数据，”如果你在团队中呆过一段时间，你就会知道谁知道代码库的哪些部分。如果你是新来的，他们会给你指出正确的方向。

如果你没有足够的信息来做决定，确切地说出来，并在你能打电话的时候与需要决定的人沟通。

# 有标准并以身作则

不要浪费任何时间挑剔风格差异。向项目添加 linter 和/或格式化程序，并将对话重定向到工具。另一个 dev 留下 15 个吹毛求疵的评论是一回事；这是另一个问题，我们是否应该增加一个新的规则。

在可能的情况下，要求代码变更与单元测试一起进行，并测量单元测试覆盖率。对我来说，在整个项目中，70%的覆盖率就足够了。

当然，有些场景更难用单元测试来覆盖，但这些都是例外。任何业务逻辑或奇怪的特定行为都需要进行单元测试——这是我们防止其他开发人员在删除他们认为可能不需要的代码时意外破坏代码的唯一方法。

# 关注大局

你必须选择你的战斗，有些更小的决定最终不值得你花费时间。如果一个开发人员在你希望它更加面向对象的时候用过程模式实现了某个东西，我会保持沉默，继续前进。

从长远来看，这些决定最终并不那么重要。重要的是确保你的团队没有将破碎的变更推向生产。这里或那里的程序片段不会影响生产。

这并不是说你不应该提供反馈。有时提问更有帮助，以确保他们提出的解决方案满足问题的需要，“你考虑过 X 还是 Y？”，而不是技术负责人的个人偏好。

你不应该告诉你的开发人员*如何*实现他们的改变(除非他们特别要求反馈)。如果你这样做得太频繁，他们可能会开始感觉更像是代码猴子在做技术领导说的事情。

有时，你可以通过询问正确的问题来指导他们采用不同的解决方案，这比你说“以这种方式构建”更能让他们拥有想法和实现其他时候，您不能，因为他们的解决方案同样满足所有需求，在两种方法之间进行选择是一个偏好问题。在这种情况下，不要为小事烦恼，继续前进。

# 结论

作为一名开发人员，您并不经常需要这些技能，尽管它们很有帮助。因此，新的技术领导通常有一点学习曲线，因为他们在平衡自己的职责的同时，会发现管理团队的人的方面。因此，如果你想在 2022 年成为更好的技术领导者，请关注以下几点:

*   做你团队的挡箭牌
*   了解其他团队在做什么
*   促进团队内部的讨论
*   有标准并以身作则
*   关注大局
