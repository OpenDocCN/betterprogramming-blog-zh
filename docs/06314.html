采访200多名工程师的经验教训

在过去的六个月里，我学会了最好的软件工程师和领导者如何管理技术债务。我采访了200多名专家，以了解成长型软件公司是如何处理技术债务的。我发现了许多公司所缺少的东西，并将我的经验汇编成一个简单但强大的框架，与大家分享。

作为我们免费Stepsize产品的客户开发工作的一部分，我需要了解那些控制住技术债务的软件公司和那些没有控制的软件公司之间的区别。

科技债务是一个情绪化的话题，它让人们谈论的话题无与伦比——而且它比政治争议少得多。问问你当地的工程师关于科技债务的问题，你就知道了。但你可能想先清空你的日程表。

管理技术债务的成本和收益

在许多情况下——如果不是大多数的话——科技债务会越积越多，直到产生无法忽视的问题。所以你解决了这些问题，继续你的生活。你接受也许事情就是这样。

但是我们真的应该接受 Stripe的研究的结果吗？该研究发现，在一般的公司，“工程师花大约33%的时间处理技术债务，这打击了团队士气，并使公司每年损失大约850亿美元”。我们不该做点什么吗？

Gartner和许多其他公司已经向我们展示了我们应该这样做。他们的研究显示，积极管理科技债务的组织出货速度至少会快50%。

我不知道有哪家软件公司不想提高50%的出货速度。

幸运的是，我确实遇到了拥有坚实的科技债务管理战略的公司。在这些采访中有许多令人恍然大悟的时刻。我最喜欢的一次是Everlane的工程经理James Rosen告诉我:

“考虑一下项目经理花了多少时间来管理要处理的功能集。现在，将这与工程师致力于为技术债务提供商业案例的时间和精力进行比较。接近零的工程能力被分配给科技债务，这有什么奇怪的吗？”

我必须承认，不，它不是。

然而，我也遇到了很多团队，他们花费了大量的时间和精力来管理技术债务，但仍然一事无成。

我的所有研究都指向一个简单的事实:成功管理技术债务的公司已经开发了适当的流程，这些流程完全融入了他们通常的敏捷仪式，并已成为习惯。

这些工程团队控制着他们的技术债务。他们出货更快，更可预测。他们的工程师更开心了，他们的客户也更开心了——见鬼，他们都赢了。

到达那里真的不需要太多。你只需要清楚你将如何处理小型、中型和大型的科技债务。

如何处理小额债务

这是一种技术债务，只要工程师在代码中发现它，就可以解决，并且在他们正在处理的问题单的范围内。这可能是像重构一个函数或重命名某个变量这样简单的事情。试着遵循罗伯特·c·马丁的 童子军规则:

"总是让代码比你发现的更好."

这些小工作不需要任何类型的计划，每个工程师都应该觉得自己有权解决这种债务，而不需要其他任何人的批准。我们已经写了关于的文章，这是一个健康的代码库所需要的文化特征，但是要确保它存在于你的工程团队中。如果没有，现在就采取措施解决它。

在这方面有很多工具可以帮助你。我们自己的针对VSCode 的 Tech Debt Metrics扩展将直接在您的编辑器中给出代码质量分数和细目分类。其他代码质量工具，如 Code Climate 或 Codacy ，你可以将它们包含在你的CI管道和代码评审过程中，也非常适合这种情况。

如何处理中型债务

这种技术债务可以在短时间内解决。它应该像任何特性工作一样经历相同的冲刺规划过程，并被同样严格地考虑。这是大多数工程团队失败的地方——记住詹姆斯·罗森的评论:

"接近于零的工程能力被分配给科技债务，这有什么奇怪的吗？"

企业理所当然地优先考虑为客户带来价值的工作。乍一看，摆脱技术债务并不能做到这一点。

但是技术债务阻碍了你向客户提供价值的能力。

为了明确说明这是如何发生的，请确定妨碍关键计划或在工程师生产力方面使业务付出高昂代价的债务，或者是影响客户体验的错误的原因。

记录技术债务并量化其成本可以让你优先考虑债务，如果解决了，将为客户带来价值——就像新功能一样。工程组织拥有技术债务。他们有责任解决这个问题，并最终为其提供商业案例。

不幸的是，这就是我们目前的工具让我们失望的地方。

“吉拉是管理项目的好地方，但却是跟踪和监控科技债务的糟糕地方。”

—杰克·佩瑟，Unqork的首席工程师

代码质量工具有助于揭示技术债务的一个方面，但是它们没有抓住大多数其他方面。

幸运的是，这正是我们为提供的免费产品。工程团队处理技术债务的时间有限，他们需要珍惜时间。Stepsize帮助他们从工作流中捕获和跟踪技术债务，以便他们可以量化业务成本，并最终确定最重要债务的优先级。

每个工程师都可以直接从他们的工作流(包括编辑器、拉取请求和Slack)中报告技术债务及其成本。这些报告都上传到Stepsize网站，在那里它们被整理成科技债务项目、描述和记录代码库中的问题。最后，每张吉拉门票都装饰有相关的技术债务项目，可以更有效地向客户提供价值。不再有从未被优先考虑的孤独的重构票——技术债务和商业价值变得真正一致。

应该如何处理大量的债务

这是无法立即解决的技术债务，甚至无法在一次冲刺中解决。我采访过的最好的公司都有季度技术规划会议，所有的工程领导都参加。工程经理的任务是强调向他们报告的大量科技债务，并为他们认为最重要的债务提供商业理由。

这个过程听起来很费力，但对Stepsize用户来说变得非常容易。他们的个人捐助者已经从前线定期报告债务。这些数据由每个团队和他们的领导不断审查和整理，他们将大部分债务以及理解他们给企业带来的成本所需的数据传递给他们的工程经理。Stepsize甚至为你的每一部《吉拉史诗》揭开科技债务的面纱。然后，领导层可以利用他们对公司更广泛的优先事项和愿景的理解，相应地对大部分债务进行排序。

一旦每一大块债务被批准，它就可以被安排到路线图上，就像特性工作一样。这样，工程领导层就拥有了监控每个技术债务项目进展所需的所有数据。

外卖食品

任何现代软件公司都应该能够将这一过程应用于小型、中型和大型的技术债务。然而，每家公司都有一点不同:商业目标。

正确管理技术债务意味着解决阻碍你实现业务目标的债务。如果你的企业是建立在正常运行时间和可靠性的基础上的，那就放弃任何让他们处于风险中的债务。如果速度是你的竞争优势，去掉任何浪费工程时间或者让新员工难以理解代码的债务。如果你想减少客户流失，解决导致债务的质量问题。

清楚处理每笔债务的商业案例。因为当你这样做时，你会更快地发布更好的软件——而且你会让你的工程师高兴。
