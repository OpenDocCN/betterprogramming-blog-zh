需求获取指的是从一个人或一组人那里收集需求。该术语通常用在软件系统的上下文中。然而，无论需求是针对一个系统还是一个过程，过程都是一样的。

具体来说，想象你被要求进行一个数据分析项目或机器学习。在你想出如何解决这个问题之前，你需要引出需求。

虽然这个过程看起来应该很简单，但它经常被描述为一种艺术形式。理解这个问题可能很棘手。

# 理解问题

解决问题的一个重要方面是理解问题。这比听起来要难。有时你会解决自己的问题，有时会解决别人的问题。

无论是哪种情况，你都必须花时间去理解这个问题。注意潜在的原因——目标是解决潜在的问题，而不仅仅是治疗症状！要做到这一点，通常需要与你试图帮助解决问题的个人进行至少三次会谈。

> "问题说得好，就是解决了一半."
> —约翰·杜威

# 澄清、核实、重复

**第一次见面**是对问题的介绍。保持好奇心，仔细听别人告诉你的事情。

如何告诉你某些细节和告诉你什么一样重要。根据 Ray Birdwhistetll 的说法，人类交流的 60%-70%是非语言的。

一旦你在第一次会议上收集了所有你能收集到的信息，花点时间回顾、研究和了解情况。这让您可以在第二次**会议**上提出后续问题，以验证您对问题的理解。

**第三次会议**将结束任何未完成的项目，并验证您对问题和情况的理解是否与您认为的完全一致。

# 提问

一般来说，人们尽可能地乐于助人。在他们试图提供帮助时，他们经常掩盖你要求他们描述的潜在问题。他们没有强调正在讨论的问题，而是提出了他们对如何解决问题的理解。

让我们看一个例子。想象一下，你正在采访一个用户，询问他们面临的问题，他们说他们需要一封电子邮件，告诉他们当天的销售情况。如果他们在 16:00 收到这封邮件，那么他们就可以走了。简单——只需编写一个程序，提取这些信息，并通过电子邮件发送给他们。

不幸的是，你只治标不治本。当被调查时，用户描述说，通过电子邮件中收到的信息，他们可以检查是否有新客户出现。如果有，他们会在系统中设置。答对了。用户真正想要的是在系统中建立新客户的能力。这打开了一个可能性的世界:也许整个过程可以自动化，无需任何用户输入。

# 5 个为什么方法

找到问题根源的一个极好的技巧是问五次为什么。每一次，你都向问题靠近了一步，你在试图识别。

```
User: I would like a report that lists all the sales for the dayAnalyst: **(1)** Why would you like this report?User: With this report I can figure out who our customers are for the day?Analyst: I see. And **(2)** why would you need to know our customers for the day?User: Oh, simple — I can use this list to cross reference it against our Client Relationship Management (CRM) database.Analyst: And **(3)** why would you need to do that?User: To figure out any new customers.Analyst: **(4)** Why do you need the new customers?User: Because I am going to need to set them up if they are not there.Analyst: **(5)** Why do you need to set them up in CRM?User: This way they can begin feeding all of our Management Information reports.
```

你有它！我们被要求解决的真正问题是确保新客户出现在管理信息报告中。

# 按优先顺序处理

有时候，你会发现自己有几十甚至上百个要求。在您致力于探索所有这些之前，了解项目的范围和预算可能是值得的。有时，没有必要解决甚至探索每一个需求。

# 帕累托分析

帕累托分析是一种将帕累托原则应用于你的工具。帕累托原理指出，80%的结果来自 20%的原因。换句话说，存在这样一种需求组合，只实现其中的 20%，就可以获得 80%的功能。

帕累托分析指的是为识别这 20%的核心需求所做的工作。
