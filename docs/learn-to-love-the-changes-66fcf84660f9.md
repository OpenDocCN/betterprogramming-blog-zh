# 对第二个敏捷原则的理解:“学会热爱变化”

> 原文：<https://betterprogramming.pub/learn-to-love-the-changes-66fcf84660f9>

## 探索敏捷原则面临的实施挑战

![](img/dde0b5d692af566183ec83b00a0006f2.png)

爱德华·豪厄尔在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片

上周，我提出了一个问题“[,你读过敏捷宣言吗？](https://andywatt83.medium.com/have-you-read-the-agile-manifesto-9785ce876078)”，并开始了一系列关于 12 个敏捷原则的帖子，并对实施挑战进行了一些挖掘。

我觉得敏捷没有兑现它的承诺。我认为失败的原因不是敏捷过程中的基本缺陷，而是因为它在大多数情况下都没有得到很好的实现。

这一次，我关注的是第二个原则，大概是这样的:

> 欢迎不断变化的需求，即使是在开发的后期。敏捷流程利用变化为客户带来竞争优势。

![](img/8e3201ca717df89be371012f3039cc15.png)

[布雷特·乔丹](https://unsplash.com/@brett_jordan?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的照片

我想我可能会对所有这些原则这样说，但是很明显，对一个产品进行改变，给客户带来竞争优势，对客户来说是一个好主意，对软件公司来说可能也是。

一个软件需要几个月甚至几年的时间来交付是很常见的，在这段时间内业务需求发生重大变化也是很常见的。确实应该预料到需求会在项目过程中发生变化，直到并包括项目的后期阶段。

如果开发公司能够管理这些变化，并在整个过程中纳入客户的反馈，那么客户的结果最终肯定会更好。这对软件公司来说应该是好的，因为他们的声誉将会受益，大概他们的底线也会随之受益。

我们将很快讨论技术和关系管理挑战，但是在实践这个原则时有一个非常明显的问题，那就是律师。

![](img/38c935c1cdcb387cbb33a5469020a397.png)

[廷杰伤害律师事务所](https://unsplash.com/@tingeyinjurylawfirm?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的照片

或者更具体地说，合同。通常，如果不是总是，会有一个合适的合同来交付做某件事情或者满足某组需求的软件。这份合同将在编码开始前达成一致，并将列出交付日期、付款里程碑、惩罚条款、性能要求，以及另一份明确的*非*敏捷的全部清单。

遵守具有法律约束力的合同没有什么“敏捷”可言。而且，永远也不会真的发生——这就是做生意的方式。

许多所谓“敏捷”项目一开始就受到一系列合同条款的束缚，这些条款是*必须*实现的。然后是技术团队的敏捷，但是只要他们不错过任何截止日期，或者跳过任何功能，或者触发任何惩罚条款，或者…你明白了。

这种契约上的无能是双向的。随着项目的进展，可能会发现某个特性超出了需求。但是，开发人员可能会发现他们自己无论如何都要创建这些特性，因为有交付它们的合同要求。同样，客户或顾客可能会发现他们无法添加他们发现他们想要的东西，因为它不在合同中，或者被明确排除在外。

在陷入合同纠纷的项目中工作对参与的团队来说是毁灭性的——对双方都是如此。没有什么比在你和客户都知道永远不会使用的应用程序中加入功能更糟糕的了，但这样做是履行双方都不会放弃的合同义务。这是一个徒劳的练习，我敢打赌很多读者都会认同！

尽管存在合同和法律问题，但在一天中的晚些时候进行更改在技术上具有难以置信的挑战性，有时甚至是不可能的。如果客户突然决定他们实际上想要一个 web 应用，而不是桌面应用，那么无论产品的架构有多好，都将有大量的工作要做。

即使是不太极端的例子，在一天的晚些时候改变需求也是具有挑战性的。随着项目接近完成，对产品功能的改变变得越来越困难。即使非常努力地坚持良好的编码实践、关注点分离等，也没有办法解决这个问题。

当然，合同中不会考虑到这些最新的变化。

为了真正应用这一原则，并按预期工作，必须在组织级别上接受这一点，这在项目开始时就得到承认。

这要求建立一个极其强大的变更管理系统，合同承认变更管理系统，并明确允许范围内的变更。*(我想任何读到这篇文章的律师可能刚刚心脏病发作)*

变更管理流程和程序的编写方式必须能够支持不交付商定的功能，并且不会导致客户要求退款。(这个很难)。同样，它要求顾问可以交付最初没有指定的功能*而不增加价格*。这也很难。

良好的变更管理始于软件组织内部以及外部与客户的公开和诚实的个人交流。例如:

*   当开发人员在某个特性上落后时，他们必须对他们的管理层开放和诚实。
*   管理层必须接受评估特性是困难的，有时开发人员会落后
*   管理层必须深刻理解合同要求，并在客户提出合同之外的要求时迅速采取行动
*   类似地，管理层必须阻止开发团队开发不在合同中的特性(开发人员有时会忘乎所以)
*   客户必须有一个渠道/论坛来表达他们不断变化的需求

必须实施一个正式的流程来记录这些讨论中产生的变更，对其进行成本核算(如果需要)并将其附加到合同中。

为了使技术团队能够整合这种“项目级变更管理”，软件必须以一种极端模块化的方式编写，这样，在程序的一个部分进行变更就不太可能在程序的另一个部分产生副作用。有许多模式、实践和编码风格在这方面有所帮助。

如果在整个开发过程中功能和代码都必须改变，那么实现健壮的自动化回归测试将是必不可少的。这将给予开发人员进行变更的信心，也应该给予管理人员和客户这样的信心，这样做不会引入任何(或许多..)新 bug。

构建合同和技术变更管理流程来支持这一原则极具挑战性。好处(当然)是显而易见的，但通往极乐世界的道路充满荆棘。

根据支持这项工作的合同，将它改造成一个棕色地块项目很可能是不可能的。

但是，如果以真正“敏捷”的方式工作是目标，那么 12 个原则中没有一个可以被忽略，因此必须努力使团队和组织作为一个整体进入这种在项目的整个生命周期中接受技术和合同变更的心态。
