# 通过大学和社区学习编程

![](img/ca8547ab7c02c889a1bd9bb462e24d05.png)

## 在过去

我们学习了计算机科学的核心，如编程范例、软件工程、数据结构、计算机系统、操作系统概念等。

特定的操作系统(如 Windows)和编程语言(如 C++)是我们自己学会的。

我们编写了文档，收集了需求分析，并准备了演示文稿——然后，我们编写了实际的程序本身。我们从图书馆找到参考资料，把论文印在思考页上，作为作业完成上交。

计算机科学学位是申请编程工作的基本要求。

## 现在

如果你有大学学位当然很好，但是候选人的选择是基于他们的编程能力。如果你没有学位也没关系，只要你优化编码。

核心计算机科学概念是很好的背景知识，但更重要的是你使用该工具的经验——例如，管理 AWS 系统或对特定编程语言有很强的掌握。

即使你没有论文或学位，如果你开发了一个被很多人使用的应用或者有一个关于开发的流行博客，你也会被认为是这个领域的专家。

# 旨在持续的代码与旨在改变的代码

![](img/e8b179820152fdf019579167a57ebc67.png)

罗斯·芬登在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的照片

## 在过去

构建软件是工程。就像在任何工程领域一样，我们希望建造经久耐用的东西。所以在过去，我们有一个很长的需求捕获阶段，希望我们不会错过未来的需求。这就是所谓的 [*瀑布模型*](https://en.wikipedia.org/wiki/Waterfall_model) *。*

面向对象的编程是最先进的。它有一个基类，大概是稳定的，不会有太大的变化。通过子类引入了微小的变化。

## 现在

客户希望快速看到产品，而需求变化很快。因此，我们构建的内容应该不断变化，以适应最新的需求。[引入敏捷编程](https://en.wikipedia.org/wiki/Agile_software_development)是为了实现更快的周转和反馈循环。

[函数式编程](https://en.wikipedia.org/wiki/Functional_programming)已经变得流行起来，因为它能够在软件核心中注入更多动态变化的函数。代码现在更加可定制，功能是重中之重。

你可以[阅读这篇](https://levelup.gitconnected.com/why-code-evolve-from-procedural-to-object-oriented-to-functional-programming-13a88292307e)来理解函数式编程如何使快速变化成为可能。

根据我的经验(尽管我现在在一个不同的组织)，有一件事是显而易见的，那就是软件发布的速度:

*   15 年前→每半年发布一次软件
*   10 年前→每三个月发布一次软件
*   5 年前→每月发布软件
*   今天→每周发布的软件

# 全面编码与简明编码

![](img/f730c127277c2f74eeaf96670b0768a0.png)

来自[像素库](https://pixabay.com/?utm_source=link-attribution&utm_medium=referral&utm_campaign=image&utm_content=1853237)的[像素](https://pixabay.com/users/Pexels-2286921/?utm_source=link-attribution&utm_medium=referral&utm_campaign=image&utm_content=1853237)的图像

## 在过去

仪式很重要。所有的需求和决策都需要被捕获。会议时间很长，会议记录被发送出去。

当我们开始编码时，我们确保:

*   最初的代码组织是最重要的，因为我们希望避免将来对它进行大的修改
*   代码可以追溯到需求的来源，或者需要对任何特殊逻辑进行注释
*   所有需要严格遵守的语法结构——例如，tab、open `{`、`;`的使用等。
*   代码审查既残酷又全面。不容忍任何违反行为。语言专家对代码的审查是必须的，以确保使用最好的 API。

代码需要尽可能全面，这样即使打印在纸上，它仍然是可读和可理解的(我们打印代码是为了在审查会议上审查，因为我们不能把我们的桌面带到会议室)。

## 现在

时间是宝贵的。我们知道代码变化很快。足够好比完美更重要，因为没有完美。

当我们开始编码时，我们确保:

*   初始结构:虽然它很重要，但它的变化是意料之中的。这些工具使我们能够更无缝地对其进行更改。
*   它简单明了，因为文档就是代码。这些工具使用户能够轻松地检索提交注释。
*   语法结构不是问题，因为这些工具有助于自动套用格式。我们不再担心制表符与空格、`{`等问题。，因为这些工具会自动修复任何格式问题。
*   代码评审关注业务逻辑；只要代码总体上有所改进，小问题就不要紧。该工具本身将提供最佳 API 的建议，并使用最佳编码实践(例如，私有与公共)。

我以前曾[写过](https://medium.com/@elye.project/how-ide-influences-my-coding-9130d1c18e34)关于工具和编程语言的发展如何改变了我们编码的方式

# 封闭系统与开放系统的代码

![](img/0ca4c01098acde9e6507786c88e11c33.png)

图片来自 [Pixabay](https://pixabay.com/?utm_source=link-attribution&utm_medium=referral&utm_campaign=image&utm_content=2836960) 的 [ComMkt](https://pixabay.com/users/ComMkt-390552/?utm_source=link-attribution&utm_medium=referral&utm_campaign=image&utm_content=2836960)

## 在过去

构建软件是一项艰巨的任务。你有的只是操作系统和带有编译器的 IDE。您必须构建尽可能多的支持系统来管理整个软件开发过程。

甚至代码修订控制系统、软件缺陷跟踪系统和持续集成系统都是由组织自己内部构建的。当时没有吉特、吉拉、詹金斯或类似的人。

一个人必须建立一个数据中心，拥有用于常规测试执行和编译的构建机器。

每个核心处理或 UI 组件都必须从头开始构建。我们在算法中写入了复杂的数学计算和内存管理系统。

## 现在

我们现在知道，从头开始构建一切不仅乏味且昂贵，而且效率也不高。我们会失去对我们商业价值的关注。

我们可以很容易地为我们的代码库、软件跟踪系统、持续系统等找到可靠的工具。(例如 GitHub、吉拉、詹金)。他们专门为这些任务开发工具，并且比公司做得更好。

我们已经脱离了数据中心，转而依赖全球公认的云解决方案，例如 AWS、阿祖拉、谷歌云等。—我们不仅可以减轻这些解决方案的维护负担，还能保证更好的灾难恢复。

我们专注于编码和开发我们业务的产品核心，同时利用开源(拥有适当的许可)所需的基本组件(例如，UI 组件、崩溃跟踪系统等)。如果开源选项有限，我们会为其改进做出贡献，而不是自己动手。

我们相信每个人都应该专注于他们的核心，让我们相互依赖，在未来建立一个更加高效的编码社区。

**结论**

与硬件不同，软件的根本特征是*变化*。这种快速变化适用于技术、生态系统、我们的工作方式以及社会对编码社区的期望。

称编码职业为*变革职业并不夸张。*

拥抱变化，继续前进！
