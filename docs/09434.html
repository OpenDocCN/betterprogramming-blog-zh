开始使用CSS width: 100%而不是width: 100vw，看看“视口单位”如何引起滚动条

水平滚动条是你的网站和应用程序“水平溢出”的标志。让我们修理它。

这是一个CSS谜语

如果文档的可视部分(其“视窗”)是100“VW”单位宽，100vw的宽度是多少？

放在CSS里，width: 100vw应该是全屏的，相当于width: 100%吧？

不对。

CSS的“盒子大小模型”有一个奇怪的错误，实际上这根本不是一个错误，但这是一件需要注意的重要事情。

视口单位包括滚动条(如果存在的话)所占据的空间。这意味着它们不像基于百分比的单位那样正常工作，导致不稳定的显示问题。

换句话说，如果你想要一个全幅的HTML元素在网站上，CSS width: 100vw 应该可以工作。但这并不是因为一些相当不幸的数学问题，我称之为“视口诡计”

水平滚动条“bug”(特性？)通常出现在当某人在构建单页应用时习惯使用100vw的时候，就像在使用 React 、 Next.js 或者你喜欢的框架时一样。

我们先来谈谈为什么100vw不工作，然后我会向你展示十个解决方案，来修复那些在你试图构建一个简单、全屏网站时无缘无故出现的水平滚动条。

使用100vh和100vw来指定一个“全屏”文档应该很简单，这是单页应用程序的常用方法。

在普通网站上，我们习惯于垂直滚动，但水平滚动并不是一种常见的用户体验。这意味着100vw应该是视窗的整个宽度，事实也的确如此，但是我们需要了解视窗实际上有多宽。

了解CSS盒子大小模型

CSS的一个基本方面是它的“盒子大小模型”，但这也是CSS中最令人困惑的事情之一。

默认情况下，在 CSS框模型中，分配给元素的width和height仅应用于元素的内容框。如果该元素有任何边框或填充，则将其添加到width和height中，以获得屏幕上呈现的框的大小。— MDN文档

这类事情最好用视觉来演示，你可以在浏览器的DevTools中轻松完成。这是Chrome打开调试CSS 的一个镜头，它会自动绘制CSS框模型:

为了将视口与CSS盒子模型联系起来，我们必须引入滚动条的概念来处理“视觉溢出”

水平滚动条=水平溢出

当我们在网站上看到滚动条时，那是因为内容的尺寸大于可见区域。这被称为溢出，类似于带有“裁切边距”的“全出血打印”

就像打印名片一样，你可以从一张大纸上剪下名片；多余的就是溢出来的。这就是全出血打印，我认为这是理解CSS溢出的一个有用的比喻。

“我们有时需要在CSS中管理的溢出被描述为可滚动溢出。这是出现在框外的内容，需要为其提供滚动机制。溢出属性是我们如何控制当内容溢出一个框时会发生什么。”— MDN文档

这是有道理的——如果在CSS中指定3000像素的固定宽度，分辨率为1920x1080的人将只能看到网站上左侧1920像素的内容。在右边，有“水平溢出”，所以你会看到水平滚动条。

移动显示器的宽度可以小于400px，而桌面显示器可以非常大(我的是2560x1440)。这就是为什么现代开发者使用响应式网页设计来构建“移动优先”的网站和应用。

连接视口和可滚动溢出

我们将viewport定义为站点或应用程序在浏览器中的可见部分，那么为什么我们会得到带有width: 100vw的水平滚动条呢？就像CSS盒子模型一样，我们需要考虑额外的空间。

虽然浏览器的用户界面不算视区的一部分，但滚动条确实算。因此，如果你同时设置了width: 100vw和height: 100vh，就像你对单页应用程序所做的那样，它就能很好地工作。

然而，如果你想要正常高度的全幅，就像互联网上几乎所有的网站一样，你会有一个垂直滚动条。这将影响视窗的宽度。

只读属性innerWidth以像素为单位返回窗口的内部宽度。这包括垂直滚动条的宽度(如果有的话)。

更准确地说，innerWidth返回窗口的可视视区的宽度。窗口的内部高度——布局视口的高度——可以从innerHeight属性获得。”— MDN文档

垂直滚动条算作视口宽度的一部分，这意味着您将有水平溢出。水平溢出意味着你会在页面上看到一个水平滚动条。

计算视口宽度与CSS盒子模型并不完全相同，但原理是相似的，因为我们的实际宽度实际上比我们预期的要宽。

对于视口计算，滚动条算作宽度。相比之下，CSS box模型的宽度包括padding & border。

(height属性在两个模型中的行为与width相同。)

从关闭overflow到改变我们在CSS中定义width的方式，有很多方法可以去除全宽网站上这些不想要的水平滚动条。

让我们来看看十种不同的修复方法，找出最适合你和你的响应网站的方法。

1)向所有父容器添加宽度:100%

首先，让我们把width: 100vw换成width: 100%。后者不涉及浏览器的视窗，永远不会显示水平滚动条。问题解决了，对吧？

“width CSS属性设置元素的宽度。默认情况下，设置内容区域的宽度，但如果box-sizing设置为border-box，则设置边框区域的宽度。— MDN文档

所有元素的默认 width为auto，这意味着浏览器会自动计算并选择指定元素的宽度。这意味着这里有点蹊跷。

问题是我们使用的width: 100%值是基于父容器的大小。这意味着如果你试图制作一个全宽元素，但是它的父元素有一个较小的尺寸，你将不会最终使用视窗的100%的width。

要解决这个问题，您必须调查每个父容器的宽度，以确保所有内容的宽度都是100%，而不是更小。

不过，总的来说，用100vw替换100%是一个很好的解决方案。然而，这并不是唯一的解决方案，所以让我们继续吧。

2)使用位置:固定锚定内容

当我在使用React构建Next.js应用程序时试图制作全尺寸响应图像时，我发现了水平滚动条错误。在我的HTML代码中，我试图“突破”父容器，它有一个在CSS中定义的max-width(最大宽度)。

当你有一个固定的导航菜单或者页脚，你想把它附加到屏幕的顶部或者底部的时候，你就需要打破一个容器。你通常会使用position: fixed来达到这种效果，尽管也有更新的position: sticky。

固定位置元素上的溢出将被自动隐藏且不可滚动，尽管100vw仍将包括垂直滚动条。在下面的截图中，我添加了一个大的红色边框，所以你可以看到这个元素是超宽的，尽管我们不再有滚动条。

CSS属性B ox-sizing: Border-box

这是一个很好的地方来提及使用CSS属性box-sizing: border-box改变CSS盒子大小模型不会修复视口问题，但这是一件值得了解的事情。

"border-box告诉浏览器考虑您为元素的宽度和高度指定的值中的任何边框和填充。如果您将元素的宽度设置为100像素，这100像素将包括您添加的任何边框或填充，并且内容框将收缩以吸收额外的宽度。这通常会更容易调整元素的大小。— MDN文档

与默认值box-sizing: content-box不同，border-box不会将填充和边框算作元素宽度的一部分。这对于某些宽度问题很有用，但是它不能修复与视口宽度相关的水平滚动条。

"content-box给你默认的CSS盒子大小行为。如果将元素的宽度设置为100像素，则元素的内容框将为100像素宽，任何边框或填充的宽度都将添加到最终呈现的宽度中，使元素宽度超过100像素。— MDN文档

我认为box-sizing: border-box应该可以工作的原因是你有时可以通过设置* { margin: 0; }来移除水平滚动条。这段CSS将删除所有HTML元素的默认边距，包括<body>，它的默认边距为8px。(通常，开发人员使用类似于 Normalize的工具。CSS 移除所有默认样式。)

以我的经验来看，移除所有边距只会在某些时候修复width: 100vw中的水平滚动条，但是重要的是要记住，在浏览器中，边距永远不会被算作元素宽度的一部分。

用默认行为(box-sizing: content-box)边框和填充算作宽度，但是我们可以用box-sizing: border-box忽略边框和填充。该属性不影响毛利。

3)使用overflow-x: Hidden隐藏滚动条

如果你不想解决水平滚动条的问题，你可以忽略它！虽然这不一定是最好的解决方案，但这是你在CSS中绝对应该知道的。

"overflow CSS 速记属性为元素的溢出设置所需的行为，即当元素的内容太大而不适合其块格式上下文时，在两个方向上都是如此。"— MDN文档

从技术上讲，我们只关心水平溢出，所以这就是CSS中的 overflow-x 属性。设置overflow-x: hidden和瞧，问题解决了…算是吧。足够好了吧？

您需要禁用<body>元素上的溢出，因为这实际上是产生水平溢出的原因。另一个创建滚动条的HTML元素是低级的<iframe>，这是一个更常见的设置overflow: hidden的元素。

同样，我包含了一个边框，所以你可以看到我们隐藏了右边滚动条后面的溢出。

4)使用宽度:100vi代替宽度:100vw

还有另一组视口样式的CSS度量单位，它们的行为与vw和vh完全不同。

vi等于包含块的初始尺寸的1%，在根元素的纵轴方向。

vb等于包含块的初始尺寸的1%，在根元素的块轴方向。”— MDN文档

这到底是什么意思？天啊，CSS已经够混乱的了，对吧？我理解你。这些新奇的长度取决于一种语言是横写还是竖写。

"块尺寸:垂直于一行内文本流向的尺寸，即水平书写模式下的垂直尺寸，垂直书写模式下的水平尺寸。对于标准英语文本，它是垂直维度。

行内尺寸:平行于一行内文本流向的尺寸，即水平书写模式下的水平尺寸，垂直书写模式下的垂直尺寸。对于标准英语文本，它是水平维度。”— MDN文档

你指出基本上所有现代语言都允许水平脚本是对的。然而，有些语言可以垂直书写，包括中文、日文和韩文。

如果我们在水平书写的页面上用vw替换它的对等物vi(比如英语)，我们将移除水平滚动条。内容将是全角的，没有任何溢出。

这有点像诱饵，因为vi是基于它的父容器，而不是视口。换句话说，width: 100vi和width: 100%是一样的，不是width: 100vw。

5)没有垂直滚动条，没问题(高度:100vw)

如果你正在制作一个真正的单页应用 (SPA ),它应该是整个视窗的大小，那么你可能会想保留width: 100vw。

毕竟，如果你正在构建一个要安装在设备上的渐进式网络应用，那么使用视口是很有意义的。

如果你还添加了height: 100vh，你应该永远不会有垂直滚动条。(不要忘记将宽度单位从vw切换到高度单位vh。)在这种情况下，您的HTML应该占据整个视窗。

这是我们确实需要去除那些内置边距的情况之一。添加CSS规则集* { margin: 0; }对我有效，尽管你可能更喜欢更强大的东西来规范CSS。例如，Tailwind CSS使用自己的工具 preflight 来设置默认样式，同时提高跨浏览器兼容性。

或者，你可以组合width: 100%和height: 100%来代替width: 100vw和height: 100vw。但是行为会有所不同，所以一定要测试这两种方法。你可以用下面截图链接的我的CodePen自己试试。

这与一开始只使用width: 100%是完全一样的，但这也是一个直观上很有意义的解决方案。
