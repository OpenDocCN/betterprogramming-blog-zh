# 我们对软件开发不够认真

> 原文：<https://betterprogramming.pub/we-are-not-serious-enough-about-software-development-b9e3222f2906>

## 为什么我们不像对待现实世界的基础设施那样对待软件呢？

![](img/4078a8bee7ed2d8f2f8120262d0e8bd6.png)

Marc-Olivier Jodoin 在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片

我在想，为什么大多数公司对待构建软件的过程与构建现实世界的基础设施如此不同。

在这篇文章中，我想提出这样一个话题:我们必须把构建软件看得更像构建现实世界的基础设施。而当我和别人说起这件事的时候，我们比以前更惊讶了！作为软件开发人员，我们为什么没有那么认真地对待我们的工作呢？

# 1.要求

让我们从我们如何处理软件开发中的需求开始。我知道有不同类型的公司从事比其他公司更大规模的软件开发，但是请听我说。

## 软件开发

在我过去十年的软件开发中，我看到了各种收集需求的方法。他们中的大多数人并没有把它看得那么严重，他们的思维非常敏捷。嗯，敏捷，实际上他们并不真正知道他们在构建什么。

他们在创建计划时遇到了问题，相反，他们只是开始编码。没有任何要求，只是一个我们需要建立什么的想法。我们必须一路走下去。由于缺乏清晰的愿景，这些项目迟早会搁浅。

## 现实世界的基础设施

当一个建筑、桥梁、购物中心或现实世界中的任何其他基础设施需要建造时，人们首先开始讨论需求。

他们心目中的最终用户是什么？需要解决的问题是什么？需要放在哪块土地上？为什么人们想要使用它？诸如此类。

要列出一个好的需求列表，需要回答相当多的问题。我想我的问题只是其中的一小部分。

但是为什么建造现实世界基础设施的人如此重视这个呢？

嗯，他们不想浪费钱。他们不想发生意外。他们想为他们的最终用户创造一个成功的故事。当然，他们想挣钱。

## 现实世界与软件开发

嗯，当我只想到这一点时，我有一种感觉，为什么我们大多数人，以构建软件为生的人，不认真对待适当的深思熟虑的需求？

我们不在乎吗？难道我们不想为我们的最终用户创造一个成功的故事吗？我们想浪费钱吗？我不这么认为！

你的老板也不会这么想。但是他们怎么会知道呢？很可能，你的老板不是软件开发人员、架构师或设计师。如果你的老板是那样的，你会听吗？

不要误解我，我不想让人们对我生气或不高兴！

但是我认为，作为软件开发人员，我们必须更认真地对待我们的需求。我们必须对自己的工作更加自豪。为了给这些人提供建议，我们必须提出这些要求。

# 2.灵活性

如果将软件开发与构建现实世界的基础设施相比较，在灵活性方面有相当多的不同。但是，让我们深入一点，看看它是否真的有那么大的不同，或者我们软件开发人员只是推动新的东西通过？

## 软件开发

几个世纪以来，我们一直在从瀑布过渡到敏捷。

使用瀑布，所有的过程都排成一行，并遵循一个顺序。编写需求，构建软件，测试软件，发布软件。

起初，这听起来并不太糟糕。但是当两者之间出现问题时，在人们可以测试软件之前，所有东西都被移动了几个月。

这并不奏效，所以一些聪明人想出了敏捷宣言。一些可以用来建立更灵活的软件开发方式的指导方针。因为在开发过程中，需求可能会发生变化。

它们可能会因为用户给出反馈或人们用软件进行测试而改变。

或者也许软件公司不知道为他们的用户解决真正的问题。因此，敏捷指南应该使应用程序的开发成为一个学习的过程，在这个过程中，敏捷是企业成功的关键。

但是当一个公司从瀑布模式过渡到敏捷模式时，他们认为他们不必再有需求，并且他们可以在他们想要的时候放弃任何改变，而没有意识到它的影响。这就产生了问题。

但是作为一个软件公司，如何敏捷地工作已经足够写一整本书(或者一系列文章)了。

当公司已经完成了它的需求，并且做得非常好的时候，影响可以被最小化。但是大多数公司不会花时间去制定他们的需求。

因此，在大多数情况下，影响比估计的要大。但是如果一个需求正在改变，或者我们需要更新一个依赖关系，或者我们从一个框架改变到另一个框架，每个人都会对此掉以轻心。

很多人没有意识到这些变化的影响！希望你不是其中之一！如果是，请继续阅读。

## 现实世界的基础设施

当现实世界基础设施的建设阶段发生变化时，每个人都知道这可能会在资金或建设何时完成方面产生巨大影响。

当一个变化必须发生时，整个过程需要暂停。这也取决于变化的影响。但是影响可能会很大。

因此，整个建筑团队、建筑师和其他人都必须从头开始。需求的变化会产生巨大的影响！团队中的每个人都会知道。

因为他们意识到每一个改变都要花费更多的时间和金钱。并且可能成为项目成功的风险。

## 现实世界与软件开发

如果我将两个世界相互比较，我认为我们必须更加意识到变化的影响。依赖性的改变。建筑变化的变化。在有人做出决定之前，应该对每一个变化进行适当的调查。

当我们更加认真地考虑它在时间和金钱方面对业务的影响时，我们就不会再轻率地考虑架构变更了。

所以，让我们让彼此意识到这一点。为你的工作感到自豪！

如果你非常清楚我们可能会造成的影响，你有责任向做决定的人提出建议。

# 3.维护

当我们想到维护时，我们知道软件开发和现实世界的基础设施都离不开它。但是在公司如何处理软件维护和现实世界基础设施之间有很多不同。

## 软件开发

我见过许多公司没有人负责维护他们的软件。更有甚者，当你要求公司在开发阶段结束后安排人员到位时，他们会感到惊讶。

他们真的看起来像他们看到水燃烧！

好吧，也许不是所有的公司都这么轻视软件维护。感谢上帝，有一些好的公司，一旦软件发布，他们会安排人更新软件。或者是修复错误的人，或者更好的，他们改进软件。

但是在小公司里，让人们这么做并不常见。

这些公司的大多数经理都是这样想的:“嗯，开发团队应该让软件经得起未来的考验，或者不应该引入任何错误。”

好吧，希望我们现在都知道人是不完美的，所以软件永远不会完成。

## 现实世界的基础设施

就道路、桥梁、房屋或其他建筑等基础设施的维护而言，经常有人负责维护。

你的房子可能时不时需要一些维护，比如粉刷墙壁、修理屋顶、修补渗漏等。或者你雇一家公司来帮你做这件事。

但是每个人都知道现实世界的基础设施需要维护，否则就会变坏。

我们的道路不时需要一层新的混凝土。我们的木材需要新的油漆，或者因为环境原因需要修理。

## 软件开发与现实世界的基础设施

令人震惊的是，很多公司根本不考虑维护这一部分。他们只是把它归咎于开发者，他们必须拥有无错误的软件，或者他们必须为未来十年构建它。

但老实说，我们不会对那些为我们建造房屋、道路、桥梁和其他现实基础设施的人说这些话。那么，为什么我们期望我们的软件是免维护的呢？

服务器获得安全补丁，框架获得漏洞修复。我们都希望我们每天使用的软件在未来变得更好、更安全，对吗？

我希望你们公司在这方面做得很好！如果是的话，击掌吧，朋友！

但是，让我们帮助其他人、软件开发人员和公司更加了解软件开发的这一部分。

# 结论

感谢阅读这里的一切！

我希望我的故事已经唤醒了一些人，并且我们作为软件开发人员更加意识到这个事实。

软件开发与构建真实世界的基础设施没有太大的区别。因此，让我们更严肃地对待我们的软件开发，更清楚地意识到我们对它的影响。

我们的认真与否可以成就或毁灭一家公司。让我们为我们的工作感到自豪，并更认真地对待它！

如果你同意或不同意我的观点，请在评论中告诉我。我期待着就如何让我们的工作做得更好进行一次愉快的交谈！
