# 衡量绩效的需要

作为工程经理，您对度量并不陌生。从代码行到正常运行时间，软件开发行业几乎太沉迷于数字了。但是当涉及到衡量一个工程团队的表现时，工程经理经常会陷入困境。

有两种人，不相信量化指标的人和相信量化指标的人。我属于第二组。这并不意味着我不关注定性反馈，但是如果你知道首先要衡量什么，定量指标可以告诉我们很多故事，并描绘出一幅美好的画面。

在这篇文章中，我将谈论你如何根据你的团队的独特情况来考虑测量什么样的指标，以及在测量和提高你的工程团队的绩效时要采取什么步骤。因为你首先想要测量它的最终原因是你可以改进它。就像俗话说的，你不能改进你没有衡量的东西。

# 度量的类型

当衡量一个软件工程团队的成功时，有两种类型的度量标准你应该看看:定性度量标准和定量度量标准。

# 定性指标

第一类指标是定性指标。通常，这可以通过与员工和客户交谈来实现。

许多软件公司使用 NPS(净推介值)来衡量客户满意度。查看客户驱动的度量标准可以确保您的软件工程团队向业务和客户交付价值。

在定性指标下，您可能还想获得对单个团队成员的 360 度反馈，以评估他们的表现。

当定性地衡量一个团队的表现时，关注团队中表现差的人是很重要的。就像俗话说的，一个烂苹果坏了一整桶。虽然可能只是十人团队中的一个人，但如果他们表现不佳，这个人可能会对整个团队的表现和士气产生重大影响。能够尽早发现表现不佳的工程师来自于经验。

他们表现不佳的原因有很多。其中一些是:

*   工作不投入(无聊，对技术不感兴趣，对公司目标不感兴趣)
*   影响他们的其他个人事务
*   感觉不到他们属于他们的团队
*   不具备所需的能力

在确定了根本原因之后，软件工程经理可以提出一个合适的计划。

可能的操作包括:

*   转移到不同的团队或项目
*   给他们时间充电
*   提供适当的培训/资源来提升他们的技能
*   将他们与其他更高级的工程师配对
*   给他们提供较小的工作范围

然后，您将观察相同的反馈循环和指标来跟踪进展。如果一段时间后以上都不起作用，那就该考虑放手了。

作为一名经理，最有收获的部分(至少对我来说)是通过指导和授权在人们的职业发展中发挥作用。我发现，让员工保持工作积极性和参与度的最佳方式是给他们提供培训、职业发展和拓展任务。

然而，这并不全是乐趣和游戏。作为一名经理，这份工作的困难在于当人们的表现没有达到预期时如何管理他们。持续的反馈很重要，可以确保你为你的员工提供支持和指导，让他们知道自己的立场。

及时可行的反馈很重要，你不应该害怕与他们进行艰难的对话。如果你粉饰业绩或行为问题，你就没有尽到经理的职责。

这只是定性反馈在识别问题和有效解决问题中有多么重要的一个例子。

# 定量度量

第二种类型的度量是定量度量。它们可以用数字表示，用科学方法获得。

可用于测量/管理软件产品开发中的质量、交付和生产率的量化指标的例子有:

## 交货时间

*   周期
*   研制周期

## 质量

*   故障率
*   测试覆盖率的百分比

## 正常运行时间(可靠性)

*   潜伏
*   恢复时间
*   事故检测时间(平均检测时间)
*   事故恢复时间(平均修复时间)

## 交货速度

*   已关闭 bug 的平均年龄
*   团队速度
*   发布次数
*   完成的故事点数量
*   故事完成率

正如我在我的书[工程经理的指南](https://gum.co/emhowto)中所写的，衡量什么是重要的，并使用与业务目标相关的指标是很重要的。否则，软件工程团队会变得离业务的原因太远。

# 如何开始

以下是为您的工程团队整合指标的建议步骤。

1.  从目标开始——目标的一个例子可能是以更快的速度交付可靠的特性。
2.  思考您想要测量哪些与目标一致的指标，以及您想要如何测量它们。一旦您定义了与您的工程团队的目标相一致的指标，监控它们并接近它们是很重要的。如果不监控和跟踪它们，就等于没有任何度量标准。
3.  分配将成为指标所有者的人员，并与他们一起工作。这些人可能是工程经理、员工工程师或者对特定的度量标准充满热情的软件工程师。
4.  整合工具和机制。信任但要核实。自动化和警报，Slack，如果是这样，则使用(ITTT)工具来设置智能警报。
5.  设定评审指标的节奏，例如，量化指标的直接会议和定性反馈的季度评审。
6.  保持监控、回顾和改进！
