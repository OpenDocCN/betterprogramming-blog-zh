## 有这么多！我选哪一个？

![](img/8fbf1a4fcb43feb51af1d75c8bd5e4b2.png)

维多利亚诺·伊斯基耶多在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的照片

我非常欣赏开源开发的现状。它可能并不完美，但我真的很重视思想和创造力的自由流动，以及这么多开发人员愿意分享他们所知道的以及他们如何解决问题。最终，我认为开源应该是这样的:解决问题和分享答案。

然而，就像生活中的几乎所有事情一样，这个硬币也有另一面:不是所有的答案都是平等的，有太多的选择几乎和没有选择一样糟糕。一旦我开始编写依赖于开源模块的软件，我很快就明白了这个事实。我最终想出了一套指导方针来帮助我做决定并减轻一些固有的风险。希望你会发现这些指南对你的搜索也很有用。

这些指南没有按照任何特定的顺序列出。我试着平等地考虑所有人，并根据总体分数做出最终决定。

(注意，虽然我的背景和观点主要基于 JavaScript 生态系统，但是这些指导方针适用于所有共享和使用开源模块的语言。这些原则几乎与语言无关。)

# 考虑作者

我们无法亲自了解所有开源作者，但有一些指标应该对我们有所帮助:

*   他们在他们的社区里很出名吗？
*   他们发布了多少模块？
*   他们的 GitHub 回购有多少明星/叉/贡献者/关注者？
*   他们贡献活动的总体水平如何？
*   他们对问题和 PRs 的反应好吗？
*   模块是否与它们所基于的语言或引擎同步更新？

这里的想法是阐明他们对开源开发的总体承诺和兴趣，以及他们与其他社区成员的关系，以及他们对周围生态系统如何发展的认识。

这是否意味着我们应该只对利用由前 1%编写的代码感兴趣？不一定。受欢迎并不等同于最好。然而，我的经验表明，在使用其他人开发的几乎任何东西时，可靠性和一致性是一个非常重要的问题。

# 考虑文档

文档是关键。好的文档将极大地改变我对开源模块的看法:

*   模块有没有[自述](https://medium.com/@meakaakka/a-beginners-guide-to-writing-a-kickass-readme-7ac01da88ab3)？
*   文档包括真实世界使用的代码示例吗，这些示例实际上可以工作吗？
*   有没有[变更日志](https://keepachangelog.com)？或者发行说明中是否充分描述了变更？
*   我对文档的理解是否足以充分利用模块？(诚然，这可能只有在您开始使用该模块之后才会变得明显。这就是生活。)
*   如果要传达大量信息，文档是否被分解成单独的文件或页面？那些来源可以理解吗？

一些开发人员可能认为代码应该是自文档化的，或者单元测试应该充分满足显示代码如何工作的需要。虽然我在理论上完全同意这个观点，但在实践中，我不想花几个小时甚至几天去阅读代码，只是为了弄清楚它是如何工作的。请为我们提供一份好的参考文档，并确保它是最新的。作为专业软件开发人员，这是我们工作的一部分，我们所有人都应该为彼此树立一个好榜样。

# 考虑代码

我没有必要花时间去阅读我考虑在项目中使用的模块中的每一行代码。但是我确实检查了基本的东西:

*   模块是否遵循编码标准？检查模块是否包含`lint`命令，或者自述文件是否表明正在遵循某个标准，这是一个好方法。

对于用 JS 编码，我确实有自己的偏好，但是我并不超级关心使用哪个标准，只要使用一个标准。这很重要，因为我很有可能在某个时间点需要查阅模块的源代码，而我不想处理格式不良的代码。

*   该模块包括自动化测试吗？那些测试是由 CI 系统自动运行的吗，README 是否包含一个标记来指示当前的构建/测试状态？

基于云的持续集成服务如今无处不在，我认为没有理由不使用它们，即使是最小的项目[。](https://travis-ci.org/github/DPassarelli/then-recursively-search)

(这里有一个我认为很重要的旁注要提一下。仅仅因为一个徽章表明最新的构建失败了，并不意味着这个构建实际上失败了。有时网络缓存会妨碍指示真实状态，所以我总是点击这些标记来亲自检查。我还发现构建矩阵可能包含我不关心的环境或版本，它们的失败可能是也可能不是问题。)

*   模块是否使用[语义版本](https://semver.org)，当前版本是否为 1.0.0 或更高版本？

永远不总是最容易正确应用的概念，但我总是奖励那些正在尝试的人。我也倾向于避免那些停留在“还没有准备好”模式的模块(意思是，他们自豪地表明他们是主要版本 0，并且最初是在几个月或几年前发布的)，即使他们至少有些受欢迎并且满足了一个重要的需求。对我来说，这是一个大红旗。还记得我上面说的可靠性和一致性吗？我得出的结论是，使用 SemVer 是尊重和考虑他人的表现。我们应该袖手旁观我们的工作，如果我们准备发布一些东西到“野外”，可以这么说，那么我们至少应该提供和传达一些关于稳定性的保证。

*   API 是否符合某种风格或模式(功能性的还是命令性的，流畅的，声明性的，等等)。)我可以轻松地对其进行编程？

API 设计有点棘手，有些 API 肯定比其他的更有意义。在不同模块之间有多种选择的好处之一是，我们经常可以选择一个符合我们偏好的模块。

# 底线

对我来说，在选择开源模块时，可靠性和一致性非常重要。事实上，有很多工具试图自动回答这些问题(特别是 JavaScript，有像 [npms.io](https://npms.io/about) 这样的网站)，这意味着其他开发人员也关心这些问题。

我喜欢这种想法，有些工作可以自动化，但在一天结束时，我是最终负责做决定的人。所以我想参与这个过程。我也不喜欢完全依赖“精选”或“最佳”名单的想法，因为我经常发现它们要么不是最新的，要么不是原创的。

最后两个想法:

1.  是的，这个过程有点耗时，但我不认为有任何办法可以绕过它。从我的角度来看，如果我的依赖于这些模块的应用程序的生命周期是以月或年来衡量的，那么花几个小时(甚至几天)做一些调查工作是值得的。
2.  这个过程并不提供成功的绝对保证。仍然存在一些风险，但是可以采取额外的防御措施，我将在以后的文章中讨论。
