## 作为一名软件开发人员，我在最初的两年里学到了什么

![](img/79f1b356ba1a2d40e027afde64f8e644.png)

在短暂的实习向公司证明我是下一个大人物并让我获得入门级职位后，我于 2019 年开始担任移动开发人员。玩笑归玩笑，我很高兴开始作为一名移动开发者工作。自从我还是一名计算机系的学生时，我就喜欢开发移动应用程序，我还为我的学士论文开发了一个基于人工智能的移动应用程序。

所以开始这份工作时，我很高兴能写出高质量的代码，学习所有关于 Kotlin 的知识，从我的同事那里学到尽可能多的知识。我每天早上 6:30 醒来，成为第一个到办公室的人(我有一个小时的通勤时间)，也是最后一个离开的人。我必须证明些什么，不是吗？我非常努力地工作，试图限制我休息的时间。我必须不辜负所有的期望，管理好压力，做好每一件事。经理会密切关注你，看你是否付出了 100%的努力，如果你不这样做，他们就会告发你。

*我想大概是这样。*

一年半后，我在一家采用不同技术的公司工作。这项技术是全能的 React Native，它已经成为我编写移动应用程序的新宠语言(除了我一直最喜欢的 Flutter)。在我到这里的六个月里，我一直在反思自己作为一名初级开发人员的行为。看到我给自己施加了这么多无用的压力，我感到震惊——在某种程度上，现在仍然如此。

让我们回顾一下我在转换公司和技术后学到的一些经验。

# 1.没人在乎你什么时候上班，也没人在乎你什么时候下班。你只需要做好你的工作

IT 行业的工作通常有灵活的时间表。当然，你需要在核心时间出现，你需要参加会议，你需要在那里支持你的同事，所以一段时间后，它很可能看起来像任何其他朝九晚五的工作。

强迫自己成为第一个到办公室的人或者很快完成任务不会给你的经理留下好印象。一个好的经理甚至会提出这样一个事实:你对自己施加的所有压力和期望都会伤害你。

你想给某人留下深刻印象吗？做好你的工作，并把它做好。这就够了！

# 2.你不应该避免休息

当我回顾我作为开发人员的第一个月的每一个晚上有多累时，我感到非常恼火。是的，我有大量的工作要做，我被驱使着去做好每一件事。然而，那是如此有害、有毒和不健康。

你应该总是允许自己休息一下。参加一些社交活动，冥想，如果你觉得需要长时间的休息，就去跑一会儿步。只要你完成了分配给你的任务，并且你对你的队友有反应，休息是必要的。你应该始终尊重你的午休时间、你的心理健康和你的幸福。

额外收获:在很多情况下，一个问题变得不可能是因为你专注于它太久了。试着休息一下或者睡一觉。你会对结果感到惊讶。晚上，我会花两个小时调试一些东西，然后第二天早上五分钟就解决了。

# 3.你不会因为工作做得好就获得加薪，这是对你的期望

我对工作场所有着如此不切实际的想象。我想象着我一生中看过的所有电影:一个努力工作、超水平发挥的员工，有一天被他们的经理带进会议室，并被授予巨额奖金。

这是不现实的。绩效评估的工作方式不同，它是一个复杂且经过深思熟虑的过程。过度劳累对你没有好处。此外，人们期望你做得很好。不要把自己累垮，而是尽力而为。你是一个团队的一部分，每个队友都应该为了更大的利益尽自己最大的努力。

# 4.除非明确要求你加班，否则你不应该加班

如果每天工作 9-10 个小时是默认的期望(而你只有 8 个小时的工资)，那么*离开*！你在有毒的环境中工作，你应该辞职。

项目有明确的时间表。它是这样定义的，团队将能够在分配的时间框架内交付产品，同时每天工作 8 小时(或您合同中的小时数)。

当然，有时会出现特殊情况，你会被要求加班。只要达成一致，事先澄清，事后补偿，那就没问题。

# 5.不要满足于低于你的价值

研究市场。玻璃门是你的朋友。明白你是一笔资产，公司需要你。

当然，第一份工作也不要太挑剔。但是有时候稍微挑剔一点会帮你省下几个月的工资过低和过度工作，这并不理想。

只是不要去一家远低于你平均水平的公司。这是一个危险信号。

# 6.有时代码审查包含审查您的代码的高级人员的个人偏好，而不是实际的错误

作为一名初级程序员，你会收到许多关于你的拉请求的代码评审。这很好。代码评审是最有效的学习方式之一——让更有经验和资历的人仔细评估你的工作。他们会发现边缘情况，你的逻辑中明显的缺陷，以及框架和编程语言的误用。

但是除了所有的学习，有时可能会有点沮丧。你知道反馈是好的。你就是这样成长的。但有时个人偏见可能会溜进入门级程序员的 PRs 中。你会在某个时候了解到，应该讨论个人偏好，整个团队应该做出决定。这些都不是理所当然的。

同样在公关主题上，与其写一篇 50 条评论的公关评论，不如完全重写特写。

# 7.过度工程并不酷

过度工程:它不迷人。有些东西很花哨，很复杂，不代表它就是好的。最好的解决方案是更简单、更优雅的方案。

伙计，我对简单问题的奇特而复杂的解决方案如此着迷。我认为让我重写特性的评论是理所当然的，唯一的原因是我没有使用一些花哨的模式和很酷的语言特性。显然，它们适用于某些情况。

但是最终，应该根据项目的性质来决定如何编写一个特性。你的主要精力应该放在编写可测试的、干净的、高质量的代码上。考虑边缘情况，编写容易被他人理解的代码，不要因为“不太可能发生”而避免处理更具挑战性的边缘情况它会发生在某人身上。QA 团队肯定会遇到这个问题，无论如何你都需要解决它。

# 8.当你觉得你有有用的意见时，大声说出来！

你的意见很重要。当涉及到软件项目时，每个输入都是有用的。可能产品负责人或团队领导没有考虑具体的边缘案例。如果你看到什么，说点什么！

你永远不应该害怕说出来。也许不会在客户会议上提及，但你至少可以私下向你的经理提及。很高兴看到人们在开始使用这些功能之前花时间仔细考虑它们。另外，从长远来看，肯定会节省时间。当你认为自己对某件事是对的时候，不要让自我怀疑让你沉默。

# 9.烧坏

小心这个。我在上面提到过，我再怎么强调也不为过:不要接受过度劳累。你的私人生活不应该受到工作的影响。是的，我们热爱编程，但它毕竟是一份工作。你需要照顾好你的个人生活，和你的家人朋友一起出去玩，并投入足够的时间来照顾你的需求。

每天工作到晚上 9 点，并在周末安排一些工作，这似乎是一个很酷的主意。我也曾经这样做过。长此以往，就不值得了。

# 10.你的经理应该关心你，确保你快乐健康

我开始在一家公司工作，这家公司向我展示了领导者的工作。他们不指挥，而是领导。他们引导你，帮助你。他们应该小心谨慎，你应该感到舒服地与他们讨论什么困扰你。如果你的经理只是对你颐指气使，给你这么多工作，却忘了你也是一个可能会累的人，那就赶紧离开。

# **结论**

作为一名软件开发人员工作是有趣和令人满意的(除了那些黑暗的日子，那时一切都在崩溃，你无法找出那个讨厌的 bug 的来源)。但从长远来看，这是值得的。你可以把 IDE 当作一块空白的画布，你可以创建对人们有用的软件。

如果你注意到了我上面提到的危险信号，作为一名开发人员，你应该很高兴。

TL；博士:尊重你自己，尊重你的团队，认真对待你的工作(但不要*太*认真)。
