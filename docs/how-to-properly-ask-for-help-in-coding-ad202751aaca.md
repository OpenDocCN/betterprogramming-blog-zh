# 什么是正确的编码和调试心态？

从本质上来说，编程就是解决问题，将繁琐烦人的任务自动化，让每个人的生活变得更轻松。就像斜面或轮子一样，编码只是一种工具，我们人类可以用它来改善我们的生活，丰富其他人的生活。

那么，为什么我们允许编码过程成为烦恼和痛苦的来源呢？永远记住你编码的真正目的是什么。

假设您正在编写一个利用数据库存储用户信息的网站。这是一个极其常见的场景。您正在创建 LAMP 堆栈(Linux、Apache、MySQL、Python/Perl ),并且很难将数据库与 web 服务器集成在一起。当涉及到后端编程时，很容易陷入细节和大量的符号和难以理解的胡言乱语中。你能做的最糟糕的事情就是只见树木不见森林。你需要时刻记住你在做什么，为什么要这么做。如果你记得你建立这个网站是为了帮助客户(或你的公司)做一些最终会节省他们时间和精力的事情，你就不会感到沮丧甚至放弃。

当涉及到处理复杂代码时，我能提出的一个建议是记录你试图找出 bug 的过程。记录您尝试修复错误的所有步骤，记录错误实际是什么，记录您寻求帮助的来源。此外，在你真正寻求帮助之前，一定要给自己一些时间(理想情况下至少 15 分钟)，在你“放弃”之前，退后一步，重新考虑一切

# 如何真正提出你的编码问题

在大多数论坛上，你可以真正地问你的编码问题，有一些适当问题的“指南”,这样人们可以专注于解决你的问题，而不是试图找出你的问题实际上是什么。最棒的是，当你花时间制定一个适当的问题时，你会被迫真正明白你的问题是什么。很多时候，答案是在重新考虑之后得出的。

您的问题的理想格式应该如下所示:

*   问题的描述——首先解释你在做什么，你在使用什么语言和框架。
*   重现你的错误的步骤是什么——复制并粘贴你的代码，详细说明你做了什么，以便找到你的错误，这样人们就不必从头开始。
*   预期结果——描述代码的预期结果，甚至展示一个真实世界的例子(假设你构建的不是全新的)。
*   实际结果。
*   环境—您运行的是什么操作系统和版本？你用的是什么包管理器和库？

# 去哪里寻求帮助

首先也是最重要的一点，你需要真正珍惜和利用人类得到的真正礼物——谷歌。在提问方面，谷歌是你最好的朋友。通常情况下，您遇到的问题可能已经被互联网上的其他人遇到过。如果发生这种情况，你很幸运。你可以简单地借用别人已经收到的编码问题的解决方案。

您可以通过谷歌或直接搜索找到不同的资源:

*   [Stack Exchange](https://stackexchange.com/) —一个丰富的网络，充满了许多社区，最著名的是 Stack Overflow，数百万开发者可以在这里提问和回答问题
*   GitHub——因为大量代码存储在 GitHub 的存储库中，也因为该网站的问题功能，许多问题都在这里被提出和回答。
*   Reddit——Reddit 上有很多事情发生，而且肯定会有一个与你的工作领域相关的子编辑。
*   quora——另一个可以回答各种问题的伟大社区。
