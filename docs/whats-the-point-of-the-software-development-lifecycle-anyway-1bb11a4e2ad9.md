好的——为什么告诉你这个故事？这个故事展示了一个[软件开发生命周期](https://www.techopedia.com/definition/22193/software-development-life-cycle-sdlc)在起作用的例子。如果你从未听说过软件开发生命周期(SDLC)，那么你来对地方了。我们正在讨论 SDLC 以及它对软件团队构建的影响。

# 什么是软件开发生命周期？

软件开发生命周期只是一个团队(或组织)开发软件产品所遵循的过程和工具的列表。很简单，对吧？

我认为生命周期这个术语在这里很重要。虽然软件不是生物，但它展示了生物的许多特征:软件生长和发展，对其环境做出反应，消耗能量，甚至可能展示出动态平衡(负荷削减)和繁殖(扩展)的形式。

这也意味着，像任何生物一样，我们的软件可能会有不健康或濒临死亡的部分。我们的软件可能即将寿终正寝，这可能会改变我们监控/维护软件的方式。在我们的软件中，我们可能会有一些我们必须注意的错误或性能问题。当然，我们有用户——甚至我们自己——想要的新功能。

就像一个花园有一个与季节和里面的植物相联系的生命周期一样，我们的软件也有一个生命周期。我们必须了解我们的软件应该做什么，如何构建它，如何正确地维护它。

# 如何构建软件的指南

最基本的形式是，SDLC 只是一个如何构建我们软件的指南。根据政策的范围和必须遵守政策的群体，这些规则可能非常具体，也可能非常模糊。例如，一个生命周期策略可能会说，“开发人员不应该能够自己编写代码并将其部署到生产环境中。”(稍后将详细介绍为什么这可能是一个规则)。它还可以说，“在产品中发现的任何 bug 都必须在 24 小时内进行记录、分类和优先级排序。”

许多文章可能会说 SDLC 只是一种方法——瀑布、敏捷等等。—然而，SDLC 远不止于此。It *可能会*指定敏捷与瀑布，但也可能不会。或者，它可能会安排额外的安全实践来满足或源代码必须存储在哪里。还有许多其他的东西，但这只是为了表明 SDLC 不仅仅是关于方法的陈述。

每个开发生命周期政策在每个组织中都是不同的。在高层次上，他们可能会解决*至少*以下问题:

*   如何收集工作需求？(需求分析、规划)
*   我们需要构建什么来满足这些需求？(规划、设计)
*   谁做这项工作？(角色和职责，规划)
*   团队是如何完成带给他们的工作的？(方法)
*   工作什么时候完成？(测试，完成的定义)
*   团队如何将完成的工作投入生产？(部署变更控制)

这些问题帮助组织在高层次上决定他们想要如何开发软件。但是，正如你可能注意到的，这些指导方针中有很多都提到了“团队”原因如下。

# 团队有发言权

在许多软件组织中，团队做出许多关于他们如何操作的决策。正如我们前面所讨论的，生命周期的构建通常考虑到了这一点。大多数团队制定他们自己的规则的一种常见方式被称为完成的[定义](https://www.scrum.org/resources/blog/done-understanding-definition-done)。done 的定义帮助个人和团队知道他们什么时候完成了一大块工作。

通常，这将包括对以下问题的回答:

*   我们如何知道工作已经满足了需求？(演示、所有者验收标准)
*   代码存储在哪里？(Github，内部部署)
*   团队如何测试他们的软件？(CI/CD，测试工具)
*   团队如何进行遥测或可观测性？(日志记录、指标、警报)
*   团队如何进行同行评审？(几个？谁啊。)

一个团队通常会有一个非常简单的“完成”的定义。你可以看一看[敏捷联盟指南](https://www.agilealliance.org/glossary/definition-of-done/)。随着你的团队的职责或成员的变化，准备好修改你对“完成”的定义。像敏捷世界中的其他事情一样，期望对你的“完成”的定义做增量的改变。

但是，在继续之前，有一个重要的注意事项:您的团队对完成*的定义通常不应该与您组织的整体 SDLC 相冲突。如果您的组织说所有代码都应该存储在 GitHub 中，那么您对 done 的定义不应该是说将代码存储在某个 SVN 中。如果出于某种原因，您的团队认为有些东西需要偏离 SDLC，请记住，它主要是一个指南。你可以和你的团队领导或经理一起工作来得到一个例外。但是你要准备好有确凿的证据来证明你为什么需要分歧。*

# 合规工具

在许多组织中，拥有书面 SDLC 的最大原因之一是法规遵从性。我不会在这里进入本质的细节。只需知道，当一家上市公司从他们编写的软件中赚钱时，他们有一系列必须遵守的规定。有了 SDLC，企业就可以对审计人员说“这就是我们编写软件的方式”。

当然，企业现在需要扭转局面，表明他们遵循自己的政策。在许多情况下，核心问题是信息安全(InfoSec ),以确保敏感数据得到适当处理，安全性得到适当考虑，以及流氓开发者不能通过过多的访问权限将一小部分钱转入他们自己的银行账户(感谢 *Officespace* )。

由于这些原因，许多组织在 SDLC 中加入了安全一词。当然，这使得术语安全软件开发生命周期(又名 SSDLC)成为可能。我个人更喜欢 SDLC 但是在大多数组织中，它们的意思是一样的。

# 有什么意义？

正如我们已经看到的，SDLC 基于利益相关者与它的交互，服务于许多不同的目的。产品所有者可能会了解他们的产品何时以及如何部署。工程师可能会了解编写代码、测试代码和审查代码的日常行为是如何发生的。团队可以用它来影响他们对 done 的定义，以澄清什么时候可以发布工作。信息安全利益相关者可以使用它来帮助公司遵守行业法规。

将所有这些总结成一句话:*SDLC 的全部目的是为软件工作如何在组织内完成创建一个指南*。像大多数好的导游一样，他们值得跟随。
