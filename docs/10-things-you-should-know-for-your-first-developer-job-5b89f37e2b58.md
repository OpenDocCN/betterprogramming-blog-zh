## 这些建议会让你领先几周

做了两年自由职业后，我又回到了企业界。现在已经一年多了，我学到了很多东西。如果我事先知道的话，会节省我很多时间和精力的事情。

也许你刚刚找到你的第一份软件开发工作，或者已经工作了几个星期。您可能对自己的编码技能和逻辑很有信心，但是对于这个职业，您可能还没有做好准备。

好吧，让我帮你减轻点负担。

根据我的经验，以下是你需要知道的 10 件事:

# 1.学习如何恰当地寻求帮助

相信我，做这件事有对错之分。

当你开始这项工作时，你不可避免地会遇到障碍。你会被卡住的。当你这样做的时候，你的初级开发人员帮不上忙，你必须找到一个中级或高级开发人员来指导。

那么，如何恰当地向他们寻求帮助呢？

让我们从错误的方式开始。

这种错误的方式是说，“嘿，吉姆，我有这个问题，我不知道该怎么办。你能帮我吗？”

这里的问题是什么？

嗯，你还没做功课。你没有给他们工作的背景。你也没有证明你为自己找到答案付出了真正的努力。

每个开发人员都应该首先尝试这项工作，*然后*询问。如果你有，你应该用这种方式接近他们:

“嘿，吉姆，我有这个任务，我似乎遇到了障碍。**下面是我试过的:** X，Y，z，有什么想法吗？”

这是关键。你需要能够展示你自己采取的方法。否则，他们将不得不从头开始解决问题，没有人喜欢这样做。他们会认为你没有充分努力完成这项任务。

关键是告诉他们你已经采取了哪些措施来解决这个问题。通常，你提供的信息会给他们一个提示，他们可以用它来帮助你摆脱困境。

记住，他们和你一样忙，如果不是更忙的话。先做工作。

# 2.编码只是你工作的一部分

我们都有这样的愿景:去工作，坐下来，整天编码。只有你，你的代码，和你的咖啡。

但这从未发生过。(事实上，我认为没有人能坚持下去！)

其他领域也需要你的专业知识。

你看，程序员是头脑风暴者，思想家，建筑师。你会发现自己在计划新项目，头脑风暴新功能，白板，不幸的是，在很多会议中。

因此，当你应该准备好你的编码技能时，一定要发展高层次的思维，准备好为公司做出你的非编程的、智力的贡献。

# 3.不要试图在第一周或第一个月逞英雄

你可能会在第一天表现得超级自信。实际上，你可能是一名出色的开发人员，能为公司做出很多贡献。你可能想炫耀你的技能，做一些事情来证明你是一个好雇员。

不要。

不要跳进去开始为重大的事情做志愿者。

当有人说，“嘿，团队，我们需要有人将所有这些数据迁移到这个其他环境中，还有这样那样的，”这是你的第一天，请**而不是**举手。袖手旁观，让别人去做吧。

你的时代会到来。你很快就能证明自己的价值。但不是在这个时候。

不要逞英雄，否则从一开始就有可能把事情搞砸。

我同事干的。他在第三天意外删除了 Azure 中的整个资源组及其内容。

从那以后，他有了一个滑稽的绰号。

卑微自己。

# 4.确保您对使用 Git 感到舒适

几乎所有的软件公司都使用版本控制，更多的时候是 git。

我保证从一开始就假设你知道它。

不管你受雇用什么语言编程，它们可能会给你一点灵活性，一点热身的时间。有时他们甚至会训练你。

好吧，有了 git 这就不会发生了。人们很可能认为你知道这件事。(如果你没有，那么你可以用你的超能力让他们惊叹。)

此外，确保你知道如何在团队环境和最佳实践中使用它，比如在推动之前提取最新的变化。

# 5.总是给自己更多的时间来完成一项任务，比你认为需要的时间要多

从事自由职业，你的预算通常会更少。这些较小的预算让你自然而然地觉得你需要更快地工作。你感到时间紧迫。(我的意思是，谁愿意免费工作？)

在企业环境中，预算通常要大得多。你有更多的时间来完成工作。

从一开始就注意这个建议:在给你的任务估算时间时，不要逞英雄。

事情永远不会像你想象的那样顺利。事情往往会在最后一分钟出问题，可能会召开一个意想不到的会议，或者系统崩溃。

养成一个习惯，给自己比你实际认为需要的时间更多的估计时间。给自己一些衬垫。如果你觉得你可能会在一天结束前完成，告诉他们你会在周末准备好。当它早点准备好的时候，我们一起庆祝！

# 6.事先熟悉敏捷/Scrum 方法论

Scrum 是管理复杂任务的敏捷过程。

你有这个团队。你有需要完成的任务。Scrum 是一种非常流行的管理方式。

你可能已经听说过它，或者至少听说过它的术语:每日站立、冲刺周期、梳理会议、回顾、pbi 等等。

简而言之，你有一个冲刺周期。假设是两周。它从 sprint 计划开始，任务被分配给开发人员，目标是在周期结束时完成这些任务。

每天你都有一个站立会议，每个人都单独报告他们的进展，并且有时间指出障碍、问题等。

你不需要成为这方面的专家——只需要获得高层次的理解，并从第一天起就准备好加入(可能需要介绍)。

# 7.如果有人比你知道如何做得更好，这并不意味着你应该去问他们如何做

这类似于上面的第二条。

如果你被分配了一项任务，并且你知道其他开发人员更擅长这项工作或者可以用一半的时间完成这项任务，这并不一定意味着你应该向他们寻求帮助或者让他们去做。

想象一下你被分配了一项任务。公司的另一个开发人员实际上构建了这个特性，但是现在你被要求以某种方式修改它。

你可能认为最好走过去，让他们给你解释，甚至让他们自己做。我的意思是，他们可以更快地完成，因为他们更好地理解它，对不对？

不对。任务已经分配给你了，这实际上是一个很好的机会，让你快速熟悉应用程序中你不熟悉的新部分。

承担努力。向前冲。为你的技能增加一项新技能。

# 8.总是提前询问有关新功能或任务的问题

这个我很纠结。

假设我正在开会，有人说，“嘿，Travis，一个客户需要对他们网站的导航栏进行更改。它需要看起来像这样那样。你能做到吗？”

出于某种原因，如果对我有意义，我会去比赛。我想我不需要更多的细节，我开始编码了。

这是一个**巨大的**问题。

优秀的开发人员会问这样的问题:

“好吧，他们为什么想要那个功能？”
“他们认为这个特性会给他们的业务或网站带来什么好处？”

或者如果这不是一个好主意，他们可能会说:

“你知道，我认为那样做会把 CTA 按钮弄混。也许把它放在这里更好。”

优秀的开发人员会提出问题以引出更多的细节和方向，优秀的开发人员也会在看到不好的实践时进行反击。

这又回到了上面的#1，编码只是工作的一部分。

在开始之前，清楚地知道到底需要做什么总是好的。

# 9.不要在下班时间发邮件或发送无聊的信息

人们对他们的手机有懈怠，通知最有可能被打开。

如果你有一个好主意，而且已经是晚上 9 点了，不要把它发出去，不要在你的同事给他们的孩子读故事让他们睡觉的时候打扰他们。等到早上。

人们有工作之外的生活。

我知道这不是一个真正受欢迎的规则，许多人不在乎，但我会鼓励你保持团队或懈怠通知只在工作时间流动。人们会欣赏它的。

# 10.负责任

学会对分配给你的任务或你被要求领导的团队完全负责。

如果出了问题，就要承担责任。不要责怪别人的工作。不要责怪你能控制的因素。

这建立了诚信。它会让同事认为你是可靠的、负责任的。人们可以信任你，相信你能把事情做好。

不要成为那个总是责怪别人的开发者。从来都不是好结果。

# 结论

好了，这 10 件重要的事情可能会帮助你适应新的软件角色。
