# 对新事物的热情怎么会让人担心呢？

这感觉像是一个悖论。当我们的程序员对新事物充满动力、积极和兴奋时，为什么要担心呢？

因为对新的解决方案、方法、技术或模式的兴奋会导致人们忽略旧的解决方案，并声称“XPTO.js 是正确的选择。我们现在就应该把整个平台迁移到这个新框架上！”

第一次出现这种情况时，您可能会想:“好吧，我们会花几个月的时间进行迁移，但最终我们会得到一个更好、更新的产品。”在六个月除了迁移什么都不做之后，您最终回到了关注与业务相关的特性上。

一年后，一个全新的、独特的、革命性的、前沿的数据库引擎出现了。开发者们窃窃私语说这是未来，如果我们不从 *QSL* 转变为*萨坎德拉*，我们就会过时。

但是这一次，迁移所有的数据会带来更多的风险。你决定冒这个险。但是你的团队花了一年多的时间来完成迁移，而不是六个月。这两种技术在范式上有一些不同。互联网上几乎没有关于它的信息，你的团队中也没有关于它的知识。因此，您的客户支持变得更差，因为工程团队专注于迁移到新技术，而不是为维持业务的客户服务。

# 做有意识的决定！

每个框架/解决方案/模式/技术都是为解决特定问题而设计的。句号。这并不会使他们比其他人更好——他们只是更好地解决了给定的问题。

作为一名专业人士，你应该经常问自己解决问题的最佳方案是什么。答案可能是最近最热门的微服务，也可能是过时的、不受欢迎的整体服务。只是看你需要解决的问题。

***世上没有银弹。***

你永远不会找到解决所有问题的方法。它根本不存在。每个解决方案都有优缺点。当你建议某人使用特定的解决方案时，确保你将问题与最合适的解决方案相匹配。

# 如何发现炒作驱动的开发者

请记住，通常，炒作驱动的开发人员不一定总是被炒作驱动。并不总是硬盘。通常，它发生在个人感兴趣的话题上。

你可以问自己几个问题，以帮助你认识到你是在炒作的基础上做决定的:

*   您知道新解决方案带来的利弊吗？
*   你知道它被设计用来解决的具体问题吗？
*   在解决与您类似的问题时，该解决方案是否证明有效？
*   您是否知道新解决方案将带来的实际好处，以及需要多长时间才能获得迁移投资回报？

如果你对这些问题的回答是否定的，你可能是在做一个感性的决定，而不是理性的决定。在这种情况下，我建议您与您的团队会面，公开迁移的利弊，并检查它是否有回报。
