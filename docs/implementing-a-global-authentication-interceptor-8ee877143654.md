在本文中，您将学习如何实现一个全局身份验证拦截器。这个拦截器的工作是当您收到未经授权的响应时刷新访问令牌。

# 介绍

将你的服务器升级到 SSL 并不一定意味着你在野外是安全的。您的服务器需要确认您在没有任何黑客拦截的情况下与用户通信(通常被称为中间人攻击)。

快速注意:在大多数情况下，使用证书锁定功能可以避免中间人攻击。你可以从这篇[文章](https://medium.com/better-programming/secure-communication-with-the-server-from-your-android-client-with-certificate-pinning-5f53cea55972)中了解什么是证书钉住，以及如何在 Android 中实现。

虽然我们有证书锁定，但最好有某种用户验证。有不同的方法来验证您正在与实际用户通信，其中之一是令牌验证。它可能是标准的 JWT 令牌，也可能是您自定义的逻辑。

如果我们在高层次上看到这个令牌验证，它将是这样的:首先，如果响应状态代码是 401，这意味着未经授权，它可能是一个黑客拦截，或者实际的令牌已过期，您将执行一个常规的服务调用。在这种情况下，我们必须请求新的访问令牌。如果生成了令牌，那么我们必须重新初始化原始请求流。如果没有，我们应该注销用户，假设连接被破坏。

# 拦截机

理论部分讲够了，让我们从什么是拦截器开始。拦截器是一种强大的机制，可以监控、重写和重试服务调用。这里有一个简单的拦截器，它记录传出的请求和传入的响应。

使用 OkHttp3 库的简单日志拦截器

`addInterceptor`是我们可以向请求构建器 OkHttp builder 添加拦截器的函数。然后我们创建了一个拦截器的新实例，在其中我们获得了作为输入的`interceptor.chain`。

从这个链中，我们可以得到`request`对象，它包含了关于`request`主体、标题等等的详细信息。一旦请求完成，我们就可以访问响应。

当在`chain`实例上调用带有`request`对象的`proceed`函数时，实际执行将开始。`proceed`函数的输出是来自服务器的响应，包括附加信息，如状态代码和其他东西。

# 认证拦截器

既然我们已经了解了如何创建拦截器及其工作原理，那么让我们开始构建身份验证拦截器。

这个拦截器的目标是当原始请求发出未经授权的访问时，调用访问令牌功能的刷新。成功检索到刷新后的令牌后，我们需要实例化请求。看一看:

认证拦截器逻辑

在第一步中，我们创建了原始的`request`对象，如果我们得到了未授权的访问，就重新初始化它。在下一步中，我们用请求对象调用了链上的`proceed`函数。

此时，请求开始执行。一旦请求完成，我们将得到响应，我们需要做的第一件事是通过比较状态代码来验证请求是否成功。如果响应中的状态代码是 200，那么请求成功执行。

如果请求成功，那么我们必须直接返回响应。如果不是，我们需要使用状态代码检查错误类型。如果状态码是 401，那么就是未授权访问；同样，如果您希望处理其他错误，我们需要在`else`块中实现它。

如果原始请求响应未经授权，那么我们需要创建一个新的请求来调用`getAccessToken`服务调用。一旦我们首先获得响应，我们需要保存最新的访问令牌以备将来使用。

之后，我们需要更新原始请求对象中的最新令牌，然后重新发起请求。
