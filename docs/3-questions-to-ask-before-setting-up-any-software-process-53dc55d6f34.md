## 不要仅仅为了“改进”而建立一个新的流程

由[路](https://unsplash.com?utm_source=medium&utm_medium=referral)上[车头](https://unsplash.com/@headwayio?utm_source=medium&utm_medium=referral)拍摄

每当我们在开发过程中面对新的东西时，我们有时会不小心掉进一个陷阱，也就是说，进行另一个过程。

不要误解我。流程很重要。没有他们，就会出现混乱。但是吃太多会有反作用。

因此，在引入任何流程之前，我们必须仔细思考。以下三个问题可能会帮助我们三思而后行。

# 1.我们真正想要的是什么？

在单口相声之前，除了每周一次的会议，我们写了一份每周状态报告。我严格遵循了流程。我每周都写一篇散文式的报告。我有长的和短的摘要，中间有链接。我每周花大约一个小时在这上面。我没有检查什么是重要的，什么是不需要的。**吸取的教训——当你不知道时就问。**

当我被赋予软件发布质量保证的职责时，我问，“在发布软件之前，我们如何确定软件的质量？”简单，让我们实施这个新规则。100%的测试覆盖率！这有点道理，但在发布后，我们仍然有错误报告。我们已经有了过程，但是我们仍然失去了这样做的目的。**经验教训——了解意图，而不仅仅是步骤。**

为了了解我们软件发布的稳定性，我们引入了一个每月记录软件发布稳定性的流程，并在每次发布时对其进行监控。整个想法是，我们可以看到它的趋势随着时间的推移。这种情况已经持续了几年。由于一些角色的改变，这些步骤错过了几个月。没人注意到。**吸取的教训——我们会用到它吗？**

# 2.会引起其他问题吗？

在任何软件发布之前，我们都会进行一些手动的健全性检查。但是有一样东西看不见，那就是分析报告。在其中一个版本中，我们发现我们不再得到任何分析报告，这是一个主要的发布后错误。马上，有人建议，让我们在预发布的手工健全检查工作中再增加一个步骤。由此产生了两个问题:

1.  后来，当发现新的问题时，提议添加另一个手动健全性检查成为了简单的解决方法。**经验教训——我们是否在正确的地方解决了问题？**
2.  随着我们向预发布测试添加越来越多的步骤，预发布测试成为一项艰巨的任务。不可扩展。**经验教训——可持续发展吗？**

在一个时间点上，当工作在一张新的卡片上时，什么都没有被记录。这通常会导致开发人员在开发时产生很多问题。不正确的假设，甚至完成了实施。为了解决这个问题，我们引入了对适当文档的需求。创建了一个又长又大的文档模板。除非文件已经写好，否则任何卡片都不能使用。这包括错误修复。

最终，开发人员不愿意报告需要修复的新的小错误，因为他们必须写一长串文档。编写文档需要更多的努力，而不仅仅是修复 bug。吸取的教训——我们是否应用得太严格了？

我在一家跨国公司工作。在那个时候，软件发布是一个主要的过程。软件的最终发布需要总监级别的批准，包括补丁。董事们都在美国。我在亚洲。有一天，我们的中国客户遇到了一个严重的问题，需要立即给我们的软件打补丁，否则他们的生产线就会停工。

从技术上讲，我们可以发布软件，但我们需要一个主管来批准它。当时是美国的午夜，所以我在午夜打电话给我的老板，请求他的批准。他告诉我，“*非常时期需要非常措施。*“我应该事先做好决定。**经验教训——我们有特殊时期的警告吗？**

# 3.参与其中的每个人都获得了什么？

曾经，作为一名经理，我需要衡量团队的生产力。我要求每个开发人员尽可能准确地定期记录他们的工作，并将它们合并到一个电子表格中。从中，我创造了一个计算一切的公式。

开发人员不太高兴，但是我引用“*说服了他们，如果你不能度量它，你就不能改进它。除了管理层可以“衡量”他们的表现之外，开发人员从这样做中一无所获。总之，最终一切都被废弃了。**经验教训——如果参与过程的人没有从中获益，那就不要做。***

为了确保我们的软件系统是健康的，我们每个季度都遵循不同的标准来评估我们系统的健康状况。其中一个标准是确保我们的 Android 系统中没有旧语言，也就是说，Android 系统应该只有 Kotlin 代码而没有 Java 代码。

目前，有一个系统有 Java 代码，但是没有添加新的 Java 代码。系统所有者争论说，因为没有添加新的 Java 代码，所以说系统不健康是不公平的。现有的都是旧码。合理。对团队来说，他们认为系统不健康是没有好处的。

然而，我们认为这个系统不健康的原因并不是为了惩罚或诋毁团队。相反，它可以用来表示我们应该给团队时间来处理它。在知道目的是获得时间来处理和删除旧的 Java 代码的情况下，团队遵循这个过程没有问题。吸取的经验教训——始终考虑对团队有什么好处。

# 这个过程是为了帮助，而不是发号施令

我已经从一个厌恶过程的立场走到了盲目引入严格过程的另一个极端。两种方式都不对。考虑流程的关键总是从这些问题开始:

1.  符合意图吗？
2.  会引起其他问题吗？
3.  这对团队有什么好处？

有时，不需要新的流程，特别是对于一次性或非定期发生的异常问题。如果维护一个过程的成本超过了它所带来的价值，那就忘了它吧。这是软件开发。我们应该允许某种形式的灵活性。要灵活。

归根结底，这个过程是为了帮助，而不是发号施令。掌握它。
