谈到编程语言，人们经常会提到速度。您总是听说跨多种语言的性能比较；哪个是最快的编程语言；Python 比 Java 慢，等等。这些陈述的唯一问题是它们实际上毫无意义。我来解释一下原因。

# 什么是编程语言，真的？

为了理解为什么比较各种编程语言的运行时性能是愚蠢的，我们需要澄清编程语言实际上是什么。

就像任何其他类型的语言一样，编程语言是一种用于交流信息的工具，在这种情况下，它的目的是让人类向计算机发出指令以供执行。

现在，你会同意像英语这样的口语——包括它的语法、句法和词汇——是一个抽象的概念，不是吗？你摸不到英语，也闻不到或尝不到它。然而，你可以争辩说，你可以看到和阅读英语文本，或者你确实可以听到英语单词。这使语言具体化了吗？

其实你读到的，听到的，都是英语的一种实现。你使用抽象的语言规则将信息嵌入到具体的媒介中，不管是声波还是纸上的墨水。

既然我们已经阐明了语言和它们的实现之间的区别，让我们回到编程语言并应用这些概念。就像你不会说英语比意大利语快一样，宣称 C 比 Python 快也是扯淡。

相反，可以比较的是语言实现。一个人说通常能比写更快地传达同样的信息。编程语言也是如此:直接执行二进制文件比解释其高级代码更快。

# 编程语言是如何实现的

既然我们已经提出了自然语言实现的概念作为类比，现在是时候讨论编程语言的具体实现了:认识一下编译器和解释器。

你可能已经知道，你的高级源代码必须在 CPU 执行之前被翻译成机器码。这种翻译是通过应用语言规则来重塑信息，同时在理想情况下保持相同的含义来实现的，就像从英语翻译成意大利语一样。

对语言速度的误解就是从这里开始的，当人们把默认实现误认为是语言本身时。你经常听说 Python 是解释型的，C 是编译型的，Java 是介于两者之间的，等等。然而，这就和声称英语本质上是口语一样，排除了它的书面形式，甚至是携带英语信息的英语思想或电信号。

关于编程语言，确实 Python 通常是通过解释器来实现: [CPython](https://en.wikipedia.org/wiki/CPython) ，而 Java 一般是由 [javac](https://en.wikipedia.org/wiki/Javac) 编译成[字节码](https://en.wikipedia.org/wiki/Java_bytecode)，然后由 [Java 虚拟机](https://en.wikipedia.org/wiki/Java_virtual_machine) (JVM)来解释。这些都是开发者向计算机传达指令的方式。

# 不同的语言实现

现在，一种媒介确实比其他媒介更有表现力。然而，语言并不局限于任何单一的交流媒介，就像英语并不是唯一的交流媒介一样。

其实你可以开发一个 Python 编译器或者 C++源代码解释器(已经有了，btw。)并使用不同的实现。这会让 C++比 Python 慢吗？绝对不是，那只是意味着一个实现比另一个更有效。

此外，无论底层的底层实现如何，您的代码都会极大地影响性能。例如，让我们假设 JavaScript 的`console.log()`能够比 C 的`printf()`函数更快地将字符串打印到控制台。这可能告诉我们，真正低效的是那个特定的函数，而不是解释器/编译器本身。

这也是为什么具有相同含义的代码片段的性能基准可能毫无意义。为了演示这一点，让我们看看 Python 中遍历列表的不同方式:

基准测试结果:

```
Foreach loop:       0.012268301301810424 seconds
While loop:         0.0842185921021155 seconds
For range loop:     0.044092743896180765 seconds
List comprehension: 0.01865486839815276 secondsThe fastest method is Foreach loop with 0.012268301301810424 seconds
```

正如您可以清楚地看到的，如果您要在不同的编程语言之间执行可靠的基准测试，您还应该考虑这样一个事实，即有更多和更少的有效方法来实现相同的结果。正因为如此，不同编程语言之间的性能比较很可能会有偏差，而且可能没有那么大的意义。

除了这些方面，另一个至关重要且不可忽视的性能因素是开发人员编写高效代码的技能。

# 结论

总结一下，我们已经阐明了语言和它们的实现之间的区别。然后我们具体讨论了编程语言实现以及它们如何影响代码的执行速度。最后，我们讨论了实现相同目标的代码片段如何比其他代码片段表现得更好，这应该是基准测试时需要关注的一个问题。

最快编程语言的神话只是一个常见的误解，如果你仔细想想，它真的毫无意义。我认为是时候根除这种错误的信念并带来清晰，尤其是对那些更有可能保持这种神话的初级程序员。
