# 为什么开发者讨厌编程语言？

## 为什么开发人员在比较语言时会出错

![](img/4f01ef31abebe608d841c6475c8c2c50.png)

[张杰瑞](https://unsplash.com/@z734923105?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍照

我最近发现了 Go 语言中的一些缺点、缺失和错误(这里是)。我从事围棋工作已经有几年了，我总是喜欢关注新的话题。看了一下这个集合，我发现了其他开发者对该语言的有趣观点。我总是发现令人惊讶的是 Go 设计背后的主要方面(没有 OOP，简单，缺乏继承等。)被认为是设计错误和缺失的功能。读到这一切，脑海中浮现出十几年前我读到的关于 Python 的第一句评论:“它很慢”“没有括号你在开玩笑吧？”“它是非类型化的”等等。不用说，Python 是最通用的语言之一。

有时这些评论揭示了对语言设计背后的原因缺乏理解。更糟糕的是，他们表现出某种非理性的仇恨。我想就这个话题发表我的意见。

# 比较编程语言

在比较编程语言时，我观察到大多数作者(不是学术作者)遵循某种模式。取一对语言，X 和 Y，在下列句子中替换它们:

> 与 Y 相比，X 更慢，性能更差。
> 与 Y 相比，X 管理数据结构的方式是一种耻辱。
> 与 Y 相比，X 管理依赖关系的方式糟糕透顶。
> 尽管 X 比 Y 好，但就 Z 而言，X 不能做这做那…

你明白了。给定任何一种语言 X，我们都可以找到另一种语言 Y，它的表现优于 X。像人类语言一样，编程语言帮助我们清晰地表达对世界的看法。语言的设计考虑到了特定的特性。很难用编程语言精确地表达我们对世界的感知([见这里](https://www.mortylefkoe.com/how-our-language-determines-our-reality/))。而这让我想到了开发者世界里“编程文化”的缺失。我发现开发人员生活在一种自我强加的语言教条中。

# 与你精通的语言进行单项比较

在互联网上，你可以找到大量的语言对比。比如搜索“Go vs Rust”，你会发现成千上万的对比。这个[例子](https://www.infoworld.com/article/3436960/rust-vs-go-how-to-choose.html)是两种语言的通用视图——我发现它很肤浅，但是没问题。对于任何好奇的开发人员来说，从通用的角度比较两种语言都是一个很好的起点。然而，你可以找到其他的比较，比如[这个](http://www.yager.io/programming/go.html)，很明显作者对其他语言感觉更舒服。这一事实导致作者指出了 X 语言与其他语言的共同特征相比的劣势。

# 忽略语言设计者的动机

定义一门编程语言不是一件容易的事情。通常情况下，*语言旨在改进其他现有语言的某些方面或改进某项任务。* Python 旨在取代 ABC 语言。Java 和它的虚拟机被设计成在任何地方运行相同的代码，而不需要额外的编译。创建 Erlang 是为了改进电话应用程序。我们不能期望这些语言是绝对适用于所有事情的解决方案。比较两种为不同目标设计的执行相同任务的语言多少有些偏见和不公平。

想想 Java。它自 1995 年以来就存在了，并且可以在与其最初目的无关的大型平台中找到。这可能是开发者讨厌它的主要原因之一。如今，用 Java 开发一个 API REST 比任何年轻的语言都复杂。然而，我们忘记了 Java 虚拟机是如何开发出一系列奇妙的解决方案的，比如 Android。

# **开发人员不愿意学习新语言**

开发人员是必须在有限的时间内完成任务的人。最好的情况是，他们使用自己熟悉的编程语言。这很有道理。你对一门语言越熟练，完成一项任务所需的时间就越短。然而，我们天生懒惰。众所周知，学习新语言会改变我们大脑的结构。*开发者不愿意学习其他语言。这需要额外的努力，你会感到痛苦，直到你达到一定的熟练程度。这就是为什么开发人员会批评他们不熟悉的语言。*

# 炒作语言的问题

营销是一个强大的工具，编程语言也不例外。时不时地，每个人都想学习某种热门语言。对这些被炒作的语言的任何批评都是不尊重的，即使其反对者对语言本身没有深刻的经验。

# 贬低他人的努力

这来自于人的心理。当一种新的语言呈现给开发人员时，他们会马上说:

> 我不明白他们为什么在这上面浪费时间。在 X 语言中，你可以用三行代码来完成。

这是一种通过贬低他人努力来保护自己信念的方法。这完全是人类。然而，重申这种想法是无知的表现，也是对其他开发者的不尊重。不能用这种推理来抛弃一种语言。

# 语言崇拜者

互联网似乎创造了静态的人群。爱贾斯汀比伯的人和恨他的人。如果你不在团队里，你就是局外人。这同样适用于编程语言。*一些开发者似乎持有一种教条，认为语言是神圣的，不允许批评。*这太荒谬了。

# 寻找最好的编程语言

软件开发不是一件容易的事情。技术在高速发展。过时的感觉很普遍，你很容易在 30 多岁时感觉自己像只恐龙。你能得到的避免这种情况的最天真的建议是:超越你的舒适区，永远不要停止学习，尝试新的语言。你已经认识他们了。然而，我认为这是一个更深刻的问题。对于是什么让编程语言变得伟大，有一种普遍的误解。

我们来思考一下下面的情况。一家公司的架构师决定使用语言 X 来实现系统 s。架构师确信 X 的许多特性将有助于 s 的开发。不幸的是，开发团队中没有人熟悉或精通 X。对此我们能有什么期望？只有问题。X 是 S 的解吗？大概是吧。但是，开发人员可能已经尽了最大努力来实现 S，用语言 Y 得到一个 MVP，然后反复迭代。在这种情况下，编程语言不是最终的解决方案。

前面的情况就是一个简单的例子。您可以找到许多其他场景，在这些场景中，语言的选择可能不是基于特性、性能或学习曲线。正是因为这个原因，基于这些因素的比较必须谨慎。这些比较只有在给定的开发环境中才有意义。因此，我们可以宣称

> 最好的编程语言是允许我们以最有效的方式执行任务的语言。

我知道要有一个完整的声明，我们应该定义什么是有效的，但这超出了本文的范围。我们会说，效率可能是指开发限制，如预算、开发人员的经验、分配的时间等。

# 摘要

开发人员倾向于减少他们不熟悉的编程语言。有时这种批评忽略了编程语言旨在解决的问题，使开发人员成为“编程语言教条”的成员我们都应该谦虚地做一个练习，来理解我们这些日子拥有的丰富的语言。我们需要认识到某些语言在抽象某些问题时比其他语言更合适。希望这可以帮助我们所有人采用一种开放的开发文化，最终将帮助我们减少错误信息并改进新思想和解决方案的传播。
