# 程序员不测试代码的 9 个借口

> 原文：<https://betterprogramming.pub/9-excuses-why-programmers-dont-test-their-code-8860a616b1b5>

## 一系列我们都能理解的借口

![](img/cc9eea1b57f220c4a24a0b99aef92557.png)

法比安·格罗斯在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片

程序员和测试——这绝对不是天造地设的一对。尽管大多数程序员理解测试的重要性以及它为什么有用，但有时我们就是不喜欢它。

有很多借口总是被用来避免一件事:写测试。

你曾经用过这份清单上的任何借口吗？

# 1.“我的代码运行良好——我为什么还要费心测试它呢？”

这种说法往往是趾高气扬的开发者说的。认为自己能写出完美的代码是天真的想法。即使是最好的程序员也会犯错。他们一直都在做。

没有人是完美的——错误发生的次数比他们应该发生的次数多得多，这实际上完全没问题，因为我们从这些错误中学习。

对你的代码有信心是可以的。但是，即使另一个开发人员审查了您的代码并批准了它，您也不应该感到满意……还没有。即使你已经手工测试了一些用例。所有这些事情仍然存在危险。

所有这些都依赖于人的因素。这正是你不应该想要的，因为人的因素有时会失败。当这种情况发生时，你需要进行适当的测试。

> “除了我之外，没有人能第一次就创造出完美的代码。但我只有一个。”
> 
> 莱纳斯·托沃兹

# 2.“这段代码不可测试”

你的意思是你真的不知道如何测试那段代码。很可能你的代码结构不良，这使得它不可测试。如果只是一段代码，就把它重构为小的可测试的部分。

当你处理那些因为设计不良而不可测试的代码时，你会遇到一些更大的问题。维护和修改不可测试代码的成本很高。最重要的是，不可测试的代码对于新团队成员来说很难学习。

如果你不测试你的代码，你最终会遇到开发人员缺乏信心部署代码的问题。缺乏测试让他们焦虑，因为他们无法确认他们的更改没有破坏任何东西。

# 3.“不知道考什么”

你对测试什么有疑问吗？全部测试！

一个好的开始是尽可能测试最常见的情况。这将告诉您代码的这一部分何时中断，这将产生最大的影响。

但是测试不仅仅是点击快乐路径场景。您应该测试代码中最复杂的部分或最有可能出错的部分的一些边缘情况。

每当你发现一个 bug，编写一个新的测试来覆盖这个测试用例被认为是一个好的实践。

让写作测试成为你日常工作的一部分，不要让它成为可有可无的。

# 4.“测试增加了开发时间，我们的时间不多了”

这是测试中最大的误解之一。每当您第一次开始编写测试时，它肯定会增加开发时间。首先，您需要学习如何编写可测试的代码，以及如何进行适当的测试。这是一项长期投资。

一旦测试成为一种习惯，它会节省你大量的时间。还有头痛。进行测试将会为您解决开发人员缺乏部署他们代码的信心的问题——当然，尽管您有能力通过点击一个按钮来运行您的整个测试套件。

# 5.“要求不好”

糟糕的需求是*而不是*跳过编写测试的借口。你有足够的选择来弄清楚需求。有时候会变得更糟。遗憾的是，未记录的需求是一个现实，我们大多数人面对的次数比我们希望的要多。

在这两种情况下，您都必须使用您能得到的任何一点点文档。试着翻翻旧邮件或你做的笔记，试着找到一些线索。

当然，如果可能的话，您还想看看应用程序的当前或旧版本。你可能会在这里发现很多隐藏的线索。别忘了看看测试。他们可能充满了隐藏的宝石，可以帮助你。

最后但同样重要的是，尝试与一些团队成员交谈。他们可能更了解您正在寻找的未记录的需求。想一想在那次你不能参加的会议中讨论过但没有记录下来的一件事。

# 6.“这段代码不会改变”

“测试不会改变的代码是没有意义的。那完全是浪费时间。”

嗯，我们在软件开发中唯一确定的是需求会改变。有时它们似乎千变万化。

人们改变他们的想法有很多原因，他们经常这样做。需求的变化可能是由许多因素引起的。一些人请求一个特性，但是他们实际上不知道他们想要什么。不幸的是，这种情况经常发生。需求可能改变的另一个原因是组织内部的政治。

每当您处理变化的需求时，您应该优先测试最常见的流程。

# 7.“如果手动测试，我可以更快地测试这种方式”

手动测试的问题是你必须一遍又一遍地做。再一次。

是的，当您手动测试一次时，您可能会更快地测试某个特性。但是当你不得不一遍又一遍地做手工测试时，手工测试是非常耗时的。手工测试是无止境的，而且非常昂贵。

编写测试需要一些时间，但是一旦你得到了你的测试，你就可以无限的运行它，并且与手工测试相比，它只需要很少的时间。运行自动化测试比打开浏览器并键入页面地址花费的时间要少。

# 8.“客户只想为可交付成果付费”

每个客户都希望看到尽可能多的可交付成果，因为这给他们一种开发团队正在做一些实际工作的感觉。在可能的情况下，他们希望尽可能降低成本。

每当开发人员不想测试时，他们就会这样告诉他们的经理。

测试是软件开发的重要组成部分，这部分代码和其他代码一样重要——所以你应该这样对待它。在您所做的评估中包括编写测试的时间。

从长远来看，编写测试可以降低维护成本和开发成本。

# 9.“这段代码很小…不会破坏任何东西”

每个开发人员都写过非常小的代码片段，以至于他们不可能破坏任何重要的东西。然而，它仍然设法打破了一些东西。你添加的两行代码设法破坏了你没有预见到的东西。

你怎么知道你的代码能完美地工作？

请让你的话得到一些实际测试的支持。彻底的测试过滤掉关键的错误，确保代码按照预期的方式工作。

# 包装它

你能认同这份清单上的任何一个借口吗？

我想我们以前都用过这个。下一次，我们可能应该只考虑编写测试，因为我们都知道不测试我们的代码会在以后的某个时候反噬我们。
