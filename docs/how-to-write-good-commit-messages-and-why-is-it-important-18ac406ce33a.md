许多开发人员在工作中学习 Git，他们可能没有意识到最重要的事情！事实上，你们中的大多数人低估了提交消息的作用，很少花时间来描述你所做的事情。尽管如此，这是一个应该一劳永逸解决的小问题！

# 什么是提交消息？

在 Git 中暂存之后，`[git commit](https://www.w3docs.com/learn-git/git-commit.html)`命令将我们的更改保存到本地存储库中。在保存更改之前，您应该告诉 Git 您想要保存哪些更改，因为您可能已经进行了大量编辑。一个很好的方法是添加一条提交消息来标识您的更改。粗略地说，提交消息是您对每个变更背后的内容和原因的注释。

# **为什么提交消息很重要？**

如果你的 [Git](http://git) 历史只有你一个人看到，你可以为所欲为。一旦多个贡献者在一个大项目上工作，这种做法对你来说有些无知和不负责任。提交消息的主要重要性在于它对将来的某个人有用。如果你想解决这个问题，但不知道如何或应该采取什么步骤，或者你意识到这个话题的重要性，并想结束它，那么这篇文章就是为你写的！

# **好的提交消息应该是什么样的？**

在这里，我们强调了编写正确提交消息的一些通用规则。如果你遵循其中的一些，你会改善你的工作流程。

## 编写提交消息需要一些时间

你们中的一些人匆忙地写了一条信息，以便完成你们的日常职责并继续前进。运行`[git log](https://www.w3docs.com/learn-git/git-log.html)`命令，查看您过去的提交消息。看着这些信息，你可能会说，“这简直是胡说八道！”您应该只花一分钟的时间来编写适当的提交消息。

## 提交消息应该包括什么和为什么

Git 提交消息必须包含关于您做了什么以及为什么要做每一项更改的信息。如果提交消息不包含这些信息，那么编写它又有什么意义呢？

## 对相关提交中的更改进行分组

更可取的做法是将事物分开，并以符合逻辑的方式将它们组合在一起。由于您的提交显示了您工作的进展，您可以创建一个历史，这将在将来对您有所帮助。

## 使用一致的术语和标记

设置一个共享术语是强制性的，以避免进一步的不便，同时也使在 [Git 库](https://www.w3docs.com/learn-git/git-repository.html)中搜索变得更容易。标准标记格式也是优选的。

# **编写正确提交消息的步骤**

运行带有`-m`选项的`git commit`来编写提交消息:

```
git commit -m “Subject” -m “Description”
```

第一个`-m`选项是标题，下一个是正文。您还可以检查如何[更改提交消息](https://www.w3docs.com/snippets/git/how-to-change-commit-message.html)。

如您所见，提交消息由标题和主体组成。

## 标题

标题应该与正文分离。它总结了您在提交中所做的更改。它应该以大写字母开头，长度不超过 50 个字符，结尾不带标点符号。

你应该像这样正确地完成句子:“这个提交将…”

## 身体

正文是对变化的简要总结。如果你只是添加了一些小的改动，并且你的信息可以通过标题来传达，那么你可以跳过这一部分。

一般来说，身体应该解释你的改变是什么和为什么——而不是如何改变。这一部分用于变化的部分对其他人来说不太明显的情况。

# 结论

> "伟大的事情不是由冲动完成的，而是由一系列小事汇集而成的."乔治·艾略特

小事总是重要的！如果你想成为一个熟练的开发人员，你应该注意那些次要的事情，并使之成为你的日常习惯。提交消息充分地呈现了关于为什么进行更改的信息，正确地理解它们可以使开发和协作更加有效。在您的日常工作流程中，应该考虑到提交消息的重要作用。
