## 白板问题烂透了。为了评估你的工程候选人，你必须亲自动手。

![](img/fe2576014ddbac6c260ea46841e9cfe0.png)

约翰·施诺布里奇在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的照片

我讨厌面试。

实际上…我收回那句话。我*喜欢*面试。我就是讨厌*被*采访。

像大多数软件开发人员一样，我可以数清自己*实际上*不得不反转一个链表，或者在一个工作环境中徒手遍历一棵二叉树的次数。但是我已经完全记不清我被要求在白板上做这些事情的次数了。

这是一个棘手的问题，虽然面试官可能会从他们那里得到一些价值——假设他们的目标是评估候选人的技能，而不仅仅是炫耀自己的知识——但这会让面试者不清楚在相关公司工作的实际情况。

因为这是很多公司在招聘过程中忘记的事情:这是一条双行道。正如你在评估每一位候选人是否适合你的组织一样，他们也在评估你(T21)的同样的事情，这就是为什么以一种互利的方式组织你的面试过程是至关重要的。

太好了，我能听到你在想，我只有几个小时去了解这个人，我该怎么做？

说白了:经过深思熟虑的策划和执行。

# 正确评估候选人

![](img/aa4f154c751b1953a07dd71802927137.png)

照片由[克里斯蒂娜@ wocintechchat.com](https://unsplash.com/@wocintechchat?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄

虽然我热衷于创建面试“小册子”来发送候选人，其中每个环节都旨在衡量特定的素质或技能，但我个人最喜欢的是**协作编码**练习。

在本节课中，你将一名工程师和一名候选人配对一个半小时，并给他们一个共同完成的项目。这不是一个在工程师评估时候选人被迫在白板上写代码的练习，而是一个他们在真实的计算机上一起工作解决方案的练习。

为了让你的员工和你的候选人处于平等的地位，从而正确模拟你在工程团队中可以找到的协作环境，我发现以下规则是一个很好的起点:

## 让候选人选择编程语言。

通过允许候选人选择构建项目的编程语言，它把他们放在了驾驶员的位置上。虽然与他们一起工作的工程师*可能*已经解决了以前候选人的这个问题，但让候选人控制技术堆栈可以确保这项工作是真正的协作，而不是指导。

## 鼓励熟悉的环境。

公司通常不应该规定他们的工程师使用的工具。Vim 或者 Emacs，CLI 或者 GUI，甚至 Linux 或者……你知道……其他的；一般都无所谓。生产力是王道，我们选择的工具*有助于*提高生产力。

因此，当你有一个候选人来参加你的协作编程练习时，鼓励他们带上他们的笔记本电脑。让他们在他们最*舒服的开发环境中工作。毕竟，你们在一起的时间不到两个小时。让某人坐在一辆不熟悉的汽车的驾驶座上不会让任务变得更容易。*

## 评估旅程，而不是目的地。

当我运行这个练习时，我最不关心的是我们是否真的解决了问题。不是每个问题都可以在这样一个固定的时间内解决，因此将完成视为通过/失败将可能拒绝许多合格的候选人。相反，关注问题解决*如何进行*。

他们是如何开始的？他们合作得如何？他们是接受输入还是忽略输入？他们是想多了，还是马虎了？没有比和某人一起工作更好的方法了。我发现在这种情况下与某人一起工作 90 分钟很能反映与某人一起工作的情况。

## **鼓励务实。**

大多数软件开发人员候选人想要展示他们知道如何做工作*和*他们知道如何正确地做*。但在这种情况下，他们没有时间这样做。我喜欢在练习的要点上尽早设定期望(见上文)，像代码质量和单元测试这样的事情既不被期望也没有益处。*

*也就是说，我发现，即使当被要求放弃这些东西时，大多数候选人也会提到他们可能会如何测试某些东西，如果他们有额外的时间，他们会如何构建解决方案，或者指出由于时间限制，他们故意走的捷径。*

# *选择项目*

*![](img/b41037c2f1f6400b1821ee3acb634d76.png)*

*由 [Alexander Schimmeck](https://unsplash.com/@alschim?utm_source=medium&utm_medium=referral) 在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄*

*理想情况下，选择的项目遵循 Joel Spolsky 关于通过电话正确评估软件开发人员的建议:*

> *理想的问题是，受访者在使用过该产品后，对其工作原理非常熟悉，但不太可能自己实施过。*

*这个练习中我最喜欢的项目是 web 服务器。在上述标准中，大多数后端 web 开发人员应该非常熟悉 web 服务器是如何工作的，*和*不太可能自己实现过它(为此，我只遇到过*一个*以前构建过 web 服务器的候选人)。*

*事情是这样的，在 90 分钟内创建一个*功能完整的* web 服务器是不可能的，但是让框架接受和响应简单的 HTTP 请求不仅是可行的，而且*足够巧妙，使它变得有趣。如果我们能够启动一个简单的套接字服务器，并正确地响应来自标准 web 浏览器的`GET`请求，那么我们就有了赢家！**

*30 分钟而不是 90 分钟就搞定了？总有成长的空间。添加对`POST`请求、mime 类型解析或缓存头的支持。这是咆哮的 20 年代。网络服务器现在变得异常复杂。发挥创造力。*
