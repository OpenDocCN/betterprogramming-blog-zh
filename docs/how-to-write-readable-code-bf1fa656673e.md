为了成为一名高级开发人员，编写可读和干净的代码是需要掌握的关键方面之一。有些人错误地认为，任何代码库都可以通过使用注释立即变得可读。注释有时可能是有用的，但只是在例外情况下。代码本身应该是自我解释的。

作为软件开发人员，我们的目标是生成简单、清晰、可读的代码。写代码没那么难。任何人都能做它。另一方面，编写简单的代码更难，需要一点实践。

什么使得代码可读？我们如何实现这一目标？做起来有多容易？

让我们来看看六个可以帮助你改进编码方式的技巧。通过遵循它们，您将产生更简单、更健壮、更可读的代码。这些提示将成为防止你编写混乱的代码的盾牌。

# 1.拥抱声明式编程

让我们首先通过看一个定义来理解什么是声明式编程。

> 在计算机科学中，**声明式编程**是一种编程范式——一种构建计算机程序的结构和元素的风格——它表达计算的逻辑，而不描述其控制流
> —维基百科

声明式编程就是使用/创建工具，将你从实现细节中抽象出来。这样一来，你就可以专注于如何解决问题了。

# 2.避免常见的代码气味

某些代码气味会阻碍可读性。通常，没有经验的开发人员更关注代码的目标。他们不关注它是如何构建的，以及它是否对其他人可读。

以下是最常见的代码气味:

*   大型方法函数——与其编写冗长的方法函数，不如创建只完成一项任务的小型方法函数。这使得它们更容易理解、测试和重用。

# 3.按应用层划分逻辑

代码架构对代码的可读性起着至关重要的作用。你应该把你的应用程序分成几层。最明显的一个就是把业务逻辑从 app 层剥离出来。

将业务和应用程序层绑定在一起不是一个好主意。你将耦合两个不应该互相了解的实体。这使得它们更难测试，并损害了可重用性。

这就是单一客体责任原则形式`**S**OLID`。

# 4.依赖抽象，而不是实现

为了对可读性产生更大的影响，客户消费者不应该关心特性的实现细节。它应该只关心提供的接口和回报:契约。

这是来自`SOLID`的依赖倒置原理。

这有什么好的？这意味着您可以在任何给定的时间更改实现，而不必更改客户端。只要它符合契约，它就不应该关心内部结构。

# 5.关注点分离

正如我们之前所看到的，你把代码放在哪里也是至关重要的。

关注点的分离意味着你把你的软件分成模块来处理单一的特性或行为。

产生的代码将是`high cohesion`和`low coupling`。高内聚指的是来自同一个域的所有代码被组合在一起。低耦合意味着模块之间不相互依赖。

```
// ✅ an Example on how to structure an application| src
**|-- auth**
|---- index.js
|---- store.js
|---- models.js
|---- utils.js
**|-- profile**
|---- index.js
|---- store.js
|---- models.js
|---- utils.js
**|-- payment**
|---- index.js
|---- store.js
|---- models.js
|---- utils.js
|-- app.js
```

因此，生成的代码更容易被发现，也更难被破解，因为它独立于其他代码。

# 6.提前计划

许多程序员会直接跳到代码中。他们没有花任何时间规划代码的依赖结构，这将对可读性产生巨大的负面影响。

没有适当的计划，所有的技术技能都是没有价值的。对任务的深刻理解对于使用合适的解决方案至关重要。

提前规划的主要原因是什么？

*   清晰的视觉需要哪些抽象:当尽可能向前看时，你会清楚地看到更大的画面。你会明白哪些抽象是需要的，哪些实现是在过度工程方面。
    你的软件应该只处理可能的用例场景。否则，一个简单的功能可能会变成一项艰巨的任务。

许多人认为当你面对问题的时候解决问题会更好。恰恰相反，积极主动比被动反应更好。否则，你可能会发现自己在修补补丁，而不是长期的解决方案。

因此，您的代码将反映您的预先计划，这将使它从长远来看更加可预测和可靠。

# 摘要

编码是一个高级开发人员应该掌握的最重要的技能。没有这一点，他们永远无法领导、引导或指导他们的同伴。

正如我们所看到的，接受实用的编程原则总是我们最好的选择。像`YAGNY`、`DRY`、`SOLID`、`KISS`这样的概念要时刻牢记在心。

我们错误地认为编码水平的提高是线性的，需要通过多年的经验来学习。这不是真的，这完全取决于你的承诺。你在代码中投入的思想越多，你进步的就越快。走出你的舒适区会有所帮助。剩下的就交给激情和奉献吧。
