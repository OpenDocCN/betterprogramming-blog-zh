# 第一条敏捷宣言原则真的像听起来那么简单吗？

> 原文：<https://betterprogramming.pub/have-you-read-the-agile-manifesto-9785ce876078>

## 你读过敏捷宣言原则吗？

![](img/2f56afed1b82c0e615abf0ad74ee3226.png)

> 举起手来，这里有多少人在敏捷团队中工作？(所有人举手)
> 
> 读过敏捷宣言的人请举手(有人举手)
> 
> 认为他们公司的管理层和董事已经阅读了敏捷宣言的人请举手(没有人举手)

敏捷宣言并不长——它实际上是一页上面有几个要点的纸。你读过吗？你明白了吗？你在日常行为中真的坚持这些原则了吗？

我认为敏捷宣言中描述的过程和原则是处理软件项目的一种奇妙方式。我也(悲哀地)认为*敏捷已经失败*。不是因为这是个坏主意——恰恰相反。我认为它失败了，因为坦率地说，没有人真的费心去阅读和*理解*敏捷宣言中的内容。

敏捷做得好可以(不是将会)改善项目的前景，这取决于项目、客户、团队和许多其他“东西”。好的敏捷是朝着正确方向的推动，而不是保证项目成功的灵丹妙药。敏捷做错了会(而不是会)让你的项目陷入管理不善、预算超支、错过最后期限和激怒团队成员的深渊！

敏捷宣言中列出了 12 条原则。在接下来的几周里，我将逐一浏览它们，并尝试思考一个团队、个人或组织需要什么才能真正贯彻这些原则。

在这篇文章中，我将从第一个开始，即:

> 我们最优先考虑的是通过早期和持续交付有价值的软件来满足客户。

![](img/8336c741e2a205857894c20ddeec8a67.png)

如同每一个敏捷原则一样，直觉上这是一个好主意。“让你的客户满意”一直是所有企业的基本建议。同样，及时交付任何产品和服务的好处也不难理解。

不幸的是，将这一基本而合理的建议应用到软件项目中，比乍看起来要复杂得多，也更具挑战性。在这篇文章中，我将谈论其中潜在的复杂性，并建议如何实践这一原则。

一个迫在眉睫的挑战是，客户通常不知道他们在项目开始时到底想要什么——随着项目的进展，客户对自己需求的理解将会成熟。此外，客户可能不知道现有技术的局限性和可能性。

最后，交付一个软件项目可能需要几年时间，而且从开始到结束，市场可能已经发生了根本性的变化。在项目开始时让客户满意的东西，在(相当重要的)交付日期可能不会让他们满意。

在上文中，还有一个假设，即软件公司定期与他们的客户交谈……但情况并非总是如此。我知道，在我的经历中，经理们只是简单地列出从未被客户运行过的规范的愿望清单，我不是唯一的一个。

你怎么能让一个你从未交谈过的顾客满意呢？

因此，虽然这听起来非常简单，但事实并非如此。要真正“实践”这一原则，组织处理业务的方式需要发生根本性的转变。

首先，如果他们还没有，那么企业必须直接与他们的客户接触，寻求指导和反馈，以某种方式记录和处理反馈，然后根据反馈改变项目的进程。

这种与系统的客户和用户的持续和开放的对话是至关重要的，但是经常被完全忽略。这不能冒险去做。必须有一个遵循某种结构的定期反馈，以便可以在项目的整个生命周期中对其进行检查、比较和监控——从编写一行代码之前开始！

最初几周的开发是基础性的，将为项目的其余部分定下基调，因此这个过程必须在开发过程开始之前就到位。该过程必须在整个项目中持续进行，并根据客户群和产品销售市场的不断变化的需求进行持续和快速的转变。

当然，如果这一过程被改造成一个棕色地块项目，一切都不会失去。原则保持不变，但是对现有代码进行更改的灵活性降低了，所以好处减少了(或者可能需要更长时间才能实现)。

我强烈建议在你的团队中推行这一原则，优先考虑招募 UX 资源。UX 经常被认为“只是”制作线框，但这是对 UX 角色的严重误解。

“UX”代表 **U** 服务 **X** 经验，因此他们的角色涵盖了用户如何与系统交互的所有方面。远远不止是线框和模型。一个好的 UX 资源将从根本上理解这一敏捷原则，并将几乎按照定义使您的团队保持一致——一个好的 UX 资源的好处不能被夸大。

即使假设雇佣了 UX 的资源，咨询了客户，收集了反馈意见，并将其整合到产品中，这个敏捷原则还有一个非常重要的“第二部分”。第二部分陈述了客户通过早期和持续交付有价值的软件而对*感到满意。这可以手动完成，但是现在满足这一要求意味着构建 CI/CD 管道。*

构建 CI/CD 管道主要是一项技术挑战。虽然现在这是一个很好理解的过程，但这并不意味着它在技术上很容易做到。建立和维护 CI/CD 渠道需要时间成本，而且通常也是持续的财务成本。

我在 CI/CD 方面遇到的最大挑战是说服管理层支持这些工作。花在管道上的时间就是没花在功能上的时间。尽管这一点非常重要，但获得花费时间的授权并不总是容易的。

传统的大致每年发布一次的模式意味着，“好主意”在进入客户手中之前会在回购中滞留长达一年。好的想法几乎一准备好就应该增值。

除了“你应该做”之外，很难对 CI/CD 提出具体的建议，因为每个项目都非常不同，实际上有一百万种方法来构建它。我肯定会建议投资 CI/CD，无论具体的实现最终是什么——根据我的经验，投资将会以满意的客户的形式得到回报。

在我看来，这是敏捷宣言中十二条清单上的第一条也是最重要的一条，也是实践中坚持得最差的一条！这里有唾手可得的水果，所以…你还在等什么！！

敏捷宣言的第二个原则是:

> 欢迎不断变化的需求，即使是在开发的后期。敏捷流程利用变化为客户带来竞争优势。
