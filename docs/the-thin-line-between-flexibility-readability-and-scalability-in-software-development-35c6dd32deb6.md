# 软件开发中灵活性、可读性和可伸缩性之间的细微差别

## 你的策略是什么？

[兰斯·安德森](https://unsplash.com/@lanceanderson?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍照。

当你的经理有问题要解决，并要求你提出解决方案时，你知道那种感觉吗？

你内心的程序员、开发人员、编码人员或书呆子会想，“是的，我要大干一场！我要用我的超级智能解决方案让所有人大吃一惊。”

如果你知道这种感觉，请继续阅读，因为这会给企业带来一些危险的情况。

在这个故事中，我想深入探讨过度工程化的危险。我想告诉您，构建足够好的解决方案和过度设计的解决方案之间只有一线之隔。

我不会告诉你最佳解决方案是什么。我想让你仔细考虑这个问题，因为这些情况和决定会对开发阶段的时间表产生很大的影响。

# 设计坚如磐石的建筑

大多数开发人员都乐于钻研一个问题，并做出最好的解决方案。我们非常致力于将我们的一切力量和能力投入到这个解决方案中。

您可能认识到您和您的团队提出的伟大架构，并且对此感到非常兴奋。

也许你激情四射，自信自己会大放异彩。毫无疑问，这两者之间存在某种联系。

然后你的经理来了，你有六个月的时间。

哎哟，计划泡汤了！

但是你和你的团队找到了一种方法，让这个架构不那么棒，但仍然足够棒。

你和你的团队开始日复一日的建设。

# 是否可重复使用

几个星期过去了，你已经制造了很多组件。

它从 API 获取数据并显示所有数据。

您正在构建一个将所有数据模型转换成表单的服务。这个超级牛逼！

所以你建立了一个可以在任何地方使用的服务。

但是当数据模型改变时会发生什么呢？它还能工作吗？还是你和你的团队会有一个大问题，因为一切都与这项服务有关？

那个截止日期越来越近了！

那么，是让所有组件都使用这个服务更好呢，还是专门构建这个服务更好呢？

# 瘦不瘦

如果你和你的团队把事情简单化，也许会更好。只是以一种非常精益的方式构建所要求或定义的东西——仅此而已。

有时候，具体开始也可以不错！只是建立一个解决方案的绝对最小值。让测试人员、管理人员或用户先测试它。

之后，你有数据来改善它。

可能流量不错，但是性能有所欠缺。那么你就可以改善这一点！

您删除不需要的代码，并可能使一些部分可重用。

这是最好的解决方案吗？还是制定出完整的架构更好？

# 灵活性、可读性和可伸缩性

大家都知道这些特质很重要。您希望保持代码的可读性，这样其他人就可以对它做出贡献。

另一方面，它需要灵活，这样你就可以长时间使用它。

其次，您希望让应用程序以多种方式可伸缩。但是当你想保持可读性的时候，你怎么做呢？

如果你在灵活性和可伸缩性上孤注一掷，这会导致大量的复杂性，降低你的可读性。你想那样吗？

如果你一味追求可读性，可能会导致性能问题，降低应用程序的灵活性和可伸缩性。你想那样吗？

你会看到它们之间有一条很细的线。这些选择决定了应用程序的成败。

# 特定或一般

就像我之前说的精益方法一样，你可以非常具体地开始，只构建最小的解决方案。

但是这会花费更多的时间。这样，您可能需要更多的迭代来改进应用程序。

您也可以选择从一个非常通用和灵活的坚如磐石的架构的伟大计划开始。

这有助于构建提高灵活性和可伸缩性的应用程序。但是代价是什么呢？

构建泛型类、函数或组件会降低可读性——尤其是对于那些刚刚涉足其中的人。

如果因为对用户体验的新认识而需要修改代码，这甚至会成为一个问题。

那一瞬间，是写得具体一点好，还是笼统一点好？

# 结论

很多开放性的问题！

构建应用程序时，可读性、灵活性和可伸缩性之间只有一线之隔。

如果你追求精益，并且你非常具体地开始，你什么时候开始改进并使代码更 D.R.Y？什么时候降低代码的可读性是可以接受的？

也许最好有些部分是特定的，有些部分是通用的。

但是最重要的是——无论你和你的团队做出什么决定——你们要互相讨论这个问题。没有解释的余地。这对团队凝聚力来说是致命的。

无论你和你的团队决定做什么，都要写下来。告诉你的经理你的决定，并按照决定去做。

如果事情不顺利，那就改变它，直到事情顺利为止。

如果你对这个话题有什么想法，请在评论中分享，这样我们可以互相学习。
