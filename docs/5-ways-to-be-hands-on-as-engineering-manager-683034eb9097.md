# 结对编程

现在，我在一个新的岗位上，技术对我来说是全新的。然而，我热衷于探索技术领域，以了解机遇和难题。我发现结对编程是更好地了解系统的一个好方法，同时我也慢慢了解了这个系统。

与和系统密切合作的人结对编程也允许理解历史，它为什么会出现的较少理解的上下文，以及以一种有趣的方式的大量上下文。

意外的奖金？没有什么比集体挠头半小时更能改善与团队成员的关系了。

# 工艺艺术品创作

技术文档可以采取多种形式。它们可以是附加到代码库的“自述”文件，也可以是有效的微型网站，还可以是由代码片段、架构图、安装手册和 FAQ 论坛组成的完整 wikis。

技术文档的棘手之处在于，无论它有多广泛，在信息量或两者方面几乎总是有改进的空间。

因此，试图在这个领域带来小小的改变，往往会产生巨大的影响，让组织中的每个人都过得更好。在清理或添加人工制品的过程中，你总会学到一些你以前不知道的东西。

# 自动化

自动化您或您的团队的一些例行任务，例如将 backlog 项目标记为陈旧，改善构建或发布管道上的开发人员体验，或者简单地自动化您自己的任务，以给你一些自由的喘息时间，这是一种有趣、省力且无阻塞的方法，可以让你继续参与游戏。

# 技术设计讨论

参与技术设计讨论，这通常会让您鸟瞰不同的系统、服务和层以及它们之间的传输协议。如果你花一些时间在系统上，你可以问一些直观的问题，这些问题指向经常被遗忘的领域，如性能、安全性或数据记录，这些可以帮助指导团队将这些需求转移到左边，并在系统的早期构建它们。对你来说，这将是一个很好的方式来帮助团队，同时利用你丰富的技术设计经验。

# 代码审查

如果你领导着一个超过 5 名直接下属的团队，你花在编码功能上的时间可能会比你希望的要少。代码审查是很多事情的重要指针。它们给你一个关于代码架构的想法，但是也给你一些关于代码质量的微妙的暗示，为控制技术债务所做的努力，以及开发人员的思维过程。

通读拉式请求中的代码评审注释也给人一种工程组织内部潜在文化的感觉。我们对低年级犯错误成长的鼓励程度如何，高级工程师是否指导而不是温和地重定向到正确的道路，以及我们的系统对编码错误的抗脆弱性如何。

如果你有时间做别的事情，我会推荐你做这个。每周至少花半天时间在团队中进行随机代码评审。它不仅仅是一个正在构建的特性的指示器，它还可以作为一个有效的指针，指示您可能需要作为一个团队工作的一些领域。
