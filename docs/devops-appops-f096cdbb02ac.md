在 IT 界，学习新的流行词汇是每天都会发生的事情。他们中的大多数就像流星一样，在你完全理解你应该学什么之前就消失了。其他一些将成为未来几年的趋势。我可以提到 DevOps 和 Frontend，只是为了举例说明我所说的内容。

所有这些介绍解释了为什么我们会被新术语轰炸，并且必须关注正确的术语。尽管标题如此，我并不相信流行语或趋势。我相信概念。即使技术和趋势过时了，概念依然存在。

这就是为什么在本文中我将讨论 AppOps 概念。然后，当你开始听到新的 AppOps 趋势时，你就会知道如何争辩。

# NoOps 趋势

NoOps 趋势旨在消除开发和运营之间的所有摩擦，顾名思义，就是消除这些摩擦。

这似乎是一个激烈的解决方案，但我们不必从字面上理解。正确的解释——也是可行的解释——是在部署和交付阶段尽可能地消除人的因素。

这种方法自然得到了云的支持，云帮助事物自行运转。如果你有兴趣发现更多关于 NoOps 的信息，你可以阅读我几年前写的一篇文章。

顺便说一句，要继续阅读这篇文章，你需要知道的是:

> NoOps 只是减少了部署管道中的人力投入，并为开发团队带来了意识。

所以，NoOps 和 DevOps 并不冲突。相反，它的目的只是在情况允许时，以一种聪明的方式使用 DevOps。

# 关注应用程序

即使你正在编写一个小的应用程序或者一个非常大的项目，如果没有一个合适的 DevOps 设置，你也不能避免这样做。

我简直无法想象 2021 年有人会通过 FTP 上传文件来更新应用程序——如果你正在这样做，请立即停止。

每个人都需要 DevOps 和基础设施。但是我们真的确定要在这上面花费时间和金钱吗？

当然，我们必须这么做。好处是如此明显！但是如果我们从投资者的角度来看问题，我们会明白这不是团队的最终目标。

由于技术是使事情成为可能的使能器，DevOps 是使事情变得高效的方法。但是我们不能忘记我们的应用程序应该做的“事情”。

> 这就是 AppOps 的目的:把注意力集中在应用范围上，包括源代码和基础设施。

# 为什么要 AppOps？

解释 AppOps 好处的最明显的场景之一是每个基于 Kubernetes 的应用程序。如果您打开每个群集，您会发现许多 pod/服务/部署设置几乎相同。

事实上，每个 PHP 应用程序都有相同的配置，除了参数。Java 也一样，。Net 或其他应用程序。问题是 Kubernetes 不知道主机应用程序的内容，所以他需要告知它每一个细节。

对于所有新的应用程序，即使技术是相同的，我们也必须从头开始。为什么？

我应该只解释一次 PHP 应用程序是如何组成的。我不会在定义如何部署 PHP 应用程序上浪费时间，因为世界上所有的 PHP 应用程序基本上都是一样的，我希望有人(可能是供应商)能给社区提供一次正确的配置。假设有一个可重用的配置。您所需要的只是简单地开发应用程序。不要再做重复性的工作了。不再重复发明轮子。此外，关注真正重要的事情，忘记细节。

到目前为止，我为 Kubernetes 介绍的内容可以扩展到虚拟机、云服务等等。这就是 AppOps 听起来如此有趣的原因。

# 有多大的可能性？

可以想象，没有任何水晶球插件可以安装在 Kubernetes 上，让集群预测您正在部署什么。Kubernetes 是一个容器编排器，它把自己确立为抽象基础设施的标准。你不能要求他做得更多。所以，我们基本上有两个选择:

1.  用老方法做，准备你可以重复使用的配方或工件(例如一套 YAML 配置)
2.  以云的方式来做，采用一种已经允许你专注于应用而忘记基础设施的解决方案。

让我们详细看看这些选项

## 用老方法

对于不要重新发明轮子来说，用老方法是最简单但最不时髦的解决方案。

如果你正在使用 Kubernetes，你可以使用像[头盔](https://helm.sh/)或 [ArgoCD](https://argoproj.github.io/argo-cd/) 这样的工具来自动设置应用程序并创建可重用的应用程序定义。

如果您正在使用虚拟机，一个好的解决方案是准备 [Ansible](https://www.ansible.com/) 脚本。

此外，您可能有兴趣探索作为一个简单的基于云的解决方案的 Terraform。

除了以上所有的解决方案都很棒之外，你仍然需要时间来制作第一个解决方案的原型，并且你将需要部署这个部分。所以我更喜欢下一种方式！

## 做云的方式

在云时代，使用现成的服务是实现事物的巨大推动力。

这就是为什么我们应该在第一次尝试时就评估易于使用的解决方案。在 Kubernetes 领域，我可以提到一些云工具，它们可以让您在应用程序级别进行部署，而无需考虑其背后的内容。

第一个工具是 [Shipa](https://www.shipa.io/) ，这是一个非常有趣的新兴解决方案，无需任何开发运维流程即可部署应用。这个解决方案让可以只专注于应用开发而忘记服务器。

第二个选择是 [Devtron](https://devtron.ai/) than，它提供了一个交付工作流实现，简化了 Kubernetes 管理，并使部署过程非常顺利。这个工具，结合一些预定义的方法，使得忘记基础设施成为可能。

我确信很快会有越来越多的解决方案以 AppOps 的方式接近 DevOps，因此，正如技术方面经常发生的那样，工具从来都不是问题。

# 带什么回家

在过去的几年里，应用程序开发和交付领域的创新取得了令人难以置信的进展。

随着我们可以在网上找到大量的资源，云给了我们很大的推动。在加速开发和减少开发与操作之间的摩擦的工具列表中，有 Kubernetes。

尽管所有这些解决方案都推动了全球的数字化转型，但我们并不满意。我们现在觉得必须减少管理这些工具和复杂性的工作量。这是强制性的，以便更专注于应用程序开发部分，并为最终用户带来价值。

AppOps 是实现这一目标的一种方法。使用从更高角度管理基础架构的工具，我们隐藏了复杂性，可以专注于对业务真正重要的事情。

这样做并非不可能。我们已经有了很多工具，可以很好地让事情朝着正确的方向发展。
