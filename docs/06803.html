合并请求模板的重要性

填写清单通常被认为是一项令人麻木的死记硬背的任务。但是如果我告诉你清单实际上对正确完成工作是必不可少的呢？

清单已被证明有助于减少复杂任务中的人为错误，如驾驶飞机或进行手术(见阿图尔·加万德的 【清单宣言】)。

那么为什么不把清单应用到软件工程中呢？在本文中，我们将研究在合并新代码之前使用合并请求(或拉请求)模板的好处。

合并请求模板

大多数软件公司将Git与GitHub或GitLab等平台结合使用，为他们的代码提供版本控制。

生产代码通常位于主(或主要)分支上。当开发人员编写新特性或修复bug时，他们从主分支创建一个分支，编写他们的代码，然后创建一个合并请求，要求将他们的新代码合并到主分支中。然后，另一个开发人员会检查代码，批准(假设代码看起来不错)，并合并新代码。

GitHub和GitLab都允许您配置您的存储库来包含一个合并请求模板，它基本上只是一个开发人员在创建合并请求时填写的表单。

合并请求模板是可选的，但是我想在这里说它们是必不可少的！一个好的合并请求模板有助于创建一个优秀的工程思维模式，并且有助于保持代码的高质量。

合并请求模板可能如下所示:

Description: what are you changing and whyTicket Info: links to tickets in your task management system like JIRA or WorkfrontTest Plan: step-by-step instructions for how someone could verify that your code worksQA Risk: low, medium, or highScreenshots or Videos: proof of your code working properlyAdditional Checklist Items:
[ ] I have written automated tests for this change
[ ] I have updated documentation as needed
[ ] The code works in all supported browsers

您的合并请求模板可能比我上面的更多或更少，但这六个字段为我们提供了一个坚实的基础。让我们来看看为什么。

描述

首先，描述。这个很简单。你是否曾经被要求审查别人的代码，却没有被告知为什么要写这段代码？知道这个新代码试图解决什么问题不是很有帮助吗？我想是的。

你不想知道工程师为什么用他们的方式解决问题，或者他们考虑了什么替代方案吗？我会的。

合并请求描述有助于为代码评审提供上下文。

门票信息

几乎每个公司都使用某种工作管理系统来跟踪他们的工作。对于软件工程师来说，拥有一个可以记录错误和请求新特性的系统尤其有用。

在您的合并请求中有一个“ticket info”字段只是提供了一个位置来包含一个指向提示编写这段代码的ticket的链接。这样，审查代码的人也可以查看原始的错误报告和票据包含的任何附加信息。保持标签和合并请求的链接可以确保不会丢失任何东西——特别是当您需要在几个月或几年后重新访问时。

测试计划

“测试计划”字段是我个人最喜欢的。在合并请求模板中拥有这个字段迫使提出合并请求的开发人员考虑他们将如何手动测试代码以确保它工作。有希望的是，这将导致他们实际上验证他们的代码也能工作，导致更高的质量，并帮助他们捕捉简单的错误。

测试计划也给了审查者清晰的步骤，如何在他们想要下载代码并在他们的机器上进行测试的情况下验证代码是否正常工作。

注意:在任何人对依赖人工测试进行QA的无效性大动干戈之前，不要担心——我们稍后将讨论自动化测试。

它还提供了编写代码的开发人员认为重要的测试内容。作为评审人员，您可能会想到测试计划中缺少的一些东西，这可能是一个很好的指标，表明编写代码的开发人员可能没有考虑到这种情况，或者没有编写代码来正确处理它。

质量保证风险

QA风险本质上是这种变化有多大的主观度量。如果出了问题，最坏的结果会是什么？文本输入会有错误的边框颜色吗？或者您的登录或结帐过程会中断吗？理解代码变更的潜在影响有助于促使评审者更彻底地进行评审。

截图或视频

如果代码审查人员没有自己下载代码进行本地测试，那么要求开发人员提供一个运行中代码的截屏或视频，这又提供了一个检查一切正常的机会。

这个字段对于影响UI的前端更改特别有用，但是对于后端更改可能没那么有用。

附加清单项目

最后，提供您的组织关心的任何其他事情的最后一个清单会很有帮助。一个常见的项目可能是验证开发人员已经为他们的代码编写了测试。如果开发人员还没有编写单元测试的话，这里的清单是对他们的一个提醒。

另一个清单项目可能是根据需要更新任何文档。如果您已经更改了一些组件的API，或者修改了先前在自述文件中记录的构建过程中的一个重要步骤，您将希望验证该文档是否仍然有意义。

对于UI开发来说，如果你的公司支持IE11这样的老浏览器，有一个关于代码在所有浏览器中工作的清单项目是一个很好的提醒去仔细检查。

批评

现在，一些缺点。工程师可以简单地忽略合并请求模板并选择不填写它吗？是的。

或者，他们是否可以在没有真正完成任何步骤的情况下，盲目地检查最终清单中的所有方框？是的。

最终，忽略清单并不比根本没有清单更好或更坏。然而，仅仅是合并请求模板的存在就有助于促使工程师彻底检查他们提供的信息。

结论

如果你仍然怀疑清单的重要性或者合并请求模板的帮助，试一试吧！我想你会对你看到的改进感到惊讶。
