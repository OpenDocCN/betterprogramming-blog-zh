团队合作行为面试问题很难回答，主要是因为人们在不同的场景下会有不同的反应，很难弄清楚面试官在寻找什么。这些年来，我参加过不同团队和不同职位的面试，我绝对感觉到了一种模式。

*免责声明:我面试的大多数人都在申请软件工程或产品经理相关的职位。其他职位在团队沟通方面可能会有稍微不同的问题。*

# **1。解决实施上的分歧**

*问题:你会如何解决实施过程中的分歧？*

我很惊讶我从受访者那里得到了这么多“老板说什么就做什么”、“产品经理说什么就做什么”或者“什么都做<insert a="" higher="" position="">说什么”的回答。这些答案引发了危险信号。</insert>

和别人交流目标比实现更容易。通常，你的队友和你有着相同的最终目标，但在如何实现这个目标上存在分歧。

这可以表现为对 MVP 特性列表、设计风格、基础设施设计、交付过程或雇佣决策有不同的想法。

对于某些问题，您可以通过邀请他人进行概念验证并获得用户的反馈来解决它们。对于 MVP 特性列表和设计，这种方法出奇地有效。最终还是要用户说了算。团队甚至可以尝试不同的想法，而不是一开始就扼杀不同的想法和创造力。

然而，并不是所有的问题都可以通过概念验证和让用户做最后的评判来解决。对于一些决定来说，一旦你做出承诺，交易就被锁定了，改变方向是困难的和昂贵的。在工程领域，一旦基础设施设计就绪，资源开始涌入，叫停是一种惩罚。对于雇佣决策，一旦团队雇佣了某人，培训就开始了。

有两种方法可以解决这个问题:在这个问题上获得更多的声音，并标准化评估指标。

当获得不同的声音时，确保你得到的是一个独立的观点。举行十人小组会议并不总是意味着你会得到十个独立的观点。请注意，大多数人不只是在会议期间跟随最大声的声音。

对于像招聘决策这样的情况，标准化评估指标意味着主动消除偏见，这样团队更有可能达成共识。标准化度量有助于团队理解团队中重视什么属性。

# **2。你的工作依赖于他人**

*问:如果你的工作依赖于团队中的其他人，你会怎么做？哦，顺便说一下，最后期限就要到了。*

我们都经历过这种情况。截止日期快到了，你还在等队友交出文件。你试图巧妙地暗示你在等着，但同时，你不想粗鲁地问“嘿…你吃完了吗？”来进行微观管理或催促别人

面试官想知道你会如何解决这个问题。

秘密在于你如何沟通你的情况。大多数时候，你可以有把握地假设你和你的队友都有相同的最终目标。友好地让对方知道你的截止日期，甚至主动提出帮助事情向前发展。很有可能，人们只是不知道你的截止日期，或者他们正在努力帮你拿到这个项目。如果你发现其他人真的很难给你完成任务所需的东西，事实上你已经传达了这一点，这是一个有效的理由，可以在团队中提出来，一起解决。

# 3.你不具备完成任务的技能

*问题:如果你没有完成任务的技能，你会怎么做？*

当我问这个问题时，受访者会感到惊讶。然而，这实际上类似于这个问题的另一种形式:“告诉我一个你不得不当场学习的时候。”有两种方法可以做到这一点:自己获得技能或者找其他有技能的人。

方法取决于技能的类型。如果你处于一个“做”的位置，比如会计或工程师，自己获得技能应该是你的首要方法。如果你处于一个授权的位置，比如经理或营销，找到其他有能力的人应该是你的首要回应。

# 4.其他人不具备完成任务的技能

*这是上一个问题的一个微小变化:如果其他人没有完成任务的技能，你会怎么做？*

有时候，直接拿起工作并完成它会更容易。对一些人来说，教别人可能很乏味。当最后期限临近时，人们开始失去耐心。

另一方面，你没有能力自己完成所有的事情。这是一个授权问题。有时，这也是一场关于风险的对话。如果你是团队中唯一有技能的人，如果你休假一个月，团队会分崩离析吗？

这是一个你需要解决的挑战。你是如何平衡自己想花多少时间去培训别人和自己能完成多少工作的？

我认识一位高级开发人员，他每天抽出两个小时与团队中的初级开发人员进行结对编程。他们将工作交给初级开发人员，为期一周，每天与他们配对，到周末，为了赶上最后期限，他们将接手任何剩余的工作。

对此没有一个通用的解决方案，情况会因时而异。这次管用的下次可能就不管用了。一个关键的要点是，培训他人也是对自己的投资。通过向他人解释这个概念，你也加强了你的理解，减轻了你未来的工作。

# **5。防止重复工作的流程**

*问:你会引入什么样的流程来防止重复工作？*

对于软件团队来说，Git 版本化并不是最终的解决方案。对于创业公司来说，敏捷也不能解决你所有的问题。尽管大多数团队都遵循相似的过程模板，但是每个团队都是独一无二的，记住这种差异非常重要。

接受新的流程。对你上一个团队有效的方法不一定对现在的团队有效。留意新的团队文化和工作实践。
