# 合作、发展，并学会享受这个过程

在代码评审中，交流和建设性的批评越多越好。你*会*成为更好的开发者。你的同事*将*变得更加擅长给出反馈和阅读代码。

最终，你、你的评论者、*和产品*都会受益。

你不应该抗拒对你的代码的反馈。这是一个已经被证明可以捕捉错误、产生更高质量的代码并提高每个相关人员的编程技能的过程。

不相信我？

我敢打赌，在你作为开发人员的历史中，至少有一次你从 CR 中学到了一些东西。

也许你的同事留下了一条评论，启发你使用一个新函数，将一个巨大的 12 行代码块减少到只有 4 行。

也许发生了一些不同的事情，一位高级工程师告诉您将您编写的聪明的小一行程序扩展成多行以获得更好的可读性。

当获得对代码的反馈时，你一定会学到一些东西——特别是如果你把你的评审交给一个更资深、更有经验的人。

如果你仍然持怀疑态度，可以看看 Atlassian(《吉拉》的创作者)关于[为什么代码审查很重要](https://www.atlassian.com/agile/software-development/code-reviews) *的这篇精彩文章。*

下面是代码评审过程中涉及到的每个人和每件事的一些好处。

## 你得到了什么好处？

*   熟悉代码库的人会对您的设计选择进行深入、有针对性的分析。
*   您可以获得关于语法决策的直接反馈。
*   您可以获得关于代码效率的建议，以及提高代码效率的潜在方法。
*   您(至少)获得了代码安全性的基本分析。
*   在某些情况下，你和你的评估者之间的工作关系会有所改善。你甚至可以交个朋友！
*   安心知道你至少有第二双眼睛在你的工作投入生产之前。

## 你的评审员得到了什么好处？

*   你的评审者在阅读和理解代码方面变得更好。
*   你的评审者了解了更多关于其他开发风格的知识。
*   您的审阅者了解了更多关于代码库的信息。
*   您的评审者有机会协作改进新的或现有的代码。
*   您的评审者获得了对代码库更多的所有权和责任。

## 产品有什么好处？

*   你的产品会有更多潜在缺陷的守门人。
*   您的产品以优化的效率接收经过深思熟虑的代码。
*   您的产品会收到(在大多数情况下)更安全的代码。
*   您的产品接收更易于维护的代码。
*   对于代码库的不同部分，您的产品获得了更多的总体所有权和可见性。

这些列表并不详尽。你不太可能在每一份 CR 中都看到所有这些好处。你会在每个 CR 中看到一些，随着时间的推移，你会开始注意到你几乎经历了所有这些。

此外，不要把你的代码审查局限于一个人。也不要总是问同一个人。如果您需要第二种意见，或者您的代码的不同部分需要不同的工程师，那么将那个人放在评审中。

# 据以生活(编码)的准则

这让我想到了一个非常重要的指导原则:总是把最接近代码的人放在评审中。

这是什么意思？

如果您正在编辑某个其他工程师编写的一段代码，让那个工程师参与代码评审是一个好主意。如果那个人不在，你就去找下一个最亲近的人——另一个接触过文件或功能的不同部分的人。

直奔源头(一语双关)。让主题专家评审变更。

不要因为方便就把一个随机的工程师或者你最好的朋友放在代码评审上。这可能会更快，但这是一个可怕的想法。

如果你问一个远离代码的人，他们甚至可能不理解它。他们甚至可能不精通它所用的语言。这导致了一个看起来不像问题的问题:他们会批准它*。*这不是你要求代码评审时想要的。你问错了人，只是因为你知道他们会给你一个匆忙的、表面的审查，并很快批准。这违背了审查的整个目的。

# 正确的感觉

将合适的人放在 CR 上启动了一个紧张但有益的反馈循环过程。代码进入循环，它被审查，你作出改变，然后重复。

有时候，这感觉很残忍。感觉永远不会结束。你只是想发布这个该死的功能。继续走。尽可能地留在圈子里。

应该是这样的感觉。你不是编程之神。你的代码从一开始就不完美。留在反馈循环中，你会得到一个更加完美、高效和安全的作品。

感谢您的阅读！
