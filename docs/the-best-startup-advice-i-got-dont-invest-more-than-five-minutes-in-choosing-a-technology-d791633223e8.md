这听起来可能违反直觉，尤其是在技术驱动的初创公司的早期阶段。然而，随着时间的推移，我开始明白这种技术背后的原因。现在，我完全接受首席执行官们倾向于坦率地告诉我应该在选择技术堆栈上投入多少时间。

让我们深入了解一下我是如何开始相信这一点的。然后，我们来看看为什么你的创业公司不应该在技术选择上浪费很多时间。最后，让我们深入研究如何解决一个糟糕的选择。

# 我是如何得出这个结论的

我当时还是个大学生，业余时间在一家公司上班。具体来说，我参与了一个小项目。然后，该公司的首席执行官决定是时候把它变成自己的创业公司了。这是因为我们的团队快速开发了粗略的原型，并且在 PHP 和 JavaScript 方面没有太多努力就获得了成功。多亏了它，我们才能够在几个月内接触到成千上万的用户。

我是团队中最年轻的成员，也是最后一个加入的。尽管如此，首席执行官还是让我负责选择开发可靠版本服务所需的技术。我很惊讶，但我没有浪费时间，并开始与我的团队成员讨论。几个小时后，我们剩下了三个候选:Spring Boot + React、Node.js + React 和 Symphony + Vanilla JavaScript。早期的原型是用后来的技术开发出来的。所以团队几乎都为他们排好了队。

相比之下，我想采用 Spring Boot 和 React，尽管团队中没有人曾经和他们一起开发过。由于我们无法达成一致，我去找首席执行官谈了谈。他注意到了这个列表，并告诉我选择技术的时间不要超过五分钟。另外，他补充说我不必担心没有人认识他们。相反，这代表了一种团队成长和提高的方式。最后，他给了我最后通牒——我有两分钟时间告诉他我的决定。

我很震惊。我深深地相信，选择正确的技术是创业公司最重要的事情。我决定坚持我的想法，我们最终采用了 Spring Boot 和反应。

这个故事听起来可能很荒谬，或者是一个孤立的事件，但事实并非如此。从那以后，我每天都与初创公司和首席执行官打交道，并一次又一次地注意到这种技术方法。曾经让我惊讶的事情现在已经司空见惯了。这是因为技术毕竟只是工具。但是让我们详细地解决这个问题。

# 没有合适的技术组合

我从与初创公司的合作中很快了解到，他们的商业模式是不稳定的。它会随着时间的推移而改变。因此，他们销售的产品和/或服务将相应地进行调整。在这种情况下，未来比以往任何时候都更加不确定。今天是对的，明天可能就是错的。

最有经验的 CEO 都非常清楚这一点。这就是为什么他们不太重视技术堆栈的选择。他们知道，现在代表完美的选择在未来可能并不完美。换句话说，您必须根据您的公司现在或在不久的将来必须满足的需求来做出与技术相关的决策。这代表了你无法避免的固有限制，这就是为什么花几个月的时间去尝试和选择技术是一个错误。

因此，我不再惊讶于我合作过的许多初创公司不会花超过几个小时来选择他们的技术堆栈。而且，你可以针对不同的产品使用不同的技术，就像很多公司做的那样。

特别是，许多与我共事过的首席执行官反复向我传达的是将技术视为工具的重要性。此外，过于依赖你的技术是一个必须避免的巨大错误。这对你的团队尤其重要，他们必须愿意在需要时采用新技术。事实上，他们必须能够像创业公司的商业模式一样快速适应。

# 如何应对一个糟糕的选择

让我们假设你选择的技术不再是最好的，因为你的创业公司的商业模式发生了变化。或者类似地，你从一开始就使用了错误的技术。虽然没有正确的技术堆栈，但即时更改它可能会过于昂贵或耗时。因此，这不是一个可行的选择。在这种情况下该怎么办？

首先，你需要明确自己的目标。然后，记住你的商业模式比你的技术更重要。记住你什么时候会想让你的产品或服务适应你的技术是很重要的。这将是一个很大的错误，你不能这样做。而是反其道而行之。让你的技术适应你的产品。这似乎很愚蠢，但实际上是可能的，因为技术不是传统工具。

事实上，相对于物理工具，技术是更加模糊的工具。换句话说，你的团队选择的工具可能是正确的，但是他们可能用错了方法。除非他们彻底研究了他们所选择的技术的真正能力，以及可用的库，否则你无法知道这一点。此外，避免这些技术的非常规方法将是一个错误。一方面，这些可能会使你的代码更难阅读和理解。另一方面，走上没有人探索过的道路可能是解决之道。

# 结论

你可能会认为选择正确的技术组合对一家初创公司来说至关重要，但事实并非如此。在过去的几年里，我一直在与几家初创公司合作，我的经验告诉我要避免花时间选择技术。这是我见过的首席执行官们在不同场合传递的信息。事实上，技术只不过是工具。但与此同时，它们比传统工具更加灵活和强大。因此，没有真正错误的技术，只有错误的用途。
