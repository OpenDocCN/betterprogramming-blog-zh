## 我从公司到创业世界的旅程

我今天的生活和一年前完全不同。2021 年 4 月，我做出了一个被一些人称为大胆的职业举动:**我从一家大公司的软件工程师转行到了一家我联合创办的初创公司的首席技术官。**我离开了世界上最好的公司之一的一份很棒的工作，以及它的无数好处，去尝试从零开始建立一些东西，感受创造新事物的兴奋。

幸运的是，我不是一个人:我加入了一个创业工作室，它从一开始就帮助了我。我甚至不知道我要建立什么，但他们提供了一个专家团队，他们已经成功地启动了几个创业公司。他们找到了一个他们认为有问题需要解决的地方，把我介绍给一位联合创始人，并每天和我们一起动手。

感谢他们，去年可能是我学到最多的一年(我的意思是，如果你不算我真正发现生活本身的那几年)。虽然我以前获得的软件工程技能派上了用场，但我必须学习一些学校里没有教的东西，而且你不能只是谷歌:如何成为一名创业创始人。作为一家早期创业公司的首席技术官，我不得不抛弃我以前的一些信念，训练我的大脑以不同的方式思考。

我将在这里描述一下，在我看来，软件工程师和 CTO 之间的主要区别，以及在没有丰富工作经验的情况下，我是如何适应这个新角色的。

# 日常活动

作为一名软件工程师，你大部分时间都在编码或者参加会议来决定编码什么。在我之前的工作中，我很幸运能够贡献我对整体架构的想法，并且在我的技术选择上相当独立。

但作为一名 CTO，这只是我的众多工作之一。通常我白天都没有时间写代码，所以我不得不在晚上或者周末写。

> 我猜在一家后期公司，编码甚至不再是工作描述的一部分。但当你处于创业的早期阶段，只有你和你的联合创始人时，就要靠你来打造你的产品了。因为我们是一家 SaaS 公司，这意味着编码。

除此之外，你还必须决定建造什么，这是最棘手的部分。在做出决定之前，你需要制定一个策略，找到客户，与他们一起迭代，然后确定优先顺序。一旦你最终决定并实际上处于构建阶段，你仍然需要花时间与客户在一起，思考接下来的步骤以及如何不断改进，保持对最新技术的了解，开始招聘，以及做无数其他事情！当然，我的联合创始人，也就是首席执行官，负责很多我不需要考虑的事情，比如销售和营销，以及大部分行政事务。但作为联合创始人，我仍然需要参与每个战略决策，这确实需要时间。然而，我不会有任何其他方式——对我来说，这就是建立一个创业公司的含义:准备好参与业务的各个方面，并了解正在发生的一切。

我尽量留出几个没有会议的大时间段来真正专注于技术方面的事情，但我承认有几个星期有太多紧急事情。在内部会议、为合格的销售线索准备演示、与客户一起去现场、创建培训或演示材料以及偶尔的 VC 介绍会之间，我唯一能在实际产品上取得进展的时间是在正常工作时间之外。

# 工作生活平衡

你可以想象，这导致了一种不同于我当软件工程师时的工作/生活平衡。我讨厌忙碌的文化，我认为以生产力的名义提倡一直努力工作是愚蠢的。但事实是——作为一名联合创始人，你的创业公司的成功很大程度上取决于你的工作效率…

不可避免的是，知道这一点后，我倾向于把个人的事情放在次要位置，花更多的时间在工作上。例如，我从 2020 年开始定期画画，我喜欢它。但是从四月起我就再也没有拿起画笔，因为这是一项相当耗时的活动，而且我知道我可以做些别的事情来代替。

当这是你的公司时，你永远不会真正“下班”，至少我是这么觉得的。但我认为，从长远来看，保持某种平衡对于可持续发展是很重要的。一旦我们发展壮大，我可以依靠更多的人来完成工作，我会确保腾出更多的时间从事个人活动。我认为花 X 年时间全速前进而不花时间去呼吸是非常不健康的。

这可能是作为一名软件工程师的最大区别之一:作为一名个人贡献者，你没有负担知道如果你不向前推进，事情将会停滞不前。你是一个整体的一部分，你可以让自己被公司的其他人所引导，即使你经常被要求采取一些主动并鞭策自己。尤其是在大公司，利害关系不一样，精神负担也不一样。

# 心态

作为一名软件工程师，我一直被训练去寻找解决方案。我喜欢寻找解决方案！这是我从小就喜欢做的事情之一。

但是有一个陷阱:独自思考解决方案可能很有趣，但实际上对业务非常不利。

由于 startup studio 的网络，当我第一次开始采访潜在用户时，我以为我在 5 分钟的提问后就有了解决他们问题的办法…

但是，作为一名首席技术官，我很难掌握的一项技能是**停止试图强加一个解决方案，而是试图找出真正的问题所在**。

我不得不学会倾听和观察，而不是提问。我有一种想要马上行动、快速行动和建造东西的倾向。我知道我可能相当不耐烦。因此，这种心态的转变实际上是我从软件工程师转型为 CTO 过程中最困难的事情之一。但现在我明白了这一点，我意识到如果没有同理心和良好的倾听技巧，“建设者”的心态对创业没有多大用处。如果这不是用户真正需要的东西，那么快速构建它有什么好处呢？

我必须重新训练我的大脑的另一件事是接受做得更少。作为一名软件工程师，我学会了生产最高质量的产品。除非代码经过单元测试，并且至少有 2 个评审员，否则 PR 不能被接受。在开始之前，我们考虑了一段时间什么架构是最优化的。我承认，如果我认为我的代码是“肮脏的”,我会感到有点惭愧，因为我害怕被评判。

现在一切都结束了。这是自相矛盾的，因为作为 CTO，当你有一个由许多需要高效工作的工程师组成的团队时，你要负责确保质量和建立各种流程。但与此同时，你也有责任尽快将客户可以使用的产品推向市场。

所以，**这是一个取舍；质量与速度**。你想要放下那些让你慢下来的过程，同时记住更广阔的图景。你想快速完成任务，同时将技术债务保持在最低水平。您希望做出能够快速实施但又不与规模不相容的选择。

雷德·霍夫曼说

> “如果你对你产品的第一个版本不感到尴尬，那你推出得太晚了”

我喜欢把这个概念叫做*“小袋鼠”:*当小袋鼠出生时，它只有一英寸长，它沿着妈妈的皮毛爬行，进入育儿袋，在那里发育，走向世界。它已经是它应该是——一只袋鼠。它出生后，只需要在安全的环境中多待几个月，就可以长到全身了。对于一个创业产品，我认为这意味着从正确的技术栈开始，有一个如何扩展和实施什么流程的计划，但在产品完全开发之前就将产品发布出去，同时确保它处于适合其增长的环境中(如早期采用者计划)。

# 管理

在转行做企业家之前，我只有 3 年的工作经验，而且之前从未管理过团队。作为首席技术官，我不得不做我的第一次招聘和人员管理。

我们仍然是一个小团队，但与我作为个人贡献者时相比，这已经是一个很大的变化。首先，我必须学会授权。作为一名 IC，我习惯于自己做所有的事情，但正如我上面提到的，作为一名 CTO，我有太多的事情要做，不能不依赖其他人。

起初，我在这方面遇到了一些麻烦，不是因为我不相信其他人能正确完成这项工作，而是因为我喜欢自己完成任务——我发现这很令人兴奋和有趣，我想成为完成这项工作的人。我已经学会接受我没有时间做所有的事情，如果别人能把剩下的事情做完，我实际上可以把我真正应该专注的事情做得更好。另外，他们可以比我做得更好，因为这就是我们雇用他们的目的！

我还认识到，花时间让新员工入职，并持续确保团队中的人不断成长，这绝对是值得的。这和花时间编码一样重要，我希望随着我们的发展壮大，这将成为我的主要活动之一。

反思你作为软件工程师所做的事情，以及你认为团队应该遵循的最佳实践是什么，以使其形式化，这很有趣。然后，新的团队成员可以有一个结构化的方式，以符合您期望的方式为产品做出贡献。

# 领导力

对我来说，领导力和管理是两码事。你可以成为一名独立的贡献者，但仍能展示你的领导力。事实上，在我以前的工作中，领导能力受到鼓励和高度重视。所以，我认为这一点可能是软件工程师和 CTO 角色分歧最小的地方！

但是，即使你可能表现出作为集成电路的领导能力，通过积极主动，对自己的选择充满信心，并说服人们相信你对某个项目的判断，你也可能不得不作为首席技术官更上一层楼。

我认为，作为一名首席技术官，最重要的是，作为一名初创公司的创始人，你需要激励人们，为此，你需要对你正在做的事情充满激情。对你来说，这不应该是一份工作，而是一项使命，每个遇到你的人都应该看到这一点。

即使我对我正在做的事情充满热情，并且确信我们的使命是值得的，我认为散发自信是作为一名首席技术官最难的部分。当我相信某件事是对的时，我知道要相信自己的直觉，我通常对自己的选择和交付能力充满信心，但我仍然不断怀疑自己。我猜这就是所谓的冒名顶替综合症？

# 一锤定音

老实说，从软件工程师到 CTO 的转变并不容易，但我认为如果创业是你真正想要的，这并不像看起来那么难。

我个人喜欢做的不仅仅是编码。我对我的个人生活和工作生活融合在一起的事实感到很舒服，因为我一直想成为一名企业家。学会改变自己的心态很难，但我很高兴我做到了。尽管我以前从未管理过员工，但我喜欢这样，而且我认为只要你关心，你就能找到帮助员工成长的方法。最终，我认为最难的部分是真正的飞跃和足够的自信。相对来说，我在这方面还是个新手，但我相信随着时间的推移，这是自然而然的事情。
