## bug 在代码库中很常见。通常开发者也是问题的原因

bug 是程序最危险的敌人。人类[是大多数错误](/the-subtle-art-of-debugging-your-code-10c0239512ba)的原因，这意味着有可能努力减少它们。当代码被创建时，避免坏习惯、坏态度和坏环境会产生令人印象深刻的影响。

你还应该记住帕累托原则。在这种情况下，人类更容易出错。绿色是表示一切正常的颜色，所以对于导致错误的错误，记住深红色:复制粘贴、匆忙、即兴创作、多任务、浅薄、过度自信、疏忽。

# 1.复制粘贴

复制粘贴引入 bug 至少有两个原因。首先，复制的代码必须适应粘贴的目的地。这很糟糕，因为很可能会遗漏一小段代码。当再次阅读时，大脑不断跳过相同的单词和命令，所以它只是*假设*它们很好。代码越长，这就变得越困难。

第二个原因:维护是困难的，并且在以后编辑这段代码时插入一个 bug 是可能的。一个副本被更改，而另一个没有。这是干原则背后的原因之一。

# 2.冲

当我们感到没有时间去做某事时，我们会感到恐慌，并倾向于关注主要方面而不是细节。这通常是我们大脑的一个有效特征，因为它允许我们首先执行最有效的任务，获得近似的结果。很公平。然而，编码是一个精度问题。所以细节也很重要，程序员必须在开发的所有阶段投入时间。

注意不要在等待想法开花的任务上徘徊。相反，有必要花费每项任务合理需要的时间。错误的结果只会花更多的时间来处理。

# 3.即兴创作

当考虑新功能的实现时，应该仔细设计。无论你运行敏捷、看板还是 Scrum，软件设计都必须明智地实施，不能即兴发挥。这个问题应该被仔细地抽象和建模。有大量的模式可以分析。经验可以带来很大的不同。高年级学生不太容易即兴发挥，但这只是一个经验法则。

此外，每当新的东西被采用或集成到软件中时，就很有可能出现集成问题。对新组件文档的分析至关重要，不能使用临时解决方案来跳过。

# 4.多重任务

当代的大谎言。人类大脑有非常有效的任务转换能力，但几乎没有实际的多任务处理能力。转换是可能的，但是环境的改变有其自身的代价。再次，首先牺牲的方面是精度和注意力，这是编程要付出的高昂成本。

你应该有一个有组织的议程，没有会议的时间，消除噪音的耳机和通知杀手。番茄工作法和其他工作方法大大有助于减少长时间保持专注的努力。当不需要零延迟时，同事之间的独立性和异步通信(不为微小的请求打扰他人)可以产生很大的不同。

# 5.浅

当设计所请求功能的实现时，应该详细分析解决方案的复杂性——即使最终目标看起来很简单。这意味着当其他数字变化时，要考虑计算时间、工作记忆和持久存储的行为。一些例子可以是用户数量、保存的条目数量、系统正常运行时间、每秒请求数等。

当远离初始条件运行使得软件达到资源极限时，忽略复杂性迟早会导致意外情况的发生。这里必须进行一些数学计算。

# 6.自负

去实现而从不质疑编码的内容是否符合需求是非常危险的。对特征的描述非常重要，因为它应该阐明所有的问题。然而，有资历的熟练程序员可以质疑自己所做的事情是否真的有意义。怀疑有助于在推出被窃听的功能之前消除误解。

同样，这是一个使用正确衡量标准的问题:太多的怀疑会让你陷入困境，而过度自信会让你偏离轨道。

# 7.疏忽

应该考虑一些外部事件干扰软件的可能性，并且应该采取一些对策。后果越严重，干预就越重要。必须通过考虑适当的解决方案来防止由网络问题、缺少对所需资源的访问或任何外部依赖性引起的错误。

注意:这是一个混合的责任，由所有参与软件开发的人共同承担，从产品所有者和 UX 设计师到软件架构师和程序员。

# 结论

深红只是导致软件错误的所有可能原因的一个子集。值得注意的是，所有为软件做出贡献的人(包括经理和人力资源)都可以发挥作用并影响 CRIMSON，从而帮助提高软件的质量。

更高的软件质量是对这些努力的巨大回报，会让开发人员、管理人员和用户更开心。
