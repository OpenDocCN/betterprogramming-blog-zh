## 许多人将帕累托原则应用于软件开发，但是这真的有用吗？

人们[经常](https://www.google.com/search?q=80%2F20+rule+software+development)尝试将 80/20 法则应用到软件开发项目中。

这可以采取多种形式，例如:

*   80%的工作占用了 20%的时间
*   20%的用户会发现 80%的错误
*   80%的用户只使用 20%的功能
*   80%的变更发生在 20%的代码中

诸如此类…

# 帕累托原则

所有这些陈述都是基于一个叫做帕累托原则的东西。这概括为:

> " 80%的结果来自 20%的原因."投资媒体

这条规则是以 19 世纪意大利经济学家维尔弗雷多·帕累托的名字命名的，他注意到意大利 80%的土地被 20%的人口所拥有。

# 软件中的帕累托

对帕累托原则在软件开发中的应用进行了大量的研究，它们都[得出了](https://citeseerx.ist.psu.edu/viewdoc/download?doi=10.1.1.204.5154&rep=rep1&type=pdf#page=78) [不同的](https://dl.acm.org/doi/abs/10.1145/2804360.2804366) [事物](https://www.proquest.com/openview/0a80d280903e44652023f6dcc560dbe8/1?pq-origsite=gscholar&cbl=25782)。

一些研究人员发现 80/20 规则适用于开发人员贡献代码的比例，其他人发现该原则适用，但比例不同(例如，有时是 60/40，等等)。

另一个研究团队发现，当控制系统规模、团队规模和项目年龄时，该原则不适用于 40–90%的 GitHub 项目。

> 根据现有的证据，我们可以得出这样的结论:这一规则有时适用，但绝不是普遍适用的。

然而，我想挑战的是，当把这个原则应用于工作和时间时，我们所说的“工作”是什么意思。

# 80%的工作从不占用 20%的时间

当人们在软件产品中应用帕累托图工作时，他们通常的意思是:

> 80%的产品价值是用 20%的时间开发出来的。

我们可以把它解释为“一个工作原型可以在 20%的时间内完成，这展示了产品价值的 80%。”这里的问题是对*值*的理解。

对于许多利益相关者(产品经理，首席执行官)来说，原型比实际工作的软件更有价值。当我们说“20%的工作占用了 80%的时间”时，言外之意是这 20%并不重要——也没有其他 80%那么有价值。

# 100%的工作需要 100%的时间

不要再说 80%的工作占用 20%的时间了。这是有害的，而不是有益的。

它所做的只是强化紧迫的截止日期，并让团队感觉他们花费大部分时间的工作并不重要。

相反，让我们宣扬这样一种观点，即这“剩余的 20%”实际上是高价值的，需要被记入并计入时间表，而不是被视为妨碍按时交付特性的不便之处。

测试和修复 bug 的周期可能在团队会议中看不到，但是如果用户遇到 bug，他们可能会删除你的应用程序并要求退款！

> 修复和特性一样重要。
