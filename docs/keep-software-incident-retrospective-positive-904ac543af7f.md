在软件开发中，我们会遇到事故。事件发生后，进行一次回顾是很重要的。我们如何处理回顾会塑造组织文化的一部分。

下面我将分享一个案例，并提供三种不同的回顾风格，以及团队成员将如何对待它们。

在分享回顾性回应之前，我先分享一下案例的例子。

# 一个示例场景

哦，事件爆发了！一个软件指标已经丢失了三个星期，并引起了注意。在一个小时内，数据人员四处寻找，然后发现丢失的数据都来自应用程序。网页数据没问题。

在接下来的三到四个小时里，更多的人被拉进来，并试图找出什么发生了变化，哪个版本的应用程序导致了这个问题。最后，在一天结束之前，确定了导致问题的应用程序的确切代码更改。

因为已经是营业时间的末尾，每个人都下了车。但是一位开发人员整晚都在调试，并找到了解决问题的方法。这不是一件容易完全测试的事情，因为修复是对一个库进行设置。所以开发者要等数据团队来测试。

第二天，一大早，开发人员发来消息要求帮助测试这一变化。数据团队对变更进行了测试，现在一切正常。一个新的构建启动并向公众发布。

使用上面的场景，我们可以有三种方法来生成回溯风格。它们都是相似的项目，但是处理它们的方式不同，这导致了不同的组织行为。

# 回顾风格 1:什么出了问题

这种方法的思路是，事件的发生一定是出了什么问题。所有事件都是不好的，需要补救。

每个人都坐在一个房间里，列出出了什么问题。以下是提出的问题:

1.  为什么没有更早发现这个问题？为什么我们需要等三个星期才注意到这个问题？
2.  为什么没有一个单元测试可以在开发过程中测试，以在一开始就防止 bug？
3.  为什么应用程序开发人员在修复后不能全面测试更改，而是需要依靠数据团队来完成？
4.  为什么更改列表没有变得更清晰，这样我们就不需要花三到四个小时来找出是什么更改导致了问题？

这些都是有效的问题。但是如果我们没有适当的处理就发布出去，人们会变得有所防备，回顾会的气氛变得紧张。

任何人都可以问，因为不需要问知识的背景。另一组的团队成员可以问另一组，反之亦然。这可以成为一个跨团队的鼓舞人心的会议。这培养了一种不良的指责文化。

这种追溯方法类似于它的旧称“事后分析”

# 回顾风格 2:学习模式的教训是什么

在这种方法中，想法是，我们有一个事件没关系。有一些教训要吸取。

有些事实已经摆在那里，没有必要问为什么。相反，每个小组陈述一些事实，并探讨学到的经验教训。以下是收集到的事实:

1.  我们在三周后才发现这个问题，因为数据趋势只有在两周后才变得清晰。该团队仅每周收集一次数据。我们现在知道，这对于快速反馈循环来说是不够的。
2.  应用程序团队不能添加单元测试，因为更改只是对库的设置。没有反馈机制让它从库中被测试。现在我们知道，库设置虽然对应用程序逻辑是透明的，但同样重要的是要测试，也许至少要手动测试。
3.  应用程序开发人员无法全面测试更改，因为开发人员无法访问指标数据。现在我们了解到，应用程序开发人员没有完全访问快速端到端测试的权限。
4.  变更列表不清楚，因为由于假期休息，几个星期没有发布。常规周会有一个更短更清晰的变化列表。我们知道没有发布的长时间开发会使问题的检测更加困难。

要提出事实并从中吸取教训，只能由负责范围本身的团队来完成。这是因为来自其他团队的成员没有那么多背景知识。

使用这种方法，更像是一种合作的方法，每个团队都认识到自己的局限性并从中学习。从经验教训中，我们可以制定行动来改进流程。

会议将以每个团队的额外任务项目结束，每个人都在改进中发挥作用。这其实并不坏，因为它培养了一种积极的学习文化。

# 回顾式 3:未来的机会是什么

这种方法的思路是，每个人在这个时间点都尽了最大努力，我们有尽可能多的最佳系统来防止问题。任何新的事件都是未来的机会，让它变得更好！

这次回顾会的开始方式是思考，让我们振作起来，我们发现了这个问题，并得到了解决。现有的系统正在工作！当然可以更好，但是先看看效果如何。

1.  我们的团队能够发现这个问题真是太好了，因为我们确实有一个常规的监控程序！事实上，我们可以深入应用程序问题，而不是网络，因为我们有一个很好的过滤机制！现在，下一个机会是，也许我们可以探索如何改进响应时间。也许可以用警报系统来自动化这个过程？
2.  该应用程序团队成员在知道这个问题后，花了一晚上的时间来解决这个问题。我们赞赏为在第二天之前解决这一问题所付出的巨大努力。干得好！！对于这种代码更改，没有单元测试可以防止问题的情况很少见，因为这只是库设置的更改。但这意味着有机会与图书馆提供商联系并合作，对图书馆进行新的改进！
3.  我们在应用程序团队和数据团队之间有很好的协作团队，形成了端到端测试循环的强大结合。伟大的团队合作。更好的是，我们可以探索工具和系统的交叉共享，这样应用程序开发人员可以访问一些数据监控工具，而数据团队成员可以有机会出于学习目的偷看应用程序代码？
4.  很高兴我们有一些更改列表，我们可以检测到导致问题的代码更改！把问题缩小到确切的地方会很有帮助。我们可以通过对变更列表有一个更好的标题约定来使它变得更好，也可以有一个更有规律的自动构建，即使是在假期。

表扬和反思有效的方法(这是真的，否则事件可能无法解决)将在回顾中提高团队的士气。它可以来自任何团队。这是有感染力的，也就是说，一个团队对另一个团队的赞扬通常会得到赞扬和感激的回报。

当士气高涨时，团队自然会探索我们如何做得更好，并主动改进事情。当团队处于积极状态时，会产生更好的想法。

这不仅仅是一个改进的机会，培养健康积极的文化，它还激励和加强团队合作，带来积极向上的氛围。

# 所有事件都是前方的机会

在所有的回顾中，从来不是指责的地方，也从来不是针对个人。所以，为什么不积极一点，把它看作一个机会，而不是一个问题。

事故之所以会发生，只是因为我们没有适当的流程和系统来预防它。软件开发围绕着过程和自动化，随着我们的学习，这些过程和自动化应该继续改进，使之更加健壮。

任何不会让我们的业务消亡的事件都是一个机会，无论是在技术上还是在分享给其他人一起学习的好故事上。
