你的估计有什么问题(以及如何改进)

原文：https://betterprogramming.pub/whats-wrong-with-your-estimates-d5c7cd143184?source=collection_archive---------18-----------------------#2022-05-18

如果你经常不能完成你的评估，这就是如何快速准确地评估任务的方法。

每个或多或少重要的项目都有其利益相关者。利益相关者总是希望你给出评估。

假设你是建筑公司的首席执行官，你的客户要求你建造一栋大楼。但是他们想让你先估算一下。他们想知道在给定的时间和预算限制下，目标是否可以实现。以下是您应该估计的示例:

  1. 需要筹集多少投资？
  2. 整个项目需要多长时间？
  3. 你会付给你的分包商多少钱？
  4. 需要多少材料？
  5. 会容纳多少人？
  6. 这座建筑将存在多久？

一开始，你永远不会知道这些问题的答案。这些答案也是你自己、你的客户、分包商、雇员、未来居民和市政当局感兴趣的。

只有把房子建到底，你才能找到这些问题的确切答案。对其余的人来说——即使过了一段时间。

软件开发也是如此。唯一的区别是我们需要更频繁地评估任务。敏捷方法要求我们有一个短的计划范围，这意味着定期的评估。

谁需要软件评估？

顾客

对于客户来说，估价就是价格。他们可以通过将他们的商业价值与他们的估计进行比较来区分特性的优先级。

项目管理人

对于PMs来说，评估是计划工具。在估算的帮助下，他可以了解截止日期，计算预算，组建团队。

开发团队

对于开发团队来说，评估是一种度量，通过它他们可以表达他们必须做的工作的数量和复杂性。他们可以衡量自己的效率，知道自己是否足够高效。

估计的主要困难

好吧，那还有什么问题？为什么评估的话题经常被提起，为什么有这么多的实现方法？

我们试图战胜的敌人是未知。

我敢打赌，你已经目睹了一项四天内完成的任务是如何在几周内都无法完成的。有时，一个可怕的bug威胁到应用程序的整个架构，被认为是“以防万一”,最终由一个新人在几个小时内成功地制造出来，而没有任何后果。

为什么会这样？因为现实和我们的预期不符。这并不是因为某人注意力不集中或缺乏经验。发生这种情况是因为估计是基于不完整的信息。这通常发生在分析阶段投入的精力太少的时候，或者更常见的是，团队根本没有准备好进行评估。

评估规则

我推导出以下规则来开始估算，除非全部满足，否则我从不估算:

  1. 需求是准备好的和完整的——不要评估你看不到的需求，它们没有完成，或者你确定它们会被改变。
  2. 只估计你所知道的具体会怎么做。如果你不知道你将如何执行某些功能，你将无法对其进行定性评估。最后分析、计划并开始评估。
  3. 估计的任务越小越好。不要试图估计一个任务太大，以至于它不容易在你的头脑中出现。总是把它分开，估计它的部分。
  4. 使用了估计框架。整个团队应该从一开始就知道评估过程。决定你将使用的测量单位、方法和途径。确定最适合你的团队的框架，并坚持下去。

测量单位

您很可能听说过，评估通常是以小时或任意单位(例如，以故事点为单位)进行的。

我希望随着时间的推移，一切都变得清晰。

同时，故事点(SP)是一个我们在日常生活中不涉及的价值。它不是Scrum的一部分，大多数团队只是凭直觉理解它。通常，把头脑中的时间转换成小时是一个常见的错误。

让我们弄清楚故事点代表什么:

  1. 故事点-是一个相对值。同样的任务，在团队里可以估算为2 SP，在另一个团队里可以估算为21 SP。两者都可能是对的。最重要的是所有团队成员对1 SP背后的东西有相同的理解。
  2. 故事点并不意味着完成任务所需的时间。它反映了任务的复杂性、工作量和不确定性。任务越大越复杂，我们知道的越少——花费的故事点就越多。
  3. 故事点是团队速度的一个很好的度量。在开始的时候，你可以假设你的团队在每一次冲刺中能够完成多少故事点(例如45个故事点)。然后当冲刺结束时，你可以回顾一下你是否猜到了，为下一次冲刺调整你的速度值。在几次冲刺中，你会得到非常接近真实值的值。这将帮助您更准确地定义您团队的sprint范围。
  4. 在评估过程中有一个计数系统是很好的。例如，最受欢迎的是“斐波纳契系统”，在该系统中，您只能使用下列值之一进行估算:1、2、3、5、8、13、21等等。像这样的系统将帮助你避免浪费时间去怀疑这个任务是9 SP还是10 SP。

评估技术

好了，我们准备好开始评估我们的任务了。但是是怎么做到的呢？

有几十种评估技术，但我将为您挑选出我自己使用的技术，它们已经在实践中证明了它们的有用性和有效性。

德尔菲法

这种方法是基于几位专家的估计。

首先，他们中的每一个人都单独地估计这个项目。然后比较所有的估计值。

如果评估中有差异(它们通常存在，这是好的)，那么它们会被公开讨论。讨论结束后，再进行一轮这样的讨论。如此循环，直到在单个估计中达成一致。

回合数是有限的，一开始就协商好了。如果达到了轮次的限制，并且在单个估计中仍然没有达成一致，那么您需要查看所有的估计。

通常，如果差异很小，则采用较大的估计值。

如果差异很大，那么团队不准备给出估计(例如，它没有足够的信息)。收集必要的信息，准备好之后，从头开始。

类比估算

这是最快的技术之一。而且在有一定历史估计和经验的情况下，也是非常准确的。

以过去最相似的已完成任务为基础。为了确定最大“相似度”，您需要考虑以下因素的相似程度:

  1. 任务大小；
  2. 任务复杂性；
  3. 执行该任务的团队的组成；
  4. 依赖性(内部和外部)；
  5. 不确定性。

找个类比。定义区别。通过差额调整你的分数。

请注意，这项技术的效果越差，任务的相似性就越少。如果您确定估计的任务与其类比之间的差异相当大，那么最好拒绝使用这种方法。在其他情况下，可以随意使用它，因为它是最快和最准确的。

工作分解结构

这里的技术是将我们的任务分解成原子子任务，每个子任务都可以很容易地估计。

任务要分解到什么程度？通常，当每项任务都很容易评估，并且在每项任务中，完成任务所需的所有步骤都很明显时，就会有这种感觉。

如果有一项任务的步骤不明显，我们就把它分开。如果我们不能评估任何任务，我们就把它分开。我们的目标是得到一组小的估计值。

请注意，对由此产生的问题的估计将不是对其组成部分的估计的简单求和。您必须调整最终结果，考虑子任务、风险和不确定性之间的依赖关系。

选择什么技术？

为了选择最合适的方法，我通常使用以下方案，包括3个问题:

  1. 我们过去做过类似的任务吗？->类比
  2. 你知道完成这项任务的所有步骤吗？->德尔菲
  3. 你能估计它包含的子任务吗？-> WBS +德尔福

你对前3个问题的回答是“否”吗？你可能没有足够的信息。如果可能，收集数据，如果仍然不够，尝试分别使用上述所有方法给出一个“粗略”的估计，并以区间的形式给出一个最终的估计(例如，13–21 SP)。

记住，评估只是工具。需要它们来简化应该完成的工作的计划。

给出一个粗略的估计并开始实现总比浪费时间精确估计但不开始实现要好。
