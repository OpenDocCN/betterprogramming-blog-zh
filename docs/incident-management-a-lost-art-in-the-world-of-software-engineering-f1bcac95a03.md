## 如何以最有效的方式处理事件

在过去的两年里，作为一名伦敦的专业软件工程师，我学到了很多关于事故管理的知识。每当实时生产系统中出现问题并对业务产生某种影响时，这就被认为是一个事件。软件工程师的职责之一是快速高效地解决事件，以便尽可能地将对业务的影响降至最低。我认为，当典型的软件工程师处理事故时，有几个方面被忽略了或者做得不够好。

# 1.敷绷带。不要去做手术

处理事件有点类似于随叫随到的医生。医生突然被传呼到急诊室，那里的病人正在缓慢(或快速)出血，正在走向死亡。同样，每当系统崩溃时，软件工程师就会收到传呼，而该系统的崩溃会导致企业损失越来越多的资金，直到企业破产。

第一个合乎逻辑的步骤是确定业务影响。一些错误的影响可能非常小，而另一些错误的影响可能是有害的。例如，如果你是一名软件工程师，你被告知用户的数据库被黑客攻击，这属于有害影响。

但是，如果您收到关于一个较小的 UI 样式不适应的页面，这在影响方面是非常小的。确定影响是很重要的，因为它会让你有一个正确的心态，知道你需要多快解决这个事件。

这里需要注意的重要一点是，如果影响是有害的或大的，最重要的是不要惊慌失措，冷静下来做出最好的决定(这当然是说起来容易做起来难)。下一步是确定如果需要，是否需要寻求帮助。记住，没有人知道所有的事情，你最终会在一个团队中工作。如果影响是有害的，此时请求快速帮助要比以后后悔好得多。

一旦你寻求帮助并确定了影响，下一步就是尽快使用最好最快的绷带。这不是所有软件工程师都具备的技能，因为许多软件工程师习惯于坐下来，诊断问题，并以非常冷静和缓慢的方式设计解决方案。

其他软件工程师更喜欢快速、频繁地动手。这些软件工程师通常在调试和事故管理方面更加熟练。软件工程师的核心技能之一是你调试系统和找到修复方法的速度和效率。

我想说的主要观点是，你需要进入“解决这个问题最快、最肮脏的方法是什么？”。因为许多软件工程师都是完美主义者，他们喜欢深入问题，尽可能地理解它，并得出最佳解决方案。当然，这是一项非常好的技能，也是非常需要的，但在事件管理的情况下，这是最糟糕的思维方式，因为它会导致非常缓慢的解决。

# 2.监控、记录和警报

值得注意的是，监控、警报和日志系统越好，事情就越容易。即使是最熟练的软件工程师也无法在没有日志/监控系统的情况下快速调试包含大量微服务的大型系统。

为了完善事件管理的艺术，请始终自问您希望能够快速、近距离地查看系统的哪些部分。哪些部件至关重要，以至于当它们停止工作时，您希望立即得到通知？

# 3.事故后验尸

事故验尸是我见过的最佳实践之一。事故验尸报告实质上是从以下方面总结事故的文件:

1.  发生了什么，为什么？
2.  决议是什么？
3.  使用了什么监控和警报系统？(触发器等..)
4.  如何防止这种情况再次发生

当然，内容可能会根据每个业务而有所不同，但整个概念是分析已经发生的错误以及它如何永远不会再次发生。如果在整个业务中完成，这是非常有用的，这样每个团队也可以从其他团队的错误中吸取教训。

# 最后的想法

我希望您已经很好地了解了软件工程师在进行事件管理时需要了解的基本技能和部分。归根结底，事件管理是一项技能；没有经过深思熟虑和持续不断的练习，没有人能掌握好技能。但是，我认为从其他人的经验中学习和阅读其他软件工程师的工作也是非常重要的。
