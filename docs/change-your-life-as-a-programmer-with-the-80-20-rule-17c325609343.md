# 什么是 80/20 法则？

> 帕累托原则(也称为 80/20 法则、重要少数法则或因素稀疏原则)指出，对许多事件来说，大约 80%的结果来自 20%的原因— [维基百科](https://en.wikipedia.org/wiki/Pareto_principle)

一般意义上，80%代表多数，20%代表少数。因此，尽管精确的百分比可能不适用于每一条规则，但同样的方法是适用的。

# 1.当选择何时工作时

我认识的大多数程序员都会犯这个大错误:尽管提供了更灵活的工作时间，他们还是朝九晚五地工作。他们认为这是他们高效工作的方式，即使他们中的许多人在下午 1 点后感到疲倦。下面是 80/20 法则的帮助:明白你 80%的工作将在你每天 24 小时的 20%中完成。所以你最有效率的时间可能是从早上 5 点到 9 点。按照这个标准，你永远不知道在几个小时内你能完成多少工作。

# 2.选择功能时

你的大多数用户只会使用你给他们的 20%的功能。因此，你应该将 80%的精力放在这些功能上，并使它们成为你的主要功能！

# 3.当对待办事项列表排序时

作为程序员，我们都创建待办事项列表来完成我们的任务。大多数情况下，20%的待办事项会占用你 80%的时间。因此，整理你的待办事项清单可能会帮助你更快地完成更多的任务，或者先完成最大的部分。根据这个规则整理你的待办事项清单将会帮助你保持更长时间的动力，并且知道你的待办事项清单需要多长时间来完成。

# 4.当开始一个项目时

花在编码项目上的 80%的时间应该花在开始的前 20%上。集思广益，创建编码结构和计划将有助于项目更快更容易地进行。因此，在尝试编写所有代码之前，确保在项目的前 20%投入足够的精力。

# 5.当学习新东西时

你只需要学会 20%的东西就可以开始使用它。所以假设你想学 Java。选择正确的 20%从中学习，会帮助你更快的掌握(即学习面向对象的原理、语法等。).一旦你掌握了这些原则，你就可以自信地开始编码了——即使你对 Java 还不了解 80%!

# 6.调试时

调试可能会花费你几个小时的编码时间。所以很多开发者说他们 80%的 bug 在 20%的代码里。因此，当您发现一个 bug 时，明智的做法是投入更多的时间来调试一个代码补丁，因为在同一个代码片段中可能会有更多的 bug。

# 7.当选择一个想法时

如果你是一名开发人员，你可能会对一个新的应用程序有上百个想法。所以知道你的想法中有 20%是值得去做的，并且是可行的，可能会帮助你选择哪个想法是最好的。不要相信外面的任何想法。

# 结论

总是有更多的。让我知道你是否有 80/20 法则的其他应用。
