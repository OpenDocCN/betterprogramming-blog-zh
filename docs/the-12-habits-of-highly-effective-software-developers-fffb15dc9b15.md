## # 3——不要自作聪明

![](img/4f5d5d321fcb5d02acc5827d06bbe565.png)

迈克尔·斯科在 [Unsplash](https://unsplash.com/s/photos/12?utm_source=unsplash&utm_medium=referral&utm_content=creditCopyText) 上的照片

我和许多软件开发人员一起工作过，他们有些刚从大学毕业，有些则是经验丰富的专业人士。这篇文章列出了这些人似乎共有的一些特征。

# 1.你不会需要它的(YAGNI)

不要写你现在不需要的代码。写一些额外的代码很有诱惑力，因为你认为你以后会需要它。这里的问题是双重的。

1.  你可能根本不需要它。现在有代码闲置在那里，没有人敢删除它，因为谁知道如果他们删除了会出什么问题。
2.  未使用的代码不会得到更新。它可能会引入原本不需要的错误和安全漏洞。

节省你自己的时间；你用不着了！

# 2.避免过早优化

就像 YAGNI 一样，过早地优化代码也很诱人。这里的风险是:

*   你的代码对其他人来说不太清楚。
*   你会花时间在一个可能不存在的问题上。

这两点我们来阐述一下。

首先，为了速度而优化代码通常会使代码更难理解。你没有走显而易见的简单路线，而是在根本不需要的地方实现了诸如缓存或循环展开之类的东西，或者任何花哨的技术。

你是在增加复杂性，而复杂性有很多不好的一面(参见:KISS)。

其次，你不知道你的代码将如何执行，直到你运行它。所以，在出现实际问题之前，请不要花时间优化它。

通常，你会发现你的代码有多快并不重要。CPU 周期便宜；工作时间不是。您也可以添加更多的 CPU 能力或等待更长的时间，以避免复杂性和潜在的错误。

# 3.不要自作聪明

比起聪明，我更喜欢清晰。一些超级酷的一行程序技巧可能很适合炫耀，但对于将来阅读您代码的人来说却是一种痛苦。不要成为那样的人。请在你的博客中发布这些很酷的技巧。

# 4.不要重复自己(干)

初学者经常重复代码，做相同或几乎相同的事情。假设您想打开一个文件并阅读其内容。你可以写几行，这样做很容易。

但是，如果您需要读取另一个文件并获取该文件的内容，不要编写相同的代码来打开这个不同的文件。或者更糟:复制粘贴它！

相反，你应该意识到你可以创建一个函数。这里有两个显著的优势:

*   代码越少，维护和调试就越容易。
*   小功能容易测试(见*单元测试*)。
*   一个命名良好的函数是文档的一种形式(参见:*记录你的代码*)。

专业提示:有些 ide 会检测重复代码并警告您，有些甚至会帮助您从重复代码中提取方法或函数。

# 5.创建单元测试

单元测试被许多人忽视。我对此也感到内疚。我经常发现自己在事后创建单元测试，如果有的话。但是即使你在事后添加了它们，也比根本不添加要好。

在最极端的形式中，您应用一种叫做测试驱动开发(TDD)的实践。使用 TDD，您首先创建一个单元测试，然后实现该功能。它迫使你测试你创建的每一个函数，并仔细考虑这个函数需要做什么以及预期的输出是什么。一本关于这个主题的好书是[测试驱动开发:举例](https://amzn.to/2MIHTRa)。

创建单元测试的另一个好处是，您或其他人可以更加自信地更改代码。变更之后，您运行所有的测试。如果没有失败，你刚刚创造了一个突破性变化的机会是渺茫的。

创建单元测试有助于您:

*   用更少的错误写出更好的代码。
*   在以后的体育场中更自信地修改您的代码。
*   迫使你创建更小的函数来做一件事。
*   通过给出使用示例来记录您的代码。

# 6.保持简单，笨蛋(吻)

这个强大的咒语不仅仅用于软件开发。本质上，它意味着“不要把事情搞得太复杂”或者“尽可能想出最简单的解决方案。”

计算机科学的先驱之一 Dijkstra 曾经说过:“简单是可靠的前提。”你的解决方案越简单，就越难引入 bug。就这么简单。

> "简单是可靠的先决条件."

# 7.坚持固定的代码风格

坚持一种编码风格。尤其是在团队工作的时候。举个例子，有些人喜欢用这样的称赞:

```
while (true)
{
  // do cool stuff
}
```

而其他人喜欢这种更紧凑的风格:

```
while (true) {
  // do cool stuff
}
```

两者都有利弊。但是无论如何，坚持一个。如果你在一个团队中，这可能意味着你需要坚持一种你不喜欢的编码风格。

在这方面，每种语言都有自己的工具和事实上的标准。你需要搜索你所选择的语言的最佳实践。此外，对于大多数语言，您可以在 IDE 中安装自动格式化程序。它可以节省你很多时间，所以一定要找到一个你喜欢的。

以下是一些帮助您入门的链接:

*   谷歌有多种语言的[风格指南](https://google.github.io/styleguide/)。
*   对于 Python，有 [PEP8](https://www.python.org/dev/peps/pep-0008/) 。许多 ide 都有 PEP8 插件，可以在你输入时检查你的代码。
*   对于 Java，您可以使用像 [Checkstyle](https://checkstyle.sourceforge.io/) 这样的工具来定义一种样式(或者使用一种现成的样式)并在编译期间强制执行正确的代码样式。在一些初始设置之后，一些 ide 也可以在这方面帮助你。

# 8.记录您的代码

有三种方法记录代码:

1.  在代码中使用注释。
2.  在单独的文档中编写文档。
3.  编写自我记录的代码。

从注释开始:稀疏地使用它们。只在需要澄清的地方使用注释，不要陈述显而易见的东西。

写文档可能是有益的。想想所有那些 GitHub 库。在项目的根中包含一个`README.md`文件已经成为事实上的标准。

这个文件描述了一些基本的东西:

*   这是什么代码？它解决什么问题？
*   如何开始编写这些代码？包括创建开发环境的具体说明，如果有的话。
*   最终用户如何使用软件？
*   提供相关链接，如更多的文档，背景信息等。
*   人们应该在哪里以及如何寻求帮助？

在某些领域，明确的方向可以拯救生命。照片由 [Calle 马卡龙](https://unsplash.com/@callemac?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄

最重要的是编写记录自身的代码。这也是最难的部分，因为它需要经验和最佳实践。为了更好地说明这一点，以下是一些有用的东西:

*   使用好的设计，这样你的代码库就容易导航并且结构合理。
*   遵循最佳实践，比如干(不要重复自己)，YAGNI(你不会需要它)，亲吻(保持简单，笨蛋)。
*   不要试图保存字符。使用变量、类和函数的全名。所以，不要用`wm`，命名为`windowManager`。不要叫它`rf`，叫它`readFileToString`。当其他人——或者你自己，在几个月不看代码之后——试图理解发生了什么时，这个名字非常有用。
*   尽可能多地提取到函数中，并让这些函数做一件事情。给它们起相应的名字，从动词开始。例如，将文件读入字符串的函数可以命名为`readFileToString(String fileName)`。不用详细阅读代码，人们就知道它是做什么的。理想情况下，你的代码是一系列像这样的函数调用，读起来几乎像人类语言。只有在需要的时候，读者才能更深入地阅读。这段代码记录了它自己！

# 9.寻求帮助:正确的方法

![](img/8f77b652bd4cbc3e131d35576db5f74d.png)

照片由 [Jametlene Reskp](https://unsplash.com/@reskp?utm_source=medium&utm_medium=referral) 在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄

专业人士只有在广泛尝试通过其他方式找到答案后才会寻求帮助。在你提问之前:

*   阅读文档—也称为 RTFM，是“阅读该死的手册”的缩写
*   如果文档不够清楚或者没有解决您的问题，请在 Google 上搜索答案。

如果这没有帮助，考虑首先从哪里寻求帮助:

*   bug 追踪器不是询问与(潜在)bug 无关的问题的地方。
*   开发人员邮件组是为开发产品的开发人员服务的，而不是使用产品的开发人员。
*   许多项目都有一页指导你如何以及在哪里提问。
*   有大型的脸书团体专门研究特定的编程语言和技术，来询问更一般的问题。根据我的经验，它们可能很乱，有很多噪音，但通常你会得到你的答案，所以值得检查一下。

最后，在写下你的问题之前，请记住以下几点:

*   要善良，要感恩。那些回答你并试图帮助你的人通常是在业余时间免费这样做的。
*   尽可能详细。提供背景:你在做什么，为什么，到目前为止你尝试过什么？
*   包括错误消息、(部分)日志文件以及引起问题的代码。不要转储整个文件。仅包括获得适当帮助所需的内容。

总的来说，尊重他人的时间。

# 10.重构

重构是在不改变代码行为的情况下重新构建代码。

![](img/b9657f517683522e1a5c482f3414a199.png)

micha Parzuchowski 在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的照片

你到底为什么要这么做？显然，是为了改进您的代码！生活中有几个事实使得重构成为必要:

*   你的软件从来不会一次就完美无缺。当然，这可能行得通，但会很混乱，而且会有重复。
*   计算机代码在不断进化。即使你从一个完美的代码库开始，它也会很快变得混乱。

当你在做一个新项目时，你首先需要做一些重大的改变，比如将你的初稿重新归类。一旦完成，重构并没有停止。

你应该养成一个习惯，每次你写代码的时候都要逐步改进。*每次都留下一段比你发现时更好的代码。*

随着时间的推移，您所做的小改进会累积成一个易于阅读和维护的代码库。

![](img/075adae0483653dbea2cbf871b7e2428.png)

不要让你的代码演变成雷区。照片由[罗伯特拜拜](https://unsplash.com/@robertbye?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄

不断重构的一个重要效果是，你的代码不太可能变成别人的雷区。

也许你以前见过这样的代码。如果你改变了一行，整个事情就会不可理解地破裂。不断地重构、重组和改进你的代码可以防止它变成一个雷区。

# 12.不断学习

专业人士在其职业生涯中不断学习。在 IT 世界中有一个不变的东西，那就是变化。

每年都会引入新的语言。新的 JavaScript 框架似乎每天都在推出。你必须不断学习，以保持你的知识与时俱进。
