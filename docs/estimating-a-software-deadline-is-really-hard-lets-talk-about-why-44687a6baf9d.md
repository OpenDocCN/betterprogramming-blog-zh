## 规划时你需要知道的 5 条法则

随着你职业生涯的进展，你可能会发现自己承担越来越多的责任，直到你发现自己处于管理角色。你将对团队输出的工作的交付和质量负责。

为了确保团队保持充分参与，并且团队成员得到有效利用，您需要为未来做好计划。

> "如果你必须预测，那就经常预测."——埃德加·费德勒

规划未来听起来容易做起来难。每一个规划决策，每一个预测的完成，都是基于大量的假设。如果这些假设中有任何一个是错误的，你就可以和你的计划说再见了。

当你在估算完成一个项目所需的每项任务所需的工作的黑暗艺术中变得有经验时，你会发现需要改进的地方，并开始实施你自己的*法则。*

也就是说，你会想出一些规则来帮助你有更准确的估计。

在这篇博客中，我们将介绍一些广为接受的定律，它们可能会帮助我们更好地制定计划。

# 1.霍夫施塔特定律

> "它总是比你想象的要长，即使你考虑到霍夫施塔特定律."

这条定律是在 1979 年提出的，它在今天和当时一样正确。这是一个在软件社区被广泛接受的法则，它强调了估算前期工作需要多长时间的难度。

为了解决这个问题，不同的团队和组织开发了许多技术。一个快速的谷歌搜索会给你提供比你所能使用的更多的想法。

我最喜欢的手法是一个叫 [*策划扑克*](https://en.wikipedia.org/wiki/Planning_poker) 的东西。规划扑克是一项基于团队的活动，需要对整个团队进行评估。

团队中的每个人都会拿到一副通常遵循斐波纳契数列的卡片，然后被要求评估一项任务。一旦每个人都选择了他们的估计，他们把他们的卡放在桌子上，面朝下。

当每个人都这样做了，每个人把他们的卡转过来，宣布他们的估计。随后是团队讨论，试图收敛到广泛接受的估计。

这种方法的优点在于它的子组件。作为一个基于团队的活动，确保了整体共识的胜利，减少了遗漏工作方面的机会，并增加了团队对评估的主人翁意识。

使用斐波纳契数列意味着当我们在数列中向上移动时，每个可用数字之间的距离会增加。这有助于满足大型工作带来的额外复杂性。

# 2.墨菲法则

> “凡是可能出错的，都会出错。”

这是一个著名的格言，你可能以前遇到过。它适用于各行各业，当然也适用于软件开发工作的评估过程。

墨菲定律和我们之前看到的霍夫施塔特定律非常吻合。当做出估计时，人们倾向于乐观，可能不会迎合任何大的惊喜。

即使我们改变我们的估计来迎合这一事实，霍夫施塔特定律提醒我们，我们仍然可能低估。

为了确保我们的估计有一个舒适的水平，我建议我们总是建立一些应急水平。一般来说，10-20%的意外开支不仅仅是建议的，很多时候也是预期的。

# 3.帕金森定律

> "工作不断扩大，以填补完成工作的时间."

帕金森定律实际上是可以在个人层面上观察到的东西。

截止日期以一种有趣的方式塑造了我们处理问题和工作的方式。当我们有一个很长的截止日期，我们可能会分心，(阅读:拖延)，我们可能会以不同的方式处理问题。

严格的截止日期往往会让我们集中注意力，并允许我们跳过不必要的部分。诚然，质量可能会受到影响，以满足最后期限，但值得记住。

那么，我们该如何利用帕金森定律呢？

根据我们之前的结论，在我们的估计中加入一个意外事件，我们需要确保我们将这个额外的时间与最初的估计分开。

如果我们把这些额外的时间分配给每一项任务，我们最终会患上帕金森定律，耗尽我们的应急能力，却没有任何益处。

# 4.斯特金定律

> “百分之九十的东西都是垃圾。”

当涉及到估算时，斯特金定律通常有助于简化事情。

每当对现有解决方案的质量或工具的可用性存在未知时，人们可以简单地参考斯特金法则。这是一个很好的起点，因为事情很少会比这更糟。

有了我们目前所看到的，我们可以轻松应对意外。为了让事情变得更简单，我们可以确保我们估计的任务尽可能的小。

我们不想做得太细，否则，我们将永远无法完成这个过程，但是我们也不想估计太大的任务。

# 5.布鲁克斯定律

> "给一个后期的软件项目增加人力会使它变得更晚."

这个定律是在流行的软件工程和项目管理书籍 [*中杜撰出来的神话人月*](https://en.wikipedia.org/wiki/The_Mythical_Man-Month) *。*在处理评估和计划如何在截止日期前完成时，这是需要记住的一点。

这一定律的基础是经济学中的收益递减定律，在某一点上，给一项任务增加新的资源，产出开始减少而不是增加。

当然，这个定律过于简单化了，因为在很多情况下它并不成立。例如，如果加入项目的资源已经熟悉了代码库和项目，他们可以立即投入工作并获得即时的收益。

此外，如果最终交付的最后期限是几个月后，额外的资源可能有时间扭转局面。通常，让新人加入项目会从现有资源中抽出时间来帮助回答问题，并让他们在一开始就安顿下来。

# 摘要

在本文中，我们从这样一个前提开始，即您将负责项目的预估和规划。然而，正如您可能已经从文章中看到的，团队中的每个人都将对过程有所贡献。

让我们快速总结一些要点:

1.  在评估即将到来的工作时，让团队作为一个整体参与进来，以增加集体主人翁意识，并更好地验证评估。
2.  总是尝试估计较小的工作(在合理的范围内)，以获得更好的估计。
3.  总是在项目工作中加入一些意外情况，以应对意外情况。
4.  将偶发事件与其余的估计分开，不要将它分配到每一项工作中。
5.  向已经延迟的项目添加额外资源时要小心，因为它们会导致项目进一步延迟。
