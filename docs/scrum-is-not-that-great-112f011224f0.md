# Scrum 并没有那么好

> 原文：<https://betterprogramming.pub/scrum-is-not-that-great-112f011224f0>

## 在一个同事在周期回顾的那天筋疲力尽之后，我意识到冲刺承诺是多么的有压力

![](img/20ed9bcb251142fb99401d4ec0dcfc61.png)

制造者在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上 [NESA 的照片](https://unsplash.com/@nesabymakers?utm_source=medium&utm_medium=referral)

几年前，在一个星期三的下午— *周期(或冲刺)评审的日子—* 我的团队正在努力合并一个 PR 并满足发布截止日期。我们已经延续了上一个 sprint 的故事，所以再拖一遍是不可行的。有太多关于公关的评论，虽然我们跳过了午餐去做它，但它还远远没有准备好。随着最后期限的临近，我的同事丢掉了他的东西，诅咒了所有人，收拾好他的笔记本电脑，怒气冲冲地离开了。

那天我有所改变。这些冲刺截止日期和围绕它们产生的虚假的紧迫感给我们的团队带来了太多的压力。在公司每个人都参加的每个周期回顾中，scrum master 展示了我们承诺的故事中有多少已经完成，以及哪些团队没有坚持他们的承诺。

让我震惊的是，只有工程师的工作才会受到如此严格的审查。公司里的其他人很少被密切监视。这让我很好奇。为什么不信任工程师来管理他们的工作和时间？为什么每个人都如此痴迷于优化我们的生产力？如果 Scrum 只是微观管理的幌子呢？

在这篇文章中，我想指出一些我认为 scrum 有问题的地方。我知道你们中的一些人会说问题不在于 scrum 本身，而是它是如何实现的，或者也许他们会责怪公司文化。对此，我要说——如果它真的那么好，那么实现起来应该不会那么难。

## 短跑对于特定的任务来说太短了

截止日期可以是刺激的，帮助你设定优先事项，并帮助你每天都更接近你的目标。但并不是对每个任务都有效。对于设计研究之类的东西，根本不管用。设计师需要时间来思考应用程序的特性和用户体验。这项工作不适合 scrum 两周冲刺。对于需要设计准备好评估并开始实现故事的开发人员来说，它们最终成为了一个瓶颈。

在我工作过的一家公司，我们是一个由几个员工组成的小团队。CTO 同时也是产品负责人和 scrum 经理，他坦率地告诉我，我的工作不是对设计给出反馈。作为一名开发人员，我的工作就是实现这个特性。问题是，在我实现了一个特性之后，他会提出一个清单，上面列有他在设计中不喜欢的东西，我必须在代码中修改这些东西。这项工作被推给了开发人员，而不是在设计阶段花更多的时间，研究和收集反馈。如果没有适当的验收标准，故事工作可能会永远继续下去。“我不明白为什么这个功能还没有出来，”他经常说，“这不是火箭科学！”。

设计研究不是唯一不适合两周冲刺马拉松的任务。为了让软件工程师变得有创造力和创新性，我们需要有一个操场去探索那里有什么和测试那里的东西。在两周冲刺的约束下，很难想出有创意的解决方案。通常，我们没有时间适当地实现想法，软件质量受到打击。

# 我们不擅长估算！

作为开发人员，我们很难判断事情实现需要多长时间。在心理学中，他们称之为“[计划谬误](https://en.wikipedia.org/wiki/Planning_fallacy)”:一种普遍存在的过度乐观和低估完成一项任务所需时间的倾向。这可以归因于几个因素。

首先，我们在估算时没有考虑自己过去的经验。我们从来不会回顾以前的故事，并认为，“哦，这看起来和我们做的另一件事相似…我们估计那个故事是 5，但它花了我们更多的时间来实现。”

第二，我们忽略了事情不会按计划进行的可能性。我们倾向于估计最好的情况，但是如果事情出错了呢？

最后，我们估计的这个数字是多少？我们似乎从来没有就此达成一致。我们评估的是复杂性、工作量，还是其他什么？

这篇由尼古拉·萨普诺夫[撰写的](https://github.com/nikolay-sapunov)[文章](https://medium.com/@forasoft/how-can-a-developer-estimate-time-and-effort-eef2904d301c)很好地解释了团队应该如何评估。

公平地说，随着时间的推移，我们确实变得更好了。然而，当一个新人加入时，或者对于我们从零开始的项目，我们的估计又不准确了。

回想起来，大部分时候我们给自己设定了不切实际的期望。当我们未能达到目标时，我们被内疚和对负面影响的自我谴责所淹没，不仅是对我们自己，也是对项目。收到经理关于我们应该如何兑现承诺的电子邮件只会让我们感觉更糟。

## 迭代太多，目的太少

在我上面提到的公司，我们每隔几个月就重新设计我们的前端应用程序。只有几个客户使用我们正在开发的应用程序，在我们收集客户反馈之前，设计要求会发生变化。我们得把所有东西都拆了，重新开始。对我来说，这种无止境的一遍又一遍地构建同样的东西的循环非常令人泄气。

我的经历并不独特。研究人员发现，我们认为有意义的工作所激发的积极情绪能让我们变得更聪明、更有创造力。例如，杜克大学心理学教授[丹·艾瑞里](http://danariely.com/all-about-dan/)和他的同事[进行了一项研究](http://faculty.chicagobooth.edu/emir.kamenica/documents/meaning.pdf)，参与者被付费建造乐高模型，其中一些模型在完成后在他们面前被拆除。尽管有相同的金钱激励，那些作品被保存下来的人比那些模型被毁掉的人平均多制作了 50%的乐高模型。正如 Ariely 博士在这个[演讲](https://www.ted.com/talks/dan_ariely_what_makes_us_feel_good_about_our_work?language=fr#t-156700)中解释的那样，当我们有目标时，我们会对工作投入更多，即使是很小的目标。

# 痴迷于用户价值

用户故事对于关注给最终用户带来价值的特性非常有用。然而，我发现产品所有者往往过于关注最终用户和用户价值，而忽略了其他一切。

以我的经验来看，代码重构和基础设施的改进一直被忽视，而更倾向于用户价值。与基础设施相关的障碍越积越多，直到让团队不堪重负。这些是每个人都知道的问题，然而，由于我们关注的是在每个故事中始终有一个用户价值，我们没有足够的时间来适当地改进它。

产品负责人建议的一个解决方案是在故事中包含所有的基础设施工作，并将其包含在评估中。我们尝试了一段时间，但是我们碰壁了。它炸了故事点，导致故事永无止境，降低了团队的满意度。

Scrum 产品所有者必须开始将基础设施和内部工具视为产品；因此，我们需要重新考虑谁是“用户”。使用内部工具的开发人员和员工也应该被视为故事的用户。

# 日常站立是没有用的

每日站立会议成为经理或 scrum masters 的状态更新，以找出谁落后于计划。正如 David Heinemeier Hansson 在他的书《远程》中所说的，“不断地询问人们正在做什么会阻止他们真正去做他们所描述的工作。“日报是为开发人员组织工作和报告障碍而准备的。是不是每个人都需要放下手头的工作加入进来？不，我们可以异步进行——被屏蔽的人可以也应该直接联系。

不要盲目地遵循 scrum 方法，记住敏捷的第一条原则: [*“个人和交互优于过程和工具。”*](https://agilemanifesto.org/) 问问自己为什么以某种方式做事——你的团队从中获得了什么好处？

解决方案是适合您的团队和您的特定需求的东西。

对于我的第二份工作，我很幸运地在一个已经建立了反馈文化的组织中工作，这使得提出这样的话题和提出适合我们需求的东西变得更加容易。对我们来说，这意味着摆脱两周的期限和我们认为无用的仪式。我们切换到看板，同时保留为我们工作的部分，比如回顾和计划。
