# 第三个敏捷原则:“经常发布，尽早发布”

> 原文：<https://betterprogramming.pub/release-often-and-release-early-bd5330d6b308>

## 我对执行这一原则所面临的挑战的想法

![](img/2cd2ee411e1e555d18bb7684ab6dc7e1.png)

雷蒙·克拉文斯在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片

这一系列文章的下一篇基于我的断言，即*敏捷已经失败了*(…没有兑现它的承诺)。

我认为敏捷宣言的简单是它失败的部分原因。列出的 12 条原则似乎都是直觉上的好主意，而且似乎很容易实现。但是所有的 12 条原则都需要一些个人/团队/组织的改变，这些改变远非简单明了。

这并不是说“实践原则”所需的努力不值得——它们肯定是值得的。在我看来，问题是每个人都跳进了敏捷的“双脚”,而没有花时间真正理解他们需要做出的改变。

在这个系列中，我将一个接一个地看每一个原则，评论为什么它很难，然后提供走向成功的建议！

在这篇文章中，我将看看敏捷宣言中的第三个原则。即:

> 频繁地交付工作软件，从几个星期到几个月，优先选择较短的时间尺度。

![](img/0868d130cab04071d9d447c65ad6619f.png)

由[卢卡斯·布拉塞克](https://unsplash.com/@goumbik?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片

这个敏捷原则可以用短语*“经常发布，尽早发布”来概括。*一个完整的特性应该尽可能快地交付给客户，而不是留在某个仓库里积灰，直到年度发布日期到来！

如今，“经常发布，尽早发布”已经成为 CI/CD(持续集成/持续部署)的同义词。虽然不是真正需要，但 CI/CD(在许多情况下)是最实用的方法——特别是对于基于 web 的软件。

这已经在[第一原理](https://andywatt83.medium.com/have-you-read-the-agile-manifesto-9785ce876078)中有所涉及，但这一条直接将其引出。没有理由让好主意在 GitHub 回购中搁置数月。

发布通常会很快将开发人员所有的好想法直接传递到用户手中。如果想法是好的，并且受到顾客的欢迎，那么他们会更快地看到好处。如果想法不太好，那么它允许一个快速的反馈过程，并防止进一步的工作建立在不稳定的基础上。

经常发布和早期范例对团队也有好处。大型发行没有“嘎吱嘎吱”声。释放一直在发生，并成为完全的例行公事。一个快乐的团队是一个更有生产力的团队，所以这自然也有利于企业。

然而，达到这一阶段并不容易。我很快会谈到一些技术上的挑战，但是最困难的是以这样一种方式来组织工作，即它可以以小块的形式发布。

很难将工作组织成非常小的增量块，这些增量块增加了价值，并且甚至在功能完全充实之前就是“可发布的”。这需要管理层和技术团队之间非常紧密的合作，也需要与客户进行非常好的对话。

这种认为产品应该经常更新的想法也忽略了这样一个事实，即这可能会扰乱某些用户的工作流程。必须对此进行分析，并了解对用户的影响。你不能每隔几周就简单地改变特性集——需要某种稳定性。要了解现有客户群是如何使用系统的，必须包括健壮的日志记录，并且还必须分析和理解这种日志记录。

以上是项目挑战，但是在“经常发布/尽早发布”的心态中当然有技术挑战。为一个中等复杂程度的 web 应用程序建立一个管道可能会有挑战，但是这些挑战对于一个桌面甚至移动应用程序来说会显著放大。必须分配资源来克服这些挑战，这最终会减缓产品的开发。

另一个巨大的技术挑战是回归。每隔几周就引入一组新的错误是绝对不可接受的。防止这种情况的唯一方法是健壮的测试。这可能会涉及到某种形式的手动测试，但是为了以非常频繁的节奏发布，项目将需要大量的自动化测试，可能需要实现 TDD 范式——这也是不容易做到的事情。

在一个新项目中增加释放管道可能是棘手且耗时的，但在棕地项目中增加管道可能极具挑战性。

根据我的经验，经常发布/早期发布很少被执行，即使在自称“敏捷”的团队中也是如此。我参与的绝大多数项目都有一个“传统的”发布周期，大约每 6-12 个月一次。这无疑导致了构建从未使用过的功能的浪费工作，也让好的想法滞留在 repos 中，远离用户长达一年之久。

我也很少看到管理团队能够将用户价值量化成足够小的块，以便经常/尽早发布。我的经验更多的是“宏大的愿景”,在完成之前，它只能提供有限的临时价值。

查尔斯·达尔文进化论的反对者质疑他，问“半只眼睛有什么用？”。当然没用，但眼睛不是这样进化的。采用这一原则要求软件产品像一个活的有机体一样进化，从非常简单的开始，慢慢地只在需要的时候增加复杂性(特性)。新功能会在环境中进行适应性测试，并很快被保留或拒绝。

这个原则成功的关键在于良好的沟通和持续的努力。必须遵守以下列表:

*   需要花费时间来建立构建和发布管道。
*   要求实现自动化回归测试。
*   要求了解客户需求。
*   要求了解产品目前的使用方式。
*   需要技术和管理团队之间非常密切的合作，以正确地组织工作。

如果你希望你的团队是“敏捷”的，这个原则不是可有可无的，对企业和客户来说好处是巨大的。但是，必须克服重大的技术和项目挑战。

好处是巨大的，我认为努力是值得的！
