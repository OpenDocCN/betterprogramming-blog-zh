# 日常工作

在每家公司，我们可以将工作分为三种类型*:

*   **产品作品** —这是客户看到的。它通常由产品所有者定义。它由特性和 bug 组成。
*   **内部 IT 工作**—基础设施或日常运营的改进。它包括创建新环境、编写自动化、改进 CI/CD、更新依赖项等。
*   **计划外工作或恢复票** —事件和问题。这是无法计划的，所以我将在这篇文章的其余部分忽略它。

*注:我从 [DevOps 手册](https://itrevolution.com/book/the-devops-handbook/)中取出四个工种，稍作改动:将*业务项目*更名为*产品工作*(所有项目均为业务项目)，并将内部 it 项目和更新合并为一个。

但是我们如何确定工作的优先顺序呢？要回答这个问题，我们需要知道公司的目标是什么，我的团队的目标是什么。

# 获得更大的图片

为了获得更大的图景，公司和团队的目标需要被传达给团队。 *OKR* 是一个框架，有助于在组织中上下级联目标。**

* *注:okr 广泛应用于科技公司:谷歌、Intuit、微软、亚马逊、英特尔、脸书、网飞、三星、Spotify、Slack、Twitter、Salesforce.com、德勤、Dropbox 等。

## 什么是 okr(目标+关键结果)？

okr 有两个部分:

*   **目标** —需要达到的目标。它们必须是重要的、具体的、面向行动的，并且(理想情况下)鼓舞人心的。
*   **关键成果** —我们如何实现。它们必须是聪明的(具体的、可衡量的、可实现的、现实的、及时的)。

# 设置您团队的 okr

获得我们 OKRs 的第一步是设定你公司的目标，然后设定你团队的目标。公司的目标(以及公司的战略)是由董事定义的。

1.了解你公司的 OKRs

2.为你的团队设定具体的 okr。

通常，团队的目标只由产品负责人设定。这样，他们就创建了他们的产品积压。这对他们来说更容易，因为他们知道业务方面，但这种方法有许多问题:

*   对业务增长至关重要的内部 IT OKRs 缺失。如果仅仅产品工作是我们的目标，服务的速度、质量和稳定性将会受到惩罚。
*   开发者不会理解他们工作的意义。团队中的每个人都应该参与设定目标，这样他们才能了解全局，并在日常工作中做出取舍。开发人员需要看到指标是真实目标的代理，正如哈佛商业评论文章“[不要让指标破坏你的业务](https://hbr.org/2019/09/dont-let-metrics-undermine-your-business)”中提到的

解决办法？团队中的每个人都应该参与定义团队的 okr。

在每个人都知道业务目标之后，他们可以坐在一起，从理论上探讨团队应该关注什么来帮助实现这个目标。最后，他们应该定义关键结果，以保证目标的实现。

# 技术 OKRs

你是否觉得很难获得 PMs 的认可？提醒他们内部工作也为客户带来价值，因此错过他们对公司来说是致命的。

以下是您可以添加的一些目标:

*   **改善交付** — [发展状况报告](https://services.google.com/fh/files/misc/state-of-devops-2019.pdf)显示了来自职场人士的 31000 份调查反馈结果。它总是指向显示开发和交付过程有效性的相同的四个度量:交付时间、部署频率、变更失败、可用性和恢复时间。这些指标将高绩效者与低绩效者区分开来。
*   **减少风险因素** — 首先，你要确定你的风险(使用便利贴与你的团队一起做)，根据其相关性进行排序，并设定其发生的概率(低、中、高)。然后，你的团队应该在桌面上写下它的 okr。

注意:你应该每年回顾一次你的风险因素。

*   **降低成本** —您是否为某个应用程序支付了过高的费用？一个新项目没有结果吗？这笔钱可以用在你生意的其他地方。我们需要通过减少不需要的费用来考虑这个机会成本。

注意:对于一些低成本公司来说，降低成本工作可能是至关重要的。

# 把所有的放在一起

一旦你的团队有了目标，每个新项目都应该与一个 OKR 相关联。如果一个项目对 OKRs 没有贡献，那么它就是在浪费团队的时间。

小贴士:

*   项目应该被视为假设，它可能会也可能不会让你更接近你的目标。例 1:如果我们允许顾客使用贝宝支付，我们将获得更多的用户。例 2:如果我们使用微服务，我们将减少交付周期。所以你需要**证实你的关键结果正在改善。**如果不是，放弃这个项目，创造新的假设。
*   目标可以持续一年或更长时间，但关键结果会随着工作的进展而变化。所以你应该**至少每年回顾一次你的目标是否仍然有意义**至少每个月或每个季度回顾一次你的关键结果。
*   **okr 应该在整个组织内透明**。这激励了部门间的沟通，避免了重复工作，因此致力于改进相同目标的团队可以一起工作。
*   在谷歌，**每个员工都有自己的 okr**。其中六到八个会来自团队，两个自己定(20%时间)。他们这样做是为了提高创新，并促进人们将他们的愿景加入公司(Gmail 是谷歌内部 20%的项目)。
*   okr 有两种类型:承诺型和期望型。当承诺的 okr 必须是团队的目标时，有抱负的 okr 会引导团队朝着相同的方向前进，但并不期望实现。事实上，**实现所有 okr 是一个信号，表明他们不够雄心勃勃**(连线“[谷歌的拉里·佩奇为什么登月重要](https://www.wired.com/2013/01/ff-qa-larry-page/)”)。
