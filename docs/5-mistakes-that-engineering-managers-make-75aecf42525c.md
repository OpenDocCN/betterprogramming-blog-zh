# 工程经理会犯的 5 个错误

> 原文：<https://betterprogramming.pub/5-mistakes-that-engineering-managers-make-75aecf42525c>

## 避免对你的团队产生负面影响

![](img/878805336ceb542897715ca5c79ec635.png)

由[埃斯特扬森斯](https://unsplash.com/@esteejanssens?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片。

在我们的职业生涯和生活中，我们都会犯错。然而，在我们的职业生涯中，有些错误比其他错误代价更大。有责任成为倍增器而不是制造者的工程经理需要理解并避免致命的错误，因为他们的错误不仅会影响他们自己和他们的职业生涯，还会影响他们所照看的团队。权力越大，责任越大。

所以事不宜迟，让我们来看看工程经理犯的五个致命错误，这样你就可以避免犯这些错误。

# 1.他们深入到代码和代码评审中，以便理解项目的复杂性

![](img/85b4d4e31378cc8fbb16df36d1f36a8a.png)

[潘卡杰·帕特尔](https://unsplash.com/@pankajpatel?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍照。

敏捷惯例的美妙之处在于计划和评估是由团队而不是个人来完成的。因此，工程经理真的没有必要为了理解项目的复杂性而查看代码库和关注代码评审。

相反，如果他们认为某件事花费的时间太长，他们应该看看其他的促成因素，如技术债务、缺乏技能和知识、次优过程等等。

# 2.他们屈服于对错过的恐惧(FOMO)

![](img/3fde541ba4d48f745863c30eacdf9957.png)

照片由[威廉·艾文](https://unsplash.com/@firmbee?utm_source=medium&utm_medium=referral)在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 拍摄。

工程经理亲眼目睹了技术是如何不断变化和发展的。几乎每个月都会有新的框架、新的库、新的编程模式和/或新的工具问世。所以，很自然地，他们会有这种恐惧，如果他们不跟上，有一天他们的技术知识会过时。结果，他们一有机会就试图回去编码。

编码和领导力需要大脑的不同部分和不同的思维模式才能成功。当你编码的时候，你是在专注于解决一个问题，做深层次的工作。领导力就是通过行动和行为以身作则，然后创造一个环境，让他人能够尽最大努力工作——而不是在实施过程中弄脏自己的手。

# 3.他们思考问题过于逻辑化，没有意识到团队成员的感受

![](img/8f86accc7505c0aa7e37634b4e547f2a.png)

凯利·西克玛在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的照片。

“我们不是训练有素的咨询师或治疗师，”我曾听到一位工程经理说。我跟大家分享一下我自己的亲身经历。我曾经和一位技术过硬、善于解决问题的经理一起工作。

但是我不喜欢和他们一起工作，因为我觉得没有同理心，没有真正的关心，也没有对他们的人的欣赏。当我向他们提到我不被欣赏或重视时，每次我见到他们，他都会开始对我大加赞赏，弄得毫无诚意。没有什么比有一个不真诚的领导更糟糕的了。

# 4.他们低估了相邻学科的重要性，如产品、设计和营销

我在[之前的一篇文章](https://medium.com/better-programming/10-signs-that-you-are-not-ready-to-be-an-engineering-manager-yet-369b4f3fecd4)中写过这个。经理们相信软件工程师让世界运转。当被问及谁是创造者、制造者和创新者时，他们会立即回答:“软件工程师。其他学科的存在是为了支持软件工程师。”他们也有这种“建造它，他们会来”的心态。

当有时间构建和创造一些东西时，他们过度关注技术细节，而不是与其他学科合作，以找出什么将为客户增加最大价值以及如何打包解决方案。

# 5.他们避开各种各样的过程

![](img/5e293bf486ec4af430a6fdd123b6435c.png)

照片由 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上的 [Daria Nepriakhina](https://unsplash.com/@epicantus?utm_source=medium&utm_medium=referral) 拍摄。

在我的文章[中，我写道，并非所有的流程都是邪恶的。是真的。十几年前，我作为一个年轻的新手 CTO，在一家没有正式流程的创业公司跑了一年的工程部。没有任何一种流程意味着我们很难知道我们是否在高效地工作。许多任务无法共享，因为不同的人甚至无法知道如何开始做某件事。文档不存在。我很幸运，我在那里的那一年没有员工流动。否则，交接和新人入职将会非常昂贵——更不用说混乱了。](http://eisabainyo.net/weblog/2020/09/05/four-things-i-wish-i-knew-as-the-new-cto-of-a-startup/)

没有任何过程的另一个问题是，我觉得自己是许多事情的看门人。我无法衡量自己。

在决定放弃所有过程之前，工程经理应该拓宽他们的视角，思考并了解过程如何帮助或阻碍他们创建高绩效团队。如果你被卡住了，我建议从[敏捷项目管理模板](https://gumroad.com/l/agilegoogledocs)开始，通过一个简单的过程来跟踪进度。

# 继续前进

我给职业生涯刚起步的工程经理们的建议是:留意你如何以及为什么出现，做你每天做的事情。当你犯了错误，承认它，从中吸取教训，继续前进。透明和真实永远是赢家。

成为领导者，而不仅仅是管理者。
