# **1。在任务完成前减少上下文切换**

这一切都合情合理，直到最后期限临近。很有可能，你总是要向经理汇报或者对客户做出回应。错过最后期限将是你业绩报告上的一个污点。每当我看到新雇佣的开发人员在离截止日期只有几天的时候还在试图理解代码库，我就不得不抑制住冲动，让他们把手头的事情推上来，自己完成。

回想一下你刚开始做初级开发人员的时候。有多少次你被分配去做一个项目，结果却被告知要转到另一个项目，而一个前辈接管了你原来的项目？

我们必须意识到，转换环境不利于培养新的开发人员。这是一个大多数人忽略的权衡。考虑与新开发人员合作，在截止日期前完成任务。

除非你有足够多的人在一个项目上工作，否则一个精益团队的生产力会一直下降，然后才能再次上升。许多项目经理或非技术业务决策者认为雇佣更多的开发人员是提高生产率的直接原因。我们应该积极沟通。

# **2。找到自给自足和提问之间的平衡**

一般的经验法则是，在向别人寻求帮助之前，花 15 分钟思考一个问题或搜索一下，但是你可以自由选择对你的团队最合适的时间。

有时，指出新开发人员为了实现某个特性或修复某个 bug 而需要了解的文件和流程比让他们去挖掘整个代码库更容易。

从长远来看，在开始阶段指导新开发人员肯定会有所帮助。

# 3.指向代码库中相似的代码结构

这也有助于保持一致的编码风格。我们真的不想要一个看起来像是由不同团队编写的代码库。以一个 web 应用程序为例。我们不想导入`lodash`并为做同样事情的函数编写自己的库。

# 4.倾听并保持开放的心态

新员工通常来自不同的背景——无论是教育背景、工作经验还是技术背景。

开发购物车应用程序的团队仍然可以从开发游戏应用程序的人那里学到很多东西。

有时，您可能会对新开发人员带来的东西感到惊讶。他们甚至可能以前遇到过类似的问题并找到了解决方案。有时候我们很容易忘记我们也需要做学生。

此外，有一双新鲜的眼睛来看待这个系统总是很好的。很有可能你的团队已经习惯了现有的结构，以至于你忽略了明显的缺陷。

# **5。记录您的流程**

我不是说在你的代码中到处留下注释。“好的代码文档本身”是正确的。有三个部分需要记录:

1.  开发:编码风格，测试驱动的开发，冲刺格式
2.  部署:测试环境和生产环境。部署流程。每个环境的目的是什么？
3.  维护流程:生产问题发生时的 SOP。

# **6。制定传授领域知识的计划**

> "你的架构应该告诉读者关于系统的信息，而不是你在系统中使用的框架."——[罗伯特·塞西尔·马丁的清洁建筑](https://amzn.to/2W6lmPQ)

尽管看起来基于任务的开发人员不需要知道业务的本质和细节，但是理解领域知识在构建内聚系统中是至关重要的。

# **7。分配重要任务**

分配重要的任务和有影响力的项目来激励新开发人员学习。这也提供了一种责任感，并激励新员工超越自我，为项目或公司愿景做出贡献。

将此转化为软件工程意味着创建一个清晰的结构，并为创新提供新颖的想法。

树立成长的心态，而不是固定的心态。

# **8。在你的团队中引入轮换制度**

当你到了入职的最后阶段，很容易就只是把类似的任务反复交给新的开发人员:开发人员 A 真的很擅长修复 CSS，所以让我们把所有的设计任务都交给他们吧。

除非你的目标是外包一个特定的任务，否则不时地把分配的任务转给新的设计师。受雇于后端开发的开发人员不一定*也不一定*擅长后端开发。很有可能，通过一些培训，你雇佣的后端开发人员可能有前端开发的诀窍，但从来没有表现出来。

许多公司都有新团队成员的轮换制度，但一旦新开发人员在团队中安顿下来，他们往往会接受将他们束缚在某个职位上的任务。

# 9.教授调试过程

特别是对于加入团队的初级开发人员来说，完成调试过程通常比让初级开发人员自己长时间解决问题更有帮助。

如果你没有专门的团队来处理生产错误，创建和共享你的 SOP 文档可以让新开发人员更快地融入团队，他们也可以更快地做出贡献。

# 10.不断的检查和反馈

当我们要求我们新雇佣的高级开发人员用两个月的时间独立开发一个特性时，我们有过这样的经历。最终的结果是，我们不得不扔掉整个代码，因为它与系统的其余部分不匹配。

无论是通过一对一的会议、反馈讨论，还是在咖啡厅的随意聊天，不断地检查和提供反馈。敏捷方法也适用于培训新人。
