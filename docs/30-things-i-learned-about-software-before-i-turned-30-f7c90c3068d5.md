## 从每天学习到团队建设到自动化到迭代

我今天 30 岁了。我不能说我很高兴离开 20 多岁，但我很期待看到下一个十年将会发生什么。为了庆祝这个重要的里程碑，我想分享我在职业生涯中学到的关于软件的 30 件最有影响力的事情。

# 1.抓住一切机会学习新东西

我不确定有哪个行业比软件发展得更快。事情在变化，新的方法和模式在开发，托管服务也在不断改进。抓住每一个机会拓展你的工具箱。

# 2.观点的不同导致创新

让你周围都是挑战你想法的人。如果你的团队 100%同意你的观点，你最终会有狭隘的想法。质疑你的思考过程不仅会让你更多地思考你的解决方案，还可能会引发一个完全不同的想法。

# 3.别往心里去

那么如果你的故事五次 QA 失败或者利益相关者讨厌你在 sprint review 中的演示呢？不是因为*你*写的。我们都是同一个团队的一部分，有着相同的目标:尽我们所能开发最好的软件。

# 4.自动化一切

每次你有一个人执行的任务，你就引入了一个失败的机会。机器不会忘记清单上的一项。一台机器将以同样的方式一遍又一遍地完成同样的任务。从测试中的[认证](https://medium.com/@allen.helton/how-to-automate-oauth2-token-renewal-in-postman-864420d381a0)到部署策略，你可以自动化一切。

# 5.拥抱失败

你永远不会比现在知道得更少。如果你试图开发一个新的软件，但是它失败了，那也没关系！下一次你尝试的时候，你会确切地知道什么不该做。利用你的经验来磨练你的技能。

# 6.用户体验就是一切

你的用户体验是你产品的卖点。客户不会买你的软件，因为后端写得好。他们会买下它[,因为它令人愉快](https://medium.com/swlh/design-to-delight-raising-the-bar-on-software-design-a1eec234188b),并且以直观有效的方式解决了他们的问题。

# 7.迭代，迭代，迭代

当构建软件时，你的目标应该是让某些东西工作起来，并尽快将[展示给人们](https://uxdesign.cc/itturat-ituratte-iterat-iterate-iterate-65199c4d5d53)。一般来说，人们很难从零开始想出点子。但是他们可以告诉你如何将你所拥有的转化为他们需要的解决方案。

# 8.没有神圣的牛

不要接受“我们这样做是因为我们一直是这样做的”政策。做某事是因为这是正确的方式，而不是因为这是熟悉的方式。如果你从来不改变任何事情，你就不能指望改变。

# 9.使用单一责任原则

汽车真的很好开。但是如果你也让一辆汽车飞起来，你会牺牲它的一些驾驶能力。它也许能做到这两点，但在任何一点上都不会很棒。你的代码也是这样。保持专注。做好一件事。

# 10.逐行调试被高估了

如果你遵循单一责任原则，你的代码应该有一个狭窄的焦点。如果它有一个狭窄的焦点，你应该能够限制出错的事情。没有什么比控制台日志和一些单元测试更快的了。嘿，在我说一些有争议的事情之前，我有 10 件事要做。

# 11.你使用的编程语言实际上是一件大事

当开始一个新项目时，想想你要解决的问题。Python 适合大数据。JavaScript 有利于一般的 web 开发。有这么多的语言是有原因的。选择最能解决你的问题的语言。

# 12.副业比其他任何东西都教你更多、更快

如果你想学习新的东西，无论是编程语言、新的架构模式还是技术栈，动手实践将极大地提高你的理解能力。理论只是理论，直到你去实践。此外，你还有机会[与他人分享你所学到的东西](https://medium.com/better-programming/how-i-built-a-serverless-geo-search-app-with-dynamodb-491879233754)。

# 13.你永远不会知道一切

带着可能有更好的方法来构建您将要构建的东西的知识进入开发。但是你会迭代，你会学习。技术变化如此之快，以至于昨天解决问题的最佳方法可能并不是今天解决问题的最佳方法。

# 14.不要建造你不需要的东西

这听起来很明显，但是它伤害了我们很多人。谁问你要那个配置了？没人吗？[硬编码](https://medium.com/better-programming/new-project-just-hardcode-it-bac72e1a231e)。如果您现在不需要构建它，那就不要构建。构建解决业务问题所需的最低要求，然后迭代。

# 15.回顾会改变游戏规则

在敏捷开发中，你在冲刺的最后做冲刺回顾和回顾。[如果你的回顾做得好](https://medium.com/better-programming/8-steps-to-facilitating-a-captivating-retrospective-2324de487706)，你和你的团队将会变成高效的代码机器。你将会团结一致，行动更快，并拥有无与伦比的团队情谊。

# 16\. 80/20

当您发布最终产品时，您可能会失望地发现，您的消费者并没有使用您开发的所有功能。事实上，他们会找到一小部分特性(大约 20%)，然后用掉它们(80%的时间)。其他 80%的特性会被使用，但只有 20%的时间。所以把你的注意力放在优化解决业务问题的特性上。

# 17.优秀的建筑师对复杂的问题做出复杂的解决方案

有些问题比其他问题更难解决。如果你有一个好的软件架构师，他们会设计一个复杂的解决方案来解决复杂的问题。

# 18.但是伟大的建筑师用简单的方法解决复杂的问题

需要一个有才华的人来提炼复杂解决方案的错综复杂之处，并把它变成每个人都可以消费的东西。他们把问题的矛头指向自己，暴露了一个简单的解决方案，而这个方案隐藏了一个困难任务的丑陋。例如，一位伟大的建筑师设计了亚马逊的“一键购买”按钮。他们将整个购买过程完全抽象化，只需点击一个按钮。

# 19.1x、10x、100x

如果你发现你写的代码有问题，现在就修复它。它需要最小的努力。但是如果你让 QA 去发现和报告，那将会花费 10 倍的努力。你有分析师的努力，加上你再次熟悉它并修复它的努力。如果它进入领域，就要花 100 倍的努力去修复。您需要消费者的努力来发现和报告，然后支持分类，然后无论您有什么流程，都需要您来修复、测试和部署。

# 20.故事和隐喻是关键

如果你想用最简单的方式解释你的软件是如何工作的，那就讲一个故事。人们通过联系他们已经熟悉的事物来联系故事和隐喻。他们会理解你想表达的意思，并能够传达给其他人。

# 21.一个健壮的、记录良好的 API 是物有所值的

今天，软件是关于集成的。我如何将这个服务绑定到那个服务？API 就是答案。如果您构建了一个具有丰富特性并有良好文档记录的 API，那么您就是一个现代英雄。开放 API 是当今的标准，[你应该构建它们](https://medium.com/better-programming/you-should-open-up-that-api-youve-been-working-on-b0313a4df9bc)。

# 22.面试官更关心你解决问题的能力，而不是你的编程理论知识

当我面试某人时，我会回答几个理论问题，但我大部分时间都在试图了解你的大脑是如何工作的。当编程是你的日常工作时，你会花一整天来想出问题的解决方案——而不是写什么是多态性。

# 23.80/20(再次)

用 20%左右的时间做 80%的工作，剩下 80%的时间做最后的 20%。想象一个正在建造的房子。在第一个月，你会看到一块土地从一片绿草如茵的土地变成了地基、脚手架和房屋框架。从那以后，事情似乎进展缓慢。软件也是一样。大部分时间将用于硬化和熨平管道和细节。

# 24.以前有人做过

开发人员开玩笑说，他们一生中从未写过一行原创代码。老实说，这可能是真的。模式是有原因的。让你的软件独一无二的是你解决商业问题的方式。不要仅仅为了独特而试图想出一个新的模式。使用有用的东西，让你的用户体验到你的游戏规则改变者。

# 25.导师至关重要

不管你是刚刚开始职业生涯还是已经工作了 45 年。有人做了你想做的事。获得他们的帮助。通常情况下，他们很乐意帮忙。当你觉得准备好了，[归还](https://medium.com/swlh/getting-started-with-mentorship-what-is-it-all-about-ce475ff4710)。

# 26.全都是 1 和 0

计算机会完全按照你的要求去做。他们没有自己的思想。你的代码不能工作，因为你是这样写的。代码里总有答案。你只需要找到它。

# 27.软件循环往复

当时我不在，但显然云开发几乎和大型机开发一模一样。今天的工具更好了。方法论也是循环的。我们曾经热衷于瀑布式开发，后来进入了敏捷，但是[瀑布式开发似乎又要卷土重来了](https://medium.com/dataseries/will-waterfall-development-ever-die-probably-not-6ab4589e061)。也许下一个 10 年将带给我们一个比 20 年前更好的工具包和流程。

# 28.存储过时，计算流行

过去，所有的成本优化策略都是围绕节省磁盘空间展开的。尽可能利用所有计算资源，尽可能少占用存储空间。但是今天，事情发生了逆转。存储成本几乎为零，而计算价格却在上涨。花时间[优化你的代码](https://medium.com/better-programming/5-steps-to-making-a-predictable-cost-model-for-aws-serverless-projects-78d78909bb82)的处理过程，尽可能地存储一切。

# 29.数据是美丽的

大数据是一件大事。它为机器学习提供动力，为我们的工作流程提供见解，并说明趋势。如今数据存储如此便宜，我们没有理由不存储所有我们能存储的东西。人们喜欢图表和趋势。用关于他们如何做某事的美丽见解让他们惊叹。

# 30.最好的工具是你的团队

如果你想要真正创新的软件，建立一个强大的团队。一个相互信任以完成工作、相互挑战并在个人层面上相互联系的团队。建立一种注重相互了解的文化将是你是否成功的最终决定因素。作为一个领导者，把你的努力投入到你的团队中。
