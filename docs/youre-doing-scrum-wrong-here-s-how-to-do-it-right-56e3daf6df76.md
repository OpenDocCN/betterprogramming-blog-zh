几乎我所在的每个团队都声称使用了某种形式的 Scrum 方法来管理他们的工作流程。它很少是有效的，甚至是真实的。

老实说，在我职业生涯的早期，严格敏捷的 Scrum 工作流的想法是可怕的。当你的日常工作是没有计划的和意想不到的时候，在规定的时间内完成工作似乎是一个很高的要求。

当你有紧急的 bug 被报告，或者有技术问题需要尽快解决，或者甚至只是有不可预见的阻碍的计划外工作，按照别人的时间表工作的想法是令人生畏的。我认为所有这些都需要处理，在预期的时间内完成一个 sprint 是不现实的。

幸运的是，至少可以说，当我在那样的环境中工作时，我们的 Scrum 工作流是松散的。我从来没有考虑太多，因为在一个 sprint 中工作似乎是不可能的，所以当工作持续地跨 sprint 进行时，这似乎是正常和合理的。

以下是一些表明 Scrum 不适合你的危险信号:

1.  大量的故事从一个冲刺发展到下一个冲刺。您的 sprint 范围已关闭。
2.  许多冲刺阶段都有故事。你的故事超出了范围。
3.  你的故事在描述中没有任何细节。你忽略了 Scrum 的计划元素。

当我得到一份声称真正敏捷的新工作，并且在面试中实际上提出了关于 Scrum 的问题时，我有点犹豫。我不认为我相信它会像他们承诺的那样有效。

他们证明我错了！

# 团队合作让梦想成真

让 Scrum 方法在你的团队中很好工作的第一件事是一个*完整的*团队，一起工作。如果你没有一个产品团队或 QA 团队，敏捷工作流是不会有效的。我认为这些人对 sprint 的成功比开发方面更重要。

我曾经在一个只有开发人员的团队中尝试过 Scrum。我们彻底失败了。我们确实完成了工作，但是 Scrum 绝对不是描述我们工作流程的准确方式。

产品的作用是计划和评估工作的优先级。如果你得到的故事没有需求，那么你可能没有一个有效的产品策略。当产品经理维护了一个详细的、按优先顺序排列的待办事项清单时，冲刺就从第一天完成的工作开始。

QA 团队成员有一个同样重要的角色——他们保持冲刺的速度。QA 工程师确保开发人员不会发布有问题的代码，这些代码会在下一次冲刺时回来咬我们，那时我们必须回去修复它。这对于保持冲刺的速度很重要。

当然，开发人员会一头扎进去，修改代码——我们在整个 sprint 中都在工作。但是如果没有产品交付给我们清晰的需求和对我们的工作进行 QA 测试，我们将会无限期地陷入困境。

当团队所有成员同步工作时，一个冲刺会以惊人的效率平稳运行！

# 公司的支持是至关重要的

产品和开发团队在整个 sprint 过程中通力合作，高效地完成计划的工作。然而，同样重要的是，企业也认同计划工作的 Scrum 概念。

我所在的团队试图将大部分即时工作融入冲刺阶段，但这并没有多大意义。Scrum 的理念是在原子层次上管理工作。项目被计划和解构，以便团队能够以迭代的方式吸收和处理它们(例如，分成小块，一次一个)。每一次冲刺，你都离你的目标更近一步。

当企业要求即时完成新工作时，这种规划很少发生。范围大而需求少的故事经常会进入并阻塞工作流。

为了让企业适应 Scrum 的理念，企业用户有一个清晰的方法来报告重要的 bug 或特性是很重要的，这样它们就可以被合并到 backlog 中并被优先化。透明度和关于优先事项和正在做什么的明确沟通是必要的。

企业永远不应该觉得他们没有任何发言权，但是他们不应该有足够的发言权来不必要地中断冲刺。

是的，我确实说过 bug 会被放到 backlog 中。

# Scrum 仪式应该紧扣主题

首先我要说，称它们为仪式感觉很傻，所以从现在开始我将称它们为会议。抱歉，如果你和那种语言结婚了。

为了让 sprint 顺利运行，会议是绝对必要的。然而，它们永远不应成为负担。每一次 Scrum 会议都有一个目的，当焦点保持不变时，会议就变得简单而有意义。当 Scrum 会议偏离主题并持续很长时间时，它们会成为负担并影响 sprint。

在我工作的团队中，我们的 Scrum 会议占用的时间经常比我实际花在开发上的时间还要多——不，我不只是在偷懒。我们的“仪式”将转化为技术讨论和计划。

不出所料，我们没有遵循上面的建议，冲刺并不顺利。混乱的会议是一个更大问题的征兆。

如果你的会议超出了范围，那么团队可能会在其他方面挣扎，并试图弥补。注意会议上讨论的内容，以确定真正的问题可能出在哪里。

然后说说 sprint 回顾会上的问题。在 Scrum 中，每件事情都有一个时间和地点！

# 当它起作用时，它是神奇的！

直到最近，我还不相信 Scrum 方法论——我从未真正看到它的工作。事实是，它并不适用于每个企业或每个团队。

Scrum 已经成为技术行业的一个标准，我认为一些团队会受益于一种不同的方法，这种方法更符合业务的期望或文化。较小的团队或者项目对时间非常敏感的团队可能无法在 Scrum 过程中取得成功。这不是一件坏事。找到适合你和你的团队的方法！

如果它对你不起作用，而你认为它应该起作用，尝试这些建议以获得更多的认同和效率:

1.  团队和公司买入
2.  广泛的规划
3.  清晰的交流

当它起作用时，它是神奇的。老实说，在一个有 Scrum down pat 的团队中工作让我大吃一惊。当 Scrum 方法被完全接受时，开发过程变得如此高效和简化，这是令人惊讶的！

你有没有经历过 Scrum 工作或者在一个团队中工作不顺利的经历？我错过了什么重要的配料吗？

祝你好运！
