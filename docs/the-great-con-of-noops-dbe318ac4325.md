# 或者是…？

NoOps 是一个愿景，一个目标，一个目的。我们遵循其模糊的原则，努力尽可能接近地模拟这一愿景。唉，还有另一种力量在起作用，就像平静水面下的水流。

操作是维护的一个子集。也就是说，需要发生的事情来保持灯亮着。在软件方面，我们还需要确保那些灯不能被黑，不能被偷，不能被砸。这需要巨大的努力，我们都在寻找方法来尽量减少我们的维护开销。谷歌将此称为[辛劳](https://landing.google.com/sre/sre-book/chapters/eliminating-toil/)。

因此，当我们转向 AWS Lambda 作为托管应用程序的解决方案时，我们突然摆脱了这些负担。

*太好了！那我们现在怎么办？事实证明，我们必须编写一些自动化程序来让我们的应用程序进入 AWS Lambda，否则，只是工程师从他们的机器上运行随机的 AWS 脚本。我们也必须把我们的秘密放进我们的 lambda 中——那将需要整理出来。我们不希望在我们的环境之间漂移，所以“基础设施作为代码”是必须的！等等……我们还必须解决监控问题。啊，地狱，在 Cloudwatch 中搜索日志是可怕的，我们需要以某种方式让他们进入基巴纳…*

## 一种完全不同的运营开销

当你炮制这些剧本的时候，你也在制造债务。随着您使用的每一个库、编写的应用程序和定义的管道，水面下的水流越来越强。

NoOps 是误导。原则是合理的:尽量减少保持软件运行所需的人工劳动。唉，这不是完整的画面。即使你完全采用 PaaS，内部自动化也是不可避免的，并且需要维护。这些开发工具都有运行开销。

这是一种奇怪的操作开销。当今存在的大多数工具和专业知识都是为了让您的生产应用程序保持活力。没有什么可以帮助你管理定制的工程工具。在足够复杂的情况下，这些工具开始类似于高度复杂的生产系统，以及随之而来的所有负担。你没有摆脱行动，你只是改变了它。

# 你坚持他们的观点

毫无疑问，云平台已经改变了游戏——否则就太可笑了。但他们也带来了自己主动提出的意见。这一切都很好，直到没有一个云解决方案可以改变你想要的方式。在那一刻，你意识到，太晚了，当你选择逃避手术时，你放弃了一定程度的控制。

在您尝试最小化运营开销的过程中，您牺牲了一些可见性和一些可延展性。从经济角度来看，这可能是一种非常明智的权衡，但这种权衡往往是在没有意识到的情况下做出的。不要梦游进入一个僵化的系统。

# Ops 不会消失

我希望它是。我做软件工程师已经很多年了，我对我需要应用的每个补丁都有怨言。我讨厌阅读关键 Docker 漏洞或 Kubernetes 升级。但最终，我是来解决问题的。这意味着定期构建和部署软件。

DevOps 思维鼓励我们将补丁视为软件的*部分*。我更赞成这种方法，而不是一些 NoOps 倡导者所主张的那种幻想中的疏忽。不要让偶然的骗子破坏了我们集体软件工程未来的美好愿景。记住，如果听起来好得不像真的，那很可能就是真的。
