## 以及为什么它不仅仅是一种构造代码的方式

克拉克·蒂布斯在 [Unsplash](https://unsplash.com?utm_source=medium&utm_medium=referral) 上拍摄的照片

微服务轰轰烈烈地闯入了 web 开发的世界。这个话题引起了激烈的争论，没完没了的帖子，许多对整体架构的比较，甚至来自大组织的用例。然而，每当这个话题出现时，开发人员往往主要关注微服务革命的技术方面，而不是它引发的文化变革。虽然围绕它发展起来的技术确实很酷，但我发现文化变革更具破坏性。

我很幸运地在 2014 年左右相对较快地赶上了这股潮流，我仍然记得将旧的 monoliths 迁移到新的又酷又性感的微服务中是多么令人兴奋。像许多人一样，我一开始就过度关注技术方面——毕竟，那是一个非常令人兴奋的时代(例如，Docker 刚刚出现)。

然而，几年后，尽管技术方面发生了很大的变化，我相信*代码所有权* 和*特性团队*对采用微服务的组织内软件的开发方式影响最大。

# **特色团队**

尽管在单一架构中有跨职能的功能团队是可能的，但是组织通常会根据他们的技术能力(例如，前端、后端、系统管理、数据库开发等)来划分团队。)因此，开发人员很少能够访问生产环境，导致生产和维护方面的职责分离。詹姆斯·刘易斯和马丁·福勒在最初定义微服务架构概念的文章[中直接指出了这个概念。](https://martinfowler.com/articles/microservices.html)

> 微服务的支持者倾向于避免这种模型，更倾向于团队应该在产品的整个生命周期中拥有产品的概念。一个常见的灵感来自于亚马逊的“你构建，你运行”的理念，即开发团队对生产中的软件负全部责任。

此外，让具有其他专长的人加入团队(例如，产品/营销)可以将范围扩大到远远超出技术方面。能够监控客户反馈、用户采用和商业价值使得团队对特性及其功能的每一个方面完全负责。通过遵循这个模型，团队可以看到他们对产品和业务本身的影响。

# 代码所有权

我总是喜欢将编程视为一种艺术形式(或者至少是一门手艺)。当我开始我的编程之旅时，我花了几年时间做一些小项目，比如游戏、桌面/网络应用和网站。将想法从我的大脑转移到代码中感觉很自由。然而，当我开始我的软件开发职业生涯时，感觉更像是在比萨饼店工作，而不是创造艺术。

除了模糊的责任之外，拥有集中的代码库和数据存储也是选择技术时的一个限制因素。关于框架、语言、设计模式或数据库引擎的决策必须是全球性的。简而言之，当一个人无法控制这些方面时，就很难有创造力。

微服务架构带来了小代码库的概念。创建几十个较小的存储库而不是一个大的存储库，使得为特定的存储库设置明确的所有者成为可能。显然，代码所有权并不意味着单个团队或个人负责维护整个代码。如果需要，其他团队也可以提交拉动请求(内部采购)。然而，代码的所有者保护解决方案的质量，并确保所有的拉请求都遵循代码标准和 API 契约。

# 结论

在我看来，拥有清晰所有权的小团队给日常开发工作带来了很多乐趣，并给开发人员带来了激发创造力的自由。此外，使用小代码库工作感觉就像再次参与较小的项目，即使是在一个大组织中工作。

不要误解我，没有灵丹妙药。虽然面向微服务的架构促进了拥有代码所有权和特性团队的可能性，但它并不适合所有人。毕竟，在单一代码库中拥有这些特性是可能的，只是需要更多的组织参与才能实现。
